# ADR-00046: Fix (FuncType)[] parser gap and enable calling closures stored in arrays/object fields

- **Date**: 2026-07-13

## Context

Third of the four bugs found in the same bug-hunting pass as `ADR-00044.md`/`ADR-00045.md`: `let counters: (() => number)[] = [];` failed with the confusing, unrelated-sounding codegen error `can only assign to identifiers or array elements`, rather than a parse error or working correctly. Flagged in `STATUS.md`'s Known Limitations table, picked up as its own piece of work this session.

## Investigation

Root cause was in the parser, not codegen. `parseTypeAnnotation`'s `LPAREN` branch (`parser/parser.go`) — which parses a parenthesized function type like `(x: number) => number`, or a plain grouping paren like `(number)` — returned immediately at each of its exit points once it finished parsing, without ever checking for a trailing `[]`. The sibling object-type-annotation branch (`LBRACE`, for `{ field: type }`) already had this check: an explicit loop consuming `[`/`]` pairs and wrapping the parsed type in a nested array `TypeAnnotation` for each one, to support `{...}[]`. The `LPAREN` branch had no equivalent.

Without that check, `let counters: (() => number)[] = [];` silently mis-split into two bogus statements: a truncated `let counters: (() => number)` (the annotation parse stopped right after the closing `)` of the function type, before the trailing `[]` was ever looked at, and with no initializer since `=` hadn't been reached yet either) followed by a second "statement" starting at the leftover `[] = [];`, which happens to parse as a grammatically valid statement on its own (an assignment whose left-hand side is an empty array literal). That second statement is what actually reached codegen and produced the misleading error — nothing in the error message pointed back to the real cause in the parser.

Confirmed via direct testing that the `LPAREN` branch actually has *two* distinct "finished parsing, about to return" exit points that both needed the same trailing-bracket check, not just one: the real curried-function-type path (`(params) => retType`, returns an `IsFuncType` annotation) and the pure-grouping-parens fallback path (a single unnamed parameter with no `=>` following, e.g. `(number)` used just to disambiguate) — confirmed the second path has the identical bug independently, via `let nums: (number)[] = [1, 2, 3]` failing the same way before the fix. The third exit point in the same branch (backtracking when a tentative `=>` doesn't pan out, used for disambiguating `(SomeFuncType)` used as a return-type annotation from a curried function type) does *not* need the check — it resets the parser position to just before the `=>` it tentatively consumed, handing control back to an enclosing arrow function's own parse, where a trailing `[]` could never legally appear at that exact position anyway.

**A second, deeper bug found while manually verifying the fix end-to-end**: once the parser correctly built `let counters: (() => number)[] = []` and `counters.push(makeCounter(0))`, actually *calling* a closure through `counters[0]()` still failed with a different, unrelated error, `only simple function calls are supported` (`codegen/llvm/emit_call.go`'s final fallback). `emitCall`'s dispatcher only recognized two callee shapes: a bare identifier (named function or closure variable) and an immediately-invoked arrow function (`((x) => x+1)(5)`) — nothing for a callee that's itself an arbitrary function-typed *expression*, like an array-index result (`arr[i]()`) or an object field (`obj.callback()`). This had never been reachable before, since the parser bug meant no array of function types could ever be declared in the first place — fixing the parser bug exposed a pre-existing gap in call-expression dispatch that had simply never been exercised.

## Decision

**Parser fix**: extracted the `LBRACE` branch's existing trailing-`[]`-consuming loop into a shared helper, `parseTrailingArrayBrackets(p *Parser, source string, ta *ast.TypeAnnotation) (*ast.TypeAnnotation, error)`, and call it from both of the `LPAREN` branch's non-backtracking return points (the curried-function-type success case and the pure-grouping-parens fallback), as well as from the `LBRACE` branch itself (replacing its inline copy of the same loop with a call to the new shared helper).

**Call-dispatch fix**: added two new cases to `emitCall` (`codegen/llvm/emit_call.go`), both reusing the existing `emitClosureCallByPtr(closurePtr string, ty Type, args []ast.Expression, pos ast.Pos)` helper (already used for immediately-invoked arrow functions and named-closure-variable calls — it just needs a header pointer and the function's `Type`, not necessarily a `Symbol`):
- `arr[i](...)`: a new check for `*ast.IndexExpression` callees, gated on `e.inferExprType(idxEx).IsFunc` (already correctly resolves an array-of-function-type element's type, via the existing `IndexExpression` case in `inferExprType`), evaluates the index expression to get the closure header pointer, then calls through it.
- `obj.field(...)`: added at the end of the existing `*ast.MemberExpression` callee-dispatch block (`emitCall`'s first, very large if-chain of hardcoded built-in method names — `push`, `slice`, `map`, etc.), after every hardcoded name has already had a chance to match, gated the same way on `e.inferExprType(mem).IsFunc` (which already correctly resolves an object field's declared type via `FieldIndex`).

## Implementation notes

- `parser/parser.go`: new `parseTrailingArrayBrackets` helper (defined just above `parseTypeAnnotation`); its two call sites in the `LPAREN` branch (curried-function-type success path, pure-grouping-parens fallback path) and its one call site replacing the `LBRACE` branch's former inline loop.
- `codegen/llvm/emit_call.go`: new `*ast.IndexExpression`-callee case right after the existing immediately-invoked-arrow-function case; new fallback inside the `*ast.MemberExpression` block, after the Map/Set method-dispatch check and before that block's closing brace.
- No changes needed to `inferExprType` itself — its existing `IndexExpression` and `MemberExpression` cases already correctly propagate a function-typed array element's or object field's `Type` (including `IsFunc`), they just weren't being consulted by `emitCall`'s dispatcher before this fix.

## Side effects discovered

- Confirmed, but explicitly out of scope: `const ops = [add, sub]` (an array literal of *named* function references, as opposed to arrow-function closures) fails with `undefined variable 'add'` — named top-level functions still can't be used as first-class values at all in this compiler, a pre-existing general limitation already tracked in `STATUS.md`'s `setTimeout`/`clearTimeout` row (documented there as "not specific to timers"), not a new gap introduced or exposed by this fix. Every case this fix actually targets (array of arrow-function closures returned from `makeCounter`-style factory functions, a closure stored in an object field) works correctly.

- Manual repros: `let counters: (() => number)[] = []` declares and tracks `.length` correctly; `counters.push(makeCounter(0))` followed by `counters[0]()` calls the stored closure and reflects its own independent captured state across repeated calls; a plain grouping-paren array type (`let nums: (number)[] = [1, 2, 3]`) parses and indexes correctly; a function-typed object field (`const h: Handler = { callback: makeCounter(10) }; h.callback()`) calls correctly across repeated calls, each call reflecting the closure's mutated captured state.

- New example `examples/closures/array_of_closures.ts`, covering an array of independently-stateful counters and a function-typed object field — ran manually, output matches expectations.
