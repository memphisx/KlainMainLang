# TDD-00011: IndexedDB-Compatible Storage API (pluggable embedded/proxy backends)

- **Status**: Not Started
- **Date**: 2026-07-17

## Context

Every program this compiler produces today is either stateless or holds state only in heap-allocated structures that vanish when the process exits — fine for a one-shot CLI, but a real gap for this project's stated long-term "microservice fronting an HTTP API" direction, where a request handler routinely needs to persist or look up data across requests and across restarts.

The actual goal, sharpened from an initial "some kind of KV store" framing: expose the **real browser `IndexedDB` API surface** (`indexedDB.open()`, `IDBDatabase`, `IDBObjectStore`, `IDBTransaction`, callback-based `IDBRequest`, eventually `IDBCursor`/`IDBIndex`/`IDBKeyRange`) as a built-in global, rather than inventing a bespoke `db.get`/`db.set` shape — so that hand-written application code already written against that idiom has somewhere to run, and (aspirationally — see the caveat below) so that existing npm packages built on top of `IndexedDB` (Dexie.js, localForage, and similar client libraries) might eventually be reusable rather than needing a from-scratch replacement.

**A SQL surface is explicitly out**: an earlier draft of this document considered mapping the store onto SQLite via a hand-rolled single-table SQL schema exposed loosely as a KV shape. That's dropped — if SQLite is used at all, it's purely an internal storage engine invisible to user code, never a SQL surface app code is expected to write against.

**Priority: deliberately low, and deliberately late.** This is scoped now (a TDD, not an implementation) so the design thinking and backend research aren't lost, but the expectation is that this gets picked up only after the compiler's core gaps are substantially closed — memory management (`TDD-00001`), classes (`TDD-00009`), generics (`TDD-00010`), and the remaining event-loop/Web-API backlog are all higher-value, more foundational work. This is a "make the compiler stand out once it's stable" feature, not a near-term one. Dependencies on other TDDs are checked below precisely so a future implementer knows what's actually a hard blocker vs. what merely matters more for the "existing npm package" stretch goal — not because this is expected to start soon.

### Caveat that has to be said plainly: literal npm package reuse is not realistic near-term

Dexie.js and localForage are both written using JavaScript features this compiler doesn't support yet: ES `class` (with inheritance) is ❌ (staged in `TDD-00009`), `Promise`-returning async generators/iterables are ❌, and at least one of these libraries' internals lean on patterns (dynamic property bags, `Proxy`-like flexibility) this compiler's fixed-shape object model doesn't offer and `STATUS.md` marks as "likely impractical." Compiling their actual source unmodified is not a realistic near/mid-term outcome — it would need `class` support at an absolute minimum, and plausibly more. The realistic win this TDD targets is narrower and still valuable: **hand-written app code that uses the `indexedDB` global directly** (the same idiom those libraries wrap, one layer down) compiles and runs. Treat "maybe those npm packages become reusable too, someday" as a bonus outcome contingent on `TDD-00009`/generators/etc. landing first, not a V1 deliverable.

## Design

Two independent axes: the **front-end API surface** (what user code writes against — this should look like real `IndexedDB`, regardless of backend) and the **storage engine backend** (what actually persists the bytes — several real options exist, compared exhaustively below since the choice matters more than usual here).

### Front-end: IndexedDB-shaped API, V1 scope

Real `IndexedDB` is callback/event-based (`request.onsuccess = ...`), **not** Promise-based — a deliberate spec quirk (it predates widespread Promise adoption) that this design should preserve rather than "improve," precisely because preserving the real shape is the entire point of this feature. This is a genuine deviation from every other async-flavored thing this compiler has (`fetch`, `async`/`await`), so it needs its own callback-dispatch design — reusing the already-proven "closure-calling-from-C-trampoline" pattern (`qsort` comparators, timer callbacks, `http.listen`'s handler dispatch) rather than inventing a new calling convention.

Concrete V1 subset (skeleton, not exhaustive — enough to start implementation from):
- `indexedDB.open(name, version)` → `IDBOpenDBRequest`, firing `onupgradeneeded` (schema/object-store creation) then `onsuccess` (handing back an `IDBDatabase`), matching the real spec's version-based migration model.
- `db.createObjectStore(name, { keyPath })` — called only from inside `onupgradeneeded`, matching the real spec's restriction that schema changes are version-transaction-scoped.
- `db.transaction([storeNames], mode)` → `IDBTransaction`, `mode` being `"readonly"` or `"readwrite"`.
- `tx.objectStore(name)` → `IDBObjectStore` with `get(key)` / `put(value, key?)` / `add(value, key?)` / `delete(key)` / `clear()`, each returning an `IDBRequest` with `onsuccess`/`onerror`.

**Deferred to V2, dependent on backend choice**: `IDBCursor` (ordered iteration), `IDBIndex` (secondary indexes), `IDBKeyRange` (range queries) — all three need the backend to support ordered range scans, which not every option below gives for free (see per-option notes).

### Backend option A: SQLite (C library, linked like `libcurl`)

Link SQLite's public-domain single-file amalgamation exactly the way `libcurl` is linked for `fetch` today — `declare` the handful of C API functions needed (`sqlite3_open`, the `prepare`/`step`/`finalize`/`bind_*` family, `sqlite3_close`), used purely as an internal engine (one table per object store, `key BLOB PRIMARY KEY, value BLOB`, values as JSON text reusing existing `JSON.stringify`/`JSON.parse`). Confirmed by research: static-linking SQLite's amalgamation on musl/Alpine is routine, unlike `libcurl`'s TLS/compression dependency chain that made `--static` a real fight (`ADR-00033`). Gives real ACID transactions (a natural fit for `IDBTransaction`) and ordered range scans (`ORDER BY key`) for free, which directly enables `IDBCursor`/`IDBKeyRange` later without a second engine. No new architectural risk — same C-dependency-linking mechanism this project already has one working precedent for.

### Backend option B: hand-rolled, zero dependency

Implement a small persistent engine directly in this compiler's own native runtime (matching `Map<K,V>`'s existing "we build everything ourselves" ethos) — crude V1 could be as simple as an append-only log file replayed on open (no crash-safety guarantees beyond "don't get killed mid-write"), upgraded later to a real B+tree or LSM structure if the crude version proves insufficient. Total control, zero cross-runtime risk, fits this codebase's existing pattern of never linking a foreign language runtime — but real storage-engine correctness (crash consistency, compaction, avoiding data loss on a killed process) is genuinely hard-won engineering, easy to underestimate, and this project would be doing it from scratch rather than reusing decades of a mature engine's bug fixes.

### Backend option C: embed a pure-Go engine via a cgo `c-archive` bridge

This is the one worth spelling out in most detail, since it wasn't fully explored in the prior draft and several concrete Go-native options were specifically requested for comparison: **BadgerDB**, **BBolt**, **Pebble**, **BuntDB**, **go-memdb**.

**Mechanism, confirmed feasible by research, not just assumed:** Go supports `go build -buildmode=c-archive -o libstore.a store.go`, which compiles a small Go wrapper package (importing whichever KV library, exposing `//export`-annotated functions like `kml_store_open`/`kml_store_get`/`kml_store_put`) into a `.a` static archive plus a generated C header — linkable into the `clang` invocation this compiler's own build already performs, the same way `-lcurl`/`-lsqlite3` would be, via this compiler's existing `LinkLibs()` plumbing. All five candidate libraries are pure Go with no cgo of their own, so the *only* cgo boundary in the whole system is the one thin wrapper file this project would write itself.

**Library comparison:**

| Library | Model | Notes |
|---|---|---|
| **BBolt** | Single-file, mmap-based B+tree | Actively maintained (etcd-io fork of the original BoltDB), boring and stable by design, single-writer/many-reader ACID transactions. Slowest of the group and largest on-disk file size in benchmarks, but **no background goroutines at all** (no compaction, no async workers) — the simplest concurrency profile of any option here, which matters a lot given the coexistence risk noted below. Ordered B+tree gives range scans natively, a clean fit for `IDBCursor`/`IDBIndex`/`IDBKeyRange`. **Recommended starting candidate** for exactly this reason — least new risk, most direct feature fit. |
| **BadgerDB** | LSM-tree (RocksDB-style) | Substantially faster writes than BBolt/Pebble in published benchmarks (large margin), slower individual reads; runs continuous background compaction goroutines — real, always-on concurrency inside the embedded engine, which is the part most worth stress-testing against this compiler's own scheduler (see below) before relying on it. Worth it only if BBolt's write throughput turns out to be a real bottleneck. |
| **Pebble** | LSM-tree (CockroachDB's engine) | Production-grade, balanced perf, but a heavier codebase/dependency footprint aimed at a much bigger production system than this project needs — likely overkill for V1. |
| **BuntDB** | In-memory, optional append-only-file persistence | Simple API, supports spatial/secondary indexing. A good fit specifically for the "in-memory, optionally durable" half of the original idea, if a lighter-weight option than BBolt's always-on-disk model is wanted. |
| **go-memdb** (HashiCorp) | Pure in-memory, radix-tree indexed, supports change watches | No persistence at all — largely redundant with this compiler's existing `Map<K,V>` unless its radix-range-scan or watch semantics are specifically needed. Lowest priority of the five. |

**The real, unresolved risk (must be a direct prototype spike before committing, not assumed either way):** the Go runtime installs a process-wide `SIGURG` handler for asynchronous goroutine preemption the moment it initializes (standard since Go 1.14), and spawns its own background OS threads (`sysmon`, GC workers) — a categorically different, actually-multi-threaded concurrency model appearing inside what has otherwise been a deliberately single-threaded, cooperatively-scheduled process (this compiler's own `ucontext.h`-based fiber scheduler and `setjmp`/`longjmp` exception model, `TDD-00006`). Whether that coexists safely — whether `SIGURG` delivery could land mid-fiber-switch or mid-`longjmp` in a way that corrupts this compiler's own state — is **not confirmed either way by this research pass**, only that it's a real, specific mechanism worth checking directly. `TDD-00006` already established the right methodology for exactly this kind of question (it directly prototyped LLVM coroutines against `setjmp`/`longjmp` before trusting an assumption, and found a real crash that way) — this option should get the same treatment: a small throwaway program linking a `c-archive` Go library alongside this compiler's actual fiber/exception mechanism, under real load, before this backend is trusted in `http.listen`.

One genuine bright side worth noting: the embedded Go library's own internal memory is managed by Go's real GC, not by this compiler's "leak everything, never free" convention — meaning this specific backend wouldn't add to the memory-growth problem `TDD-00001` tracks, unlike every other allocation this compiler currently makes.

Cross-compilation note: `c-archive` output is OS/architecture-specific, same as `clang`'s own target concerns — a Linux Docker build stage would need `GOOS=linux` set for the Go build step too, not just for the compiler's own `go build`.

### Backend option D: proxy to an external Redis (RESP protocol)

RESP is a deliberately simple, line-based text protocol (5 data types, CRLF-terminated) — realistically hand-buildable directly on the raw socket primitives already declared for `http.listen`/the event loop (`@socket`, `@read`, `@write`, `@close`), needing **exactly one new primitive**: outbound `connect()` isn't declared anywhere yet (today's socket use is entirely accept-side). This needs **no new C library and no cgo bridge at all** — encoding a command is string-building a byte buffer this compiler already does routinely; decoding a reply is branching on the first byte, comparable in size to the `.split()`/`.replaceAll()` hand-written IR helpers already in `runtime.go`.

To fit behind the same `IndexedDB`-shaped front-end, an object store's operations would map onto a Redis hash (`HSET storeName key value` / `HGET` / `HDEL`) — giving `IDBObjectStore`'s basic CRUD surface directly, though not naturally ordered (Redis hashes aren't range-scannable the way a B+tree or SQLite index is), so `IDBCursor`/`IDBIndex`/`IDBKeyRange` would need either a second data structure (a Redis sorted set alongside the hash, for ordering) or would simply stay unsupported when this backend is selected — a real, backend-specific feature gap worth surfacing to the app (or documenting) rather than silently returning unordered results.

### Which to build first — effort/risk ranking across all four

Ranked lowest to highest architectural risk, to directly answer "whichever is simpler, staged":

1. **Redis/RESP proxy (D)** — lowest effort and risk: no new dependency, no cgo, sockets already exist, only `connect()` is missing. Can ship independently of any embedded-engine decision, and in either order relative to it.
2. **SQLite (A)** — next: one proven, low-risk C dependency (this project already has exactly one precedent, `libcurl`), gives ACID + ordered scans for free, no foreign-runtime coexistence question at all.
3. **Hand-rolled native engine (B)** — moderate-to-high effort depending on where the crash-safety bar is set for a "good enough" V1; zero foreign-runtime risk, but real storage-engine correctness work that's easy to underscope.
4. **cgo-embedded Go library (C)** — highest architectural uncertainty (the `SIGURG`/threading coexistence question is genuinely unverified, not just unbuilt), but offers the most mature, actively-maintained engine choices (BBolt recommended first) if the prototype spike above comes back clean. Should not be started before that spike, regardless of how appealing BBolt's feature fit is on paper.

None of these four are mutually exclusive long-term — a pluggable-backend design (one `IndexedDB`-shaped front-end, swappable storage engine chosen via config, matching how `Store<K,V>`-style typing might look) is plausible and even desirable, but picking *one* backend to ship first is the right way to start rather than building the abstraction before any concrete implementation exists to abstract over.

### Not reconsidered here: MongoDB proxy

Still deferred, unchanged from the prior draft: `OP_MSG` + BSON is a meaningfully heavier lift than any of the four options above (either the full `mongo-c-driver` dependency tree, or real hand-rolled binary-protocol work), and nothing in this round of feedback changed that calculus. Revisit only if a concrete need to talk to an existing MongoDB deployment specifically comes up.

## Open questions

- Which single backend to actually build first: the effort/risk ranking above favors Redis (D) or SQLite (A) as the least-risky starting points, with the Go-embedded option (C) gated behind an unresolved prototype question — not yet decided which of A/D goes first, since they're independent of each other.
- Whether the front-end API should be exactly spec-shaped `IndexedDB` (real event/callback semantics, `IDBRequest` objects) or a lightly simplified variant that keeps the *idiom* (callback-based, versioned schema, object stores) without matching every spec detail — leaning toward staying close to the real spec, since fidelity is the entire point of the "existing code/packages might port" goal, but not decided how far fidelity should extend (e.g. whether `IDBKeyRange`'s full API is worth matching exactly vs. a reduced range-query surface).
- Whether a pluggable-backend abstraction should exist from the start (one front-end, swappable engine via config) or whether the first implementation should hard-code its one chosen backend and only generalize once a second backend is actually built — leaning toward the latter (avoid designing the abstraction before there's a second real implementation to abstract over), not decided.
- The `SIGURG`/fiber-coexistence question for backend C is a genuine unknown, not a decided risk assessment — this document deliberately stops at "needs a direct spike," matching how `TDD-00006` treated the LLVM-coroutine question before it was actually tested.
- Whether backend D (Redis) should attempt `IDBCursor`/`IDBIndex` support via a secondary sorted-set structure, or ship without ordered-scan support at all and document that as a backend-specific limitation — not decided.
