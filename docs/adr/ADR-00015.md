# ADR-00015: Implement Date.parse(string)

- **Relations**: Extended by ADR-00017
- **Date**: 2026-07-11

## Context

Picked this up as the direct follow-on to `Date` itself (`ADR-00014`) — `Date.parse` was the one piece of that feature explicitly called out as not-yet-implemented there.

## Investigation

- **What scope is worth supporting.** Real JS's `Date.parse` accepts a wide grab-bag of formats (full ISO 8601 with timezone offsets, RFC 2822, and various engine-specific fallbacks). Implementing all of that means a general date-string grammar parser — disproportionate for what this is actually needed for in practice, which is almost always parsing a string that this same compiler's own `toISOString()` produced (a round-trip), or a similarly-shaped literal a program author writes by hand. Scoped to: the exact `toISOString()` shape (`YYYY-MM-DDTHH:mm:ss.sssZ`), the same shape without milliseconds (`YYYY-MM-DDTHH:mm:ssZ`), and a bare date (`YYYY-MM-DD`, interpreted as UTC midnight — this matches real JS's own documented rule for date-only strings). Non-UTC offsets (`+02:00`-style) are out of scope, consistent with `Date` itself being UTC-only throughout.
- **How to turn calendar fields back into a timestamp — the real design question.** The existing `Date` getters (`ADR-00014`) go the other direction (timestamp → calendar fields) via `gmtime()`, which returns a pointer to a library-owned buffer — no allocation concerns. The inverse, `timegm()`, is the opposite: it requires the *caller* to allocate a `struct tm`-shaped buffer, read and write it, and return a timestamp. Its problem is that `struct tm`'s exact byte size differs across libc's (both glibc and Darwin's BSD-derived libc append extra fields — `tm_gmtoff`, `tm_zone` — that `ADR-00014`'s decompose helper never had to account for, since it only *reads* the first 7 fields of an already-correctly-sized buffer the library allocated). Getting that byte layout wrong on one platform, silently, would be exactly the kind of platform-dependent bug this project has been deliberately avoiding (`ADR-00014`'s `clock_gettime`-over-`gettimeofday` choice was the same consideration in the other direction). Instead of `timegm()`, implemented Howard Hinnant's `days_from_civil` algorithm (a well-known, widely-used piece of public-domain integer arithmetic: http://howardhinnant.github.io/date_algorithms.html) directly as hand-written LLVM IR — pure integer math, no libc struct dependency at all, and it works correctly for any proleptic-Gregorian year including pre-1970 (negative timestamps), which `timegm` would also need extra care to get right depending on `time_t` signedness assumptions.
- **How to detect which of the three supported shapes a string is, and reject anything else cleanly.** Reused `sscanf` (not previously used anywhere else in the codebase, so this is the first `ensureSscanf()`), trying each of the three formats in order (full → no-millis → date-only) and checking the field-count return value for an exact match, rather than trying to hand-write a scanner. Any string that fails all three returns a sentinel.
- **What the sentinel should be.** Real JS's `Date.parse` returns `NaN` for unparseable input. This compiler's `Date`/`number` default representation is a plain `i64` with no NaN bit pattern to borrow (unlike `float64`, which does have one) — inventing a NaN-like sentinel value for `Date` specifically, that every downstream consumer would then need to special-case, is a bigger design change than this feature needs. Chose `-1` as a plain documented sentinel instead (a valid-looking but implausible almost-epoch value that's cheap to check for with a plain comparison) and called this out explicitly wherever it's documented (this ADR, `STATUS.md`, the example file) rather than leaving it as an implicit surprise.

## Decision

`Date.parse(dateString: string): number` — a static call (like `Date.now()`), returning a plain `number` (not a `Date`), matching real JS where the typical use is `new Date(Date.parse(s))`. Supports exactly the three shapes above, UTC-only. Returns `-1` for anything else.

## Implementation notes

- `codegen/llvm/emit_date.go`: `emitDateParse(args, pos)` — evaluates the single string argument, calls `@__kml_date_parse`, returns `Value{Ty: TypeI64}`.
- `codegen/llvm/emit_call.go`: dispatch alongside the existing `Date.now()` static-call check (`id.Name == "Date" && mem.Property == "parse"`).
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s `Date.now()` special case gained a sibling for `Date.parse` → `TypeI64`, so untyped declarations (`const a = Date.parse(...)`) and function-return-type propagation both work without any further changes — verified directly rather than assumed, since `ADR-00014` found three separate gaps in these exact code paths for `Date`/`Error` and I didn't want to repeat that blind spot.
- `codegen/llvm/runtime.go`: three new `ensure*()` helpers — `ensureSscanf()` (bare `declare`), `ensureDaysFromCivil()` (the Hinnant algorithm, translated field-by-field into `sdiv`/`select`/arithmetic IR, matching C's truncating-division semantics exactly since LLVM's `sdiv` also truncates toward zero), and `ensureDateParse()` (the three-attempt `sscanf` cascade feeding into `__kml_days_from_civil`, format strings declared once via the existing `internString()` helper).
- `codegen/llvm/emitter.go`: three new `used*` singleton-guard flags (`usedSscanf`, `usedDaysFromCivil`, `usedDateParse`), following the established `ensure*()` pattern exactly.

## Side effects discovered

None — verified the three `ADR-00014`-era type-inference gaps (untyped declaration, function-return-type propagation, chained-field-access fallback) were already generic enough to cover `Date.parse` for free, rather than assuming so.

- Manual repro matched against the same known reference values used in `ADR-00014`: `Date.parse("1970-01-01T00:00:00.000Z")` → `0`; `Date.parse("2023-11-14T22:13:20.000Z")` → `1700000000000`; the no-millis and date-only variants of the same string both also resolve correctly (the latter to UTC midnight, `1699920000000`); `Date.parse("not a date")` → `-1`; `new Date(Date.parse(...)).toISOString()` round-trips exactly.
