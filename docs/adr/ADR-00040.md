# ADR-00040: Support JSDoc @type overrides on interface fields

- **Date**: 2026-07-13

## Context

Tracked in `STATUS.md`'s Known Limitations & Bugs, the last of the three remaining "Medium" bugs after `docs/adr/ADR-00039.md`. An interface field annotated only `number` always resolved to `i64`, with no way to pin it to `float64`/`float32` — unlike a variable declaration, where `/** @type {float64} */` already works. This also meant `JSON.parse`'s float-field support (`ADR-00007`) was correctly implemented but practically unreachable through any interface, since there was no way to declare a float-typed interface field to parse into in the first place.

## Investigation

- `parseVarDecl` (`parser/parser.go:275`) already supports this for variables: it calls `doc := p.takeDoc()` before parsing the TS annotation, then overrides it afterward if `doc.GetType()` is non-empty.
- The JSDoc-capture mechanism itself (`Parser.peek()`, `parser.go:30`) is already fully generic and positionally-agnostic — it lazily stores the most recently skipped `JSDOC` token into `p.pendingDoc` regardless of what follows it, and `p.takeDoc()` just reads and clears that field. Nothing about it is specific to variable declarations.
- `parseInterfaceDecl`'s field-parsing loop (`parser.go:493`, at the time) simply never called `p.takeDoc()` at all — a JSDoc comment placed before a field got silently captured into `pendingDoc` (by the loop condition's own `p.check(lexer.RBRACE)`, which calls `peek()` internally) and then just as silently discarded, overwritten the moment the next JSDOC token (if any) was encountered, without ever being consulted.
- Confirmed `codegen/llvm/emitter.go`'s `resolveType` (used for both variable declarations and interface fields, via `Field{Name: af.Name, Ty: e.resolveType(af.Type)}`) needed **no changes at all** — it already falls through to the shared `ResolveTypeName(ta.Name)` for any plain, non-object/array/interface-registry name, which already handles `float64`/`float32`/`int8`..`int64`/etc. regardless of whether the annotation came from a `ts` or `jsdoc` source. The only actual gap was the parser never producing a `jsdoc`-sourced annotation for a field in the first place.

## Decision

Mirror `parseVarDecl`'s existing pattern exactly: capture `doc := p.takeDoc()` at the start of each field-parsing iteration in `parseInterfaceDecl`'s loop, then after parsing the field's TS type annotation, override it with `&ast.TypeAnnotation{Name: doc.GetType(), Source: "jsdoc"}` if the JSDoc comment specified a `@type`. No codegen changes needed, since `resolveType`/`ResolveTypeName` already handle a `jsdoc`-sourced annotation identically to a `ts`-sourced one.

## Implementation notes

- `parser/parser.go`, `parseInterfaceDecl`: added `doc := p.takeDoc()` before parsing each field's name, and the override check after parsing its type annotation — the same two lines `parseVarDecl` already has, applied per-field instead of once per declaration.
- `STATUS.md`: removed the bug row from Known Limitations & Bugs and the Roadmap's "Next up" table (no other rows needed updating — the existing "JSDoc extended floats ✅" row's wording was already general enough to not need correcting).

## Side effects discovered

None — this was a pure gap-fill; nothing about the existing mechanism needed to change, only be reached from one more place.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including two new tests below) all green; `make examples` 55/55.
- Manual repros, both directions: an interface field declared `/** @type {float64} */ score: number` correctly holds and prints `9.5` (not truncated to `9`), and `JSON.stringify`s as `9.5`; `JSON.parse('{"score":9.5}')` into that same interface correctly reads back `9.5` — confirming the originally-blocked `JSON.parse` float-field path (`ADR-00007`) is now actually reachable through an interface, not just theoretically implemented.
- New tests in `compiler_test.go`: `TestE2EInterfaceFloatField`, `TestE2EInterfaceFloatFieldJSONParse`.
- Extended `examples/interfaces/interfaces.ts` with a `Measurement` interface exercising both directions; manually ran the compiled binary and confirmed correct output across the full example, including every pre-existing line (no regression).
