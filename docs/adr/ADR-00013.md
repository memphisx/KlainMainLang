# ADR-00013: Implement object spread { ...obj, key: val }

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the first Phase 3 item, following the completion of Phase 2 (`ADR-00001` through `ADR-00012`).

## Investigation

- **Existing precedent to reuse.** Array spread (`[...a, ...b]`) already has a dedicated AST node, `ast.SpreadElement{Arg Expression}` (`ast/nodes.go`). Rather than inventing a separate node for object spread, I reused the same `SpreadElement` as an `ObjectProperty`'s `Value`, with an empty-string `Key` sentinel to distinguish it from a regular (or shorthand) property — `ObjectProperty{Key: "", Value: &ast.SpreadElement{Arg: expr}}`. This meant the parser change (`parseObjectLiteral`, right alongside last session's shorthand-property addition) was a single new branch checking for a leading `...` token, with zero new AST types.
- **The real design question was the merge/override semantics, not the parsing.** JS object spread has two properties worth getting exactly right: (1) a key mentioned more than once (via a later explicit property or a later spread) takes its *last* assigned value, and (2) a key's *position* in the object is fixed by its *first* occurrence — reassigning it later doesn't move it. I confirmed this by tracing what real JS does for `{ x: 100, ...p }` where `p = {x:1,y:2}`: `p`'s spread happens *after* the explicit `x: 100`, so `p.x` (`1`) wins over `100`, even though the explicit property was written first, textually — the win goes to whichever appears *later in source order*, not whichever is more "explicit."
- **Confirmed the codegen doesn't need explicit merge bookkeeping at value-assignment time.** Since a field's final byte offset in the constructed struct is fixed (via `inferObjectType` computing the merged field list once, up front), and codegen processes properties strictly in source order, a later property/spread naturally overwrites an earlier `store` to the *same* struct offset just by being emitted later in the instruction stream — I don't need to track "who wins" explicitly in the value-filling pass, only in the field-*layout* pass (`inferObjectType`), which needs an upsert (overwrite-in-place, not append) when a name repeats, to get JS's first-occurrence-position rule right.

## Decision

1. **Parser** (`parser/parser.go`, `parseObjectLiteral`): when the next token is `...`, parse the spread argument and append `ObjectProperty{Key: "", Value: &ast.SpreadElement{Arg: arg}}` — sitting alongside the existing shorthand/explicit property branches, so `{ ...a, b, c: 1, ...d }` parses freely mixed.
2. **`inferObjectType`** (`codegen/llvm/emit_exprs.go`): rewritten around an `upsert(field)` closure — if a field name already exists in the accumulated list, overwrite its `Ty` in place (preserving position); otherwise append. For a spread property, `upsert` is called once per field of the spread source's inferred object type (recursively resolved via the existing `inferExprType`, so a spread source can itself be a nested object literal, an interface-typed variable, etc.).
3. **`emitObjectLiteral`** (`codegen/llvm/emit_objects.go`): the final (already-merged) field layout comes from `inferObjectType` as before; a small `storeField(name, val)` closure does the coerce+GEP+store that used to be inlined per-property. For a spread property, load each of the spread source's fields (via `GetElementPtr` into the *source* struct) and call `storeField` for each, in the source object's own field order; for an explicit/shorthand property, evaluate its expression and call `storeField` once. Because properties are walked in the literal's source order and `storeField` always targets the same GEP index for a repeated name, later stores simply clobber earlier ones — exactly the semantics confirmed in Investigation, with no extra logic needed at this layer.
4. **Spreading a non-object value** (e.g. `{ ...someNumber }`) is a clean compile-time error (`spread in object literal requires an object value`), not silently broken codegen.

## Implementation notes

Files touched: `parser/parser.go` (`parseObjectLiteral`), `codegen/llvm/emit_exprs.go` (`inferObjectType`), `codegen/llvm/emit_objects.go` (`emitObjectLiteral`).

## Side effects discovered

None. Object spread's shallow-copy semantics (a nested object field is shared by pointer between the source and the spread result, not deep-cloned) fall out for free from this compiler's existing object representation (objects are always heap pointers; copying a field's value copies the pointer, not the pointee) — confirmed this matches real JS exactly via a manual repro (mutating a nested field through the spread copy is visible through the original), rather than assuming it.
