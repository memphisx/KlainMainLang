# ADR-00043: Fix forEach/HOF callbacks with console.log bodies or non-numeric elements

- **Date**: 2026-07-13

## Context

Found while verifying `docs/adr/ADR-00042.md`'s parameter-typing fix against real higher-order-function (HOF) usage: `nums.forEach((n) => console.log(n))` failed at the `clang` stage (`value doesn't match function result type 'i64'`). Confirmed pre-existing via `git stash` against the commit just before `ADR-00042` (reproduces identically) and unrelated to that fix — flagged in `STATUS.md` rather than fixed at the time, then picked up as its own piece of work once `ADR-00042` shipped.

## Investigation

- **First bug: `inferExprType` never recognizes `console.*` calls.** `emitArrowFunctionWithHints`'s expression-bodied branch (`af.Body != nil`) infers the closure's return type via `e.inferExprType(af.Body)`. For `(n) => console.log(n)`, `af.Body` is the `console.log(n)` call expression itself. `inferExprType`'s `*ast.CallExpression` case has an extensive list of recognized `*ast.MemberExpression` callees (`String.*`, `Number.*`, `Math.*`, `JSON.*`, `Date.*`, `performance.now`, `fs.*`, `process.*`, `crypto.*`) but never checked for `console` at all — so every `console.*` call fell through to the function's final, universal fallback, `TypeI64`. Confirmed the real evaluator disagrees: `emitConsolePrint` (backing `log`/`info`/`debug`/`error`/`warn`, and `emitConsoleDir` which delegates to it) and every other `console.*` emitter (`emitConsoleGroup`/`GroupEnd`/`Time`/`TimeEnd`/`Count`/`CountReset`/`Assert`) all return `Value{Ty: TypeVoid}`. So the closure got built expecting an `i64` return (from the wrong static inference) that its actual body — correctly evaluating to `void` — never produces.
- Confirmed no existing test or example ever exercised this shape: every `forEach`/`map`/`filter` usage in the corpus either used a block body, or an expression `inferExprType` already handles correctly (e.g. a `BinaryExpression` like `x * 2`).
- **Second, deeper bug, found while fixing the first**: once the return-type mismatch above no longer masked it, `nums.forEach((n) => console.log(n))` on a `string[]` still failed — a *different* error (`'%t16' defined with type 'ptr' but expected 'i64'`), this time in the callback's *parameter*. Traced to `emitArrayForEach` calling the plain `e.resolveCallback(args[0])`, which never supplies the array's element type as a hint for an unannotated callback parameter — unlike `emitArraySort`, which already calls `resolveCallbackWithHints` with `[]Type{elemTy, elemTy}` for exactly this reason. Checked every other HOF method and found the same gap in **all of them**: `map`, `forEach`, `filter`, `reduce`, `find`, `some`, `every`, `findIndex` all used the plain, hint-less `resolveCallback`. This had been invisible until now because every existing test/example only ever ran these methods over `number[]` arrays — an unannotated parameter's `i64` default (marked `Inferred: true` as of `ADR-00042`) happens to coincidentally match a numeric element type, masking the gap completely. The instant a `string[]` (or any non-numeric element type) is involved, an unannotated callback parameter silently gets the wrong type.
- `reduce`'s callback has a different, two-part signature (`(accumulator, currentElement) => newAccumulator`) where the accumulator's type comes from the *initial value* argument (`args[1]`), not the array's element type — needed its own hint computation rather than reusing the `[]Type{elemTy, TypeI64}` shape the other seven methods share. Resolving the callback (`args[0]`) happens *before* evaluating the initial value (`args[1]`) in the existing code, matching real JS's left-to-right argument evaluation order — computing the accumulator hint via `e.inferExprType(args[1])` (a pure, side-effect-free static check, not an early evaluation of the expression) preserves that ordering while still getting the right hint in time.

## Decision

Added a `case "console"` to `inferExprType`'s `*ast.MemberExpression`-callee handling, returning `TypeVoid` unconditionally (every `console.*` method returns void, confirmed above) — following the same one-line-per-namespace pattern already used for `String`/`Number`/`Math`/etc.

Changed all eight array HOF methods (`emitArrayMap`, `emitArrayForEach`, `emitArrayFilter`, `emitArrayReduce`, `emitArrayFind`, `emitArraySome`, `emitArrayEvery`, `emitArrayFindIndex`) from `e.resolveCallback(args[0])` to `e.resolveCallbackWithHints(args[0], []Type{elemTy, TypeI64})` (element type for the callback's first parameter, `number` for the optional index parameter) — mirroring `emitArraySort`'s already-correct pattern. `emitArrayReduce` uses `[]Type{accTyHint, elemTy}` instead, where `accTyHint := e.inferExprType(args[1])` is computed before resolving the callback but the actual evaluation of `args[1]` stays in its original position.

## Implementation notes

- `codegen/llvm/emit_exprs.go`: new `console` case in `inferExprType`'s `*ast.CallExpression` handling.
- `codegen/llvm/emit_arrays.go`: `emitArrayMap`, `emitArrayForEach`, `emitArrayFilter`, `emitArrayFind`, `emitArraySome`, `emitArrayEvery`, `emitArrayFindIndex` — each one's `resolveCallback` call replaced with `resolveCallbackWithHints(args[0], []Type{elemTy, TypeI64})`. `emitArrayReduce` — same replacement, but with `[]Type{accTyHint, elemTy}` and `accTyHint` computed via `inferExprType(args[1])` ahead of the callback resolution.
- `STATUS.md`: removed the now-fixed row from Known Limitations & Bugs and the Roadmap's "Next up" table.

## Side effects discovered

None new — both bugs were fully understood and fixed within their own scope; no further gaps found while verifying.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including nine new tests below) all green; `make examples` 55/55.
- Manual repros for every HOF method over a `string[]` with unannotated callback parameters: `forEach` (including the original `console.log` callback shape), `map`, `filter`, `find`, `some`, `every`, `findIndex`, and `reduce` (both a numeric accumulator built from `.length` and a string-concatenation accumulator) — all produce correct output where they previously failed at the `clang` stage.
- New tests in `compiler_test.go`: `TestE2EArrayForEachConsoleLogCallback`, `TestE2EArrayForEachUnannotatedStringParam`, `TestE2EArrayMapUnannotatedStringParam`, `TestE2EArrayFilterUnannotatedStringParam`, `TestE2EArrayFindUnannotatedStringParam`, `TestE2EArraySomeEveryUnannotatedStringParam`, `TestE2EArrayFindIndexUnannotatedStringParam`, `TestE2EArrayReduceUnannotatedStringAccumulatorAndElement`.
- Extended `examples/arrays/higher_order.ts` with a `string[]` section exercising every HOF method with unannotated callback parameters, including the `console.log` callback shape; manually ran the compiled binary and confirmed correct output across the full example, including every pre-existing line (no regression).
