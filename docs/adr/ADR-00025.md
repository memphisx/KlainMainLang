# ADR-00025: Implement process.execFileSync(file, args?) (V1: no options object)

- **Date**: 2026-07-12

## Context

Looked into how difficult Node's `child_process.execFileSync` would be to add to this compiler. Comparable effort to `fetch`/`fs.readFileSync`: no new type or IR shape needed, the exception/throw path reuses the existing `Error` mechanism; the one genuinely new piece is that this is the compiler's first multi-process operation â€” everything so far, `fs` and `fetch`, does I/O within a single process. Decided on a lean V1 (file + args only, string-only stdout, plain thrown `Error` on failure, no options object) and built it.

## Investigation

- **Read `emit_process.go` and `emit_fs.go` first** to confirm the existing conventions before writing anything: `process.*`/`fs.*` are recognized as pseudo-namespaces (a plain `MemberExpression` on a bare identifier, dispatched in `emit_call.go`'s big per-namespace `switch`), not real importable modules. A failed OS-level operation throws a catchable `Error` via the existing `@__kml_throw` mechanism (`emit_exceptions.go`) â€” `ensureFsThrow` builds its message from `strerror(errno)`; this feature needed its own message (an OS exit code or signal number, not `errno`), but the same throw mechanism.
- **Reused three existing patterns rather than inventing new ones**: the growable `{ ptr, i64, i64 }` buffer (`data`/`len`/`cap`), grown via `realloc` doubling, that `__kml_fetch`'s curl write callback already uses for accumulating a response body of unknown length â€” read straight off `runtime.go`'s `__kml_curl_write_cb`/`__kml_fetch` and copied almost verbatim for accumulating a child's stdout instead of a curl chunk. `resolveArrayForHOF` (`emit_arrays.go`) for resolving the optional `args: string[]` argument into a `(ptrReg, lenReg, elemTy)` triple â€” already handles both a named-variable array and an arbitrary array-valued expression, so no new array-resolution code was needed. And `process.argv`'s own representation (`emitProcessArgv`, `emit_process.go`) confirmed that a KML `string[]` is *already* laid out in memory exactly like a C `char**` (a contiguous block of `ptr`s) â€” meaning building `execvp`'s `argv` from a KML `args` array is just a `malloc` + one `store` (the file itself, at slot 0) + one `memcpy` (the args block, verbatim) + one `store null` (the terminator), not a real marshaling problem.
- **Chose real POSIX `pipe()`+`fork()`+`dup2()`+`execvp()`+`waitpid()`, not `popen()`.** `popen()` is the obvious first thing to reach for and is already what a hand-rolled `execSync`-equivalent would use, but it unconditionally runs the command through `/bin/sh -c "..."` â€” exactly the shell-interpolation behavior `execFileSync` (unlike `execSync`) is specifically supposed to avoid. Real `execFileSync` semantics require passing `argv` straight to `execve`-family with no shell in between, which `popen()` cannot do. Verified directly, with a real repro (see Verification), that this matters: an argument like `"$(whoami); rm -rf /tmp/nothing"` must come back out of the child's stdout completely unexpanded, not executed as an embedded shell command.
- **Wait-status decoding needed hand-rolled bit logic, not real C macros.** `WIFEXITED`/`WEXITSTATUS`/`WIFSIGNALED` are C preprocessor macros, not linkable functions, so they can't be `declare`'d and called from hand-written LLVM IR. Reimplemented the underlying bit pattern instead: the traditional Unix wait-status encoding (identical on Linux and Darwin/BSD) puts the low 7 bits at `0` for "exited normally" (with the exit code in bits 8-15), and a non-zero killing signal number in those same low 7 bits otherwise. Confirmed this is exhaustive, not just an approximation, given the actual call site: `waitpid` is called with no `WUNTRACED` flag, so a child can only ever be reported as exited or signaled â€” never stopped â€” meaning a plain "are the low 7 bits zero" branch correctly covers both real cases with nothing missed.
- **Deliberately did not capture stderr.** Multiplexing two pipes (stdout and stderr) without risking a deadlock needs `select()`/`poll()` over both file descriptors, real added complexity for a first pass. Chose to let the child's stderr fall through to this program's own stderr (inherited, not redirected) instead â€” visible on the terminal live, same as an uncaptured stream in a plain shell pipeline, just not available as a string afterward. Documented as a V1 scope narrowing, not a silent gap.
- **Confirmed `execvp`'s `$PATH` search happens for free**, not something needing separate implementation â€” passing a bare command name (no `/` in it) to `execvp` is the C library's own job, verified directly by calling `process.execFileSync("echo", ...)` (no path) successfully.

## Decision

`process.execFileSync(file: string, args?: string[]): string` â€” `fork()`s and `execvp()`s `file` with `argv = [file, ...args]` directly (no shell), captures stdout into a string via a pipe (grown with the same realloc-doubling pattern `fetch` already uses for its response body), and `waitpid()`s for the child to finish. Throws a catchable `Error` if the child exits non-zero (`"Command failed with exit code N: <file>"`) or dies to a signal (`"Command was terminated by signal N: <file>"`). No options object in this pass â€” `cwd`/`env`/`timeout`/`stdio` overrides are all deferred; stderr is inherited, not captured.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureExecFileSync()` â€” declares `pipe`/`fork`/`dup2`/`close`/`execvp`/`_exit`/`read`/`waitpid`, then hand-writes `@__kml_exec_file_sync(ptr %file, ptr %argsdata, i64 %argslen) -> ptr`. Builds `argv` via `malloc` + `store` + `memcpy` + `store null`; the child branch closes the read end, `dup2`s the write end onto fd 1, `execvp`s, and `_exit(127)`s if `execvp` itself failed (e.g. binary not found â€” the same `127` a real shell uses for "command not found", which falls out naturally here rather than being special-cased); the parent branch closes the write end immediately (so a fork failure, or a child that never writes, still terminates the read loop via EOF rather than hanging) before looping on `read()` into a 4096-byte stack buffer, appending each chunk into the growable heap buffer.
- `codegen/llvm/emitter.go`: added the `usedExecFileSync` singleton flag.
- `codegen/llvm/emit_process.go`: `emitProcessExecFileSync` â€” validates 1 or 2 arguments, coerces `file` to `TypePtr`, resolves the optional `args` array via `resolveArrayForHOF` (rejecting anything other than a plain `string[]` â€” object/array/function/dynamic element types all rejected with a clean compile error), defaults to a `null`/`0` pointer+length pair when `args` is omitted (safe: the IR's `hasargs` check skips the `memcpy` entirely when `argslen` is `0`, so the unused `null` `argsdata` is never dereferenced).
- `codegen/llvm/emit_call.go`: dispatch added as `case "execFileSync":` inside the existing `process` namespace switch.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s `process.readLineSync` single-case check was widened into a `switch` also covering `execFileSync` (both return `TypePtr`) â€” `emitVarDecl`'s untyped-inference path needed no separate change, since its `default:` branch for a `MemberExpression`-callee call already delegates to `inferExprType`.

## Side effects discovered

None â€” no pre-existing bugs surfaced while building or testing this feature.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, including the 10 new tests below), `make examples` all green (one `examples/fetch/fetch.ts` failure during a run was the already-documented httpbin.org flakiness, unrelated, and passed cleanly on retry).
- Manual repros, each actually compiled and run: `/bin/echo` with and without `args`; a bare `"echo"` (no path) resolving via `$PATH`; an argument containing `$(whoami); rm -rf /tmp/nothing` printed back completely verbatim (confirming no shell is involved); `/usr/bin/false` (non-zero exit) and `/bin/sh -c 'kill -9 $$'` (signal death) both throwing the correct, distinct messages; `/no/such/binary` throwing with exit code `127`; a `for`-loop generating 50,000 bytes of stdout (comfortably past a single pipe read and the growable buffer's initial capacity) coming back with the exact right length, confirming the realloc-doubling path.
- New tests in `compiler_test.go`: `TestE2EExecFileSyncCapturesStdout`, `TestE2EExecFileSyncNoArgs`, `TestE2EExecFileSyncResolvesViaPath`, `TestE2EExecFileSyncDoesNotInvokeAShell`, `TestE2EExecFileSyncNonZeroExitThrows`, `TestE2EExecFileSyncSignalDeathThrows`, `TestE2EExecFileSyncMissingBinaryThrows`, `TestE2EExecFileSyncLargeOutputGrowsBuffer`, `TestE2EExecFileSyncWrongArgCountRejected`, `TestE2EExecFileSyncNonStringArrayArgsRejected`.
- New example `examples/process/exec_file_sync.ts`, covering the same scenarios as the tests (capture, optional args, `$PATH` resolution, no-shell semantics, non-zero exit, and an explicit `/bin/sh -c` invocation to show a shell is still reachable when asked for).
