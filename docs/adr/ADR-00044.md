# ADR-00044: Fix array index out-of-bounds reads/writes with a runtime bounds check

- **Date**: 2026-07-13

## Context

Found during a bug-hunting pass: `const arr = [1, 2, 3]; arr[10] = 99;` left `arr.length` at 3 (no growth/resize), but a subsequent `arr[10]` correctly read back `99` — confirmed, via write-then-read-back, that the write physically landed past the allocated 3-element buffer in heap memory. Flagged in `STATUS.md`'s Known Limitations table as the highest-priority of four bugs found in that pass (real, unchecked heap corruption, not just wrong output), then picked up as its own piece of work this session.

## Investigation

`emitIndexPtr` (`codegen/llvm/emit_exprs.go`) is the single function backing both array reads (`emitIndex`) and array-element writes (`emitAssign`'s `arr[i] = ...`/`arr[i] += ...` path) — confirmed via `grep` that it has exactly these two call sites, so a fix here covers both directions at once. It computes a raw `getelementptr` off the array's data pointer at the given index, for both the named-symbol (`Symbol.Ptr`/`Symbol.LenPtr`) and aggregate-value (`{ptr, i64}` extractvalue) forms, with no comparison against the stored length anywhere.

The open design question flagged in `STATUS.md` was what happens on violation: hard abort vs. a catchable `Error`. This compiler already has a working setjmp/longjmp exception mechanism (`emit_exceptions.go`'s `@__kml_throw`/`try`/`catch`) used for `fs`/`fetch`/`process` runtime failures — reusing it for an out-of-bounds access is consistent with those existing failure modes and lets user code catch and recover from it, the same way real JS's `RangeError` would be catchable (even though a `RangeError` is the wrong specific error class to bother modeling — plain `Error` is what every other runtime-failure path already throws).

## Decision

Added a runtime bounds check directly inside `emitIndexPtr`, after computing both the index and the array's length (loaded from `sym.LenPtr` for the named-symbol form, `extractvalue {ptr, i64} ..., 1` for the aggregate form) but before the `getelementptr`:

- Single comparison `icmp uge i64 <index>, <length>` — since the index is coerced to a signed `i64` but compared as unsigned, a negative index also satisfies this check (it wraps to a huge unsigned value), so one comparison catches both "negative index" and "index >= length" without a separate sign check.
- On violation: branch to a block that mallocs an 8-byte `Error` struct (mirroring `emitNewError`'s exact shape), stores an interned `"Array index out of bounds"` message pointer into it, calls `@__kml_throw`, and terminates with `unreachable` — reusing the existing exception plumbing rather than a hard `abort()`, so it's catchable via `try`/`catch` like every other runtime failure in this compiler.
- On success: falls through to a second label that does the original `getelementptr` and returns, unchanged from before.

## Implementation notes

- `codegen/llvm/emit_exprs.go`: `emitIndexPtr` — added `lenReg` computation (both branches), the `icmp uge`/branch/throw/label sequence, before the pre-existing `getelementptr` line. No changes needed to `emitIndex` or `emitAssign` themselves, since both already just consume `emitIndexPtr`'s returned `gepReg`.
- No new `Symbol`/`Type` fields — the length was already tracked (`Symbol.LenPtr` / the aggregate's second field), just never read at this call site.

## Side effects discovered

None — this was a self-contained fix to a single function, as anticipated in `STATUS.md`'s original bug entry.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including five new tests below) all green; `make examples` 56/56 (new example added, see below).
- Manual repros: out-of-bounds read (`arr[5]` on a 3-element array) throws instead of returning garbage; out-of-bounds write (`arr[5] = 99`) throws instead of corrupting heap memory past the buffer, confirmed the write never happens by reading the array back afterward; negative index (`arr[-1]`) throws; all three are catchable via `try`/`catch` and print the `"Array index out of bounds"` message via `e.message`; an uncaught out-of-bounds access exits with a non-zero code; in-bounds reads/writes (index 0, last valid index, an interior write) are unaffected.
- New tests in `compiler_test.go`: `TestE2EArrayIndexOutOfBoundsReadThrows`, `TestE2EArrayIndexOutOfBoundsWriteThrows`, `TestE2EArrayNegativeIndexThrows`, `TestE2EArrayIndexOutOfBoundsUncaughtExitsNonZero`, `TestE2EArrayInBoundsAccessStillWorks`.
- New example `examples/arrays/array_bounds.ts`, covering in-bounds access, a caught out-of-bounds read, a caught negative-index write, and a subsequent successful in-bounds write — ran manually, output matches expectations.
- `STATUS.md`: removed the now-fixed row from Known Limitations & Bugs and the Roadmap's "Next up" table.
