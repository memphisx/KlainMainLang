# ADR-00005: Implement String.prototype.trimStart() / trimEnd()

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the next Phase 2 "quick win" from `STATUS.md`'s roadmap, following the empty-string-argument bug fixes in `ADR-00004`. Completes the trim family alongside the existing `.trim()`.

## Investigation

- **Existing `.trim()`.** `emitStringTrim` (`codegen/llvm/emit_strings.go`) delegates to `@__kml_trim` (`codegen/llvm/runtime.go`): a forward `skip_lead` loop finds the first non-whitespace byte, then a backward `trail_loop` (operating on the *already-leading-trimmed* substring) finds the last non-whitespace byte.
- **A reusable trick in `skip_lead`.** It never explicitly checks "have I reached the end of the string" — it doesn't need to, because a NUL byte (`0`) never matches any of the six whitespace-byte comparisons, so the loop naturally stops there anyway, with no separate bounds check. The same trick applies directly to a standalone `trimStart`.
- **Why `trim()`'s `trail_loop` can't be reused as-is for a standalone `trimEnd`.** `trail_loop` scans backward without an explicit lower-bound check, and that's safe *only* because it always operates on `start_p` — the substring left over *after* leading whitespace has already been stripped — which is guaranteed (by construction) to have a non-whitespace byte at position 0, so the backward scan can never walk past it. A standalone `trimEnd` has no such preceding step; scanning backward from the raw original string's last byte, an all-whitespace or empty string would walk the index below `0` with no explicit check — the same class of bug identified in `ADR-00004` (an operation implicitly assuming some input can't be "all edge case"), except caught here during design rather than via a runtime crash, having just been sensitized to look for it.

## Decision

Implemented two new, independent C-level LLVM helpers rather than trying to share code with `@__kml_trim` or with each other — a forward "skip and copy remainder" shape and a backward "scan with an explicit bounds check" shape are different enough that sharing would add more indirection than it would save:

1. **`@__kml_trim_start`** (`runtime.go`): the same `skip_lead` forward scan as `.trim()`, then copies everything from the first non-whitespace byte (or the NUL terminator, if the whole string is whitespace — the loop already stops there) to the end, including the terminator, in one `memcpy`. No separate empty-string case needed: if the remainder length is `0`, the general path already produces a correct 1-byte (just-NUL) buffer.
2. **`@__kml_trim_end`** (`runtime.go`): an explicit `is_empty` check up front (empty input → return an empty copy immediately), then a backward scan with an explicit `check_bound` block: if the current byte is whitespace *and* the index is `0`, the whole string is whitespace, so return an empty copy; otherwise decrement and continue. This avoids the out-of-bounds walk that a naive backward scan (mirroring `trim()`'s `trail_loop` without adaptation) would have.

Wired via new `emitStringTrimStart`/`emitStringTrimEnd` (`codegen/llvm/emit_strings.go`, same shape as `emitStringTrim`), gated by new `ensureStringTrimStart()`/`ensureStringTrimEnd()` (`runtime.go`, standard singleton-declare convention), dispatched from the existing string-method if-chain in `codegen/llvm/emit_call.go`.

## Implementation notes

Files touched:
- `codegen/llvm/emitter.go` — new `usedStringTrimStart`/`usedStringTrimEnd` fields.
- `codegen/llvm/runtime.go` — new `ensureStringTrimStart()`/`ensureStringTrimEnd()` and the `@__kml_trim_start`/`@__kml_trim_end` IR definitions.
- `codegen/llvm/emit_strings.go` — new `emitStringTrimStart`/`emitStringTrimEnd`.
- `codegen/llvm/emit_call.go` — `trimStart`/`trimEnd` dispatch alongside the existing `trim` entry.

## Side effects discovered

None new. The one relevant risk (out-of-bounds scan on an all-whitespace/empty input) was avoided by design here, directly informed by `ADR-00004`'s findings, rather than discovered via a failure this time.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including the extended `examples/strings/string_methods.ts`.
- Manual repros: leading+trailing whitespace (`"  hello  "` → `"hello  "` / `"  hello"`), no whitespace to strip (`"hello"` unchanged both ways), whitespace-only (`"   "` → `""` both ways), empty string (`""` → `""` both ways), and mixed whitespace characters (tabs, newlines, carriage returns) — all correct.
- New test `TestE2EStringTrimStartEnd` in `compiler_test.go` covers the above cases.
