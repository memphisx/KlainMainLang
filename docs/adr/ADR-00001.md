# ADR-00001: Fix closure capture-by-value bug (share mutable state with enclosing scope)

- **Date**: 2026-07-11

## Context

This was found while implementing `Array.prototype.forEach` (a Phase 2 roadmap item from `STATUS.md`). It is a real, pre-existing bug, not something introduced by `forEach` — it just happened to be the first feature whose natural test (a `forEach` callback accumulating into an outer variable) exercised it.

### How it was found

The E2E test for `forEach` was originally written as:

```ts
const nums: number[] = [1, 2, 3]
let sum: number = 0
nums.forEach((n: number) => {
    sum += n
})
console.log(sum)
```

This printed `0` instead of `6`. Minimizing the repro to remove `forEach` entirely confirmed the bug was in closures generally, not in the new `forEach` codegen:

```ts
let sum: number = 0
const inc = (n: number): void => { sum += n; }
inc(1); inc(2); inc(3)
console.log(sum) // printed 0, expected 6
```

And even without escaping the enclosing function at all:

```ts
function outer(): () => void {
  let total: number = 0
  const middle = (): (() => void) => { ... } // (see also the two limitations discovered below)
}
```

A simpler same-function repro also failed:

```ts
let sum: number = 0
const inc = (n: number): void => { sum += n; }
inc(1); inc(2)
console.log(sum) // printed 0
```

## Investigation

Root cause traced to `codegen/llvm/emit_func.go`, `emitArrowFunctionWithHints` and `emitClosureFunc`. At closure-creation time, each captured variable's **current value** is `load`ed from its stack `alloca` and copied into a field of a heap-allocated env struct (`envStructIR`/`envStructSize`, previously ~line 145-160). Inside the closure body, that env-struct field's address itself becomes the captured variable's storage (`%vcap_name = getelementptr ..., ptr %env, ...`, previously ~line 406-413).

Consequences of this design, confirmed by tracing the code and by manual repros:
- Mutations inside a closure are visible across **repeated calls to that same closure instance**, because the env struct persists on the heap across calls. This is why the pre-existing `TestE2EClosure` counter pattern (`let count = 0; return () => { count++; return count }`) already passed — it never reads `count` back through the *original* enclosing-scope alloca after the closure is created, only through repeated calls to the one returned closure.
- Mutations **never** propagate back to the enclosing scope's own variable (confirmed above).
- Two separate closures capturing the same outer variable get **independent copies**, not a shared cell (a JS closure would share one binding).

This diverges from JS/TS closure semantics, where all captures of a given binding must share one mutable cell.

Supporting research, used to design the fix without breaking existing behavior:
- `Symbol` (`codegen/llvm/emitter.go`, `Symbol{Ptr, LenPtr, Ty}`) is used completely generically for `load`/`store` everywhere (`emit_exprs.go` assignment codegen, ~line 429 at the time: `store <ty> <val>, ptr <sym.Ptr>`). Nothing downstream cares whether `Ptr` names a stack `alloca` or a heap `malloc`'d cell — confirmed by reading every read/write site.
- Variable declarations (`emit_exprs.go` VarDeclaration, ~line 1460) and function/arrow parameters (`emit_func.go`, ~line 82 and ~line 398) all just do `alloca` + `e.define(name, Symbol{Ptr: ...})` — no special-casing needed there.
- `gatherCaptures` (`emit_func.go`) already walks an arrow function's body (`scanExprFV`/`scanStmtsFV`) to compute exactly which enclosing-scope variables it references — this existing mechanism runs once per closure-creation site and was reused as-is.
- `CapturedVar{Name, Ty, Sym}` — `Sym` was only read once (at the value-copy site), so changing its meaning there was low-risk.
- No existing pre-pass determines "which variables will ever be captured by any nested closure" ahead of variable-declaration time. Building one (to decide `alloca` vs `malloc` up front) would require a generic whole-body AST walk to find every `ArrowFunction` node anywhere in a function, at any nesting depth, before any code is emitted.

### Alternative considered and rejected

A static whole-body pre-pass that decides, at each variable's declaration site, whether to `malloc` instead of `alloca` because some closure *later in the body* will capture it. Rejected because:
- It requires a new generic "find all arrow functions anywhere in this subtree" walker (existing `scanExprFV`/`scanStmtsFV` compute one arrow function's own free variables — they don't need to answer "does this whole body contain a closure that captures X").
- It doesn't compose as elegantly for nested closures (closure-inside-closure transitively capturing a grand-enclosing variable) — that case falls out for free under the alternative chosen below.

## Decision

**Lazy promotion at first capture.** Instead of deciding ahead of time, promote a variable to a shared heap cell only when a closure actually captures it — inside the existing capture-population loop in `emitArrowFunctionWithHints`, which already runs exactly once per closure-creation site, per `CapturedVar`.

1. **`Symbol` struct** (`emitter.go`): add `Boxed bool` — true once `Ptr` points to a heap cell shared with any closure(s) that capture it (as opposed to a private stack `alloca`).

2. **`Emitter.updateSymbolInPlace(name string, sym Symbol) bool`** (new, `emitter.go`, next to `lookup`): walks `e.scopes` innermost→outermost like `lookup`, and for the first scope whose map contains `name`, **overwrites that map entry in place** rather than shadowing it in the current (innermost) scope. This is the key correctness point: if promotion instead called `e.define` (which always writes to the innermost scope), the boxed replacement would vanish on `popScope()` when the enclosing block exits, and code after the block would silently fall back to the stale, pre-promotion alloca entry. Since `scope.syms` is a `map[string]Symbol` (reference type) even though `scope` itself is stored by value in `[]scope`, mutating the found map in place is safe and immediately visible to every future lookup at any scope depth.

3. **Capture-population loop in `emitArrowFunctionWithHints`** (where captures are copied into the env struct): for each `cap`:
   - If `!cap.Sym.Boxed` (first closure to capture this name): `malloc` a new cell sized `cap.Ty.Align()` bytes (reusing the existing `elemTy.Align()`-as-byte-size convention already used in `emit_arrays.go`'s `emitArrayMap`), `load` the variable's current value from `cap.Sym.Ptr`, `store` it into the new cell (one-time migration of whatever value existed at promotion time), then `updateSymbolInPlace(cap.Name, Symbol{Ptr: newCell, Ty: cap.Ty, Boxed: true})` so every future access to that name — from the enclosing scope, or from any later closure that captures it — goes through the same cell.
   - If already `Boxed`: reuse `cap.Sym.Ptr` directly, no re-copy (a previous closure, or an enclosing closure this one is nested inside, already owns the shared cell — re-copying here would silently revert to a stale value).
   - Store the **cell pointer** (not a value copy) into the env struct slot.

4. **`envStructIR` / `envStructSize`** (`emit_func.go`): since every env slot now always holds a cell pointer regardless of the captured variable's own type, both are simplified to be type-agnostic — every field is `ptr` (8 bytes): `envStructIR` returns `{ ptr, ptr, ... }`, `envStructSize` returns `len(caps) * 8`.

5. **`emitClosureFunc`'s captured-variable scope setup** (`emit_func.go`): previously GEP'd the env slot and used that address directly as the variable's storage. Changed to GEP the slot, `load ptr` from it to get the shared cell's address, then `e.define(cap.Name, Symbol{Ptr: <loaded cell ptr>, Ty: cap.Ty, Boxed: true})`. This is what makes **nested/transitive closures** work automatically: if this closure's own body creates another arrow function capturing the same name, `gatherCaptures` sees `Boxed: true` already and step 3 just reuses the pointer — one shared cell across arbitrarily deep nesting, with no extra nested-closure-specific code.

### Explicitly out of scope

- Arrays remain uncapturable — the existing `gatherCaptures` restriction (arrays can't be captured at all yet) is untouched.
- True ES6 per-iteration-fresh-`let`-binding semantics for a closure created inside a loop body (each iteration should get its own independent binding) was not specifically engineered for. There was no existing test/example for this before the fix, and getting it exactly right depends on this compiler's loop codegen structure (allocas live once in the entry block; only the loop body's straight-line instructions re-execute per dynamic iteration). Not addressed here.

## Implementation notes

Files touched:
- `codegen/llvm/emitter.go` — added `Boxed` field to `Symbol`; added `updateSymbolInPlace`.
- `codegen/llvm/emit_func.go` — `envStructIR`, `envStructSize`, the capture-population loop in `emitArrowFunctionWithHints`, and the captured-var scope setup in `emitClosureFunc`.

Nothing needed to change in variable declaration sites, assignment codegen, or `lookup`/`define` themselves — confirmed during investigation that they're all already generic over what `Symbol.Ptr` points to.

## Side effects discovered

Three other issues surfaced during this work, none of which are part of this ADR's fix:

1. **Void-callback crash (fixed as a prerequisite, not part of this ADR).** While first testing `forEach`, callbacks with a block body and no `return` statement (e.g. `(x) => { console.log(x); }`) were unconditionally typed as returning `i64` in `emitArrowFunctionWithHints`, and their body fell through to LLVM `unreachable` since there was no `return` to hit — causing a runtime crash (exit 133) for any void callback. Fixed by adding a `blockHasReturn`/`stmtHasReturn` recursive scan (`emit_func.go`) that defaults block-bodied arrow functions with no reachable `return` to `TypeVoid` instead of `TypeI64`. This fix is prerequisite plumbing for `forEach` and is not re-described in detail here; see the `forEach` work in `compiler_test.go`/`emit_arrays.go` for that context.
2. **Calling a closure returned from an unannotated arrow-typed `const` isn't resolved as callable** — e.g. `const middle = () => { ...; return innerFn }; const inner = middle(); inner()` fails with `undefined function or closure 'inner'`, but works fine if `middle` has an explicit `function middle(): () => void { ... }` declaration instead. Not fixed here — tracked in `STATUS.md` under "Known Limitations & Bugs".
3. **Parenthesized function-type return annotations aren't parsed** — e.g. `(): (() => void) => { ... }` gives a parse error (`expected IDENT, got return`); only unparenthesized function-type annotations (`(): () => void`) work. Not fixed here — tracked in `STATUS.md` under "Known Limitations & Bugs".

Both (2) and (3) were discovered while trying to construct a nested-closure test case using the natural "closure returns a closure" pattern; the nested-closure verification below instead uses a pattern that avoids both (an inner arrow function defined and invoked directly inside an outer arrow function's own body, both closing over a grand-enclosing variable), which is unaffected by either limitation.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass, including the pre-existing `TestE2EClosure` (repeated-call counter semantics unchanged).
- `make examples` — all 40 examples pass, including `examples/closures/closures.ts`.
- Manual repro confirms the originally-reported bug is fixed:
  ```ts
  let sum: number = 0
  const inc = (n: number): void => { sum += n; }
  inc(1); inc(2); inc(3)
  console.log(sum) // now prints 6
  ```
- Manual repro confirms independent closure instances stay independent (two separate `makeCounter()` calls don't share state — `c1()=1,2`, `c2()=1`).
- Manual repro confirms nested/transitive capture works:
  ```ts
  let total: number = 0
  const outer = (): void => {
    const inner = (): void => { total += 10; }
    inner(); inner()
  }
  outer(); outer()
  console.log(total) // 40 (2 outer calls x 2 inner calls x 10)
  ```
- New tests added to `compiler_test.go`: `TestE2EClosureMutatesOuterScope`, `TestE2EClosureIndependentInstances`, `TestE2ENestedClosureCapture`.
- `TestE2EArrayForEach` restored to its natural form (`sum += n` inside the callback) now that the bug it originally exposed is fixed, instead of the `console.log`-only workaround used while the bug was still open.
- Examples updated: `examples/closures/closures.ts` (added a "Mutating a captured variable is visible in the enclosing scope too" section) and `examples/arrays/higher_order.ts` (added a `forEach` accumulator section).
