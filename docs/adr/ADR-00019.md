# ADR-00019: Implement Date formatting (toDateString / toLocaleDateString)

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the next `Date` follow-on after arithmetic (`ADR-00018`) — `toISOString` (`ADR-00014`) was already implemented, but the two other common formatting methods weren't.

## Investigation

- **`toDateString`'s format is fixed and locale-independent in real JS** — it's one of the few `Date` string methods with no locale variance at all, so full fidelity to real JS's exact shape (`"Www Mon DD YYYY"`, e.g. `"Thu Jan 01 1970"`) was achievable without any scope-narrowing beyond the already-established UTC-vs-local decision (`ADR-00014`). Double-checked the exact shape carefully rather than assuming from memory: the day-of-month field is zero-padded to 2 digits (`"01"`, not `"1"`) — caught and fixed this after an initial implementation used the unpadded form, verified by re-checking against the well-known real JS behavior before finalizing.
- **`toLocaleDateString` is a genuinely different problem** — real JS's version is locale-dependent by design (`Intl.DateTimeFormat` under the hood), varying not just in field order but in punctuation, calendar system, and even digit script depending on the locale argument or host locale. Implementing that properly would mean bundling locale/calendar data this compiler has no other use for — a much bigger undertaking than anything else in the `Date` feature set, and disproportionate to the value for a personal, exploratory compiler project. Scoped down to exactly one fixed, deterministic format: the default (no-locale-argument, en-US-shaped) `"M/D/YYYY"` output (e.g. `"11/14/2023"`, unpadded month/day) — no locale parameter is accepted at all. This mirrors the same "one fixed deterministic shape, not full spec fidelity" scope-narrowing already used for `Date.parse` (`ADR-00015`/`ADR-00017`) and the setters (`ADR-00016`).
- **Needed a genuinely new codegen pattern: a runtime string lookup table.** `toDateString` needs weekday (`"Sun"`..`"Sat"`) and month (`"Jan"`..`"Dec"`) abbreviations, but the index into those name lists is only known at run time (the decomposed `weekday`/`month` fields from `__kml_date_decompose`, `ADR-00014`) — there's no way to resolve this with a Go-side compile-time `switch` the way `emitDateCall`'s method dispatch does. Solved with a small, reusable pattern: two global LLVM arrays of string pointers (`@__kml_weekday_names`, `@__kml_month_names`), each built once via the existing `internString` helper for the individual name literals, indexed at the call site with a plain `getelementptr` + `load` using the runtime index register. This is the first place in the codebase needing a runtime-indexed string table (every other Date formatter so far — `toISOString` — only needed numeric `sprintf` fields), so it's introduced here as a small, targeted, reusable primitive rather than something built into `emitDateToDateString` itself.

## Decision

- `toDateString()`: `"Www Mon DD YYYY"`, exact real-JS shape, UTC.
- `toLocaleDateString()`: one fixed `"M/D/YYYY"` format (unpadded), no locale argument, UTC. Documented as intentionally not implementing `Intl`-style locale support.

## Implementation notes

- `codegen/llvm/emit_date.go`: `weekdayAbbrevs`/`monthAbbrevs` (the two Go-side name lists, backing the runtime lookup tables); `emitDateToDateString` (decompose → GEP+load into both name tables → `sprintf` with `%s %s %02lld %04lld`); `emitDateToLocaleDateString` (decompose → `sprintf` with `%lld/%lld/%lld`, month converted from the 0-indexed decomposed value to 1-indexed the same way `toISOString` already does). Both added to `isDateMethodName` and `emitDateCall`'s dispatch switch.
- `codegen/llvm/runtime.go`: new `ensureDateNameTables()` — builds the two `internString`-backed global pointer arrays once, guarded by a new `usedDateNameTables` singleton flag (`emitter.go`).
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s existing `"toISOString"` case (Date methods returning `TypePtr`) extended to include the two new method names.

## Side effects discovered

None.

## Verification

- `go build .`, `go vet ./...`, `go test ./...`, `make examples` (all pass, examples 43/43).
- Manual repros against two reference dates (epoch, and the `1700000000000` reference timestamp used throughout the other Date ADRs): `toDateString()` → `"Thu Jan 01 1970"` / `"Tue Nov 14 2023"`; `toLocaleDateString()` → `"1/1/1970"` / `"11/14/2023"`. Caught and fixed the unpadded-day bug in `toDateString` during this verification pass, before considering the feature done.
- Confirmed both work through untyped variable inference and as a function's declared return value, not just as an immediate `console.log` argument — the same two `Date`-as-`TypePtr` return-type paths already established for `toISOString`.
- New tests in `compiler_test.go`: `TestE2EDateToDateString`, `TestE2EDateToLocaleDateString`, `TestE2EDateFormattingUntypedInferenceAndFunctionChain`.
- Extended `examples/date/date.ts` with a Date-formatting section.
- `STATUS.md`: added `toDateString`/`toLocaleDateString` rows to the Performance & Timing table.
