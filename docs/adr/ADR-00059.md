# ADR-00059: Fix Map/Set method calls, .size, and for...of only recognizing a plain named variable

- **Relations**: Extends ADR-00058
- **Date**: 2026-07-14

## Context

Follow-on to `ADR-00058` (fixing `Map<K,V>`/`Set<T>` silently resolving to `i64` as a type annotation) ‚Äî once an interface field could correctly be typed `Map<K,V>`, the next natural check was whether *using* that field actually worked. It didn't: `interface Container { scores: Map<string, number> }`, then `c.scores.get('a')` failed with `"only simple function calls are supported"` ‚Äî the exact same class of confusing, unrelated-looking error `ADR-00058`'s own investigation already saw once for the type-annotation bug.

## Investigation

Traced to three separate dispatch points in `codegen/llvm`, all sharing the identical shape of gap: each one only recognized a Map/Set receiver when the receiver expression was a **plain identifier** (`m.get(...)`, where `m` is directly a named variable), via a `mem.Object.(*ast.Identifier)` type assertion followed by `e.lookup(id.Name)`. Anything else ‚Äî a field access (`c.scores`), an array index (`arr[0].scores`), a function call's result ‚Äî isn't an `*ast.Identifier` at all, so the assertion fails and the whole Map/Set branch is silently skipped, falling through to an unrelated error further down:

- **Method calls** (`emit_call.go`): the `Map`/`Set` dispatch block gated on `mem.Object.(*ast.Identifier)`.
- **`.size`** (`emit_exprs.go`, in both the real evaluation path and `inferExprType`): same identifier-only gate.
- **`for...of`** (`emit_stmts.go`'s `emitForOf`): the non-identifier (`else`) branch only checked `arrVal.Ty.IsArray`, with no `IsMap`/`IsSet` case at all ‚Äî a Map/Set-typed non-identifier iterable fell straight to a generic `"for...of requires an array, Map, or Set value"` error.

This is the exact same "named-variable-only" limitation `ADR-00016` already documented for `Date`'s setters, and the exact problem `resolveArrayForHOF` (`emit_arrays.go`) already solved for arrays years earlier in this codebase's history ‚Äî arrays have long supported both a named identifier (reusing its existing `Ptr`/`LenPtr` allocas) and an arbitrary expression (evaluating it and extracting the `{ptr, i64}` aggregate directly). Map/Set method dispatch had simply never been generalized the same way, since every existing test/example happened to always call Map/Set methods on a directly-named variable.

## Decision

Added `resolveMapOrSetForCall(objExpr ast.Expression, pos ast.Pos) (Type, string, error)` (`emit_collections.go`), mirroring `resolveArrayForHOF`'s exact two-branch shape: a plain identifier loads the map/set's heap pointer from its alloca (the named-variable case, unchanged behavior); anything else is evaluated directly via `emitExpr`, since a field-access GEP+load (or an array-index GEP+load, or a call's return value) already yields the heap pointer itself with no separate alloca indirection to unwrap.

Refactored `emitMapCall`/`emitSetCall` (`emit_collections.go`) and `mapOrSetValuesArray` (same file) to accept an already-resolved `(Type, ptr string)` pair instead of a `Symbol` ‚Äî the "load from `sym.Ptr`" step moved out to each call site (now `resolveMapOrSetForCall`, or `emitForOf`'s own identifier branch, which still needs its own load since `mapOrSetValuesArray` no longer does it internally).

Updated the three dispatch points to use a cheap `inferExprType` pre-check (no side effects ‚Äî the same idiom `"slice"`/`"indexOf"`/`"at"` already use to disambiguate array vs. string receivers) before committing to the real evaluation:
- `emit_call.go`'s Map/Set method dispatch: `if objTy := e.inferExprType(mem.Object); objTy.IsMap || objTy.IsSet` instead of the identifier-only assertion.
- `emit_exprs.go`'s `.size` (both the evaluation path and `inferExprType`'s own mirror of it): added an `else if` branch for the non-identifier case.
- `emit_stmts.go`'s `emitForOf`: the non-identifier branch's `switch` gained an `IsMap || IsSet` case alongside the existing `IsArray` one, routing through `mapOrSetValuesArray` the same way the identifier branch already does.

## Implementation notes

- Files touched: `codegen/llvm/emit_collections.go` (`resolveMapOrSetForCall`, `emitMapCall`/`emitSetCall`/`mapOrSetValuesArray` signature changes), `codegen/llvm/emit_call.go` (method-call dispatch), `codegen/llvm/emit_exprs.go` (`.size`, both call sites), `codegen/llvm/emit_stmts.go` (`emitForOf`).
- Both existing callers of `emitMapCall`/`emitSetCall` were the two spots already listed above ‚Äî a small, fully contained refactor, not a wide-reaching signature change across the codebase.

## Side effects discovered

None beyond the three dispatch points themselves ‚Äî all three shared the identical root cause and were found together by directly testing the same interface-field scenario end to end (method calls, `.size`, `forEach`, `for...of`), not discovered one at a time across separate sessions.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` ‚Äî all green.
- Manual repros, each confirmed broken before the fix and correct after, on a `Map<string, number>`/`Set<string>`-typed interface field: `.get()`/`.set()`/`.has()`/`.size`/`.clear()`/`.forEach()` all called through `c.scores.*`/`c.tags.*`; `for (const t of c.tags)`; and, as a further check beyond the original interface-field trigger, the same Map accessed through an array index (`arr[0].scores.get(...)`) ‚Äî confirming the fix is general, not narrowly patched for interfaces specifically.
- New E2E tests in `tests/collections_test.go`: `TestE2EMapFieldAccessMethodCalls`, `TestE2ESetFieldAccessMethodCalls`, `TestE2EMapFieldAccessForEach`, `TestE2ESetFieldAccessForOf`, `TestE2EMapArrayIndexMethodCall`, `TestE2EMapFieldAccessClear`.
- `make examples` ‚Äî full suite still passes (one transient, unrelated `httpbin.org` network flake on the first run, clean on immediate re-run ‚Äî the same known flakiness already documented in `examples/fetch/fetch.ts`'s own comment).
