# ADR-00050: Non-blocking await fetch(...) via libcurl's multi-interface (TDD-00006 Part 2, second real slice)

- **Relations**: Extends ADR-00049. Implements TDD-00006 (Part 2 — `fetch()` slice; general `Promise.all`/`.race`/`.allSettled` still not started)
- **Date**: 2026-07-14

## Context

Picked up as the direct continuation of `ADR-00049`'s concurrent HTTP connection handling: that ADR closed the "connections block each other" gap, but `fetch()` itself was still a single blocking `curl_easy_perform` call — an HTTP handler that called an upstream API via `await fetch(...)` would still freeze the *entire* process (every fiber, not just its own) for the whole duration of that call, since fibers on one OS thread don't preempt a blocking syscall. Closing this was the explicit, scoped continuation of `docs/tdd/TDD-00006.md`'s Part 2 after its fiber mechanism was proven — the real value target the whole event-loop effort was building toward: a microservice handler that calls an upstream API without stalling every other client.

## Investigation

- **libcurl's multi-interface is a real, unfamiliar API surface** — checked its exact function signatures and struct layouts directly against `curl.h`/`multi.h` (both present on this dev machine), the same "never trust from memory" standard `ADR-00048`'s socket constants already established, rather than assuming API shapes. Confirmed: `curl_multi_init`/`curl_multi_add_handle`/`curl_multi_perform`/`curl_multi_fdset`/`curl_multi_info_read`/`curl_multi_remove_handle`; `CURLOPT_PRIVATE`=10103, `CURLINFO_PRIVATE`=1048597, `CURLMSG_DONE`=1 (derived from the header's own type-tagged constant scheme, e.g. `CURLINFO_STRING`(0x100000) + 21, matching exactly how the pre-existing `CURLINFO_RESPONSE_CODE`=2097154 was already derived); `CURLMsg`'s field offsets (`msg` at 0, `easy_handle` at 8, the `result`/`whatever` union at 16) read as raw bytes rather than trusted to a hand-written LLVM struct type, avoiding any doubt about compiler-inserted padding.
- **Prototyped the concurrency claim in isolation before touching the compiler**: a small C program driving two `curl_multi` transfers (one against a 1.5s-delay local test endpoint, one against a 0.1s one) via `curl_multi_perform`/`curl_multi_fdset`/`select()`/`curl_multi_info_read` — confirmed the fast transfer completed at ~0.11s while the slow one was still in flight, and total wall-clock was ~1.5s (the max), not ~1.6s (the sum) — genuine overlap, not just API plumbing that happens to compile.
- **Ported that design into the real compiler, then hit a real linking bug**: the event loop (`__kml_event_loop_run`) needs to reference curl-multi symbols (`curl_multi_fdset`/`perform`/the drain helper) *unconditionally* in its own IR, because whether a program's `http.listen` handler will ever call `fetch()` isn't knowable at the point the event loop itself gets built (the handler's body, where a `fetch()` call might live, is only emitted afterward by `buildHTTPDispatcher`). Fixed by having `ensureHTTPRuntime` always pull in the full async-fetch machinery (and, transitively, libcurl) rather than trying to make it conditional — a real, if modest, new dependency for any program using `http.listen`, even one that never calls `fetch()` itself.
- **Found and fixed a second, larger, fully independent bug while making the target scenario (an async HTTP handler awaiting fetch) actually work**: async *arrow* functions (`async (req) => { ...; return ...}`) silently returned their value unwrapped instead of via the malloc'd Promise slot every caller expects — `emitClosureFunc` never set up the async prologue/epilogue at all, unlike `emitFunctionDecl` (named functions), which already did. Diagnosed by isolating a minimal repro (a plain async arrow-function `http.listen` handler that returns immediately, no `fetch` involved) after seeing the full scenario crash with `curl exit 52` (empty reply) right after the handler's own body had visibly finished — confirmed it wasn't fetch-related at all before looking further. This bug had been invisible until now because named top-level functions can't be passed by reference either (a separate, already-tracked limitation) — meaning an async arrow function used to be the *only* way to get an async callback in this language at all, and nothing had ever exercised one.
- **`http.listen`'s handler-type validation needed to accept an async handler** (`Promise<T>` return type, not just `T` directly) for the target scenario to be expressible at all — `await` can only appear inside an async function, so an HTTP handler that awaits `fetch(...)` must itself be async.

## Decision

**`ensureFetchAsync`** (`runtime.go`, new): declares the multi-interface functions and a global `@__kml_curl_multi` handle (created lazily on first use), and defines three hand-written IR helpers:
- **`__kml_fetch_async(ptr url) -> ptr`**: builds the easy handle exactly like the old blocking `__kml_fetch` (same setopts: URL, write callback/data, follow-location, timeout, nosignal), attaches a new 40-byte pending-fetch struct (`{ ptr easy, ptr buf, i64 done, i64 httpStatus, i64 curlResult }`, every field ptr/i64 — the same no-padding-ambiguity convention the timer queue and connection array already use) to the easy handle via `CURLOPT_PRIVATE`, adds it to the multi handle, calls `curl_multi_perform` once to kick the transfer off, and returns immediately.
- **`__kml_curl_drain_messages()`**: drains `curl_multi_info_read`'s completed-transfer queue; for each `CURLMSG_DONE`, retrieves the pending struct via `CURLINFO_PRIVATE`, records the HTTP status and `CURLcode` result into it, removes+cleans up the easy handle, and marks `done`. Shared by the event loop (called after every `select()` wake) and the busy-spin fallback below.
- **`__kml_await_fetch(ptr pending) -> { i64 status, ptr body }`**: loops until `pending->done`. If running inside a connection fiber (`@__kml_current_conn_idx >= 0`): stores `pending` into that connection's own array entry (a new fourth field, `pendingFetch`) and `swapcontext`s back to `@__kml_main_ctx` — reusing `ADR-00049`'s exact yield/resume mechanism, not a new one. Otherwise (top-level code, no event loop to yield into): busy-spins `curl_multi_perform`+drain directly, behaviorally equivalent to the old blocking wait. Once done, throws a catchable Error on a transfer-level failure (identical shape to the old `__kml_fetch`'s own error path) or returns the final status/body.

**Event loop** (`__kml_event_loop_run`, extended, not replaced): the `fd_set`-building step now also allocates write/exception fd_sets (previously always `null`, since only reads mattered before) and merges in libcurl's own three fd_sets via `curl_multi_fdset` whenever `@__kml_curl_multi` is non-null. After `select()` wakes, for any reason, an unconditional `curl_multi_perform`+drain step runs before the existing listener/connection-resume logic. The resume-scan loop's condition is now two-way: a connection with `pendingFetch == null` resumes on fd readiness exactly as `ADR-00049` already had it; a connection with `pendingFetch != null` resumes when *that specific fetch* is done, regardless of its own fd's readiness (it isn't waiting on its own socket right now). **Deliberately not queried: `curl_multi_timeout()`** — curl's own fds are already in the shared `fd_set`, so `select()` already wakes promptly on real curl activity; skipping the extra timeout call is the same "simplicity now, revisit only if it becomes a real bottleneck" call this project already made for `select()` over `epoll`/`kqueue`.

**`emitFetch`/`emitAwait`** (`emit_fetch.go`/`emit_async.go`): `fetch()` itself shrank to "kick off `__kml_fetch_async`, wrap the returned pending handle in a Promise slot" — building the actual `Response` object moved to `emitAwait`'s new `promiseTy.IsResponse` branch (fetch is the only producer of `Promise<Response>` in this language, so this check is unambiguous), which calls `__kml_await_fetch` and only then builds the `Response` struct, mirroring exactly what `emitFetch` used to do inline before the transfer could even be known to have finished.

**`emitClosureFunc`** (`emit_func.go`, bug fix, not new scope): given the same treatment `emitFunctionDecl` already has for named async functions — sets up `e.isAsync`/`e.currentPromiseTy`/`e.coroRetLabel`/`emitAsyncPrologue()` when `af.IsAsync`, and routes both the block-bodied and expression-bodied cases through `emitAsyncEpilogue()` instead of a plain `ret`.

**`emitHTTPListen`** (`emit_http.go`): now accepts a handler whose return type is `Promise<T>` (an async handler) in addition to a plain `T`, unwrapping to `T` for the status/body field validation either way. The dispatcher's handler-call result gets one extra `load ptr` indirection when the handler is async — `Promise<T>` and a plain object `T` share `IR="ptr"`, so the call instruction itself needs no changes, only how its result is interpreted afterward.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureFetchAsync` (new); `httpNonblockFlag`/`httpEagainErrno` reused unchanged from `ADR-00049`; `ensureFiberRuntime` extracted as its own singleton (previously inlined in `ensureHTTPRuntime`) so `fetch()`-only programs (no `http.listen`) still get the fiber globals `__kml_await_fetch` unconditionally references, without pulling in unrelated socket declares; `ensureHTTPRuntime` now calls `ensureFetchAsync` unconditionally (see the linking-bug note above); connection-array entries grew from `{ i64 fd, ptr ctx, ptr stack }` (24 bytes) to `{ i64 fd, ptr ctx, ptr stack, ptr pendingFetch }` (32 bytes) — every existing GEP referencing the old 3-field shape (in `__kml_http_append_conn` and `__kml_event_loop_run`) updated to match.
- `codegen/llvm/emit_http.go`: `buildHTTPDispatcher`'s self-lookup GEP updated to the 4-field struct shape; new `isAsyncHandler` parameter and the extra promise-unwrap step described above.
- `codegen/llvm/emit_fetch.go`, `codegen/llvm/emit_async.go`: as described above.
- `codegen/llvm/emit_func.go`: `emitClosureFunc`'s async wiring, mirroring `emitFunctionDecl`'s existing pattern field-for-field.
- `codegen/llvm/emitter.go`: new `usedFetchAsync`/`usedFiber` flags.

## Side effects discovered

The `emitClosureFunc` async bug (described above) — fixed within this same piece of work, not deferred, since it was a hard blocker for the target scenario (an async HTTP handler calling `await fetch(...)`) to be expressible at all, not an unrelated tangent.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including all pre-existing `TestE2EFetch*`/`TestE2EHTTPListen*`/`TestE2EAsync*` tests plus the new ones below) all green throughout; `make examples` 58/58 (no example changes needed).
- All 10 pre-existing `fetch()` tests — including the network-failure-throws and redirect-following cases, the two most likely to reveal a behavioral difference between `curl_easy_perform` and the multi-interface — pass unchanged against the new implementation, confirming the two curl driving mechanisms are wire-level equivalent for this project's purposes.
- The decisive manual verification, run against the real compiled output (not just the scratch prototype): an `http.listen` server whose async handler does `await fetch(upstream + req.path)`, hit with two concurrent requests — one resolving against a 1.5s-delay upstream path, one against a ~50ms path. The fast request completed in ~66ms while the slow request's own fetch was still in flight; both received correct bodies.
- New tests: `tests/http_test.go`'s `TestE2EHTTPListenAsyncHandlerAwaitFetch` (single-request correctness: an async handler's `await fetch(...)` result reaches the client correctly) and `TestE2EHTTPListenConcurrentAwaitFetch` (the concurrency proof — a slow-upstream request and a fast-upstream request fired concurrently at the server; the fast one must complete in well under half the slow upstream's own delay, or the test fails, catching a regression back to serialized behavior). `tests/async_test.go`'s `TestE2EAsyncArrowFunctionBlockBody`/`TestE2EAsyncArrowFunctionExpressionBody` (regression coverage for the `emitClosureFunc` fix, independent of HTTP/fetch).
- `STATUS.md`: `fetch(url)`'s row updated to describe the non-blocking rework; `async`/`await` rows noted (async arrow functions now correctly supported; `await` on `fetch()`'s promise really waits). Roadmap's Event Loop entries updated to reflect Part 2 now covering both HTTP concurrency and fetch concurrency, with `Promise.all`/`.race`/`.allSettled` named as the remaining gap. `docs/tdd/TDD-00006.md` updated to match (Status line, Decision paragraph, Prerequisites table, Open Questions); `docs/tdd/README.md`'s index updated.
