# ADR-00030: Implement Memory.free(x) (Stage 1 of the manual-memory-management plan)

- **Relations**: Implements TDD-00001
- **Date**: 2026-07-12

## Context

Looked into how complicated `Memory.free(x)` — Stage 1 of the staged manual-memory-management plan already written up in `STATUS.md`'s "Memory Management" section — would be to build. Concluded the core cases (string, plain object, array) are genuinely easy (a single compile-time type dispatch, reusing machinery that already exists), with the real decision being shallow vs. deep free, and `Map`/`Set` needing their own dedicated free helper since their internal backing buffers aren't reachable through the generic dispatch. Decided on shallow free (matching Stage 1's own "no analysis, no safety net, exactly like C" framing) and built it, giving `Map`/`Set` their own helper as part of the same pass.

## Investigation

- **Confirmed the exact heap shape for every freeable type before writing any dispatch code**, rather than assuming:
  - `string`/plain `object` values are already a single `ptr` — `Value.Ref` (or a `Symbol`'s loaded `Ptr`) *is* the heap pointer directly. Trivial.
  - `array` is the one type with a *dual* representation this compiler already has to handle everywhere (`Symbol{Ptr, LenPtr}` two-alloca form vs. a `{ptr, i64}` aggregate `Value` form) — resolving either into the actual data pointer needed the same `extractvalue`/two-alloca handling `resolveArrayForHOF` already does elsewhere, confirming the STATUS.md write-up's claim that "this dispatch-by-type already exists in the codegen for other reasons."
  - `closure` (`emit_func.go`'s `emitArrowFunctionWithHints`) turned out to have **three** levels of heap allocation, not one: the `{funcPtr, envPtr}` header struct (`hdr`, what `Value.Ref` actually points to), a separate environment struct (`env`, holding one cell-pointer per captured variable, only allocated if there are any captures), and the captured variables' own heap-promoted cells (`ADR-00001`) — which can be *shared* across multiple closures and the enclosing scope. Freeing a closure therefore has to stop at freeing `hdr` and `env` (both always uniquely owned by this one closure instance) and must never touch the shared cells `env` merely points to.
  - `Map`/`Set` are a 32-byte header (`{i64 size, i64 cap, ptr keys, ptr vals}`, confirmed identical for both string- and number-keyed maps by reading `ensureMapStrHelpers`/`ensureMapNumHelpers`, and confirmed `Set` is literally a `Map` with unit values under the hood by reading `emitSetCall`) plus two separately-malloc'd backing arrays (`keys`, `vals`) at fixed offsets 16/24 — none of which are reachable by a generic "free this one pointer" dispatch, hence the need for a dedicated `__kml_map_free` helper.
  - `Promise<T>` (`emit_async.go`) is just a single malloc'd slot — as trivial as a plain object, so included for free (so to speak).
- **A real crash, caught by the very first smoke test, that exposed a design gap rather than an implementation bug.** `Memory.free(s)` on `let s: string = "hello world..."` (a string *literal*) crashed with a `malloc` corruption abort. Root cause: string literals are interned as compile-time global constants (`e.internString`, a `private unnamed_addr constant` global) — never malloc'd — so `free()`-ing one is undefined behavior, exactly like C's own `free("literal")`. This isn't a bug to fix (there's no way to distinguish "this `string`-typed value happens to be a literal" from "this `string`-typed value is a heap pointer" at the type level, the same way a `char*` in C doesn't carry that distinction either) — it's an inherent, C-shaped footgun of what `Memory.free` fundamentally is, documented loudly rather than silently risked.
- **Reused an existing singleton rather than declaring a duplicate.** `@free` was already declared once, by `emit_async.go`'s `ensureFree()` (used to free a `Promise`'s slot the moment `await` reads it) — a first attempt at declaring a second, independent `ensureFree()` in this new code collided immediately at `go build` (`usedFree redeclared` / `ensureFree already declared`), the same class of duplicate-singleton mistake `ADR-00023`/`ADR-00026` already found and fixed for other C symbols. Fixed by deleting the duplicate and calling the existing `ensureFree()` directly.

## Decision

`Memory.free(x)`, dispatching on `x`'s statically-known type: `array` → free the data buffer; `Map`/`Set` → free both backing buffers then the header (`__kml_map_free`, new); closure → free the environment struct (if any captures) then the header (`__kml_closure_free`, new); `object`/`Promise`/plain `string` → free the single pointer directly. Anything else (`number`, `boolean`, `Date`, `any`/`unknown`, `null`/`undefined`) is rejected with a clean compile-time error — nothing heap-allocated to free, or (for `any`/`unknown`) no safe way to tell what's inside without runtime tag inspection this builtin doesn't do. Shallow only: never recurses into what a freed value merely points to. For a named variable, also nulls out the variable's own storage after freeing (a subsequent read at least sees `null` rather than freed memory — cheap, and makes freeing the same variable twice harmless, though freeing the same allocation through two different aliases is still unsafe by design).

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureMapFree` (`__kml_map_free`, two `free`s for the backing buffers at fixed offsets 16/24 plus one for the header); `ensureClosureFree` (`__kml_closure_free`, loads the env pointer from the header's second word, frees it if non-null, then frees the header).
- `codegen/llvm/emitter.go`: `usedMapFree`/`usedClosureFree` flags (no new `usedFree` — reused the existing one from `emit_async.go`).
- `codegen/llvm/emit_memory.go` (new file): `emitMemoryFree` (the identifier-vs-general-expression split, and the array-specific data-pointer extraction for each); `freeResolvedPointer` (the shared type-dispatch switch for everything that isn't an array).
- `codegen/llvm/emit_call.go`: `Memory.free` dispatch added alongside the existing `crypto`/`fs`/`process` bare-identifier-namespace blocks.

## Side effects discovered

None — no pre-existing bugs surfaced. The string-literal/`free()` interaction described in Investigation is an inherent property of what this feature is, not a bug; documented prominently in the example, `STATUS.md`, and this ADR rather than treated as something to silently work around.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, including the 11 new tests below), `make examples` all green (54/54).
- Manual repros, each actually compiled and run: a concatenation-built string (freed successfully, variable reads back `null` afterward); an array (freed, `.length` reads back `0`); a plain object; a closure with no captures; a closure *with* a captured variable, confirming the captured cell is still correctly readable/incrementable after the closure itself is freed (the shared-cell-survives case, the one most likely to be gotten wrong); `Map<string, number>` and `Set<string>`; a directly-evaluated expression (not just a named variable); freeing the same named variable twice (harmless, via the null-then-`free(NULL)`-is-a-no-op path); a plain `number` rejected with a clean compile error; a bare string *literal* crashing exactly as documented (confirming the caveat is accurate, not theoretical).
- New tests in `compiler_test.go`: `TestE2EMemoryFreeString`, `TestE2EMemoryFreeArray`, `TestE2EMemoryFreeObject`, `TestE2EMemoryFreeClosureNoCaptures`, `TestE2EMemoryFreeClosureLeavesSharedCaptureIntact`, `TestE2EMemoryFreeMap`, `TestE2EMemoryFreeSet`, `TestE2EMemoryFreeGeneralExpression`, `TestE2EMemoryFreeDoubleFreeViaSameVariableIsSafe`, `TestE2EMemoryFreeUnsupportedTypeRejected`, `TestE2EMemoryFreeWrongArgCountRejected`.
- New example `examples/memory/memory_free.ts`, covering every supported type plus the string-literal footgun as a prominent, explicit warning in its header comment.
