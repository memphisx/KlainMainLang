# TDD-00010: Generics on user-defined functions and interfaces

- **Status**: Not Started
- **Date**: 2026-07-14

## Context

`STATUS.md` tracks "Generics on user functions/interfaces" as ❌ ("Only built-in generics (`T[]`, `Promise<T>`)"), filed as High complexity in the roadmap. `TDD-00009` (Classes/OOP) explicitly deferred generic classes (`class Box<T>`) as out of scope, tracked together with this gap rather than duplicated there — this document is that tracking landing.

Direct testing (not assumed) turned up two different things, worth keeping separate since they have different fixes:

1. **A real, confirmed bug in existing, shipped code**, unrelated to whether user-defined generics ever get built. The parser has a special case for `Promise<T>` in type-annotation position (`parser.go:426`, parses the type argument for real) but everything else — `Array<T>`, `Map<K,V>`, `Set<T>` used as a plain type annotation, not the `new X<T>()` constructor form — hits a "skip other generics" branch (`parser.go:438`) that just consumes and discards the `<...>` entirely, keeping only the bare name. Confirmed directly: `function f(x: Array<number>): number { return x.length }` compiles, but fails at `.length` with `"only supported on arrays and strings"` — `x`'s resolved type silently fell through `ResolveTypeName`'s unknown-name default (`types.go:314`, `return TypeI64`) instead of becoming an array type, because the discarded `<number>` never reached type resolution at all. A `Map<K,V>` parameter/return type cascades into an even more confusing, unrelated-looking error (`"only simple function calls are supported"`) at the first `.get()`/`.set()` call, rather than a clean message pointing at the actual cause. This only bites `Array`/`Map`/`Set` used as a *type annotation* (a function parameter, a return type, an interface field) — `new Array<T>(n)`/`new Map<K,V>()`/`new Set<T>()` construction already parses correctly via their own dedicated AST nodes, untouched by this bug. Fixed on its own, ahead of and independent of the rest of this document — see `docs/adr/ADR-00058.md`; it was a real, silent, orthogonal defect in already-shipped built-in generics, not a consequence of user-defined generics being unimplemented, so fixing it doesn't move this TDD's own (still not-started) scope forward.
2. **User-defined generic declarations fail cleanly today**, which is the honest, correct failure mode for a genuinely unimplemented feature — confirmed directly: `function identity<T>(x: T): T {...}` gives `parse error: ... expected (, got <`, and `interface Box<T> { value: T }` gives `parse error: ... expected {, got <`. Neither silently misbehaves; both are clean, if unhelpful, parse errors. This document is about closing that gap for real, not about a hack that needs undoing.

## Design

### The key existing precedent: built-in generics already are monomorphized, just ad hoc

`Array<T>`, `Map<K,V>`, `Set<T>`, and `Promise<T>` all already exist today, and none of them use a shared runtime representation for `T` — `ArrayOf(elem Type)`, `MapType(key, val Type)`, `SetType(elem Type)`, `PromiseOf(inner Type)` (`types.go`) are plain Go-side functions that build a distinct, fully concrete `Type` value per instantiation. `number[]` and `string[]` are two different `Type`s with two different `ElemType`s, and codegen emits genuinely different machine code for each (different GEP element types, different element sizes, different comparator functions for `.sort()`, etc. — see `emitArraySort`'s element-type `switch`, `codegen/llvm/emit_arrays.go`). In other words: **this compiler already monomorphizes generics, once per built-in type, entirely by hand.** The real question for user-defined generics isn't "should this be monomorphized," it's "should the same idea be generalized into a real mechanism, or should user-defined generics instead lean on the (separately already-built) `any`/`unknown` type-erasure machinery instead." Both are real, viable options here, not a foregone conclusion.

### Option A: Monomorphization (recommended direction, not yet decided)

Generate a distinct, fully concrete, name-mangled copy of a generic function/interface's codegen per distinct combination of type arguments actually used anywhere in the program — extending the exact pattern the built-ins already use by hand into a general mechanism.

- **Type-argument inference, not explicit syntax, drives instantiation for V1.** `identity(5)` infers `T = number` from the argument's own inferred type, reusing the `inferExprType` machinery already used throughout `emit_exprs.go` for unannotated parameters/return types (`ADR-00041`/`ADR-00042` precedent). Explicit call-site type arguments (`identity<number>(5)`) are deliberately **not** in V1's scope: `<` immediately after a call-target identifier is a genuine grammar ambiguity in a TS/JSX-shaped language (`a<b>(c)` could be a generic call with an explicit type argument, or two chained comparisons `a < b > (c)`) — real parsers for TypeScript and JSX need lookahead/backtracking heuristics to disambiguate this, which this compiler's parser has no machinery for today. Scoping V1 to inference-only sidesteps the ambiguity entirely and matches how the overwhelming majority of real-world generic calls are actually written (argument-inferred, not explicitly instantiated) — the same "ship the common case cleanly, document the narrower gap" shape this project already uses elsewhere.
- **Two-pass structure, enabled specifically by whole-program compilation.** `resolver.ResolveProgram` already merges the entry file and everything it transitively imports into one AST before codegen ever runs (`docs/adr/ADR-00022.md`) — there are no separately-compiled modules, so "every call site of every generic function is visible before codegen starts" is a real, decidable, whole-program fact here, not an open-world unknown the way it would be in a language with separate compilation units. Pass 1 walks the merged AST, finds every call site of every generic function/every use of a generic interface, and infers each site's concrete type argument(s) (reusing `inferExprType`); pass 2 emits one specialized, name-mangled function/struct-type per distinct `(declaration, concrete type args)` pair actually collected (memoized — a repeated instantiation, e.g. `identity(5)` called twice, doesn't get emitted twice), and rewrites each call site to target its own instantiation's mangled name instead of the generic one.
- **Generic interfaces** (`interface Box<T> { value: T }`) follow the identical shape, keyed off *use* sites (`const b: Box<number> = {...}`) instead of call sites — `Box<number>` and `Box<string>` become two different, concretely-typed `ObjectType()` values under the hood, exactly analogous to how `ArrayOf(TypeI64)` and `ArrayOf(TypePtr)` are already two different `Type` values today, just generalized from "hardcoded per built-in" to "derived from whatever the program actually instantiates."
- **Generic classes**, once `TDD-00009`'s Stage 1 (methods + constructors, no inheritance) exists, follow the same instantiation-collection-then-specialize shape as generic interfaces, applied to class instance construction sites — this is exactly the gap `TDD-00009` deferred and pointed here.
- **Cost**: real, new compiler engineering — an instantiation-collection pass and a name-mangling scheme don't exist in any form today. Not free, and the actual cost (how many distinct instantiations a realistic program generates, what that does to compile time and binary size) is an empirical question this document can't answer from reasoning alone — worth a direct, throwaway prototype before fully committing, in the same spirit `TDD-00006` used before picking fibers over LLVM coroutines (that TDD's own recommendation was later confirmed, not just assumed, by five small hand-written test cases).

### Option B: Type erasure via the existing `any`/`unknown` machinery

Compile a generic function's body exactly once, treating every occurrence of a type parameter as `TypeAny` (`emit_dynamic.go`) internally — box each actual argument via the already-existing `emitBoxValue` before the (single, shared) call, unbox/dispatch inside the function body wherever the value actually needs to be inspected (printed, compared, etc.) via the already-existing `emitDynamicToString`/`emitDynamicTypeof`/`emitAnyEquals`.

- **Genuinely cheaper to build**: no instantiation-collection pass, no name-mangling scheme, no "how many copies" question — this reuses infrastructure that's already built, tested, and shipped (`ADR-00008`).
- **Arguably more faithful to real TypeScript**, worth stating explicitly since it cuts against the instinctive "monomorphization is obviously the better/more correct option": real TS generics are *themselves* fully erased at compile time — compiled JavaScript output carries no runtime type information for `T` either. A type-erasure implementation here isn't a compromise relative to "real" generics, it's arguably a more direct port of how TS's own generics actually behave at runtime.
- **Real ceiling, not just a performance cost**: a small but nonzero runtime tag-dispatch overhead on every generic value touched is the least of it — the actual limit is that the existing `any`/`unknown` machinery already **rejects arithmetic and most other operators** on a dynamic value at compile time (`ADR-00008`'s Staged V1 scope). A fully generic `function add<T>(a: T, b: T): T { return a + b }` would hit that exact wall unmodified — erasure alone only cleanly covers *pass-through* generics (containers and identity-shaped functions that store/retrieve `T` without operating on it), not anything that needs real arithmetic/comparison over the erased type. Monomorphization has no such ceiling, since every instantiation is fully concrete by construction.

**Leaning, not a decision**: monomorphization, both because it's the more direct generalization of a pattern this compiler already trusts (the built-ins), and because whole-program compilation specifically removes monomorphization's usual worst cost (unbounded instantiations across modules this compiler can't see all of at once) — but this isn't settled, and a direct prototype (not just this document's reasoning) should decide it, the same way `TDD-00006` was actually decided.

## Open questions

- Monomorphization vs. type erasure — leaning monomorphization (see Design), but not decided; worth a direct, throwaway prototype (in `TDD-00006`'s own spirit) measuring realistic instantiation counts/compile-time cost before committing, rather than deciding from reasoning alone.
- ~~Should the confirmed built-in-generic-type-annotation bug (`Array<T>`/`Map<K,V>`/`Set<T>` silently defaulting to `i64` as a plain type) be fixed as its own small, immediate, standalone fix regardless of when/whether the rest of this document gets built?~~ Resolved: yes — fixed in `docs/adr/ADR-00058.md`, independent of this TDD's own (still not-started) scope.
- Explicit call-site type arguments (`identity<number>(5)`) — deliberately deferred past V1 (see Design); worth revisiting only if inference-only proves insufficient in practice, not scheduled now.
- Type parameter constraints (`<T extends Base>`), multiple type parameters (`<K, V>`), and default type parameters (`<T = number>`) — not designed here at all; likely natural follow-ons once whichever base mechanism (Option A or B) actually ships, not blockers to shipping a first, narrower slice (single unconstrained type parameter).
- If Option A (monomorphization) is chosen, should the name-mangling scheme be considered a stable, inspectable detail (useful for reading `--emit-llvm` output while debugging) or a pure, unstable implementation detail free to change? Not decided.
