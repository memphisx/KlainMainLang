# ADR-00017: Extend Date.parse to support "+HH:MM"/"-HH:MM" timezone offsets

- **Relations**: Extends ADR-00015
- **Date**: 2026-07-11

## Context

Picked this up as a direct follow-on to `Date.parse` (`ADR-00015`), which explicitly scoped out anything other than a `Z` (UTC) suffix or a bare date. Timezone-offset ISO strings (`"2023-11-14T22:13:20.000+02:00"`) are common enough — many real-world APIs and logs emit them — that leaving them unhandled was worth closing.

## Investigation

- **Found a real, pre-existing bug while confirming the starting behavior.** Before writing any new code, ran the existing `Date.parse` against an offset string to see exactly what "unsupported" meant in practice: `Date.parse("2023-11-14T22:13:20.000+02:00")` returned `1700000000000` — the same value as the equivalent `Z` string, i.e. the `+02:00` was silently *ignored*, not rejected. Root cause: `ADR-00015`'s `__kml_date_parse` checks success via `sscanf`'s return value (count of successfully-assigned `%`-conversions), but a literal character in a format string (like the `Z` in `"%d-%d-%dT%d:%d:%d.%dZ"`) that fails to match does **not** reduce that count — it just stops scanning at that point. For the offset input, all 7 numeric fields (year..millis) still matched before the literal `Z` check failed against the actual `+`, so `sscanf` returned exactly 7, and the existing `ok1 := icmp eq i32 %n1, 7` check reported success anyway. This wasn't a hypothetical edge case: it was actively lying about what it had parsed. Confirms the "fix bugs found during development" rule applies here — noted the mechanism (an unmatched literal never rolls back the field count), not just the symptom.
- **How to detect an offset vs. a bare "Z" string without that ambiguity.** The fix an offset-aware format needs a *higher* required field count (year..millis..offH..offM = 9, vs. 7 for `Z`-with-millis), so trying the offset patterns *first* — before the `Z` patterns — and checking for an exact match on the higher count resolves it cleanly: a genuine `Z` string can never reach 9 matched fields against an offset pattern (the scan stops at the literal `+`/`-` position, which isn't present), so it correctly falls through and is picked up by the (now second-in-line) `Z` pattern. This ordering constraint — most-specific-pattern-first — is the crux of the fix, not just adding more patterns.
- **How to parse the sign correctly, including a genuine edge case.** Considered parsing the offset hour as a signed `%d` (letting `sscanf` consume an optional leading `+`/`-` itself) — rejected after tracing through `"-00:30"` (a negative offset whose hour component happens to be zero): `%d` parsing `"-00"` yields the integer `0` (since `-0 == 0` for a fixed-width integer), silently losing the sign entirely. Considered capturing the sign via `%c` into a separate byte before the numeric fields — rejected because `%c` in `sscanf` always succeeds and consumes exactly one byte, valid or not (it doesn't check the byte is actually `+` or `-`), which would require a second post-match validation step to reject garbage, adding complexity without a clean failure path if that validation failed after the fields were already deemed "matched". Settled on four separate format strings, one per (millis-present, sign) combination, each with the sign baked in as a **literal** character (`"...+%d:%d"` / `"...-%d:%d"`) — a literal mismatch cleanly fails that whole attempt (no partial/ambiguous state), and which pattern succeeded is itself the source of truth for the sign, sidestepping both problems: no signed-zero ambiguity (the hour/minute magnitudes are always parsed as plain non-negative numbers), and no post-hoc validation of an unconstrained byte.
- **Verified the fix against the exact bug found above**, plus deliberately re-tested the `-00:30` zero-hour-negative-sign case specifically (since that's precisely the scenario a naive signed-`%d` implementation would have gotten wrong) to confirm the chosen approach actually avoids it, not just assumed to.

## Decision

Extended `Date.parse` to additionally accept `"...+HH:MM"` and `"...-HH:MM"` suffixes, with or without milliseconds, converting to UTC (`UTC = local − offset`, so a `+` offset subtracts and a `-` offset adds). Four new `sscanf` patterns, tried before the existing `Z`/date-only patterns (highest field-count first): full+millis+`+`, full+millis+`-`, full+millis+`+` , full-no-millis+`-`. Everything else about `Date.parse`'s scope (`ADR-00015`) is unchanged — still ISO-shaped strings only, still `-1` for anything unparseable.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureDateParse()` rewritten with the four new offset-cascade stages (`try_off_minus_ms`, `try_off_plus_s`, `try_off_minus_s`, each falling through to the next on a field-count mismatch) ahead of the renamed `try_z_ms`/`try_z_s`/`try_date` stages (previously `try2`/`try3`). Each successful offset match computes a signed offset in milliseconds (`(offH*60 + offM) * 60 * 1000`, then negated for the `-` patterns) and stores it into a new `offset_ms_a` accumulator (an `i64` alloca, defaulted to `0` at function entry) rather than branching to separate compute logic — all seven possible successful paths (4 offset + 2 `Z` + 1 date-only) converge on the same shared `compute:` block, which now does one extra step at the end: `%totalms = sub i64 %localms, %offset_ms` (a no-op subtraction of `0` for the non-offset paths, so their behavior is provably unchanged).
- No changes needed in `emit_date.go`, `emit_call.go`, or `emit_exprs.go` — this is purely a runtime-string-parsing extension behind the existing `Date.parse` call site; the Go-level dispatch, argument handling, and type inference from `ADR-00015` already cover it unchanged.

## Side effects discovered

The pre-existing silent-ignore bug for offset strings, described in Investigation — fixed as part of this same change (not deferred), since it's the direct predecessor bug this feature closes.
