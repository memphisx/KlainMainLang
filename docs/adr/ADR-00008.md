# ADR-00008: Implement any/unknown as a runtime-tagged value (Staged V1)

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Found via the same audit as `ADR-00006`/`ADR-00007`: `any`, `unknown`, and general union types beyond `T | null` all silently resolved to `i64` (`ResolveTypeName`'s fallback), so `let x: any = 5; x = "hello"` would try to store a `ptr` into a slot the compiler had already committed to `i64` — producing invalid LLVM IR caught only by `clang`'s final IR verifier with a confusing internal error, not a clean compiler diagnostic. I chose real runtime-polymorphic support over a "fail cleanly" fix, with the scope explicitly narrowed in two passes given the size of the undertaking (see Decision).

## Investigation

- **Scope-defining finding: general unions need parser changes, separate from this work.** `ast.TypeAnnotation` has no field to hold more than one type name plus a `Nullable` bool. Tracing the parser (`parser/parser.go:338-352`) confirmed why: its loop for `T | U` only actually handles `U` being `null`/`undefined` (setting `Nullable`); for any other `U`, the code's own comment says *"we silently accept the syntax but use the first type"* — `string | number` parses fine but `number` is discarded entirely during parsing, before codegen ever runs. So `any`/`unknown` (single keyword type names, no parser involvement) could be tackled independently of general unions (which need the parser fixed first, to even retain both member types) — narrowed this ADR's scope to `any`/`unknown` only.
- **Blast radius of full support, before scoping down.** A research pass across assignment codegen, `console.log`/template-literal formatting, binary operators, function signatures, and `typeof` found roughly 400–600 call sites would need touching for *full* polymorphic support (every arithmetic operator's type-based dispatch, every function call's argument packing, etc.) — comparable in size to implementing `class` declarations, not a session-scale task. This is why I chose "Staged V1" (declare/assign/reassign, printing, `typeof`, `===`/`!==` — arithmetic and function/array/object positions explicitly deferred with a clean error) over full support, given this concrete estimate.
- **The closest existing precedent (`T | null`) doesn't generalize.** Nullable types work today by keeping `T`'s normal storage type and using the pointer's own null value as the sentinel — this only works because `T` is `ptr`-shaped; there's no way to represent "a null number" this way, since `i64` has no spare bit pattern reserved for it. Confirmed no other runtime-dispatch-on-a-tag mechanism exists anywhere in the codebase (`IsGroupMap` was the closest analog, but it's a compile-time-only flag affecting codegen decisions, never a runtime branch).
- **Design validation pass, before writing code.** A dedicated validation pass (given the scale) confirmed: (1) an anonymous/literal LLVM struct type `{ i8, i64 }` works correctly as a single `alloca`'d value (not just inside `StructIR()`-style GEP contexts, which is the only place literal struct syntax was previously used in this codebase, for objects) — verified by direct experiment, including a `bitcast double to i64`/back round-trip (no prior occurrence of same-size `bitcast` existed in the codebase, but it's standard, well-defined LLVM IR); (2) `emitOptionalMember`'s existing "alloca a result slot, branch to N blocks each storing into it, merge and load" shape (`emit_exprs.go:604-674`) was confirmed as the best existing pattern to generalize from 2 branches to a per-tag dispatch; (3) found and fixed a design gap before implementation: `console.log` does **not** go through the same formatting function as template literals (`emitConsolePrint`/`emitConsolePrintVal` in `emit_call.go` format directly via `Type.PrintfFmt()`, a separate path from `emitValueToString`) — both needed the same dynamic-dispatch fix, not just one; (4) found a second gap: an uninitialized `let x: any;` (no initializer) would leave the tag byte as garbage, which is far more consequential for a tagged value (driving real runtime branching in print/typeof/equality) than the general, pre-existing, unrelated lack of zero-init for plain scalars; (5) found the most important gap: function parameters/returns and array/object element types needed an **active guard**, not just a documentation note — without one, they'd silently accept the new `TypeAny` and produce exactly the kind of broken IR this feature exists to fix, just at a different call site.
- **Two bugs found and fixed during implementation, not just design.** First, my initial `emitFunctionDecl`/`emitArrowFunctionWithHints` guards checked `pty.IsDynamic` directly, which correctly rejects `x: any` but misses `x: any[]` or `x: { y: any }` (only the *element*/*field* is dynamic, not the parameter's own top-level type) — confirmed by testing `let arr: any[] = [1,2,3]` compiled successfully and produced broken IR *before* adding a recursive `containsDynamicElement` check; fixed by adding that helper and using it everywhere a param/return/array/object type is validated, not just the direct `IsDynamic` field. Second, testing `const p: Point = JSON.parse(...)`-style object-typed `any` fields via a similar path surfaced nothing new here (that was `ADR-00007`'s bug), but confirms the guard needed to be recursive through `Type.Fields`/`Type.ElemType`, not just the top level.

## Decision

**Representation**: `Type` gets a new `IsDynamic bool` field; `TypeAny = Type{IR: "{ i8, i64 }", IsDynamic: true}` (`types.go`) — a literal 16-byte struct (1-byte tag, padded, 8-byte payload), added to `ResolveTypeName`'s switch for `"any"`/`"unknown"` (both resolve identically — this compiler doesn't do static-only type checking that would distinguish them meaningfully anyway). Tags (Go-side constants in the new `emit_dynamic.go`, no AST/parser involvement): `0`=int, `1`=float (payload holds the `double` bit-cast into the `i64` slot), `2`=string (payload is `ptrtoint` of the string pointer), `3`=boolean (`zext` of `i1`), `4`=null, `5`=undefined, `6`=object (`ptrtoint`, included for completeness though no in-scope V1 path produces one).

**New file `emit_dynamic.go`**: `emitBoxValue` (idempotent — boxing an already-dynamic value is a no-op passthrough, so every call site can call it unconditionally); `emitUnboxTagPayload` (two `extractvalue`s); `emitDynamicToString` and `emitDynamicTypeof` (both generalize `emitOptionalMember`'s branch/merge shape to 7 tag-checks); `emitAnyEquals` (boxes whichever operand isn't already dynamic, then calls a new `@__kml_any_eq` runtime helper); `containsDynamicElement` (the recursive array/object guard helper, described above).

**`@__kml_any_eq`** (`runtime.go`, standard `ensure*()` singleton pattern): compares two boxed values. Equal tags compare directly per the tag's meaning — critically, string/string compares via `strcmp` after `inttoptr`, not raw payload/pointer equality, and object/object compares by pointer (matching JS reference equality); an int/float tag *mismatch* (either order) is still treated as a real numeric comparison (`sitofp` the int side, then `fcmp oeq`) so `(5 as any) === (5.0 as any)` is correctly `true`; any other tag mismatch is `false` — including `null !== undefined`, matching real JS strict-equality semantics.

**Wire-up** (`emit_exprs.go`/`emit_call.go`/`emit_func.go`): `emitVarDecl`'s scalar path boxes the init value instead of `coerce`-ing when the declared type `IsDynamic`, and explicitly boxes `undefined` when there's no initializer; `emitAssign`'s identifier `=` path does the same, and rejects compound assignment (`+=` etc.) on a dynamic-typed symbol with a clean error; `emitUnary`'s `"typeof"` case checks `inferExprType(arg).IsDynamic` and, only then, actually evaluates the argument and delegates to `emitDynamicTypeof` — every other `typeof` case is untouched, staying fully compile-time; `emitBinary` checks either operand for `IsDynamic` immediately after evaluating both (before the existing unconditional `coerce`), routing `===`/`!==` to `emitAnyEquals` and rejecting every other operator with a clean error; `emitValueToString` and `emitConsolePrint` both delegate to `emitDynamicToString` when the value `IsDynamic`.

**Guards** (`emit_func.go`, `emit_exprs.go`): `emitFunctionDecl`'s param loop and return-type resolution, and `emitArrowFunctionWithHints`'s equivalent, reject `pty.IsDynamic || containsDynamicElement(pty)` (and the same for return types) with a clean error; `emitVarDecl` rejects `containsDynamicElement(ty)` before delegating to the array/object declaration paths. This is why `let arr: any[]` and `interface Foo { x: any }` both fail cleanly today, not just bare `let x: any` used somewhere it can't be supported.

## Implementation notes

Files touched: `codegen/llvm/types.go` (`IsDynamic`, `TypeAny`, `ResolveTypeName` case), `codegen/llvm/emitter.go` (`usedAnyEq` flag), `codegen/llvm/emit_dynamic.go` (new), `codegen/llvm/runtime.go` (`ensureAnyEq`/`@__kml_any_eq`), `codegen/llvm/emit_exprs.go` (`zeroRef`, `emitVarDecl`, `emitAssign`, `emitUnary`'s typeof case, `emitBinary`, `emitValueToString`), `codegen/llvm/emit_call.go` (`emitConsolePrint`), `codegen/llvm/emit_func.go` (param/return guards in both `emitFunctionDecl` and `emitArrowFunctionWithHints`).

## Side effects discovered

- `console.log`ing an `any`-boxed boolean prints `"true"`/`"false"`, while a plain (non-`any`) `boolean` prints `1`/`0` in this compiler's existing convention (confirmed via `examples/strings/string_methods.ts`'s own comments). This is because `emitDynamicToString` is shared by both `console.log` and template literals, and mirrors the template-literal convention (which already used `"true"`/`"false"` for concrete booleans) rather than adding call-site-specific formatting to also match `console.log`'s different, narrower convention. Kept as a deliberate, minor, documented simplification rather than adding complexity to make the two call sites diverge for this one case.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 42/42 pass, including new `examples/types/any_unknown.ts`.
- Manual repros, all correct: reassignment across int/float/string/bool/null/undefined with printing at each step; template literals picking up the current runtime value; `typeof` correctly reflecting the *current* value after reassignment (not the declaration-time type), including the `typeof null === "object"` quirk; an uninitialized `let x: any;` correctly showing `undefined`; equality against another `any` variable, against a plain literal, cross-numeric (`5 === 5.0` → true), cross-type (`5 === "5"` → false), and `null !== undefined`.
- Negative-path repros, all correctly rejected at compile time (not silently broken or a clang crash): arithmetic (`+`) on an `any` operand; `any` as a function parameter type (both `function`-declared and arrow); `any` as a function return type; `any[]` as an array element type; `{ x: any }` as an object field type.
- New tests in `compiler_test.go`: `TestE2EAnyReassignAcrossTypes`, `TestE2EAnyTemplateLiteral`, `TestE2EAnyTypeofRuntime`, `TestE2EAnyEquality`, `TestE2EUnknownFloat`, plus three negative tests (`TestE2EAnyArithmeticRejected`, `TestE2EAnyAsFunctionParamRejected`, `TestE2EAnyArrayElementRejected`) using a new `parseAndCompile` test helper (parse+codegen only, no `clang`, returning the error for assertion rather than failing the test).
- New example `examples/types/any_unknown.ts`.
- `STATUS.md`: `any`/`unknown` flipped to ✅ with the Staged V1 scope noted explicitly; general unions beyond `T | null` remain ❌ with a note pointing to the separate parser limitation; removed the now-fixed "can emit invalid LLVM IR" entry from Known Limitations, replaced with the boolean-formatting inconsistency noted above; coverage numbers updated.
