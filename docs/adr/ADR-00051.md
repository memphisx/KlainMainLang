# ADR-00051: Fix ucontext_t's size/layout being hardcoded to this dev machine's platform

- **Relations**: Extends ADR-00049, ADR-00050
- **Date**: 2026-07-14

## Context

GitHub Actions CI (`ubuntu-latest`, x86-64) started failing every `TestE2EHTTPListen*` test right after `ADR-00049`/`ADR-00050` landed — `connection reset by peer` and `connection refused` errors, and one run that appeared to hang entirely. Every one of these tests had passed locally throughout both of those pieces of work. The discrepancy pointed at exactly one thing: this dev machine is arm64 Darwin, and nothing in the fiber implementation had ever been verified anywhere else.

## Investigation

- **`ucontext_t`'s size and field offsets were hardcoded from a single macOS probe and never revisited** — `ADR-00049`'s own prototyping got `sizeof(ucontext_t) == 880`, `uc_stack` at offset 8 (`ss_sp`/`ss_size` at +8/+16 within it), `uc_link` at offset 32, via a throwaway C program compiled on this machine. Correctly labeled "on this machine (arm64 Darwin)" in that ADR's own text, but the actual `runtime.go` code that used those numbers had no platform branch on them at all, unlike every *other* platform-specific constant in the same feature (`SOL_SOCKET`/`SO_REUSEADDR`/`sockaddr_in`'s byte layout, `O_NONBLOCK`, `EAGAIN`), which all correctly branch on `runtime.GOOS`. This one slipped through.
- **Confirmed directly, not assumed, using Docker to reach the platforms this dev machine can't run natively** — the same `sizeof`/`offsetof` C-probe technique already used for the Darwin numbers, this time run via `docker run --platform linux/amd64|linux/arm64 ubuntu:24.04` (matching `ubuntu-latest`'s glibc, not Alpine's musl, which this project's *other* Dockerfiles use for unrelated reasons and would have given a third, still-wrong answer). Results: Linux x86-64 `sizeof(ucontext_t) == 968`; Linux arm64 `sizeof(ucontext_t) == 4560`; both completely different from Darwin's 880, and different from *each other* — but with identical field offsets between the two Linux architectures (`uc_stack` at 16, `ss_sp`/`ss_size` at 16/32, `uc_link` at 8), only the overall struct size differing (presumably a differently-sized register/FPU save area later in the struct, which this project never needs to touch directly since it only reads/writes the stack/link fields near the front).
- **Root cause, precisely**: two places wrote a full `ucontext_t` into a buffer sized for Darwin's 880 bytes regardless of the actual host platform — the `@__kml_main_ctx` global (`[880 x i8]`) and `__kml_http_append_conn`'s per-connection `%ctx = call ptr @malloc(i64 880)`. On Linux, `getcontext`/`makecontext`/`swapcontext` wrote the real (larger) struct past the end of these too-small allocations, corrupting whatever memory happened to sit next — a global on the `@__kml_main_ctx` side, heap metadata or an adjacent allocation on the `malloc`-based side. This is exactly the kind of memory corruption that manifests as unpredictable, hard-to-reproduce symptoms (a reset connection here, a hang there) rather than a clean crash — matching what CI actually showed.
- **Confirmed the fix works on both real target architectures before considering this done** — built the compiler and ran the full `http.listen`/`fetch` test suite inside Docker containers for both `linux/amd64` (exactly matching `ubuntu-latest`) and `linux/arm64`, not just re-running the existing macOS-only suite. All 12 `TestE2EHTTPListen*` tests, and the full suite, pass on both.

## Decision

Added `ucontextLayout() (size, ssSpOff, ssSizeOff, ucLinkOff int64)` (`runtime.go`), branching on `runtime.GOOS`/`runtime.GOARCH` (Darwin: 880/8/16/32; Linux arm64: 4560/16/32/8; Linux amd64 and other 64-bit Linux targets: 968/16/32/8), and used it everywhere a `ucontext_t`-sized buffer gets allocated or its fields get addressed: `@__kml_main_ctx`'s declared size, and `__kml_http_append_conn`'s per-connection `ctx` malloc size and its three field-offset GEPs. No other call site touches `ucontext_t`'s internal layout directly — every other `swapcontext`/`getcontext` call just passes an already-correctly-sized pointer through opaquely.

Given `__kml_http_append_conn`'s IR is a large hand-written backtick string (not run through `fmt.Sprintf`, for the same "hundreds of `%name` LLVM values would need escaping" reason `ADR-00049` already documented for `__kml_event_loop_run`), the four dynamic values are spliced in via the same plain-Go-string-concatenation technique that ADR already established for `O_NONBLOCK`.

## Implementation notes

- `codegen/llvm/runtime.go`: `ucontextLayout()` (new); `ensureFiberRuntime`'s `@__kml_main_ctx` declaration now uses `fmt.Sprintf` for its size instead of a literal `880`; `__kml_http_append_conn`'s ctx `malloc` size and its `ss_sp_p`/`ss_size_p`/`uc_link_p` GEP offsets now splice in the computed values instead of literals.
- No changes needed anywhere else — `emit_http.go`'s dispatcher and the event loop's resume-scan logic only ever pass `ctx` pointers around opaquely, never computing offsets into them themselves.

## Side effects discovered

None beyond the bug itself — this was a self-contained platform-portability gap in exactly two call sites, both already identified precisely by direct probing before writing any fix.

## Verification

- `go build .`, `go vet ./...` on this dev machine (arm64 Darwin) — unaffected, since Darwin's own branch of `ucontextLayout()` returns the exact same numbers as before.
- **The actual regression test**: built the compiler and ran `go test ./tests/... -run TestE2EHTTPListen -v` inside `docker run --platform linux/amd64 ... ubuntu:24.04` (glibc, matching GitHub Actions' `ubuntu-latest` exactly, not this project's other Alpine/musl-based Dockerfiles) — all 12 tests failed before this fix (matching CI's own reported failures) and all 12 pass after it. Repeated the same check under `--platform linux/arm64` (a second real Linux target this project claims to support) — same result, all pass.
- Ran the complete `go test ./...` suite (not just the HTTP subset) inside the `linux/amd64` container to confirm no other test regressed under the corrected layout — full suite green (298s under this machine's QEMU-emulated x86-64, versus ~90-120s natively; slow but not a concern, this was a one-off verification run, not something CI itself needs to do differently).
- Full local macOS suite (`go build`, `go vet`, `go test ./...`, `make examples`) re-run after the fix, unaffected as expected.
