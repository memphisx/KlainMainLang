# ADR-00011: Implement for...of over Map and Set

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the last remaining Phase 2 roadmap item, following labeled `break`/`continue` (`ADR-00010`).

## Investigation

- **Real JS semantics for `Map` don't fit cleanly.** `for (const v of set)` unambiguously iterates a `Set`'s elements — no design question there. But real JS's `for (const x of map)` iterates `[key, value]` *entry pairs*, which requires destructuring the loop variable per iteration. I checked the for-of grammar directly (`parser/parser.go`'s `parseForStatement`): it only ever accepts a single bare `IDENT` as the loop variable (`peekNth(2).Type == lexer.IDENT && Literal == "of"`), with no support for a `[k, v]` destructuring pattern in that position at all. Adding that would be a real parser change, not something this pass's scope covers.
- **Scope decision, confirmed rather than assumed.** Given the above, I confirmed the fork explicitly: bare `for (const x of map)` iterates *values* (same shape as `Set`, and combined with the already-working `for (const k of map.keys())` gives full practical coverage) rather than leaving bare Map iteration unsupported. This is a deliberate, documented deviation from real JS's entry-pair default — not a limitation I stumbled into, a call made up front because the alternative (implementing destructuring-in-for-of) is materially more work than this roadmap item's scope.
- **Existing infrastructure made this straightforward.** `Map.keys()`/`Map.values()` and `Set.values()` (`emitMapCall`/`emitSetCall` in `codegen/llvm/emit_collections.go`) already resolve to `{ptr, i64}` array aggregates via existing C helpers (`@__kml_map_str_keys`/`@__kml_map_num_keys`/`@__kml_map_str_vals`/`@__kml_map_num_vals`) — a `Set`'s elements are stored as map keys internally, so `Set.values()` already reuses the *keys* helper. `emitForOf` (`emit_stmts.go`) already has a generic path for "any expression producing a `{ptr,i64}` aggregate" (used today for e.g. `Object.keys(obj)`, `arr.slice(1)`) — meaning `for (const k of m.keys())` already worked before this change, with zero code changes needed; the only gap was the *bare* `for (x of m)`/`for (x of s)` form, which currently hits `emitForOf`'s named-identifier branch and requires `iterSym.Ty.IsArray`, erroring on a Map/Set symbol instead.

## Decision

Added a case to `emitForOf`'s named-identifier branch (`emit_stmts.go`): when the iterable symbol is `IsMap` or `IsSet` (not `IsArray`), resolve it to the appropriate values array via a new `mapOrSetValuesArray` helper (`emit_collections.go`) — for a `Set`, the keys helper (matching `Set.values()`'s own existing implementation exactly); for a `Map`, the values helper. Extracted the existing "split a `{ptr,i64}` aggregate into fresh allocas for the loop to keep reloading" logic (previously inlined only in the generic-expression branch) into a small shared `splitArrayAggregate` helper, reused by both the new Map/Set case and the pre-existing generic-expression case, rather than duplicating those ~10 lines.

## Implementation notes

Files touched: `codegen/llvm/emit_collections.go` (new `mapOrSetValuesArray`), `codegen/llvm/emit_stmts.go` (`splitArrayAggregate` helper; `emitForOf`'s named-identifier branch restructured into a `switch` covering array / Map-or-Set / neither).

## Side effects discovered

None.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 42/42 pass, including the extended `examples/map/map.ts` and `examples/set/set.ts`.
- Manual repros: `for (const v of set)` over both a `Set<number>` and `Set<string>` iterates elements correctly; `for (const v of map)` over both `Map<string,number>` and `Map<number,string>` iterates values correctly; an empty `Set` iterates zero times; `for (const k of map.keys())` (already-working generic-expression path) still works unchanged; labeled `break`/`continue` (`ADR-00010`) works correctly inside a Map/Set for-of, since it reuses the exact same `emitForOf` machinery; a non-array/Map/Set variable in a for-of gives a clean "not an array, Map, or Set" error.
- New tests in `compiler_test.go`: `TestE2EForOfSet`, `TestE2EForOfSetString`, `TestE2EForOfMapValues`, `TestE2EForOfMapValuesExplicit`, `TestE2EForOfEmptySet`, `TestE2EForOfMapLabeledBreak`.
- `examples/map/map.ts` and `examples/set/set.ts` extended with for-of sections.
- `STATUS.md`: merged into the existing `for…of` row (already ✅ for arrays) with a note on the Map-iterates-values scope decision; removed from the "not implemented" table and the Phase 2 roadmap list. No coverage-count changes, since this expanded an already-✅ row rather than flipping a new one.
