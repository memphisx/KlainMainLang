# ADR-00058: Fix Array\<T\>/Map\<K,V\>/Set\<T\> silently resolving to i64 as a plain type annotation

- **Date**: 2026-07-14

## Context

Found while researching and writing `docs/tdd/TDD-00010.md` (generics on user-defined functions/interfaces), not during ordinary feature work — directly testing `function f(x: Array<number>): number { return x.length }` surfaced a real bug in already-shipped code, unrelated to whether user-defined generics ever land. The program compiled, but failed at `.length` with `"only supported on arrays and strings"`: `x`'s resolved type wasn't an array at all.

## Investigation

Traced it to the parser's type-annotation path (`parser/parser.go`, around line 425 at the time): a special case already existed for `Promise<T>` — parsing the type argument for real — but every *other* generic name followed by `<...>` (`Array<T>`, `Map<K,V>`, `Set<T>`) fell into a "skip other generics" branch that consumed and discarded the `<...>` entirely, keeping only the bare name (`"Array"`, `"Map"`, `"Set"`). `ResolveTypeName` (`codegen/llvm/types.go`) has no case for any of those three bare names, so each one fell through to that function's own unknown-name default — `i64`.

This only affects a *type annotation* use (a function parameter, a return type, an interface field) — `new Array<T>(n)`, `new Map<K,V>()`, `new Set<T>()` construction is untouched, since each already has its own dedicated, correctly-parsed AST node (`NewArrayExpression`/`NewMapExpression`/`NewSetExpression`) that never goes through this code path at all. That's exactly why this went unnoticed for as long as it did: every existing test and example either constructs a collection directly (`new Map<K,V>()`) or declares an array with the far more idiomatic `T[]` suffix, never `Array<T>` as a bare parameter/return annotation.

A second, separate bug surfaced while verifying the fix, not introduced by it: `emit_exprs.go`'s `emitVarDecl`, when inferring an unannotated variable's type from a function call's return type (`const m2 = identity(m)`, no explicit annotation on `m2`), only trusted the callee's registered return type when it was `IsArray`/`IsObject`/`IsFunc`/`IsDate` — `IsMap`/`IsSet` weren't in that allowlist, so even after fixing the parser, `identity`'s correctly-resolved `Map<string, number>` return type was silently dropped back to `i64` the moment it got assigned to an unannotated variable. Confirmed this is genuinely a different code path from `inferExprType`'s own equivalent `CallExpression` case (`emit_exprs.go:1338`), which already trusts `sig.RetType` unconditionally with no allowlist at all — only `emitVarDecl`'s separate, more conservative check needed the fix.

## Decision

**Parser** (`parser/parser.go`): replaced the "skip other generics" branch with real parsing for the three built-in generics that were being discarded:
- `Array<T>` and `Set<T>` — single type parameter, same shape `Promise<T>` already had; both now set `Name` and `ElemType` on the resulting `ast.TypeAnnotation`.
- `Map<K,V>` — two type parameters, comma-separated. `ast.TypeAnnotation` gained a new `KeyType *ast.TypeAnnotation` field (mirroring `NewMapExpression`'s existing `KeyType`/`ValType` naming) to hold `K`; `ElemType` holds `V`, consistent with how it already doubles as "the wrapped type" for `Promise<T>` and for structural array types (`{...}[]`).
- Any other/unrecognized generic name still falls into the original skip-and-discard behavior — user-defined generics are still genuinely unimplemented (`docs/tdd/TDD-00010.md`), and skipping is still the right, honest failure mode for those; this fix only touches the three built-ins that already have full runtime support and were being mis-typed for no good reason.

**Type resolution** (`codegen/llvm/emitter.go`, `resolveType`): added explicit `Map`/`Set` cases, positioned before the existing generic "`ElemType` present → treat as array" fallback (the same position `Promise<T>`'s own check already occupies, and for the same reason — that fallback is a blanket rule any type setting `ElemType` falls into unless intercepted first). `Array<T>` needed no new case at all: since it now sets `ElemType` like a structural array type does, it resolves correctly through the *existing* fallback without any special-casing — only `Map`/`Set` needed to be pulled out ahead of it, since an unintercepted `Map<K,V>`/`Set<T>` would otherwise silently resolve as `ArrayOf(V)`/`ArrayOf(T)` instead of a real map/set.

**`emitVarDecl`'s call-return-type inference** (`codegen/llvm/emit_exprs.go`): added `IsMap`/`IsSet` to both places that gate whether a function call's registered return type is trusted for an unannotated variable declaration (the named-function case and the closure-variable-call fallback right below it) — otherwise the parser fix alone wasn't observable through `const x = someMapReturningFn()`, only through an explicitly-annotated `const x: Map<K,V> = ...`.

## Implementation notes

- Files touched: `ast/nodes.go` (`TypeAnnotation.KeyType` field), `parser/parser.go` (generic-type-argument parsing), `codegen/llvm/emitter.go` (`resolveType`'s `Map`/`Set` cases), `codegen/llvm/emit_exprs.go` (`emitVarDecl`'s call-return-type allowlist, two call sites).
- No changes needed to `emit_call.go`'s `inferExprType`, which already trusted a callee's return type unconditionally — confirmed directly by reading it before assuming a parallel fix was needed there too.

## Side effects discovered

The `emitVarDecl` allowlist gap described above — found while verifying this fix, not a pre-existing known issue, and fixed in the same pass since it directly blocked confirming the parser fix actually worked end-to-end for the common "assign a Map-returning call to an unannotated variable" shape.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green.
- Manual repros, each confirmed broken before the fix and correct after: `Array<number>` as a parameter type (`.length` now works); `Array<number>` as a return type, including indexing the result; `Map<string, number>` as both a parameter and return type on the same function, round-tripped through an unannotated `const`; `Set<string>` as a return type. Also re-verified, as a regression check, that the untouched construction forms still work: `new Array<T>(n)`, `new Map<K,V>()`, `new Set<T>()`, and `Promise<T>` (an `async` function declared `Promise<number>`, awaited).
- New E2E tests: `tests/arrays_test.go`'s `TestE2EArrayGenericTypeAnnotationParam`, `TestE2EArrayGenericTypeAnnotationReturnType`, `TestE2EArrayGenericTypeAnnotationStringElem`; `tests/collections_test.go`'s `TestE2EMapGenericTypeAnnotationParamAndReturn`, `TestE2ESetGenericTypeAnnotationReturnType`, `TestE2EMapGenericTypeAnnotationNumberKey`.
- `make examples` — full suite still passes (58/58).
