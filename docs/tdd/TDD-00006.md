# TDD-00006: Event Loop

- **Status**: Partially Implemented (ADR-00048, ADR-00049) — Part 1 done; Part 2's fiber mechanism is real and used for concurrent HTTP connection handling, but `async`/`await`/`fetch()` itself still runs synchronously (needs libcurl's non-blocking multi-interface too — not started)
- **Date**: 2026-07-13

## Context

Tracked as this project's single biggest structural gap relative to its stated microservice direction. Written up in full because the reason the gap exists at all matters for how to close it: `async`/`await` (`emit_async.go`) was deliberately built as a fast, syntax-compatible stand-in for `Promise`-shaped code — malloc a slot, run synchronously, hand back a pointer to it — specifically to get real programs writing `async`/`await`/`Promise<T>` working quickly, not as a faithful implementation of what those keywords actually mean. The file's own header comment says so plainly: *"No LLVM coroutine intrinsics are used."* That was the right call at the time (getting the syntax working end-to-end mattered more than getting concurrency right immediately), but it means every "async" program this compiler has ever produced has actually run in strict, blocking, top-to-bottom order — this document exists because that stand-in now needs to be replaced with the real thing.

### Why this is two problems, not one

That splits "event loop" into two pieces of very different size:

1. **An I/O-multiplexing wait loop** (`select`/`poll`/`epoll`/`kqueue`) that lets the process wait on sockets and timers together instead of blocking on one thing at a time. This alone is enough to make `TDD-00004` (HTTP Server)'s own V1 scope (single-threaded, one connection at a time) actually buildable, and reuses a lot that already exists.
2. **Real suspension** — actually letting `await fetch(...)` yield control back to the loop while a request is in flight, letting `Promise.all` run several awaits concurrently, letting one slow HTTP request not block every other client. This needs a fundamental rewrite of how `async`/`await` is compiled, not an addition next to it.

Part 2 cannot do anything useful without Part 1 (something still has to notice "this socket is ready now" and know which suspended computation to resume) — so Part 1 is the correct thing to build first regardless of which direction Part 2 eventually takes, and is worth doing even if Part 2 never happens (it still unlocks the HTTP server).

## Design

### Part 1: I/O-Multiplexing Loop

- **Mechanism**: plain POSIX `select()` for a first cut, not `epoll`/`kqueue`. `select()` is available identically on both macOS and Linux with zero platform-conditional codegen needed (unlike most of this project's other platform differences), and its `FD_SETSIZE` (typically 1024) / O(n)-scan limitations are a non-issue at the scale this project actually operates at — the same "simplicity now, revisit only if it needs to scale" reasoning `TDD-00002` (Timers) already used for a linear-scan queue instead of a real heap. `epoll` (Linux) / `kqueue` (macOS/BSD) are the natural follow-up once/if a program genuinely needs more than a handful of concurrent connections.
- **Merges with the existing timer queue, generalizing the drain loop** Timers already added: instead of "sleep until the next timer is due," compute that same next-due timestamp but pass it to `select()` as a timeout instead of `nanosleep`, alongside a set of file descriptors to watch. Whichever wakes `select()` first — a timer coming due, or a socket becoming readable/writable — gets handled, then the loop repeats. One unified loop, not two competing ones.
- **fd registration is an internal runtime concern, not a new user-facing TS API for V1** — no direct "register this socket for readiness" call exposed to user code, the same way the timer queue itself was never a user-visible data structure either. The HTTP server's accept loop (and, later, a non-blocking `fetch`) would register file descriptors internally.
- **Calling back into user code on readiness reuses the same closure-trampoline convention** already proven three times over (`qsort` comparators, timer callbacks, and `TDD-00004`'s own planned dispatch) — no new calling-convention work needed here specifically.

**Recommended sequencing**: buildable independently of Part 2 — the first piece to pick up regardless of Part 2's eventual direction, since it's needed either way and is far more tractable.

### Part 2: Real Async Suspension

Without this, Part 1 alone still leaves every "async" program running fully synchronously under the hood — `await fetch(...)` blocks the entire process exactly as it does today, `Promise.all` has nothing meaningful to parallelize, and an HTTP server built on Part 1 alone can multiplex *accepting* connections but still has to finish one request's handler, synchronously, top-to-bottom, before starting the next. Real concurrency needs a function to actually stop executing at an `await`, hand control back to the loop, and resume later, picking up exactly where it left off — a fundamentally different execution model than "call a function, it returns when it's done," which is the only model this compiler (and LLVM IR's own `call`/`ret` instructions) work with today.

**Options considered:**

1. **LLVM coroutine intrinsics** (`llvm.coro.id`/`coro.begin`/`coro.suspend`/`coro.end`, the mechanism Rust's `async`/`await`, Swift's concurrency, and clang's own C++20 coroutines are built on). LLVM's own coroutine-splitting pass does the hard part — transforming a function into a resumable state machine with a heap-allocated frame — automatically, once the right intrinsic calls are emitted at the right points. The most battle-tested, well-documented option, and doesn't add a new *external* dependency (it's part of LLVM itself, already required) — but it's a wholly new corner of LLVM IR generation this project has zero prior experience with, and every existing `await` call site and the entire async codegen path would need rewriting from scratch. **Ruled out** by direct prototyping (see below): incompatible with this compiler's `setjmp`/`longjmp` exception model as-is.
2. **Continuation-passing-style (CPS) transformation done by this compiler itself**, not by LLVM: split an async function's body at every `await` into separate continuation functions, each capturing exactly the local variables still needed — conceptually similar to this compiler's *own* existing closure-capture/heap-boxing machinery (reusing a pattern it already knows), rather than learning a new LLVM subsystem. But doing this correctly needs a real compiler pass this project has never built: a control-flow-graph-aware split with genuine live-variable analysis (which variables are alive across which suspend point) — a serious, standalone piece of compiler engineering, comparable in kind to the borrow-checker-scale effort `TDD-00001` (Memory Management) already declined to build for exactly this reason. **Not prototyped** — de-prioritized on effort grounds alone once option 3 below was confirmed to work with none of CPS's downsides.
3. **Hand-rolled fibers/green threads** (a separate stack per concurrent task, switching between them by saving/restoring the stack pointer and registers — the same shape as `boost::context`, `libco`, or Lua's coroutines). Originally scoped as needing "a small amount of architecture-specific assembly," but prototyping found POSIX `ucontext.h` (`getcontext`/`makecontext`/`swapcontext`) does the actual context-switch with **no hand-written assembly at all** — plain C-ABI functions, `declare`d and `call`ed exactly like every other libc dependency this compiler already has. **Recommended — confirmed working by direct prototyping, see below.**

**Prototyped directly, not decided from the design doc alone** (per this TDD's own original recommendation): five small, throwaway hand-written `.ll` files, compiled with this project's exact `clang -O2` invocation on this dev machine (arm64 Darwin), each testing one specific question rather than a full toy implementation:

- **Basic LLVM coroutine mechanics work correctly** on this exact toolchain: a hand-written switch-based coroutine (`llvm.coro.id`/`size`/`begin`/`suspend`/`end`, plus `llvm.coro.resume`/`destroy`) compiled and ran with the expected suspend/resume output, no surprises.
- **The exact risk this TDD originally flagged as "unstudied" turned out to be a real, confirmed crash**: a `try { await g() } catch {}`-shaped test — `setjmp` called before a coroutine's suspend point, `longjmp` back to that same `jmp_buf` from code running after a `resume()` call — segfaults (confirmed against a passing control test using the identical setjmp/longjmp pattern with no coroutine involved, isolating the crash to the suspend/resume boundary specifically). Root cause: LLVM's switch-based lowering unwinds the *C stack frame* of whichever invocation (the initial call, or a given `resume()`) hits a suspend point — `setjmp`'s saved stack pointer becomes stale the instant that frame returns, and a later `longjmp` to it is undefined behavior (in practice, here, a segfault) regardless of where the `jmp_buf`'s own bytes live (this compiler's real `@__kml_jmp_stk` is a global array, not a stack alloca — tested that shape specifically, not a strawman). Fixing this would need the exception system itself redesigned to be coroutine-frame-aware (jmp_bufs re-established at the top of every `resume()` that re-enters a live try block) — a second, comparably-sized rewrite on top of the coroutine rewrite itself.
- **`ucontext.h`'s deprecation guard is a C-header/preprocessor-only concern** — `_XOPEN_SOURCE` gates whether `<ucontext.h>` compiles at all, but this compiler never includes C headers (every C dependency is a hand-written `declare`), so the guard never applies. Confirmed the functions link and work directly from raw IR.
- **`ucontext_t`'s exact size and field offsets** (needed to build one by hand, with no header/`sizeof`/`offsetof` available at IR-emission time) were obtained the same way this project's real `fs.readdirSync` already gets `struct dirent`'s field offset — a tiny throwaway C program using `sizeof`/`offsetof`, compiled and run once to read off the numbers, not guessed or trusted from memory: `sizeof(ucontext_t) == 880`, `uc_stack` at offset 8 (`ss_sp` at +8, `ss_size` at +16 within it), `uc_link` at offset 32, on this machine (arm64 Darwin).
- **The identical setjmp/longjmp-across-suspend test, redone with a fiber yield (`swapcontext`) instead of a coroutine suspend, works correctly** — no crash, correct output, because a fiber's own OS stack stays allocated and untouched for the entire time it's parked in `swapcontext`, unlike a coroutine invocation's C stack frame (which is unwound the moment it suspends). This is the deciding factor: fibers need **zero changes** to the existing exception system, while LLVM coroutines need a from-scratch redesign of it just to reach parity with what fibers already give for free.

**Decision: fibers (option 3) — implemented for `http.listen`, not yet for `async`/`await` itself.** The only option confirmed (not just assumed) to compose safely with this compiler's actual, unusual `setjmp`/`longjmp` exception model, and cheaper to build than originally scoped (no custom assembly needed). `docs/adr/ADR-00049.md` used exactly this mechanism to make `http.listen` handle connections concurrently: each accepted connection gets its own fiber (a `ucontext_t` + a 64KB stack, matching the prototype's own default — fixed, not grow-on-demand, per the Open Questions leaning below), a global "current fiber" scheduler slot the fiber body reads to find its own connection-array entry, and a fd `-1` sentinel a fiber sets on itself right before returning to signal "I'm done" to the scheduler (the same tombstone-not-compact idiom the timer queue already uses). Only the `read()` side is fiber-aware (yields on `EAGAIN` after `fcntl(O_NONBLOCK)`); `write()` stays a single blocking call, a deliberate V1 simplification since small HTTP responses essentially never block a local socket write.

Not yet done: wiring a user's own `async function`/`await` through this same mechanism, which needs `fetch()`'s blocking libcurl call replaced with the non-blocking multi-interface first (a separate, unscoped piece of work — this compiler has zero experience with `curl_multi_*` yet) before suspension would have anything real to compose with. The memory-management caveat below still applies unchanged: fiber stacks/contexts, like everything else this compiler allocates, are never freed.

## Prerequisites

### Part 1

| Dependency | Status | Notes |

| Timer queue + drain loop (`TDD-00002`) | ✅ already done | Generalizes directly — same loop, `select()`'s timeout replaces `nanosleep`, fds get added alongside the existing timer-due check |
| Closure-calling-from-C-trampoline pattern | ✅ already proven | `qsort` comparators and timer callbacks are direct precedents |

### Part 2

| Dependency | Status | Notes |
|---|---|---|
| Part 1 (I/O-multiplexing loop) | ✅ done (`ADR-00048`) | A hard prerequisite — nothing here is useful without something to notice readiness and resume the right task |
| Fiber context-switch primitive | ✅ confirmed via prototype | `ucontext.h`'s `getcontext`/`makecontext`/`swapcontext`, called directly via `declare`/`call` — no hand-written assembly needed, contrary to this TDD's original assumption. `ucontext_t`'s size/offsets for this machine (arm64 Darwin) obtained via a throwaway `sizeof`/`offsetof` probe, the same technique `fs.readdirSync` already uses for `struct dirent`. |
| Exception system (`setjmp`/`longjmp`) compatibility with suspension | ✅ confirmed safe (fibers only) | Directly prototyped: a `try`/`catch` spanning a fiber-based suspend point works correctly, unmodified — the fiber's own OS stack stays live for the entire suspend. The same test against an LLVM-coroutine-based suspend segfaults (confirmed, not assumed) — ruling out option 1 without an exception-system rewrite. |
| Non-blocking replacement for `http.listen`'s per-connection `read()` | ✅ done (`ADR-00049`) | Each accepted connection is fcntl'd `O_NONBLOCK` and runs on its own fiber; a would-block read yields (swapcontext back to the scheduler) instead of stalling every other connection. Confirmed directly, not just by unit test: a connection that never sends its request line does not block a second, concurrent connection from being accepted and answered. |
| Non-blocking replacement for `fetch()`'s blocking libcurl call | ❌ not started | `fetch()` still calls libcurl's blocking `curl_easy_perform` — real concurrent `await fetch(...)` needs the non-blocking multi-interface (`curl_multi_*`) wired into the same event loop, a separate, not-yet-scoped piece of work |
| `async`/`await` itself compiled as real suspension (rather than `emit_async.go`'s synchronous malloc-a-slot stand-in) | ❌ not started | The fiber mechanism now exists and is proven (used by `http.listen`), but nothing wires a user's own `async function`/`await` through it yet — every `async` program still runs fully synchronously under the hood |
| Garbage collection / memory management (`TDD-00001`) | ⚠️ same caveat as `TDD-00004` | Long-running concurrent tasks that never free anything is the same accumulating-memory problem, just with more tasks (and now per-task stacks) alive at once — `http.listen`'s per-connection fiber stacks/ctx buffers are never freed either, tombstoned (`fd = -1`) in the same array forever, matching the timer queue's own established "leak everything, don't compact" convention |

## Open questions

- Whether Part 1 should target `select()` permanently for this project's realistic scale, or whether `epoll`/`kqueue` should be built from the start — leaning toward `select()` first, revisit only if it becomes a real bottleneck.
- Whether `async`/`await`/`fetch()` should be wired onto this same fiber mechanism next (needs `curl_multi` too), or whether that's better scoped as its own follow-on now that the underlying primitive is proven — leaning toward a separate pass, since libcurl's multi-interface is its own unfamiliar API surface this project has zero experience with yet.
- Per-fiber stack size (64KB, fixed, used both in the prototype and the real `http.listen` implementation) vs. grow-on-demand — unchanged from the prototype's leaning: fixed for now, revisit only if it becomes a real limitation.
