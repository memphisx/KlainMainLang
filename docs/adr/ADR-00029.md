# ADR-00029: Implement console.time/timeEnd, count/countReset, group/groupEnd, dir

- **Date**: 2026-07-12

## Context

Picked up the remaining `❌` rows in `STATUS.md`'s `console` table: `.table()`, `.time()`/`.timeEnd()`, `.count()`/`.countReset()`, `.group()`/`.groupEnd()`, `.dir()`.

## Investigation

- **`.table()` deliberately deferred, not attempted.** Real `console.table` needs a genuinely new algorithm — dynamic per-column width computation and box-drawing header/index rows over arbitrarily-shaped input (arrays of objects with different keys, nested values, etc.) — not a small extension of the existing single-value print path every other `console.*` method already rides on. Left `❌` with the reason spelled out rather than attempted as a half-implementation.
- **`.time()`/`.timeEnd()`**: reused the existing `performance.now()` machinery (`ADR-00024`) directly — no new clock code. Scoped to a single global monotonic-time slot rather than a real per-label map, a deliberate V1 narrowing (calling `time()` again just overwrites the one running timer, regardless of label) — the same incremental-scope pattern this project's other features have started with.
- **`.count()`/`.countReset()`**: checked whether a per-label map was actually within reach before scoping it down the same way — it was. This compiler already has full `Map<string, number>` runtime support (`__kml_map_str_create/get/set`, from the language-level `Map<K,V>` feature), so `.count()` reuses those helpers directly via a hidden, lazily-created global map, never exposed as a KML-level value. Matches real Node's actual per-label semantics exactly, unlike `.time()`'s single-slot narrowing above — the difference is simply that the machinery for the fuller behavior happened to already exist.
- **`.group()`/`.groupEnd()`**: added a hidden global indent-depth counter, and centralized indent-printing in `emitConsolePrint` (the one shared function every `console.log`/`.error`/`.warn`/`.info`/`.debug`/`.trace` call already funnels through), so every existing output method gets correct indentation for free. Since this compiler prints each `console.log` argument on its own line (an existing, unrelated convention — not real JS's single space-joined line), the indent is applied once before the first line (before an optional prefix like `"Warning: "`, or before the first argument if there's none) and again before every argument after the first, since each one starts a fresh line of its own.
- **`.dir(obj, options?)`**: implemented as a thin call into the existing single-argument `console.log` path, ignoring `options` (real Node's depth/color controls) — the core use case (print a value) is already fully served by machinery that exists.
- **A genuine bug in the new indent-loop code, caught by an existing, unrelated test failing.** `TestE2EProcessExit` (`console.log("before"); process.exit(42); console.log("after")`) started failing at the `clang` step with `use of undefined value '%t7'` the moment `.group()`'s indent logic was added — not a logic error in the feature itself, but a structural one: `emitLabel` unconditionally resets the emitter's `blockDone` dead-code flag regardless of whether it was already set, so a value computed *before* the indent loop's own labels (and correctly dropped as dead code, per this codebase's existing "instructions after a terminator are silently dropped" convention) ends up referenced by code that looks "live" again purely because a label reset the flag — an LLVM verifier violation, not a runtime bug, since the referencing block is never actually reachable but LLVM still requires every SSA value's use to be dominated by a real definition. Fixed by bailing out of the indent helper immediately if the block is already dead, mirroring `emitInstr`'s own behavior at the granularity of the whole helper rather than per-instruction.
- **A second, deeper, entirely unrelated bug, found while writing a regression test for the one above.** A hand-written regression test (`console.log("before"); return; console.log("after")`, inside a plain function, no `process.exit` involved) *still* printed `"after"` even after the fix above. Traced with `--emit-llvm` down to the raw IR: the `return` statement's `ret void` appeared at the very end of the function, *after* `"after"`'s own printf — meaning `return`'s value-expression parsing, not codegen, was at fault. Confirmed in `parser.go`'s `parseReturnStatement`: it only checked for `;`/`}`/EOF before deciding whether to parse a return value, with **no check for a line terminator between `return` and what follows** — missing JS's actual ASI restriction that a line break there forces a bare `return`. `return\nconsole.log("after")` was parsing as one statement, `return console.log("after")`, not two — meaning "after" was never dead code in the first place, at the AST level. Confirmed no existing test or example anywhere in the repo relied on the old (incorrect) behavior (a multi-line return *value* starting on the line after `return`) before fixing it.

## Decision

`console.time(label?)`/`console.timeEnd(label?)`, `console.count(label?)`/`console.countReset(label?)`, `console.group(label?)`/`console.groupEnd()`, `console.dir(obj, options?)` — implemented with the scope narrowings above. `console.table()` — deliberately left unimplemented, with the reason recorded. Also fixed, as bugs found along the way: the `emitConsoleGroupIndent` dead-code verifier crash, and the parser's missing ASI restriction on `return`.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureConsoleGroupDepth`/`ensureConsoleTimer`/`ensureConsoleCountMap` (thin global-declaration singletons; the timer one pulls in `ensurePerformanceNow`, the count-map one pulls in the existing `ensureMapStrHelpers`).
- `codegen/llvm/emit_call.go`: `emitConsoleGroupIndent` (the indent loop, built with `freshLabel`/`freshReg`/`emitAlloca` like ordinary control flow elsewhere in the compiler, not hand-written IR text — and guarded against already-dead code, see Investigation); `emitConsolePrint` restructured to call it once up front and again before every argument after the first; `emitConsoleGroup`/`emitConsoleGroupEnd` (depth increment/decrement, floored at 0); `emitConsoleDir` (delegates to `emitConsolePrint` with a truncated single-element arg list); `consoleLabelArg` (shared optional-label-argument resolution, defaulting to `"default"`); `emitConsoleTime`/`emitConsoleTimeEnd` (store/read the global timer slot, `sprintf` the `"<label>: <elapsed>ms"` message); `emitConsoleCountMapEnsure` (lazily creates the backing map, using the same alloca+store-in-each-branch+load-after-merge shape `emitOptionalMember` already established, rather than hand-tracking `phi` predecessor labels); `emitConsoleCount`/`emitConsoleCountReset`.
- `parser/parser.go`: `parseReturnStatement` gained the `p.peek().Line == tok.Line` check described above.
- `docs/adr` convention followed for both bugs: documented in this ADR's Side effects section rather than a separate one, consistent with `ADR-00023`/`ADR-00026` folding bugs found during a feature's own implementation into that feature's ADR.

## Side effects discovered

Two bugs, both described in Investigation above, fixed in this same pass:
1. `emitConsoleGroupIndent` crashing the `clang` verification step for any program with dead code after a terminator, once `.group()`'s indent loop was in place — fixed by an early `blockDone` bail-out.
2. The parser's missing ASI restriction on `return` — a real, pre-existing, previously-undiscovered bug (not introduced by this work), only surfaced because bug 1's fix made a `console.log`-after-`return` test actually exercise the difference between "dead code correctly dropped" and "silently kept alive by a mis-parsed return value." Fixed in `parseReturnStatement`.

- Manual repros, each actually compiled and run: `.dir` on a string and a number; `.time`/`.timeEnd` around a loop (confirming the `"<label>: ...ms"` shape, not an exact value — including hitting the known, harmless `-O2` loop-idiom-collapse artifact from `ADR-00024` where a reducible loop measures as exactly 0ms); `.count`/`.countReset` across multiple labels including the default one, confirming independent per-label counting and a correct reset; `.group`/`.groupEnd` nested two levels deep, an unbalanced extra `groupEnd()` not going negative, and multi-argument `console.log` calls each getting their own indented line while inside a group; `--emit-llvm` inspection of the exact IR that exposed both bugs in Investigation, confirmed fixed.
- New tests in `compiler_test.go`: `TestE2EConsoleDir`, `TestE2EConsoleDirWrongArgCountRejected`, `TestE2EConsoleTimeEnd` (checks the `"label: ...ms"` shape via `strings.HasPrefix`/`HasSuffix` on captured stdout, not an exact value), `TestE2EConsoleTimeEndWrongArgCountRejected`, `TestE2EConsoleCount`, `TestE2EConsoleCountWrongArgCountRejected`, `TestE2EConsoleGroupIndentsSubsequentOutput`, `TestE2EConsoleGroupMultiArgIndentsEveryLine`, `TestE2EConsoleGroupEndUnbalancedDoesNotUnderflow`, `TestE2EConsoleGroupEndWrongArgCountRejected`, `TestE2EConsoleGroupWithLogAfterDeadCode` (the bug-1-and-2 regression test), `TestE2EReturnASI` (a narrower, `console.group`-independent test of the parser fix alone).
