# ADR-00037: Fix parenthesized function-type return annotations

- **Status**: Completed
- **Date**: 2026-07-12

## Context

Tracked in `STATUS.md`'s Known Limitations & Bugs: `(): (() => void) => { ... }` failed to parse — only the unparenthesized form (`(): () => void => { ... }`) worked as a return-type annotation. Picked up as the next low-effort item after `docs/adr/ADR-00036.md`.

## Investigation

- Reproduced directly: `const makeCounter = (): (() => void) => { let n = 0; return () => {...} }` failed with `2:3: expected IDENT, got let` — not the `expected IDENT, got return` shape `STATUS.md`'s original entry described, since the exact failure text depends on what follows, but the same root cause.
- Traced through `parseTypeAnnotation` (`parser/parser.go:316`), specifically its `lexer.LPAREN` branch. This branch **always** assumes an opening `(` begins a function type's parameter list (`(param: type, ...) => retType`), unconditionally requiring `=>` immediately after the matching `)`. Given input `(() => void) => { ... }`: the outer `(` starts this branch; inside its param-collecting loop, the recursive call for the first "parameter" is itself given `() => void)`, which the same LPAREN branch happily parses as a **complete, self-contained function type** (`IsFuncType: true, FuncParams: [], FuncRetType: void`) — becoming `funcParams[0]` of the *outer* call, which was never the intent. Back in the outer call, the loop then sees the outer `)`, consumes it, and unconditionally expects `=>` next — which happens to exist, but it's the *enclosing arrow function's own body arrow*, not a second curry level. The outer call then tries to parse a return type starting at `{ let n = 0 ...`, hits the object-type-annotation branch (`tok.Type == lexer.LBRACE`), and fails expecting a field name.
- Confirmed real currying (`(x: number) => number` used as, say, a parameter or return type — no parenthesized wrapping) still works and remains unaffected by design, since parenthesized-grouping and true single-parameter curried function types are genuinely ambiguous at the grammar level up to the point where the return type is parsed — they only diverge based on whether a valid type actually follows the `=>`. A `(): (() => void) => { ... }`-style body block can never be a valid type, but a real curried return type always is, so this is resolvable with a small amount of lookahead/backtracking rather than requiring a grammar change.
- Also confirmed, while isolating a clean repro for the fix, that this bug is entangled with a **second, separate, pre-existing bug**: even with parentheses removed entirely (`const makeCounter = (): () => void => { ... }`), calling the returned closure (`inc()`) still fails with `undefined function or closure 'inc'`. This reproduces with or without this ADR's fix — confirmed by testing the identical shape both ways — so it isn't something this fix caused or could accidentally paper over. It turns out `STATUS.md`'s pre-existing "closure return-type inference gap" entry, which claimed an explicit return-type annotation resolves the issue, was only ever verified using a plain `function middle(): () => void { ... }` **declaration** — re-tested here and confirmed that form does still work correctly — not a `const`-bound **arrow function** with the same annotation, which does not. Updated that `STATUS.md` entry to reflect the narrower, now-confirmed scope of the workaround (arrow functions are affected regardless of annotation; only `function` declarations are helped by one) rather than leaving the previously-untested, broader claim in place.

## Decision

In `parseTypeAnnotation`'s `LPAREN` branch, after consuming the closing `)`: if an `ARROW` follows, tentatively consume it and attempt to parse a return type. If that succeeds, treat the whole thing as a genuine (possibly curried) function type, exactly as before. If it *fails* and there was exactly one parameter that was unnamed (no `ident:` prefix consumed), backtrack `p.pos` to just before the tentatively-consumed `=>` and return that single parameter's type directly — i.e. treat the outer parens as pure grouping, leaving the arrow for whatever comes next (the enclosing arrow function's own body arrow, in this case) to consume. If there was no `ARROW` at all after the closing `)`, apply the same single-unnamed-parameter fallback directly (handles `let x: (number)` and similar non-function grouped types too, as a natural side effect of the same logic, without a separate code path).

Restricting the fallback to *exactly one, unnamed* parameter deliberately avoids misinterpreting a genuinely broken multi-parameter or explicitly-named-parameter function type (e.g. `(cb: () => void, bad syntax here) => number`) as silent "grouping" — those cases still propagate their real parse error instead of being swallowed.

This is the first place in this parser that backtracks (`Parser.pos` is a plain `int`, saved and restored locally) — no new general backtracking mechanism was introduced; the save/restore is entirely local to this one branch.

## Implementation notes

- `parser/parser.go`, `parseTypeAnnotation`'s `LPAREN` branch: added a `singleUnnamed bool` tracked per parameter (set to `false` the moment an `ident:` prefix is consumed for any parameter), and the tentative-arrow/backtrack logic described above, replacing the previous unconditional `p.expect(lexer.ARROW)` immediately after the closing `)`.
- No codegen changes — this is a pure parser fix; the resulting `*ast.TypeAnnotation` for a parenthesized function type is identical in shape to what an unparenthesized one already produced (`IsFuncType`/`FuncParams`/`FuncRetType`), so nothing downstream needed to change.
- `STATUS.md`: removed the "Parenthesized function-type annotations" row from both Known Limitations & Bugs and the Roadmap's "Next up" table; refined the wording of the still-open closure-return-type-inference row (see Investigation) to reflect what was actually confirmed, rather than leaving the previously broader, partially-incorrect claim in place.

## Side effects discovered

- The separate arrow-function closure-return-type-inference bug described in Investigation — not fixed here (out of scope, tracked as its own, now-more-precisely-worded `STATUS.md` row), but `STATUS.md`'s existing description of it was corrected to remove an inaccurate claim (that an explicit return-type annotation fixes it universally — it doesn't, only for `function` declarations, not arrow functions) discovered while isolating a clean, unconfounded repro for this ADR's actual fix.
