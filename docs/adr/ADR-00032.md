# ADR-00032: Add a --static CLI flag for statically-linked binaries (Linux only)

- **Date**: 2026-07-12

## Context

Looked into compiled executables' runtime dependencies — specifically, whether a compiled binary would run in a Docker image built `FROM scratch` (nothing else in the image at all). Found no: every compiled binary is dynamically linked, closer to typical C/C++ toolchain output than a normal Go binary's usual self-contained default — which matters specifically because avoiding exactly this gap was one of the reasons this project's own compiler is written in Go in the first place. Added a way to close that gap.

## Investigation

- **Confirmed the current linking behavior directly, not from memory.** `main.go`'s clang invocation (`clangArgs := []string{"-O2", llFile, "-o", outBin}`, plus conditional `-l<lib>` flags) has no `-static` anywhere. Ran `otool -L` (macOS's `ldd` equivalent) against an actual compiled example: every binary links `/usr/lib/libSystem.B.dylib` unconditionally, and anything using `fetch` additionally links `/usr/lib/libcurl.4.dylib` (`ADR-00020`'s conditional linking). On Linux the equivalent would be `libc.so.6` (glibc) plus a program interpreter entry, and `libcurl.so` (plus its own transitive deps — OpenSSL, zlib) for `fetch`-using programs.
- **Confirmed macOS categorically cannot do this at all, directly rather than assumed.** Compiled a trivial `hello world` C program with `clang -static` on this dev machine: `ld: library 'crt0.o' not found`. This is not a missing-package issue fixable by installing something — Apple's toolchain has never shipped a static libSystem, and treats its ABI as a rolling-release interface not meant to be statically linked against. There is no workaround on macOS; the constraint is architectural, not a version/installation gap.
- **Checked whether the actual Linux static-linking path could be verified directly in this environment — initially, it could not.** No Linux cross-toolchain, no musl-gcc, no Linux sysroot, and no Docker available on this dev machine at first (confirmed: `which musl-gcc x86_64-linux-gnu-gcc zig` all empty, and `clang -target x86_64-linux-gnu` fails immediately at `#include <stdio.h>` with no Linux libc headers present to find). This compiler also doesn't cross-compile (it always runs `clang` on the same machine that ran the KML compiler itself, per its own stated scope) — meaning `--static` genuinely has to be exercised by running KlainMainLang itself on a real Linux machine or a Linux Docker build stage. Docker Desktop turned out to have been present but broken (uninstalled, likely by an interrupted Homebrew upgrade, leaving a stale `/usr/local/bin/docker` symlink and an orphaned `com.docker.vmnetd` helper process behind) — reinstalled, and the Linux path below was verified directly once it was working.
- **Verified the Linux path directly via a real multi-stage Docker build (`docker/Dockerfile`)**: stage 1 builds the KlainMainLang compiler itself on `golang:1.26-alpine`; stage 2 uses that compiler plus Alpine's `clang`/`musl-dev` to `--static`-compile `docker/hello.ts`; stage 3 is `FROM scratch` with nothing copied in but the resulting binary. Chose Alpine/musl over glibc specifically because musl's static linking has none of glibc's NSS/`getaddrinfo` rough edges — moot for this non-`fetch` demo program either way, but the smoother default. Confirmed success two ways, not just "the build didn't error": `docker run --rm klainmainlang-demo` actually executed inside the empty `scratch` image and printed correct output (`process.pid`/`process.platform`/`process.env` all working) — impossible for a dynamically-linked binary, since there'd be no loader present to even start the process; and `readelf -d`/`readelf -l` on the binary directly confirmed "there is no dynamic section in this file" and no `PT_INTERP` program header at all, the definitive ELF-level proof of true static linking.
- **Decided against making `--static` the default**, given the above: since macOS cannot do this at all, defaulting to it would break every existing build on macOS immediately. Opt-in via a new flag is the only option that doesn't regress the platform this project is actually developed on day to day.
- **Decided to fail fast with an explanation, not let clang's own linker error surface.** Given the macOS failure mode is a categorical, permanent constraint (not a transient/fixable error), checking `runtime.GOOS` in the Go compiler itself before ever invoking clang (the same "check the Go compiler's own OS, since it also runs clang moments later" reasoning `errnoAccessor`/`monotonicClockID`/`nodePlatformName` already rely on) and refusing immediately with a clear, specific explanation is far more useful than a cryptic `ld: library 'crt0.o' not found` with no context.

## Decision

`--static` (or `-static` — Go's `flag` package treats both prefixes identically): if set and `runtime.GOOS != "linux"`, `main.go` refuses immediately with an explanation (why macOS can't do this at all, and what to do instead — run KlainMainLang itself on Linux). Otherwise, `-static` is appended to the clang invocation. No attempt to special-case `libcurl`'s own static-linking requirements (a static `libcurl` — and its own dependencies — would need to be available on the Linux build machine separately for a `fetch`-using program's `--static` build to fully succeed) — documented as a caveat, not solved automatically, since it's a genuinely separate, environment-dependent concern outside what this flag itself controls. (Investigated concretely in `ADR-00033`: it does work, but needs curl's full dependency chain listed explicitly and, on Alpine/musl, a `gcc`-not-`clang` final link step — specific enough to one distro's build that this compiler still doesn't try to automate it.)

## Implementation notes

- `main.go`: new `static` flag (`flag.Bool("static", ...)`); the `runtime.GOOS` guard, placed immediately after argument parsing (before any parsing/codegen work happens, so the failure is instant); `-static` appended to `clangArgs` when set (before the `-l<lib>` flags, matching typical linker flag ordering conventions).
- `README.md`: documented alongside the existing `--emit-llvm`/`-o` flags, plus a short paragraph explaining the general dynamic-linking-by-default fact this flag addresses and the `libcurl` caveat.
- `docker/Dockerfile`, `docker/hello.ts`, `.dockerignore` (new): the three-stage verification build described in Investigation — also a reusable template for the multi-OS/architecture release-binary CI this project will eventually want (build stage → compile stage → minimal runtime stage is the same shape either way).

## Side effects discovered

None — no pre-existing bugs surfaced.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, unaffected — no existing test exercises `main.go`'s own CLI flag parsing at all; every existing test calls the underlying `parser`/`llvm`/`resolver` packages directly and invokes `clang` itself, bypassing `main.go` entirely, so there was nothing to regress here and nothing existing to extend either), `make examples` all green (55/55, unaffected).
- Manual repros, each actually run: a normal compile with no flags, unaffected; `--static` and `-static` both correctly refused on this (macOS) dev machine with the full explanatory message and exit code 1, confirmed *before* any clang invocation happens (no cryptic linker error reaches the user).
- **Linux path verified end-to-end via `docker/Dockerfile`** (see Investigation): `docker build -f docker/Dockerfile -t klainmainlang-demo .` succeeds; `docker run --rm klainmainlang-demo` executes successfully inside a `scratch` container and prints the expected output; `readelf -d`/`readelf -l` on the compiled binary directly confirm no dynamic section and no `PT_INTERP` header. Final image: 79.2kB content, 292kB on disk — consistent with "nothing but one small static binary."
