# ADR-00006: Fix JSON.stringify(boolean[]) crash and JSON.stringify(object[]) garbage output

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Found via a systematic audit of every `❌`-marked item in `STATUS.md` (triggered by discovering `Map.size`/`Set.size` were stale-marked `❌` despite already working correctly). I compiled and ran a minimal repro for every remaining `❌` item rather than trusting the docs. Two of the findings were in `JSON.stringify`:

- `JSON.stringify([true, false, true])` — compiles cleanly but **segfaults at runtime** (exit code 139).
- `JSON.stringify([{x:1,y:2}, {x:3,y:4}])` — compiles and runs without crashing, but silently prints wrong data (garbage like `["",""]`) instead of the correct JSON.

This ADR fixes both. (Two other findings from the same audit — `JSON.parse` into a typed object, and `any`/`unknown`/union types emitting invalid IR — are tracked separately, since they're larger, distinct pieces of work.)

## Investigation

- **Root cause, both bugs.** `emitJSONStringify`'s array-handling branch (`codegen/llvm/emit_call.go`) only recognized two cases: `argTy.ElemType.IR == "i64"` (numbers) → one C-level helper, and an unconditional `else` for everything else, assuming the element type must be a string `ptr`. A `boolean[]`'s element type is `i1`, not `i64`, so it fell into the string branch, which called a helper that treats each element as a `char*` and calls `strlen`/copies bytes from it — but an `i1` (0 or 1) reinterpreted as a pointer is either null or address `0x1`, an invalid pointer dereference → crash. An `object[]`'s element type is a `ptr` to a heap-allocated struct, which also matched the `else` branch's "must be a string" assumption — reading a few bytes of the struct's raw memory as if it were a null-terminated C string happened to produce something that looked like an empty string on this build/data layout, rather than crashing, which is why it "worked" but produced garbage instead of failing loudly.
- **A working precedent already existed.** `emitArrayJoin` (`codegen/llvm/emit_arrays.go`, backs `.join(sep)`) already solves the general "runtime-length loop that builds up a separator-joined string, with special handling for the first element to avoid a leading separator" problem correctly, for *any* element type — it converts each element via `emitValueToString`, which already dispatches correctly on every scalar type. `emitJSONStringifyValue` (`emit_call.go`) is the JSON-specific analog of that per-element conversion — already correctly handling `i1` (→ `true`/`false`), `ptr` (→ quoted string), objects (→ recursive `{...}`), and numbers — but it was only ever being called for a single scalar argument or a single object's fields, never per-element inside the array branch.
- **The fix is therefore a reuse, not a new algorithm**: replace the two-case (`i64` vs. "everything else is a string") special-cased C helpers with a general Go-level runtime loop, modeled directly on `emitArrayJoin`'s shape, that calls the already-correct `emitJSONStringifyValue` per element instead of `emitValueToString`, and wraps the result in `[`...`]` with commas instead of a single separator.
- **A bug in the first draft of this exact fix, caught before shipping.** The initial implementation copied `emitArrayJoin`'s `firstL` block verbatim — `store ptr <element>, ptr <accumulator>` — which is correct for `.join()` (whose accumulator starts as `""`, so the first element simply becomes the whole accumulator) but wrong here, since the JSON-array accumulator is pre-seeded with `"["` before the loop starts. Verified by running `JSON.stringify([true,false,true])` and getting `"true,false,true]"` — missing the opening bracket, because the first-element branch was *overwriting* the pre-seeded `"["` instead of concatenating onto it. Fixed by loading the current accumulator (which holds `"["`) in the `firstL` block and concatenating the first element onto it, rather than storing the element alone.

## Decision

Replaced the array branch of `emitJSONStringify` with a new `emitJSONStringifyArray` (`codegen/llvm/emit_call.go`): a runtime loop over the array (via the existing `resolveArrayForHOF`, which already handles both the named-symbol and aggregate array representations) that, per element, calls `emitJSONStringifyValue` (already correct for every scalar/object type) and accumulates into a `[`-seeded string pointer, concatenating a comma before every non-first element, closing with `]` after the loop. This uniformly and correctly handles number, string, boolean, and object array elements in one code path, instead of two special-cased C-level helpers that only recognized two element shapes.

Removed the now-dead `ensureJSONStringifyNumArr()`/`ensureJSONStringifyStrArr()` and their `@__kml_json_str_num_arr`/`@__kml_json_str_str_arr` C-level IR definitions (`runtime.go`) entirely, along with their `usedJSONStringifyNumArr`/`usedJSONStringifyStrArr` gate fields (`emitter.go`) — confirmed via `grep` that nothing else referenced them before deleting, rather than leaving them as orphaned dead code.

## Implementation notes

Files touched:
- `codegen/llvm/emit_call.go` — new `emitJSONStringifyArray`; `emitJSONStringify`'s array branch now just calls it.
- `codegen/llvm/runtime.go` — removed `ensureJSONStringifyNumArr()`/`ensureJSONStringifyStrArr()` and their IR definitions.
- `codegen/llvm/emitter.go` — removed the two now-unused `used*` fields.

## Side effects discovered

None new beyond the self-caught bug described in Investigation (missing-bracket accumulator bug in the first draft, fixed before landing).

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including the extended `examples/json/json_methods.ts`.
- Manual repros: `JSON.stringify([true,false,true])` → `[true,false,true]` (previously segfaulted); `JSON.stringify([])` (any element type) → `[]`; `JSON.stringify([{x:1,y:2},{x:3,y:4}])` → `[{"x":1,"y":2},{"x":3,"y":4}]` (previously printed garbage); existing `number[]`/`string[]` cases re-verified unchanged.
- New tests in `compiler_test.go`: `TestE2EJSONStringifyBoolArray`, `TestE2EJSONStringifyObjectArray`.
- `examples/json/json_methods.ts` extended with boolean-array and object-array cases.
- `STATUS.md`: `JSON.stringify(boolean[])`/`JSON.stringify(object[])` flipped to ✅; removed their entries from Known Limitations & Bugs (the `JSON.parse` → object entry remains, tracked separately); coverage numbers updated.
