# ADR-00054: Implement Object.assign(target, ...src)

- **Date**: 2026-07-14

## Context

`STATUS.md`'s Object/Collections table and its "High priority / low complexity" roadmap both tracked `Object.assign(target, ...src)` as ❌ — the next natural quick win after `ADR-00053`'s `Map`/`Set` cleanup pass, since object spread (`{ ...obj }`, `ADR-00013`) and `Object.entries/keys/values` already existed and cover most of the same ground field-copy-wise.

## Investigation

Read `codegen/llvm/emit_objects.go`'s `emitObjectLiteral` (`ADR-00013`'s object-spread implementation): its `storeField` closure plus the `SpreadElement` branch already do exactly the field-copy work `Object.assign` needs — GEP into a source struct by field index, load, then GEP+store into a destination struct by field index. The only structural difference is *what* gets written into: object-literal spread always writes into a **freshly malloc'd** struct being built field-by-field; `Object.assign` needs to write into an **already-existing** target object in place, then return that same pointer.

The one real design question was what happens when a source object has a field target's type doesn't. Real JS's `Object.assign` would just add the field to `target` dynamically. This compiler's objects are fixed-shape heap structs — an interface's field list, and therefore its `malloc` size and GEP layout, are fixed at compile time — so there is no way to "add a field" to an already-allocated `target` at the call site. Decided this is a clean-rejection case, not a silent-drop or a crash: the same posture already established for spread's own boundary (a spread source contributes fields to the *literal's own inferred type*, which is computed from all its sources up front — spread never needs to reject because the target type is derived, not fixed in advance) and for `JSON.parse`→object's nested-field rejection (`ADR-00007`).

## Decision

Added `emitObjectAssign(args []ast.Expression, pos ast.Pos) (Value, error)` (`codegen/llvm/emit_objects.go`): evaluates `target` (must be `IsObject`), then for each remaining source argument (evaluated left to right, matching real JS's argument order and its last-write-wins overwrite semantics when two sources share a field), for each of the source's fields, looks up that field by name in `target`'s own type via `FieldIndex` — a miss is a compile error ("source has field '%s' not present on target's type"), a hit does a GEP-load from the source and a GEP-store into `target`'s existing struct pointer (through `e.coerce`, same as every other field-store in this codegen). Returns `target` itself (unchanged pointer, unchanged type), matching real JS's "returns the mutated target" contract.

Wired into two dispatch points, mirroring how `Object.keys`/`values`/`entries` are already wired: `emit_call.go`'s `Object.<method>` switch (new `case "assign"`) and `emit_exprs.go`'s `inferExprType` `Object.<method>` switch (`case "assign"` returning `e.inferExprType(ex.Args[0])`, i.e. target's own type — needed so `const merged = Object.assign(target, source)` infers `merged`'s type correctly).

## Implementation notes

- Files touched: `codegen/llvm/emit_objects.go` (new function), `codegen/llvm/emit_call.go` (dispatch case), `codegen/llvm/emit_exprs.go` (type-inference case).
- No new runtime IR helpers needed — this is pure Go-side codegen reusing the exact GEP/load/store shape `emitObjectLiteral`'s spread branch already established, just retargeted at an existing pointer instead of a freshly allocated one.
- `Object.assign(target)` with zero sources is accepted (a no-op returning `target` unchanged), matching real JS, which also allows a sources-less call.

## Side effects discovered

None — self-contained, no interaction with any other feature found during implementation or testing.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green.
- Manual repros: single source overwriting every target field; a partial-field source (fewer fields than target, only those touched); three sources with the middle one's field later overwritten by the last, confirming argument-order last-write-wins; a source field absent from target's type, confirming a clean compile error (not a crash or silently-dropped field) — `KlainMainLang: codegen error: 5:14: Object.assign: source has field 'z' not present on target's type`.

- `examples/objects/object_static.ts` extended with a `Object.assign` section (mutating in place, a partial-field source, multiple sources) — each line's inline expected-output comment verified against actual `./KlainMainLang` output.
- `make examples` — full suite still passes.
