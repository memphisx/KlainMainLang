# ADR-00033: Verify --static with fetch/libcurl on Alpine/musl

- **Status**: Completed
- **Date**: 2026-07-12

## Context

`ADR-00032` shipped `--static` but flagged one open caveat: a program using `fetch` also needs a statically-linkable `libcurl` (and its own dependencies) available, not solved automatically by that flag alone. Investigated this directly with real Docker builds — a couple of test images, one dynamic-`fetch` baseline and one attempting `--static` with `fetch` — to find out concretely what it actually takes, rather than leave it as a documented guess.

## Investigation

- **Baseline confirmed first**: a normal (dynamic) `fetch` build on Alpine/musl (`docker/Dockerfile.fetch-test`'s `dynamic-fetch-demo` target) compiles and runs correctly against a real HTTPS endpoint (`httpbin.org/ip`), establishing that any problems found afterward are specific to `--static`, not to Alpine/musl or this compiler's `fetch` implementation in general.
- **`KlainMainLang --static` alone (its single clang compile+link invocation) fails once libcurl's full dependency chain is involved.** `ldd /usr/bin/curl` on Alpine lists ten shared library dependencies beyond libc: `libz`, `libcares`, `libnghttp2`, `libidn2`, `libpsl`, `libssl`+`libcrypto`, `libzstd`, `libbrotlidec`+`libbrotlicommon`, `libunistring`. Static archives, unlike shared libraries, don't record or auto-pull their own transitive dependencies — the final link command has to list every one of them explicitly (`ADR-00020`'s existing `-lcurl`-only `LinkLibs()` logic, built for dynamic linking, doesn't know to do this). Confirmed by the exact failure mode: `undefined reference to nghttp2_session_callbacks_del` / `DES_ecb_encrypt` (OpenSSL) / etc. — real, resolvable symbols, just never asked for.
- **Even with the full dependency list passed explicitly, clang's linker still failed** on two specific archives: `libpsl-static` and `brotli-static` (`plugin needed to handle lto object`). Alpine builds these with GCC's LTO; clang/lld can't consume GCC's LTO bitcode format — a toolchain interop gap, not a missing package. Verified this diagnosis directly rather than guessing further: compiling the same LLVM IR down to a plain object file (`clang -O2 -c fetch-demo.ll -o fetch-demo.o`, no LTO/linking involved at that step) and then linking that object file with **`gcc`** instead of clang — passing `-static` and the full dependency list — succeeded immediately. GCC's own linker driver knows how to invoke the LTO plugin its own toolchain produced those archives with; clang's doesn't.
- **The resulting binary is a "static-PIE"**, not a fully static non-PIE binary the way the simpler `docker/Dockerfile` demo (`ADR-00032`) produced — confirmed via `readelf -d`: a small `Dynamic section` is present, but it contains only self-relocation metadata (`RELA`/`RELACOUNT`, `FLAGS_1: NOW PIE`), zero `NEEDED` entries, and `readelf -l`'s program headers have no `PT_INTERP` at all. This is GCC's modern default for `-static` on this Alpine/musl setup, not a partial failure — no external library or dynamic loader is needed at runtime either way, which is what actually matters for running in `scratch`.
- **First real run inside `scratch` still failed** — not from an exec/loader problem (the process started and ran its own code, proving the static link itself was sound), but with `Uncaught: Problem with the SSL CA cert (path? access rights?)`. Traced directly: Alpine's OpenSSL/curl build has `/etc/ssl/cert.pem` compiled in as its default CA bundle path — a symlink, on a normal Alpine install, to `certs/ca-certificates.crt` — not the flat `/etc/ssl/certs/ca-certificates.crt` path this project's earlier, non-`fetch` `docker/Dockerfile` demo never needed to think about at all. `scratch` has no symlink structure (and no `/etc` at all) to resolve that indirection, so the bundle has to be `COPY`'d directly to the literal `/etc/ssl/cert.pem` path the binary actually queries. Fixed, and confirmed working: real HTTPS request to `httpbin.org/ip` inside the `scratch` container, actual `200`/`ok: true`/real response data back.

## Decision

Documented the working recipe (`docker/Dockerfile.fetch-test`, `static-fetch-demo` target) rather than changing `KlainMainLang` itself to automate it. This is a deliberate scope boundary, not an oversight: the exact dependency list, the GCC-vs-clang LTO wrinkle, and the CA bundle path are all specific to *this* distro's (Alpine/musl) package build choices — a glibc-based distro could easily need a different list, a different (or no) LTO workaround, and a different CA path convention (`/etc/ssl/certs/ca-certificates.crt` proper on Debian/Ubuntu, for instance). Baking one distro's specifics into the compiler itself would silently produce a broken `--static` invocation on any other distro — worse than the current, honest "you'll need to figure out your own distro's static-curl recipe" state, which at least fails loudly (a linker error) rather than quietly succeeding with the wrong assumptions baked in.

This whole investigation raised a bigger, separate question worth tracking on its own: whether `libcurl` was the right foundation for `fetch` at all, given this compiler's own toolchain is Go and `net/http` sidesteps this entire class of problem by default. Scoped, not started, in `STATUS.md`'s [Alternative fetch Backend](../../STATUS.md#alternative-fetch-backend-a-go-helper-instead-of-libcurl--scoping-not-started-low-priority) section — deliberately deprioritized for now, not urgent this early in the project.

## Implementation notes

- `docker/Dockerfile.fetch-test` (new): `compiler-builder` (shared with `docker/Dockerfile`), `dynamic-fetch-builder`/`dynamic-fetch-demo` (the baseline), `static-fetch-builder`/`static-fetch-demo` (the two-step `clang -c` + `gcc -static -l<full dependency list>` recipe, plus the `/etc/ssl/cert.pem` copy).
- `docker/fetch-demo.ts` (new): a minimal program exercising `fetch`+`.json()`, deliberately not reusing `docker/hello.ts` from `ADR-00032` since this one specifically needs network access and a typed response.

## Side effects discovered

None new — the CA-cert-path and LTO-plugin issues described above are exactly what this investigation set out to find, not incidental surprises.

## Verification

- `docker build -f docker/Dockerfile.fetch-test --target dynamic-fetch-demo -t kml-fetch-dynamic .` then `docker run --rm kml-fetch-dynamic`: `status: 200`, `ok: true`, `origin length > 0: true`.
- `docker build -f docker/Dockerfile.fetch-test --target static-fetch-demo -t kml-fetch-static .` then `docker run --rm kml-fetch-static` (a genuine `FROM scratch` image, nothing else in it): same successful output, real HTTPS round trip.
- `readelf -d`/`readelf -l` directly on the extracted static binary: a `Dynamic section` present but containing only self-relocation (`static-PIE`) metadata, zero `NEEDED` entries, no `PT_INTERP` — confirmed self-contained, not just "the build didn't error."
- Image sizes for context: `klainmainlang-demo` (no `fetch`, `ADR-00032`) 292kB; `kml-fetch-dynamic` 21.8MB (Alpine base + shared `libcurl` etc.); `kml-fetch-static` 35MB (curl + OpenSSL + everything statically embedded, but still `FROM scratch` — no base OS layers at all).
