# ADR-00010: Implement labeled break/continue

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the last remaining Phase 2 roadmap item, following on from the `Math` trig/hyperbolic additions (`ADR-00009`). An earlier audit this session confirmed the parser had zero label-statement grammar at all (`outer: for (...) { break outer; }` failed to parse, choking on the stray `:` token), while plain unlabeled nested `break`/`continue` already worked correctly — so this was a real gap, not a stale doc entry.

## Investigation

- **Existing loop/break/continue machinery.** Every loop form (`emitFor`, `emitWhile`, `emitDoWhile`, `emitForOf`, `emitForIn` — `codegen/llvm/emit_stmts.go`) pushes its end label onto `Emitter.breakStack` and its continue-target label onto `Emitter.continueStack` before emitting its body, and pops both via `defer` on return; `emitBreak`/`emitContinue` just branch to the top of the respective stack. This is a clean, already-correct foundation to extend — I didn't need to touch the core loop-emission logic at all, only add a second, named parallel mechanism alongside it.
- **`BreakStatement`/`ContinueStatement` had no field for a label**, and `ast.TypeAnnotation`-adjacent statement types had no `LabeledStatement` node at all — this needed both a small AST addition and parser support, not just codegen.
- **A bug in my own first draft, caught by the existing test suite.** My first version of `parseBreakStatement`/`parseContinueStatement` unconditionally consumed a following `IDENT` as a label. Running the full suite immediately failed `TestE2EForInBreak`, which has exactly the pattern `if (key === 'y') break` (no semicolon) followed on the next line by `console.log(key)` — my code was consuming `console` as `break`'s label, then choking on the stray `.log(key)` that followed. This is JS's "no LineTerminator here" restriction between `break`/`continue` and its label, which I'd missed initially — fixed by checking the following identifier token's `Line` against the `break`/`continue` keyword's own line, only treating it as a label when they match. Added `TestE2EBreakStillWorksWithoutSemicolon` as a regression guard for exactly this shape.

## Decision

1. **AST** (`ast/nodes.go`): added `Label string` to `BreakStatement`/`ContinueStatement` (empty means unlabeled); added a new `LabeledStatement{Label string, Body Statement}` node, generic over any statement (matching real JS grammar — `outer: { ... }` or `outer: if (...) {}` parse fine), even though only the five loop forms register a label for practical use (see below).
2. **Parser** (`parser/parser.go`): `parseStatement` checks for `IDENT` followed by `COLON` (placed after the existing contextual-keyword checks for `interface`/`type`/`enum`, so those aren't mistaken for labels) and routes to a new `parseLabeledStatement`. `parseBreakStatement`/`parseContinueStatement` optionally consume a same-line trailing `IDENT` as the label.
3. **Codegen** (`codegen/llvm/emit_stmts.go`, `emitter.go`): added a parallel, named mechanism alongside the existing unnamed stacks — `Emitter.pendingLabel` (set by `emitStmt`'s new `LabeledStatement` case just before emitting its body, cleared right after whether or not it was consumed) and `Emitter.namedLabelStack` (a stack of `{name, breakL, continueL}`). A new `pushPendingLabel(breakL, continueL) func()` is a no-op returning a no-op cleanup when there's no pending label, so every loop form can call it unconditionally (`defer e.pushPendingLabel(endL, incL)()`, right alongside the existing `breakStack`/`continueStack` push+defer-pop lines) without special-casing "is this loop labeled." `emitBreak`/`emitContinue` check `s.Label`: if set, search `namedLabelStack` innermost-first and branch to the matching entry's target (erroring cleanly on an unrecognized label); otherwise, fall back to the pre-existing unnamed-stack behavior unchanged.
4. **Scope: only the five loop forms register labels.** A label placed on anything else (a block, an `if`, a `switch`) still parses, but is never pushed onto `namedLabelStack` (only the loop-emitting functions call `pushPendingLabel`), so `break`/`continue` referencing such a label fails with a clean "undefined label" error rather than emitting broken IR. Labeling `switch` for `break`-only (which real JS allows) was deliberately left out — it's additional scope beyond the roadmap's stated scope ("labeled break/continue" in the context of loops), and `emitContinue` already guards against a label with an empty `continueL` regardless, so adding it later is a small, isolated extension rather than something this decision forecloses.

## Implementation notes

Files touched: `ast/nodes.go` (`Label` fields, `LabeledStatement`), `parser/parser.go` (`parseLabeledStatement`, label parsing in `parseBreakStatement`/`parseContinueStatement`), `codegen/llvm/emitter.go` (`pendingLabel`, `namedLabelStack` fields), `codegen/llvm/emit_stmts.go` (`namedLabel` type, `pushPendingLabel`, `lookupNamedLabel`, the `LabeledStatement` case in `emitStmt`, the `pushPendingLabel` call in each of the five loop forms, and the label-aware `emitBreak`/`emitContinue`).

## Side effects discovered

The line-terminator bug described in Investigation was caught and fixed before landing, via the existing test suite — not a separate, deferred issue.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass (including the pre-existing `TestE2EForInBreak`, which exposed and confirmed the fix for the line-terminator bug).
- `make examples` — 42/42 pass, including the extended `examples/control_flow/control_flow.ts`.
- Manual repros, all correct: `break outer` from a doubly-nested `for` stops both loops entirely after exactly one inner-loop iteration; `continue outer` correctly resumes the *outer* loop's increment step (not just breaking the inner loop), verified across nested `for`, `for...of`, and `while`; a bare unlabeled `break`/`continue` continues to behave exactly as before; an undefined label produces a clean compile-time error, not broken IR.
- New tests in `compiler_test.go`: `TestE2ELabeledBreak`, `TestE2ELabeledContinue`, `TestE2ELabeledContinueWhile`, `TestE2EBreakStillWorksWithoutSemicolon` (the regression guard).
- `examples/control_flow/control_flow.ts` extended with a labeled break/continue section.
- `STATUS.md`: the `break`/`continue` row updated to explicitly cover labeled loops; removed from the "not implemented" table and the Phase 2 roadmap list; also corrected a long-standing `9/10` undercount in the Control Flow Statements coverage row to `10/10` now that every listed control-flow construct is genuinely done.
