# ADR-00026: Implement process.cwd/chdir/pid/platform/kill

- **Date**: 2026-07-12

## Context

After `execFileSync` (`ADR-00025`), looked at what other Node.js APIs would be quick wins. Came up with a prioritized list split by size — a trivial `process.*` tier (`cwd`, `chdir`, `pid`, `platform`, `kill`, each a single syscall or, for `platform`, no runtime code at all) and a slightly bigger `fs.*` tier (`mkdirSync`, `renameSync`, `copyFileSync`, `readdirSync`) — and built the entire `process.*` tier in this pass.

## Investigation

- **Confirmed `getcwd(NULL, 0)`'s auto-allocating behavior before relying on it**, rather than assuming from memory: compiled and ran a standalone C program calling `getcwd(NULL, 0)` on this dev machine. Both glibc (Linux) and Darwin/BSD libc document this as the recommended, portable way to call `getcwd()` without first guessing a buffer size — avoids the usual "grow a fixed buffer in a loop until it fits" pattern entirely.
- **Reused the existing "path-taking operation, throw on OS failure" shape** for `chdir`: `ensureFsThrow` (`runtime.go`, originally built for `fs.*` in `ADR-00023`) already builds a generic `"<opDesc> '<path>': <strerror>"` message and throws it — nothing about it is actually `fs`-specific, so `process.chdir`'s failure path calls it directly rather than duplicating the message-building logic.
- **`kill`'s wait-status-style bit tricks weren't needed here** — unlike `execFileSync`'s child-exit decoding (`ADR-00025`), `kill()` just returns `0`/`-1` with `errno` set on failure, the same shape as `chdir`/`unlink`. The one difference: the failure message needed to embed a numeric `pid`/`signal`, not a path string, so this got its own small dedicated throw path (`__kml_process_kill`) rather than reusing `ensureFsThrow` (which is hardcoded to a `path` argument specifically).
- **A real bug, caught by the very first combined smoke test, before it ever reached the test suite.** Testing `process.chdir` and `process.kill` together in the same program failed at the `clang` step with `invalid redefinition of function '__error'` — `ensureFsThrow` (used by `ensureProcessChdir`) and the new `ensureProcessKill` each independently emitted their own `declare ptr @<errnoAccessor>()` line with no shared singleton guard, so using both in one program emitted the same declaration twice. This is the exact same bug class `ADR-00023` already found and fixed once for `fopen`/`fclose`/`fwrite` — evidently the lesson ("any C declaration two independent driver functions might both need has to be its own singleton-guarded `ensure*()`, not inlined into each") hadn't been generalized into a rule followed automatically the next time a new failure-throwing helper was added. Also found `@strerror` had exactly the same latent problem (declared inline by both `ensureFsThrow` and `ensureProcessKill`), just not yet triggered by an existing test combination. Fixed both properly: extracted `ensureErrnoAccessor()` and `ensureStrerror()` as their own singletons, and pointed both `ensureFsThrow` and `ensureProcessKill` at them instead of inlining the `declare`s.
- **`process.platform` needed no runtime code at all** — this compiler doesn't cross-compile (it always runs `clang` on the same machine that ran the KML compiler itself, the same reasoning `errnoAccessor`/`monotonicClockID`/`stdinGlobalName` already rely on), so the platform string is knowable at KML-compile time, straight from the Go compiler's own `runtime.GOOS`. Mapped to Node's exact platform strings (`"win32"` for Windows; every other `GOOS` value, `"darwin"`/`"linux"`/etc., already matches Node's own convention verbatim).
- **`process.pid` is a property read, not a call** — `process.argv` already established this shape (`emitMember`, not `emitCall`) for a `process.*` value that isn't invoked with `()`; `pid` and `platform` both follow the same pattern, `pid` calling `getpid()` at codegen time and `platform` resolving to a compile-time constant.

## Decision

Five new `process.*` members: `cwd(): string`, `chdir(path: string): void` (throws on failure), `pid: number` (property, not a call), `platform: string` (compile-time constant), `kill(pid: number, signal?: number): void` (signal defaults to `15`/`SIGTERM`, matching real Node; throws on failure). All follow the existing "throw a catchable `Error` built from the OS's own reason" convention already established for `fs.*`/`fetch`/`execFileSync`.

## Implementation notes

- `codegen/llvm/runtime.go`: `nodePlatformName()` (the `runtime.GOOS` → Node-platform-string mapping); `ensureProcessCwd`/`ensureProcessChdir`/`ensureGetpid`/`ensureProcessKill`; and the bug-fix extraction, `ensureErrnoAccessor()`/`ensureStrerror()` as their own singletons, with `ensureFsThrow` and `ensureProcessKill` both updated to call them instead of inlining the `declare`s.
- `codegen/llvm/emitter.go`: `usedProcessCwd`/`usedProcessChdir`/`usedGetpid`/`usedProcessKill`/`usedErrnoAccessor`/`usedStrerror` flags.
- `codegen/llvm/emit_process.go`: `emitProcessCwd`, `emitProcessChdir`, `emitProcessPid`, `emitProcessKill`.
- `codegen/llvm/emit_call.go`: `cwd`/`chdir`/`kill` added to the existing `process` dispatch switch.
- `codegen/llvm/emit_exprs.go`: `emitMember`'s `process` handling widened from a single `argv` check into a `switch` also covering `pid` (calls `emitProcessPid`) and `platform` (an interned compile-time string); `inferExprType`'s two `process` cases (the `MemberExpression` one and the `CallExpression` one) both extended to cover the new members' return types.

## Side effects discovered

The `errnoAccessor`/`strerror` duplicate-declaration bug described in Investigation — found and fixed in this same pass, before it reached the test suite or a real user. Re-verified the fix doesn't change `fs.*`'s existing behavior by re-running the full test suite after the `ensureFsThrow` refactor.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, including the 9 new tests below, both before and after the `ensureFsThrow` refactor), `make examples` all green (one `examples/fetch/fetch.ts` failure during a run was the already-documented httpbin.org flakiness, unrelated, and passed cleanly on retry both times).
- Manual repros, each actually compiled and run: `cwd()`/`chdir()`/`pid`/`platform`/`kill()` all together in one program (the combination that surfaced the duplicate-declaration bug); `chdir` to a nonexistent directory throwing with the expected message prefix; `kill(pid, 0)` on the program's own pid succeeding silently (the POSIX "existence check" convention); `kill` on a pid unlikely to exist (`999999999`) throwing with the expected message prefix.
- New tests in `compiler_test.go`: `TestE2EProcessCwdAndChdir` (using `t.TempDir()` + `filepath.EvalSymlinks`, since macOS's `/tmp` is itself a symlink and `getcwd()` reports the resolved path), `TestE2EProcessChdirNonexistentThrows`, `TestE2EProcessChdirWrongArgCountRejected`, `TestE2EProcessPidIsPositive`, `TestE2EProcessPlatform` (compares against Go's own `runtime.GOOS`, mapped the same way, so the test is portable across the machines it might run on), `TestE2EProcessKillSignalZeroOnSelfSucceeds`, `TestE2EProcessKillDefaultsToSigterm` (compile-only — deliberately never runs `kill(1, 15)` for real), `TestE2EProcessKillNonexistentPidThrows`, `TestE2EProcessKillWrongArgCountRejected`. Error-message assertions check a deterministic prefix (`e.message.startsWith(...)`) rather than the OS-provided `strerror` suffix, following the existing `fs.*` test convention of not asserting exact OS-provided wording.
- Extended `examples/process/process.ts` with a full round trip (`cwd` → `chdir` elsewhere → `chdir` back → confirm restored) plus the nonexistent-directory throw, `pid`, `platform`, and both `kill` cases.
- `STATUS.md`: added five rows under Node.js APIs → Process / CLI I/O; updated the Node.js APIs coverage table (Process / CLI I/O 6/6 → 11/11; Node.js total 11/15 → 16/20, 80%).
