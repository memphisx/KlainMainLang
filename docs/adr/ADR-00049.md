# ADR-00049: Concurrent HTTP connection handling via fibers (TDD-00006 Part 2, first real slice)

- **Relations**: Extends ADR-00048. Implements TDD-00006 (Part 2 — first real implementation slice; `async`/`await`/`fetch()` itself is not yet wired through this mechanism)
- **Date**: 2026-07-13

## Context

`docs/tdd/TDD-00006.md`'s Part 2 (real async suspension) had, at the start of this piece of work, a *direction* — a prototyping spike (documented in the TDD itself) had already confirmed hand-rolled fibers via `ucontext.h` compose safely with this compiler's `setjmp`/`longjmp` exception model, ruling out LLVM coroutine intrinsics (which segfault under the same test). What hadn't happened yet was a real implementation of anything using that mechanism.

Two directions were possible for a first real slice: rework `fetch()`/`async`/`await` to actually suspend (needs libcurl's non-blocking multi-interface, an unfamiliar API surface this project has never touched), or apply the now-proven fiber mechanism to `http.listen` (`docs/adr/ADR-00048.md`), whose own V1 scope explicitly documented "a slow handler blocks every other client" as a known, deliberate limitation. Picked the second: smaller, fully scoped by two already-existing TDDs, and directly fixes a documented limitation rather than opening a second unscoped sub-project (`curl_multi`) in the same pass.

## Investigation

Given the size and the amount of new interacting LLVM IR (a generalized `fd_set` covering the listener *and* an arbitrary number of live connections, plus per-connection fiber lifecycle management), prototyped the entire scheduler shape in a scratch `.ll` file before touching the real compiler — the same "prove it in isolation first" approach the Part 2 spike itself used, just for the concrete scheduler this time instead of the underlying primitive.

- **The full multi-connection scheduler prototype worked on the first fully-corrected pass**: a growable `{ i64 fd, ptr ctx, ptr stack }` connection array (24 bytes/entry, same realloc-doubling shape and `fd = -1` tombstone-not-compact convention the timer queue already established), a generalized `fd_set`-building loop that ORs in the listener's bit plus every still-active connection's bit while tracking the overall highest fd for `select()`'s `nfds`, and — after `select()` returns — an accept-and-spawn-a-fiber step for the listener plus a resume-if-ready step for every connection already in the array.
- **Confirmed empirically, not just by unit test, that this actually delivers concurrent connection handling**: a raw TCP connection that connects but deliberately withholds its request line for 2 seconds does not block a second, concurrently-issued `curl` request from completing — measured directly (28ms and 36ms in two separate runs, once against the scratch prototype and once against the real compiled output), a result that would have taken over 2 seconds (or hung past a test timeout entirely) against the previous single-connection-at-a-time implementation.
- **Non-blocking sockets need two more platform-specific constants**, verified via the same throwaway `sizeof`/`offsetof`/`printf`-a-C-probe technique `ADR-00048` already used for `SOL_SOCKET`/`SO_REUSEADDR`/`sockaddr_in`'s layout: `O_NONBLOCK` (Darwin `0x4`, Linux `0x800` on both x86-64 and arm64 — not independently compiled on a Linux machine in this sandbox, same caveat `fs.readdirSync`'s ADR already noted for its own Linux-side offset) and `EAGAIN`/`EWOULDBLOCK` (Darwin `35`; Linux `11`, where the two names are the same value). `F_GETFL`/`F_SETFL` (`3`/`4`) turned out identical on both platforms, needing no branch.
- **Confirmed `ucontext_t`'s size/field-offsets are unaffected by adding non-blocking sockets to the picture** — the same `sizeof(ucontext_t) == 880`, `uc_stack` at offset 8 (`ss_sp`/`ss_size` at +8/+16), `uc_link` at offset 32 already established during the Part 2 prototype still apply unchanged; no new struct-layout risk introduced by this feature specifically.

## Decision

Replaced `ADR-00048`'s single global dispatcher (called directly, once per event-loop wakeup, doing a blocking accept+read+respond+close all in one go) with a per-connection fiber model:

- **`__kml_http_append_conn(i32 fd)`** (new, `runtime.go`): appends a new connection-array entry, builds its `ucontext_t` (a fresh 880-byte buffer + a 64KB stack, `uc_link` pointed at a new `@__kml_main_ctx` "scheduler" context so the fiber function returning normally resumes the scheduler automatically), `makecontext`s it to `@__kml_listen_dispatch`'s stored pointer (the per-call-site dispatcher, unchanged mechanism from `ADR-00048` for *finding* the right function — just now used as a fiber entry point instead of a plain callback), and immediately `swapcontext`s into it once to launch it.
- **`__kml_event_loop_run`** (rewritten, `runtime.go`): the `fd_set`-building step now scans the connection array (in addition to the listener), tracking the overall max fd for `select()`'s `nfds`. After `select()` returns: accept + `fcntl(O_NONBLOCK)` + `__kml_http_append_conn` if the listener is ready; then, separately, resume (`swapcontext`) every connection whose fd came back ready in the same `fd_set` select() already populated. Both steps reuse the exact bit-level `FD_SET`/`FD_ISSET` arithmetic (`byte = fd/8, bit = 1 << (fd%8)`) `ADR-00048` already established — no new struct/word-size assumptions, just applied to more than one fd now.
- **`buildHTTPDispatcher`** (rewritten, `emit_http.go`): the per-call-site dispatcher is now a fiber *body*, not a one-shot callback. Finds its own connection-array entry via a new `@__kml_current_conn_idx` global (set by the scheduler immediately before every `swapcontext` into a given fiber — safe since fibers are cooperative and never preempted). Loops on a non-blocking `read()`: real data → parse the request line and proceed exactly as before (build `Request`, call the handler, extract `status`/`body` via the same Go-known-field-offset GEPs `ADR-00048` already used, `__kml_http_send_response`); `read() == 0` or a non-`EAGAIN` error → close and mark done; `EAGAIN` → `swapcontext` back to `@__kml_main_ctx` (yield) and retry the read once resumed. Marks itself done (`fd = -1` in its own array entry) right before returning, so the scheduler can tell "genuinely yielded, keep watching" apart from "finished" — both cases return control to the exact same point in the scheduler's own `swapcontext` call, so this explicit self-tombstoning is the only way to tell them apart.
- **`__kml_http_bind_and_listen`, `__kml_http_send_response`, `ensureHTTPThrow`, and the `Request`/response-field-reading machinery are all unchanged** — this feature only replaces *how* a connection gets serviced (concurrently vs. one at a time), not the request/response shapes or error handling around them.

## Implementation notes

- `codegen/llvm/runtime.go`: `httpNonblockFlag()`/`httpEagainErrno()` (new Go-side platform-constant helpers, same shape as `httpSockConstants()`/`httpSockaddrFamilyBytes()`); new globals `@__kml_main_ctx`, `@__kml_conn_data`/`@__kml_conn_len`/`@__kml_conn_cap`, `@__kml_current_conn_idx`; new declares `fcntl`/`getcontext`/`makecontext`/`swapcontext`; removed `__kml_http_accept_and_read` (its accept+read+parse responsibilities moved into the per-call-site fiber body, since accept now happens in the event loop and read needs to be retryable/yieldable, neither of which fit a single generic hand-written helper anymore); new `__kml_http_append_conn`; `__kml_event_loop_run` rewritten as described above.
- `codegen/llvm/emit_http.go`: `buildHTTPDispatcher` rewritten as a fiber body (read-retry loop with an explicit yield point) instead of a single straight-line accept+read+respond sequence; `emitHTTPListen` itself is unchanged (still binds, still stores the handler/dispatcher/listenfd globals, still calls `__kml_event_loop_run` and marks `unreachable`).
- One raw Go string-concatenation subtlety worth flagging for future editors of this function: `__kml_event_loop_run`'s hand-written IR is a plain backtick string, *not* run through `fmt.Sprintf` (the function is hundreds of lines with `%name` LLVM values everywhere — escaping every one as `%%` for `Sprintf` would be extremely error-prone). The one genuinely dynamic piece (`O_NONBLOCK`'s platform-specific value) is spliced in via plain Go string concatenation (`` `...` + fmt.Sprintf("%d", httpNonblockFlag()) + `...` ``) around just that one spot, leaving the rest of the giant IR blob as an untouched raw string.

## Side effects discovered

None beyond what `TDD-00006.md`'s own Prerequisites table already flagged as a known, accepted caveat (memory management) — every fiber's `ctx`/`stack` buffers, like every other allocation in this compiler, are never freed; a tombstoned (`fd = -1`) connection-array entry just sits there permanently, the same "leak everything, don't compact" convention the timer queue already established for its own cancelled entries.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including nine pre-existing HTTP tests unchanged plus one new test below) all green; `make examples` 58/58 (no example changes needed — the existing `examples/http/http_server.ts` only ever exercises a single connection within its 300ms self-exit window, unaffected by the scheduler change underneath it).
- The decisive manual verification, run twice (once against a standalone scratch prototype, once against the real compiled `http.listen` output): a raw TCP connection that connects but never sends its request line does not block a concurrently-issued `curl` request — the fast request completed in 28ms/36ms respectively, versus what would have been a 2+-second stall (or an outright test timeout) against the previous one-connection-at-a-time implementation.
- New test in `tests/http_test.go`: `TestE2EHTTPListenConcurrentConnections` — opens a raw `net.Dial` connection that deliberately withholds its request line, waits 100ms, then fires a normal `http.Get` in a goroutine and asserts it completes within a 2-second `select`/`time.After` race (rather than blocking indefinitely behind the still-open slow connection). All nine pre-existing `TestE2EHTTPListen*` tests (basic GET, method/path fields, multiple sequential requests, custom status, `setInterval` coexistence, bind-failure throwing, and the three argument/type-validation rejections) pass unchanged against the new scheduler, confirming the externally-visible behavior for every case they already covered is preserved.
- `STATUS.md`: `http.listen`'s row updated to describe concurrent connection handling and cite this ADR; the Roadmap's "Structural priorities" Event Loop/HTTP Server entries updated to reflect Part 2 now having a real, shipped slice (HTTP concurrency) while `async`/`await`/`fetch()` itself remains unstarted. `docs/tdd/TDD-00006.md`'s Status line, Part 2 Prerequisites table, Decision paragraph, and Open Questions updated to point at this ADR and record the fixed 64KB-per-fiber-stack choice and the deliberately-unaddressed `fetch()`/`async`/`await` gap.
