# TDD-00009: Classes / OOP (methods, constructors, inheritance)

- **Status**: Not Started
- **Date**: 2026-07-14

## Context

`class` has sat in `STATUS.md`'s "High priority / high complexity" bucket (alongside `RegExp`, `Symbol`, generators, decorators, `Proxy`/`Reflect`) with no special urgency flag — one item among several, not called out as foundational to anything else. An audit done directly against the current file (not from memory) found that undersells it: `instanceof` (❌, "needs type tag stored on heap objects") and `Error` subtypes (❌, "tagged error values") both need exactly the kind of runtime type-identity infrastructure a class implementation would introduce anyway; decorators are meaningless without classes to decorate; and a real, user-extensible iterable protocol (blocking `Array.from(iterable)`, tracked as needing "a general iterable protocol this compiler doesn't have") needs objects with methods, which nothing in this compiler currently has.

More concretely: several already-shipped features are special-cased "objects with fake methods" precisely because there is no general mechanism for "a type with both data and behavior." `Date` is the clearest example — it's a plain `i64` millisecond timestamp, not a reference type, so `setFullYear()` and friends only work when called on a *named variable* (not a field, not a call result — see `docs/adr/ADR-00016.md`), because there's no heap location to mutate otherwise. `Map`/`Set`/`Response` all work fine for built-in use, but only because the compiler special-cases their static type (`sym.Ty.IsMap`/`IsSet`/`IsResponse` checks scattered through `emit_call.go`/`emit_exprs.go`) and dispatches to hand-written runtime functions — closed to extension, since there's no way for user code to define its *own* type that call syntax would recognize the same way. Every one of these was solved independently, one special case at a time, rather than with one reusable mechanism.

Worth recording explicitly, since it shapes the staging below: I have a strong preference for functional programming and have been burned by inheritance-heavy OOP codebases before — this isn't being scoped because OOP is suddenly the goal, it's being scoped because the *absence* of any type-with-methods mechanism is now blocking things that have nothing to do with OOP taste (a real iterator protocol, `instanceof`, typed errors). The staging is deliberately ordered so the highest-leverage, lowest-risk piece (methods on non-inheriting types) lands first and stands entirely on its own — inheritance is the last, most expensive, most "classical OOP" stage, and I can reasonably stop before it for a good while and still have closed most of the gaps above.

To be clear, full OOP (including inheritance) is still a real goal for this compiler, not something being written off because of that preference — it's just genuinely low priority relative to everything else on the roadmap right now, for the reasons above. That could change: if I eventually need to compile real third-party TypeScript/JavaScript code that leans on classes, that's a concrete forcing function to pull Stage 3 forward, not just an abstract nice-to-have — worth remembering the next time this document's priority is reassessed.

## Design

Four stages, each independently valuable and each closing specific gaps already tracked in `STATUS.md`, ordered cheapest-and-highest-leverage first. Full inheritance (vtables, `extends`) is deliberately last: it's the only stage that actually needs new runtime dynamic-dispatch machinery. Stages 1–2 get most of the "no more special-cased hacks" value using patterns this compiler already has proven, working precedent for.

### Stage 0: Front-end groundwork (lexer/parser/AST)

Shared prerequisite for every stage below. Confirmed directly (not assumed) that none of it exists yet: `grep`ing the lexer and parser for `class`/`extends`/`this`/`super`/`private`/`public`/`protected`/`static`/`implements`/`abstract` turns up nothing — none of these are reserved keyword tokens today (`lexer/token.go` only has `new`, already reused by the existing `new Error(...)`/`new Map()`/`new Set()`/`new Array()`/`new Date()` forms). Needed:

- New keyword tokens: at minimum `class` and `this` for Stage 1; `extends`/`super` deferred to Stage 3; `private`/`protected`/`static`/`implements`/`abstract` deferred to Stage 4.
- A new `ClassDeclaration` AST node: name, a list of typed fields, one constructor, a list of methods (each shaped like a `FunctionDeclaration` body but with an implicit receiver).
- A **generic** `NewExpression` AST node for `new UserClass(args)`. This is a real gap, not a formality: every existing `new X(...)` form — `NewArrayExpression`, `NewMapExpression`, `NewSetExpression`, `NewErrorExpression`, `NewDateExpression` (`ast/*.go`) — is its own separately-parsed, hardcoded node keyed on the literal callee name. There is no shared "construct anything" node to extend; the parser's `new`-handling needs a fallthrough case for names that aren't one of the existing built-ins.
- `this` as an expression valid inside method/constructor bodies, resolving to the implicit receiver (see Stage 1).
- No new module-system work: a class is just one more top-level declaration kind, subject to the same "unique name across every merged file" constraint the Modules section already documents for functions/interfaces/enums.

### Stage 1: Methods + constructors, no inheritance

The cheapest stage by a wide margin, because it reuses two patterns that are already fully built and proven rather than inventing new runtime machinery:

1. **Instance representation is exactly today's object representation.** A class instance is a heap-`malloc`'d struct, fields accessed by GEP index — the same `ObjectType`/`StructIR`/`FieldIndex` machinery (`codegen/llvm/types.go`) already backing every interface-typed object. No new heap layout needed at this stage.
2. **Method calls reuse the closure-calling convention, minus the indirection.** `emitClosureCallByPtr` (`codegen/llvm/emit_func.go:794`) already compiles a closure call as `call RETTY (ptr, ARGTYS...) FPVAL(ptr ENVVAL, ARGS...)` — the closure's env pointer is passed as a hidden first argument to whatever function `FPVAL` points at. A method call is the identical shape, with `this` standing in for `ENVVAL` — except every instance of the same class shares the exact same method function (unlike a closure's per-instance captured env), so there's no need for a heap-allocated `{funcPtr, envPtr}` header or an indirect call at all: `instance.method(args)` compiles directly to `call RETTY @ClassName_methodName(ptr thisVal, ARGS...)`, a plain static call — cheaper than a closure call, and close enough to how `emitFunctionDecl` already compiles a named function (`codegen/llvm/emit_func.go:12`) that a method is essentially "a named function with an implicit first parameter."
3. **Constructors reuse the object-literal construction pattern.** `emitObjectLiteral` (`codegen/llvm/emit_objects.go`) already does "malloc a struct, then run per-field initialization code" for object literals; a constructor is the same shape driven by explicit `this.field = value` statements in a user-written body instead of a literal's property list. `new ClassName(args)` compiles to: malloc the instance, call `@ClassName_constructor(ptr instance, ARGS...)`, return the instance pointer.

What this closes, concretely: a real reference-type object whose methods work on *any* expression, not just a named variable — the exact restriction `Date`'s setters carry today. It's also the direct prerequisite for Stage 1a below, and it closes the "special-cased, closed to extension" gap `Map`/`Set`/`Response` all currently have, since user code can now define its own type with real methods for the first time.

**Stage 1a (same-stage add-on): a real `for...of` protocol.** Once instances can have methods, `emitForOf` (`codegen/llvm/emit_stmts.go:305`) — which today special-cases exactly three receiver shapes (array, `Map`, `Set`) — can gain one general case instead of a fourth special one: at compile time (this is a fully statically-typed compiler, so this is a structural check over the receiver's declared class, not a runtime `Symbol.iterator` lookup — cheaper than what real JS actually does), check whether the receiver's class declares a method matching an iterator shape and, if so, generate a loop calling it repeatedly. A simple `next(): T | null` sentinel form (mirroring the possibly-null convention `process.env`/`.find()`/`process.readLineSync()` already use) is cheaper than a real `{done, value}` struct per call and is the leaning here, not a final decision. This is what actually closes `Array.from(iterable)` (❌ today, specifically blocked on "no general iterable protocol") and the earlier-discussed gap between this compiler's `for...of` and a real iterator protocol.

### Stage 2: Runtime type tags + `instanceof`

Reserves field index 0 of every class instance's struct as a hidden tag (`i64`, a small integer ID assigned per class at compile time) — a new `ClassType()` constructor mirroring `ObjectType()`/`ResponseType()`'s existing shape (`types.go`), just always prepending the tag field before the user-declared ones. `instanceof` becomes `load i64 tagfield; icmp eq i64 tag, <ClassName's compile-time ID>`.

`Error` subtypes have two viable paths, deliberately left open here rather than picked:
- **Narrow**: give the already-existing `Error` struct (`new Error(msg)`, no dependency on user-defined classes at all) its own tag field distinguishing `Error`/`TypeError`/`RangeError`/etc. Buildable independently of Stage 3, smaller in scope.
- **General**: once Stage 3 exists, model `TypeError extends Error` as a real built-in class hierarchy, which subsumes the narrow path. More consistent with the rest of the design, but gated on inheritance landing.

### Stage 3: Inheritance (`extends`), vtables, `super`

The expensive stage, and the only one that genuinely needs dynamic dispatch: a base-typed variable can hold different concrete derived instances, and a call to an *overridden* method on it has to resolve to the right implementation at runtime, not compile time.

- **Field layout**: base class's fields first (after the Stage 2 tag), derived class's own fields appended after — the standard base-first struct-prefix layout (the same technique many C-ABI object systems use), so a `Derived*` can always be treated as a `Base*` for field access or a call to a *non-overridden* base method, at zero runtime cost and no pointer adjustment.
- **Dispatch**: only genuinely overridable methods need indirection.
  - Full vtable: one hidden `ptr` per instance (plausibly reusing the Stage 2 tag slot as a vtable pointer instead of a separate field — vtable identity doubles as type identity for `instanceof`, one hidden field instead of two) pointing at a per-class static table of function pointers.
  - **Static dispatch for methods never actually overridden anywhere in the whole program** is a real, safe optimization here specifically because this compiler already does whole-program compilation (`resolver.ResolveProgram` merges every file before codegen — no separate compilation units, no open-world "some other translation unit might override this" uncertainty the way a linker-based language would have). `STATUS.md`'s existing `class` row already hedges with *"VTable **or** static dispatch"* — this whole-program visibility is exactly why that hedge is realistic rather than wishful.
- **Constructors / `super(...)`**: a derived class's constructor calls the base class's constructor as an ordinary subroutine call (same instance pointer) before running its own field-initializing code — no new mechanism beyond Stage 1's constructor-as-function shape.
- **Single inheritance only**, matching real TS/JS exactly (`extends` takes one base class) — no reason to support more, since TS/JS has no multiple-inheritance concept to be compatible with.

### Stage 4: further out, not designed in depth

Listed so it isn't forgotten, not scoped here: `implements` / structural interfaces-with-methods (compile-time-only structural check — this compiler already does structural matching for data-only interfaces; extending that to method signatures is a bounded follow-on, not a new mechanism); `static` members (belong to the class, not an instance — no `this`, effectively a namespaced global, the simplest possible case); `private`/`protected` as compile-time-only visibility checks (no runtime enforcement needed — arguably the *correct*, minimal match, since real TypeScript's own privacy is erased at compile time with no runtime enforcement either); `abstract` classes/methods (compile-time-only "must be overridden" check).

## Open questions

- Should built-in `Date` be reimplemented as a real class once Stage 1 lands, closing its named-variable-receiver-only gap for real — or left as its current plain-`i64` representation (simpler, faster, already shipped and extensively tested) since the existing workaround is narrow and documented, not silently wrong? Leaning toward leaving it as-is unless a concrete need for reference semantics shows up — migrating a shipped, tested feature purely for architectural consistency carries real regression risk for no currently-demonstrated user-facing benefit.
- Should `Map`/`Set` be rewritten as (dogfooded) classes once Stage 1 lands, or stay hand-written runtime helpers? Same reasoning as `Date` — likely not worth the migration risk, since Stage 1 already closes the actual gap (user code being able to define its *own* Map-like type), without needing to touch the built-ins at all.
- Should Stage 2's type tag and Stage 3's vtable pointer be the same hidden field, or two separate ones? Leaning toward merging (one hidden field, not two) but not decided until Stage 3 design work actually starts.
- Exact iterator shape for Stage 1a — the simpler `next(): T | null` sentinel form, or a real `{done, value}` struct closer to actual JS semantics at the cost of an extra allocation per iteration step? Leaning toward the sentinel form; not decided.
- Whether Stage 4's `private`/`protected` should be enforced at compile time at all is basically decided above (yes, compile-time-only, matching real TypeScript's own erasure) — the genuinely open part is whether it's worth building before `implements`/`static`, or after.
- When to actually pick up Stage 3: it's not a "never," just low priority given today's FP leaning and given Stages 1–2 already close most of the concrete gaps on their own — so the honest current answer is "not scheduled yet, not abandoned either." The concrete trigger to revisit this, rather than a vague future revisit, is needing to compile real third-party TS/JS code that depends on class inheritance — worth checking against that specific condition each time this document's priority is reassessed, rather than re-litigating FP-vs-OOP taste from scratch.
