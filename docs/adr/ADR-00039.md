# ADR-00039: Implement the multi-argument new Date(year, month, ...) constructor

- **Status**: Completed
- **Date**: 2026-07-13

## Context

Tracked in `STATUS.md`'s Known Limitations & Bugs, the next pick after `docs/adr/ADR-00038.md`: `new Date(2023, 10, 14)` failed with a parse error (`expected ), got ,`), since `parseNewDateBody` only ever accepted zero or one constructor argument. Real JS's calendar-field form (`year, month, day?, hours?, minutes?, seconds?, ms?`) wasn't accepted in any form.

## Investigation

- `parseNewDateBody` (`parser/parser.go:1578`) parsed at most one argument into `ast.NewDateExpression.Millis`, then immediately expected `)` — a second argument's leading comma triggered the parse error.
- `emitNewDate` (`codegen/llvm/emit_date.go:75`) only ever handled the no-arg (current time) and single-arg (timestamp or, since `ADR-00038`, ISO string) forms.
- `__kml_date_compose` (`runtime.go:1131`, already used by `Date.parse` and the Date setters) already does exactly the calendar-fields-to-milliseconds math this constructor form needs — its `month` parameter is documented as **1-indexed** (matching ISO date strings), while real JS's `new Date(year, month, ...)` is **0-indexed** (matching `getMonth()`'s own convention) — the same adjustment `emitDateToISOString` already makes in the reverse direction, so a `+1` on the month argument was the only translation needed before calling it.
- Verified real JS's actual default-filling and indexing behavior directly with `node`, rather than assuming: omitted trailing fields default (day → 1, everything after that → 0), and `getMonth()` reads back the same 0-indexed value passed in.
- **Found, while verifying with `node`, that real JS's multi-argument constructor interprets its fields as *local* time, not UTC** — confirmed directly by comparing `new Date(2023, 6, 14, 10, 0, 0).toISOString()` under `TZ=Europe/London` (a summer date, so BST/UTC+1 applies): real JS produces `2023-07-14T09:00:00.000Z` (fields shifted by the local offset), while this compiler, which always treats every Date field as UTC (`ADR-00014`'s pre-existing, deliberate, project-wide decision), produces `2023-07-14T10:00:00.000Z` unchanged. This is not a bug introduced here — it's the exact same UTC-only convention every other Date operation in this compiler already follows, applied consistently to the new constructor form too, rather than carved out as a special case. Documented explicitly in `STATUS.md` so it isn't a surprise later.
- Deliberately did not replicate real JS's "two-digit year (0–99) means 1900+year" historical quirk for this constructor — not called out anywhere this bug was tracked, and not worth adding speculatively as a surprising, rarely-used special case.

## Decision

Extended `ast.NewDateExpression` with a new `Args []Expression` field (`Millis` stays for the existing no-arg/single-arg forms, to avoid touching any existing call site), populated by `parseNewDateBody` when it collects 2–7 comma-separated arguments (0 args → current time, 1 arg → the existing timestamp-or-string form, 8+ args → a clean compile-time error rather than silently ignoring the extras). `emitNewDate` dispatches to a new `emitNewDateMulti`, which evaluates each provided argument (coercing to `i64`), fills in real JS's defaults for any omitted trailing arguments, adds 1 to the month, and calls the existing `__kml_date_compose`.

## Implementation notes

- `ast/nodes.go`: `NewDateExpression` gains `Args []Expression` (non-nil only for the 2+ argument form); new constructor `NewNewDateExpressionMulti(args, pos)` alongside the existing `NewNewDateExpression(millis, pos)`.
- `parser/parser.go`, `parseNewDateBody`: rewritten to collect a comma-separated argument list, then dispatch to the appropriate constructor by count (0, 1, 2–7, or a clean `>7` error).
- `codegen/llvm/emit_date.go`: `emitNewDate` checks `n.Args != nil` first, delegating to the new `emitNewDateMulti`; the new function fills `[7]int64{0, 0, 1, 0, 0, 0, 0}` defaults (year, month, day, hour, min, sec, ms) for any argument beyond what was actually supplied, adds 1 to the (0-indexed) month, and reuses `ensureDateCompose`/`@__kml_date_compose` exactly as `Date.parse` and the setters already do — no new runtime helper needed.
- No changes to `inferExprType` or any other `*ast.NewDateExpression` call site — all of them only care that the node is Date-typed, never touching `.Millis`/`.Args` directly, so the new field is additive with zero blast radius elsewhere.
- `STATUS.md`: removed the bug row from Known Limitations & Bugs and the Roadmap's "Next up" table; updated the `Date` feature-matrix row to document the new form, its defaults, and the UTC-vs-local-time distinction found during verification.

## Side effects discovered

None new — the UTC-vs-local-time difference described in Investigation is a pre-existing, already-documented project-wide decision (`ADR-00014`), not a new issue; it's called out here only because this is the first place it might visibly surprise someone porting real JS code that relied on local-time semantics for this specific constructor form.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including three new tests below) all green; `make examples` 55/55.
- Cross-checked against real `node` output directly (not assumed): `new Date(2023, 10, 14)`, `new Date(2023, 10, 14, 22, 13, 20, 500)`, and `new Date(2023, 0)` (day-defaulting case) all match `node`'s `toISOString()`/`getFullYear()`/`getMonth()`/`getDate()` output exactly for the UTC-equivalent (winter, UTC+0 local) case; the summer/local-time case was used specifically to confirm and document the deliberate UTC-vs-local divergence, not to find a bug to fix.
- Manual repro of the new `>7`-argument rejection: `new Date(2023, 10, 14, 1, 2, 3, 4, 5)` fails at parse time with a clear message, not a crash.
- New tests in `compiler_test.go`: `TestE2EDateMultiArgConstructor`, `TestE2EDateMultiArgConstructorFullFields`, `TestE2EDateMultiArgConstructorDefaultsDay`.
- Extended `examples/date/date.ts` with all three cases; manually ran the compiled binary and confirmed correct output across the full example, including every pre-existing line (no regression).
