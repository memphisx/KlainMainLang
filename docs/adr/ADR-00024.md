# ADR-00024: Near-zero-effort roadmap batch (NaN/Infinity, performance.now, atob/btoa, encodeURI(Component)/decodeURI(Component), crypto.getRandomValues/randomUUID, process.readLineSync)

- **Date**: 2026-07-12

## Context

Picked this up as a themed batch straight from `STATUS.md`'s own "Near-zero effort (C stdlib, no new dependencies)" roadmap tier — eight small, self-contained items, each individually a quick win, grouped into one pass and one ADR (the same batching precedent `ADR-00002` already set for `process.argv`/`exit`/`env`).

## Investigation

- **`NaN`/`Infinity` as bare globals immediately surfaced a real bug in their own first manual test.** `const x = NaN` silently typed `x` as plain `i64` (the untyped-declaration switch's `*ast.Identifier` case only checked `e.lookup`, with no fallback for an unresolved name that happens to be one of these two) — storing a `double`'s NaN bit pattern into an `i64` slot corrupts it, so `isNaN(x)` came back `false`. The exact same shape of gap this project has hit repeatedly (`Date`/`Error`/`fetch` in earlier ADRs): a value's *runtime* behavior was correct in isolation, but an *untyped declaration* of it silently picked the wrong storage type. Fixed by adding the fallback in both `emitIdent` and the untyped-inference switch, checked only *after* a real local-variable lookup fails — so a user-declared variable of either name still shadows the built-in constant, matching how real JS treats these as ordinary (shadowable) globals, not reserved keywords.
- **`performance.now()` (and, while verifying it, `Date.now()`) initially looked broken during testing, but weren't.** A first test — read the clock, run a summation loop, read the clock again, subtract — reported an elapsed time of exactly zero even after 100 million loop iterations. Traced this to the actual generated `-O2` IR rather than assuming a bug: the optimizer had legitimately collapsed the entire loop into a single precomputed constant (LLVM's loop-idiom recognition solving `sum(0..n)` in closed form), leaving the two clock reads executing back-to-back with genuinely nothing in between — the same "need `volatile` or a real side effect to stop the optimizer from erasing your benchmark loop" pitfall C code hits without `volatile`, just with no direct analog here. Confirmed by re-testing with a workload that can't be reduced to a closed form (building an array via repeated `.push()`), which correctly showed a positive elapsed time. No bug in either clock function — a testing-methodology lesson worth recording, not a compiler fix.
- **`CLOCK_MONOTONIC`'s numeric value needed the same cross-platform verification as `Date.now()`'s `CLOCK_REALTIME` already got in `ADR-00014`.** Checked the actual system header rather than assuming: Darwin's value is `6`; glibc's is the well-known, kernel-UAPI-stable value `1` — different per platform, unlike `CLOCK_REALTIME`'s `0`, which happens to match on both. Handled with the same `runtime.GOOS` conditional already established for `errnoAccessor`/`arc4random`-vs-portable-fallback.
- **`atob`/`btoa` and the percent-encoding functions were prototyped in C first**, following the same practice used for `fetch`'s libcurl integration (`ADR-00021`) — verified the base64 algorithm against known reference encodings (e.g. `"hello"` → `"aGVsbG8="`) before translating to hand-written LLVM IR, catching nothing that needed fixing but de-risking the translation considerably.
- **A real bug found and fixed during the `encodeURI`/`decodeURI` IR translation, before it ever reached a test.** The shared `ensurePercentDecode` generator uses a Go-level flag to decide whether to emit the extra "is this decoded byte one of the reserved URI characters" check `decodeURI` needs but `decodeURIComponent` doesn't — the *simple* (non-reserved-checking) path's control-flow label bookkeeping pointed at a label (`%pctwrite`) that only exists in the *other* (reserved-checking) path, causing an "undefined value" compile failure the moment `decodeURIComponent` was actually exercised. Fixed by correcting the label name to the block that's actually shared by both paths (`%pctvalid`).
- **`decodeURI` needed to specifically preserve reserved-character escapes, not just decode everything `decodeURIComponent` does — verified this exact case, not just the common path.** `decodeURI("http://example.com/path%3Fa=1&b=2%20space")` must leave `%3F` (`?`, reserved) as literal text while still decoding `%20` (space, not reserved) — checked directly against this precise input/output pair (initially misread the correct output while scanning quickly and suspected a bug that wasn't there; re-ran the isolated case to confirm it was actually right before spending further effort chasing a phantom).
- **`crypto.getRandomValues`/`randomUUID` deliberately do *not* reuse `Math.random()`'s portable fallback.** That fallback (plain C89 `rand()`) is explicitly *not* cryptographically secure — `crypto.*` needs a real CSPRNG, so it goes straight to `arc4random_buf` (BSD/macOS) or `getrandom()` (Linux), matching the roadmap note's own wording and never touching the weaker path even as a fallback.
- **`crypto.getRandomValues(arr)` needed a concrete decision about what "buffer" even means here.** This compiler has no `ArrayBuffer`/`TypedArrays` (0% implemented, tracked separately). Rather than block on that, reused the existing `number[]` array type as the stand-in "buffer" — filling each `i64` element with one random byte value (0–255) via a small runtime helper. A real, documented deviation from the actual API (which fills a `TypedArray` byte-for-byte), but it makes the feature usable today instead of waiting on a much larger prerequisite.
- **`process.readLineSync()`'s `stdin` reference hit a third instance of the same cross-platform-symbol-name class of issue** already seen for `errno` (`ADR-00023`) and `CLOCK_MONOTONIC` above. Checked directly with the preprocessor rather than assumed: on Darwin, the `stdin` macro expands to `__stdinp`, a differently-named global `FILE*` — not literally `stdin` at the link level at all. glibc exposes it as the plain symbol `stdin` itself. Handled with the same `runtime.GOOS`-conditional pattern.
- **A real, separate, pre-existing bug found while running the full example suite after this batch, unrelated to any of it.** `make examples` started reporting a segfault in the (already-shipped) `fetch` example — traced with `lldb` to a crash inside `JSON.parse`'s object-parsing path (`emitJSONParseObject`, from `ADR-00007`, several sessions ago), triggered because `httpbin.org` (the example's target) happened to be returning a degraded `503` HTML error page instead of real JSON at that exact moment. Reproduced in complete isolation (`JSON.parse` of non-JSON text into an interface with a `string` field whose key is entirely absent) to confirm the root cause was unrelated to `fetch`/network conditions themselves: a missing field's default value came from `zeroRef`, whose general `ptr`-type default is `null` — correct for an object/array/closure field (which genuinely has no better zero value), but wrong for a plain `string` field, since *every other string operation in this compiler* (concatenation, `.length`, `console.log`, template literals) assumes a `string` value is never `null`. Passing that `null` to `printf`'s `"%s"` the moment it was printed is undefined behavior, which `clang -O2` exploited aggressively enough to reduce parts of `main` to an unconditional trap. Found no existing test coverage for a *missing string field* specifically (only a missing *number* field was tested, from `ADR-00007`) — the gap had been sitting there since that ADR without being noticed. Fixed and covered with a new regression test in this same pass, since the standing rule is to fix bugs found during development, not defer them just because they're unrelated to the current task.

## Decision

All eight roadmap items implemented as scoped in `STATUS.md`'s own roadmap notes, plus the `JSON.parse` missing-string-field crash fixed as an unrelated bug found along the way.

## Implementation notes

- `codegen/llvm/emit_exprs.go`: `emitIdent` and the untyped-declaration switch's `*ast.Identifier` case both gained the `NaN`/`Infinity` fallback (checked after a real lookup miss, so shadowing works); `inferExprType`'s `*ast.Identifier` case gained the same fallback for type inference. Several new bare-global/namespaced-call `inferExprType` and untyped-declaration-switch cases added for the same reason established throughout this project's history — `performance.now`, `btoa`/`atob`/`encodeURI(Component)`/`decodeURI(Component)`, `crypto.getRandomValues`/`randomUUID`, `process.readLineSync` — each checked to confirm it was actually needed (all were; every one of these would have silently defaulted to `i64` in an untyped declaration otherwise, the exact bug class found for `NaN` above). The `JSON.parse` missing-string-field fix: `emitJSONParseObject`'s missing-field default now checks whether the field is a plain string type (`ptr` and not object/array/func) and uses `internString("")` instead of `zeroRef`'s general `null`.
- `codegen/llvm/runtime.go`: `ensureClockGettime` extracted as its own shared singleton (`Date.now`'s and `performance.now`'s `clock_gettime` declaration used to live only inside `ensureDateNow`, which would have hit the exact `fopen`/`fclose`/`fwrite` double-declaration bug `ADR-00023` already found and fixed once, if both functions were ever used in the same program — caught and fixed *before* it could happen this time, not after); `monotonicClockID`/`ensurePerformanceNow`; `ensureBase64Encode`/`ensureBase64Decode` (`__kml_btoa`/`__kml_atob`, a 64-entry alphabet table and a 256-entry reverse lookup table, both Go-computed constants); `ensureHexDigits`/`ensureHexDecodeTable` (shared by the percent-encode/decode family); `ensurePercentEncode`/`ensurePercentDecode` (shared Go-level generators parameterized by the safe-character set / whether to preserve reserved-character escapes, producing `encodeURIComponent`/`encodeURI`/`decodeURIComponent`/`decodeURI`'s four runtime functions from two Go functions); `ensureCryptoRandomBytes`/`ensureCryptoFillNumberArray`/`ensureCryptoRandomUUID`; `stdinGlobalName`/`ensureReadLineSync`.
- `codegen/llvm/emit_call.go`: new shared `emitStringToStringBuiltin` helper (used by all six `string → string` builtins — `btoa`/`atob`/`encodeURIComponent`/`decodeURIComponent`/`encodeURI`/`decodeURI` — avoiding six near-identical dispatch functions); `emitCryptoGetRandomValues` (requires a named `number[]` array variable, the same restriction `emitPush` already has for array mutation — there's no heap location to write into otherwise) and `emitCryptoRandomUUID`; dispatch wiring for all eight items plus `process.readLineSync`.
- `Makefile`: found and fixed a real, separate hang risk while building `process.readLineSync`'s example — the `examples` target's run step never redirected stdin at all, so any stdin-reading example (this one specifically) blocks the *entire* `make examples` run indefinitely in a shell without an already-closed/redirected stdin. Fixed by redirecting `< /dev/null` for every example's run step — matches the property every other example already has (fully automated, no interactive input assumed) and costs nothing for examples that don't touch stdin.

## Side effects discovered

The `JSON.parse` missing-string-field crash and the `Makefile` stdin-hang risk, both described above — both found and fixed in this same pass, not deferred, since both are real correctness bugs (a crash and a CI-hang risk respectively) surfaced directly by this batch's own testing.

## Verification

- `go build .`, `go vet ./...`, `go test ./...`, `make examples` (all pass, examples 49/49 — two new example directories, `examples/globals/` and `examples/crypto/`, plus an extension to the existing `examples/process/process.ts`).
- Manual repros for every item, each checked against a known-correct reference value or behavior: `isNaN(NaN)`/`-Infinity < 0`/`Infinity > 1000000`, including the untyped-declaration case; `performance.now()` via a non-optimizable workload (array-building, not a closed-form-reducible loop); `btoa`/`atob` round-tripping through several known base64 test vectors (`"hello"` ↔ `"aGVsbG8="`, etc.); `encodeURIComponent`/`decodeURIComponent` round-tripping through reserved/special characters; `encodeURI`/`decodeURI`'s reserved-character-preservation behavior specifically (the one real difference from the `*Component` forms); `crypto.getRandomValues` filling every element of a `number[]` with a value in `0..255`; `crypto.randomUUID()` producing a 36-character string with dashes at the correct positions and the version nibble fixed to `4` (checked across 20 generated UUIDs that the variant nibble is always one of `8`/`9`/`a`/`b`, per RFC 4122); `process.readLineSync()` reading real piped multi-line input correctly, handling a final line with no trailing newline correctly, and returning `null` at genuine EOF — confirmed with `lldb` that the `JSON.parse` fix actually resolves the real-world crash (not just the isolated repro) by re-running the exact `fetch`-plus-`.json()` sequence that originally crashed against `httpbin.org`'s degraded response.
