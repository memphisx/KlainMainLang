# ADR-00023: Implement fs.readFileSync/writeFileSync/appendFileSync/existsSync/unlinkSync

- **Status**: Completed — extended by ADR-00027
- **Date**: 2026-07-12

## Context

Picked this up as the next feature after `import`/`export` (`ADR-00022`) — File I/O is one of this project's three named priority areas (alongside REST API interaction, done via `fetch`, `ADR-00021`, and terminal/process interaction, already covered by `process.*`), and `STATUS.md`'s existing "File API" section was, on inspection, framed entirely around browser-style `File`/`FileReader`/`FileSystemFileHandle` — objects that model browser sandbox/permission concepts (a file picker, a `Blob`) with no equivalent in a native program that already has direct filesystem access. The real intended scope (per this project's own stated direction) was always Node-`fs`-shaped synchronous file I/O, the same reframing "Process / CLI I/O" had already needed for itself.

## Investigation

- **Confirmed the browser-vs-Node framing mismatch by reading `STATUS.md` before writing any code**, rather than assuming scope from the section title alone — the existing "File API" table only listed `File`/`FileReader`/`FileSystemFileHandle`, but a separate row already sitting under "Process / CLI I/O" (`"Simple synchronous file read/write (fs.readFileSync/writeFileSync-style)"`) and the Phase 5 roadmap's own "Low effort" tier both already described the *actually* intended feature in Node terms. Rewrote the File API section to match reality rather than implement the browser-shaped API nobody actually wanted.
- **Namespace choice**: recognized `fs.*` as a pseudo-namespace the same way `Math`/`JSON`/`process` already are — a plain `MemberExpression` on the bare identifier `fs`, dispatched in `emit_call.go` exactly like `process.exit`'s existing dispatch. Not a real importable module (no `import fs from 'fs'`); the newly-built `import`/`export` system (`ADR-00022`) only resolves user-authored relative files, and inventing a "built-in virtual module" concept for this would be a meaningfully bigger, separate design question not needed here.
- **Scope: which `fs` functions, and in what shape.** Chose the smallest set that covers the stated motivating use case (config, data, logs): `readFileSync`, `writeFileSync`, `appendFileSync`, `existsSync`, `unlinkSync`. All synchronous — there's no event loop in this compiler (a documented structural gap tracked separately), so there's no non-blocking variant to offer; the `Sync` suffix is honest about that, not just naming-convention mimicry. Directory operations (`mkdirSync`/`readdirSync`) were deliberately left out of this pass and logged as the natural next step, rather than trying to fit everything into one change.
- **Encoding: text-only, no `Buffer`/encoding parameter.** Real Node's `readFileSync` returns a `Buffer` unless an encoding is given; this compiler has no `ArrayBuffer`/`TypedArrays` (0% implemented) to represent raw bytes at all, so the only sensible representation is a plain string, UTF-8/byte-transparent-until-a-null-byte, matching how every other string in this compiler already works. This carries over the exact same limitation `fetch`'s response bodies already have (`ADR-00021`): a file containing embedded null bytes reads back shorter than its real size, because `.length` is `strlen`-based. Not a new gap introduced here — the same pre-existing, already-documented fact about this compiler's string representation, just reachable from a second source now.
- **Error handling: throw a catchable `Error`, built from the OS's own reason, on failure** — directly following `ADR-00021`'s precedent for `fetch`'s network failures, rather than inventing a different convention for this feature. The one deliberate, documented exception: `existsSync` does *not* throw for a missing path, matching real Node's own `fs.existsSync` — one of the few `fs` functions where "doesn't exist" is itself the normal, expected answer (`false`), not a failure.
- **Getting the actual OS reason for a failure required a real cross-platform decision, verified rather than assumed.** `strerror(errno)` needs `errno`'s current value, but `errno` is a macro, not a portable global symbol — glibc (Linux) exposes it via `__errno_location()`, while Darwin/BSD (macOS) exposes it via a differently-named `__error()`. Found the exact precedent for handling this already in the codebase: `emitMathRandom` already switches on `runtime.GOOS` to choose between `arc4random()` (BSD/macOS) and a portable fallback (elsewhere) for `Math.random()`. Since this compiler doesn't cross-compile (it targets whatever architecture the host `clang` defaults to, and always builds and runs `clang` on the same machine that ran the KML compiler itself), checking `runtime.GOOS` in the *Go compiler's own* process — not at the generated program's run time — is the correct fix, not a hack: whichever OS is running the KML compiler right now is also the OS that will run `clang` moments later. Followed this exact same pattern for the errno accessor.
- **A real bug, caught immediately by the very first manual end-to-end test, not left for the test suite to eventually surface.** The first hand-compiled test program (write, read, append, delete, exists-check, in sequence) failed at the `clang` step with `invalid redefinition of function 'fopen'` — `ensureFsReadFile()` and `ensureFsWriteFile()`/`ensureFsAppendFile()` each independently emitted their own `declare ptr @fopen(...)` line, and since each has its *own* singleton guard (`usedFsReadFile`, `usedFsWriteFile`, `usedFsAppendFile`), using more than one of them in the same program meant the shared `fopen` declaration got emitted more than once. Same root cause hit `fclose` and, on the next build attempt, `fwrite` too (shared between the write and append variants, which also each have their own independent singleton flag). Fixed properly rather than patched around: extracted dedicated `ensureFopen()`/`ensureFclose()`/`ensureFwrite()` singleton helpers (each with its own `used*` flag), so any C-level declaration shared across multiple `fs.*` driver functions is only ever emitted once regardless of which combination of `fs` functions a given program actually calls. Verified the fix by re-running the exact combination that had failed, then confirming the full test suite (which exercises every function together) stayed green.

## Decision

`fs.readFileSync(path)`, `fs.writeFileSync(path, data)`, `fs.appendFileSync(path, data)`, `fs.existsSync(path)`, `fs.unlinkSync(path)` — all synchronous, all text-only, all except `existsSync` throwing a catchable `Error` (message built from `strerror(errno)`) on failure. Recognized as a pseudo-namespace (`fs.*`), not a real module.

## Implementation notes

- `codegen/llvm/runtime.go`: `errnoAccessor()` (the `runtime.GOOS` switch); `ensureFopen`/`ensureFclose`/`ensureFwrite` (shared, properly singleton-guarded C declarations — the fix for the bug above); `ensureFsThrow` (builds the "`<opDesc>` '`<path>`': `<strerror>`" message via `sprintf` into a generously-sized malloc'd buffer, then calls the existing `@__kml_throw`, ADR-00021/emit_exceptions.go); `ensureFsReadFile` (`fopen`/`fseek`/`ftell`/`fread`/`fclose`, sized via `ftell` after seeking to the end); `ensureFsWriteFile`/`ensureFsAppendFile` (both delegate to a shared `ensureFsWriteLike`, differing only in `fopen` mode string, generated function name, and error message).
- `codegen/llvm/emit_fs.go` (new file): `emitFsReadFileSync`, `emitFsWriteFileSync`/`emitFsAppendFileSync` (both delegate to a shared `emitFsWriteLikeCall`), `emitFsExistsSync`, `emitFsUnlinkSync` — each validates its own argument count, evaluates and coerces arguments to `TypePtr`, and calls the matching runtime helper.
- `codegen/llvm/emit_call.go`: dispatch added under `id.Name == "fs"`, following `process`'s exact existing dispatch shape.
- `codegen/llvm/emit_exprs.go`: `inferExprType` gained `fs.readFileSync` (→ `TypePtr`) and `fs.existsSync` (→ `TypeBool`) cases, so untyped declarations (`const content = fs.readFileSync(...)`) infer correctly — `writeFileSync`/`appendFileSync`/`unlinkSync` return `void` and don't need a case (nothing meaningful to assign).

## Side effects discovered

The `fopen`/`fclose`/`fwrite` duplicate-declaration bug described in Investigation — found and fixed in this same pass, before it ever reached the test suite or a real user.

## Verification

- `go build .`, `go vet ./...`, `go test ./...`, `make examples` (all pass, examples 47/47).
- Manual repros, each actually compiled and run: full write → exists → read → append → read → delete → exists sequence, matching expected output exactly at each step; `writeFileSync` correctly truncates on a second call rather than appending; reading a nonexistent file throws and is catchable via `try`/`catch` (with a real, helpful `strerror`-based message: `"cannot open file for reading '<path>': No such file or directory"`), and exits non-zero with `"Uncaught: ..."` when not caught; writing to a nonexistent directory throws with an analogous message; deleting a nonexistent file throws; `existsSync` correctly returns `false` (not a thrown error) for a path that was never created, and for one already deleted.
- New tests in `compiler_test.go`: `TestE2EFsWriteReadAppendUnlink`, `TestE2EFsWriteFileSyncOverwritesExistingContent`, `TestE2EFsReadFileSyncUntypedInference`, `TestE2EFsReadFileSyncNonexistentThrows`, `TestE2EFsReadFileSyncNonexistentUncaughtExitsNonZero`, `TestE2EFsUnlinkSyncNonexistentThrows`, `TestE2EFsWriteFileSyncWrongArgCountRejected`, `TestE2EFsReadFileSyncWrongArgCountRejected` — using `t.TempDir()` for isolated, auto-cleaned file paths per test.
- New example `examples/fs/fs.ts`, demonstrating the full read/write/append/exists/unlink/error-handling cycle against a real file, cleaning up after itself.
- `STATUS.md`: rewrote the "File API" section to match the actual (Node-`fs`-shaped) scope instead of the stale browser-`File`-API framing; flipped the pre-existing "Process / CLI I/O" row that already (correctly) anticipated this feature; updated coverage numbers and the Phase 5 roadmap.
