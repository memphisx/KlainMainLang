# ADR-00028: Implement String.prototype charAt/codePointAt/search/localeCompare

- **Date**: 2026-07-12

## Context

Picked up the remaining `❌` rows in `STATUS.md`'s String Methods table: `.charAt(i)`, `.codePointAt(i)`, `.normalize()`, `.match()`/`.matchAll()`, `.search()`, `.localeCompare()`, and `String.raw`.

## Investigation

Audited each remaining row for whether it's a genuine quick win or something that needs infrastructure this compiler doesn't have, rather than implementing all seven uniformly:

- **`.charAt(i)`**: real JS semantics differ from `.at(i)` (already implemented, `ADR`-untracked, pre-existing) in one specific way — `.charAt` never treats a negative index as "from the end" (`.charAt(-1)` is always `""`), while `.at(-1)` wraps to the last character. A genuine, small, self-contained method — implemented directly.
- **`.codePointAt(i)`**: this compiler's strings are plain byte sequences, not real UTF-16 like actual JS strings (confirmed by re-reading `emitStringCharCodeAt`, `emit_strings.go` — it's a raw byte load, no encoding awareness at all). Real `codePointAt` differs from `charCodeAt` only for astral (non-BMP) characters requiring surrogate-pair decoding — a distinction this compiler's byte-string model has no way to represent in the first place. Implemented as a direct alias of `charCodeAt`, since for the only inputs this compiler can actually represent (ASCII/Latin-1 byte values), the two are numerically identical — not a partial/wrong implementation, but the entire correct behavior for the representable input space.
- **`.search(pattern)`**: real JS coerces `pattern` to a `RegExp`. Confirmed this compiler has no `RegExp` type, no regex literal syntax (`/foo/`) anywhere in the lexer/parser, and no way to construct one — meaning a plain string is the *only* value any KlainMainLang program could ever pass to `.search()`. Implemented as a direct alias of `.indexOf`, for the same reason as `.codePointAt`: not a partial implementation of a bigger feature, but the complete reachable behavior given this compiler's actual type system today.
- **`.localeCompare(other)`**: no locale/`Intl` infrastructure exists anywhere in this compiler. Implemented via plain `strcmp` byte-order comparison, normalized to exactly `-1`/`0`/`1` (real JS's spec only requires negative/zero/positive, but a fixed small set is more useful to assert against) — the same documented scope narrowing `toLocaleDateString` (`ADR-00019`) already established: a single fixed, non-locale-aware behavior under the real API name.
- **`.normalize()`, `.match()`/`.matchAll()`, `String.raw`: deliberately NOT implemented, and not attempted as fakes.** `.normalize()` needs real Unicode normalization tables (NFC/NFD/NFKC/NFKD) this compiler has zero infrastructure for; a no-op identity implementation would look like it works for ASCII input while silently producing wrong output for any actual composed/decomposed Unicode text reaching it — exactly the "silent wrong output instead of a clear unsupported error" failure mode this project's own bug-tracking criteria call out as worse than not implementing something at all. `.match()`/`.matchAll()` need a real `RegExp` engine (tracked separately, unrelated to this pass). `String.raw` needs general tagged-template-literal syntax (`` tag`...` ``) that this compiler's parser doesn't support at all — a distinct, meaningfully larger language feature, not a small addition on top of existing template literals. All three left `❌` in `STATUS.md` with the reason spelled out, rather than silently skipped.
- **A naming collision, caught by the compiler itself, not by inspection.** `emitStringCharAt` already existed — not implementing `.charAt()`, but backing `s[i]` bracket-index syntax on strings (a separate, pre-existing feature). `go build` failed immediately with a duplicate-method-declaration error and a call-site arity mismatch, making the collision impossible to miss. Renamed the new method's implementation to `emitStringCharAtMethod` to keep both call sites unambiguous.

## Decision

`.charAt(i)`, `.codePointAt(i)`, `.search(pattern)`, `.localeCompare(other)` — implemented, all with the scope narrowing described above spelled out in `STATUS.md`. `.normalize()`, `.match()`/`.matchAll()`, `String.raw` — deliberately left unimplemented, each with its blocking reason recorded rather than left as an unexplained gap.

## Implementation notes

- `codegen/llvm/emit_strings.go`: `emitStringCharAtMethod` (bounds-checked, non-wrapping — clamps the extraction start to `0` when out of range to avoid ever computing a negative-offset `getelementptr`, the same safety margin `.at()`'s own clamping already relies on); `emitStringCodePointAt` (a one-line delegation to `emitStringCharCodeAt`); `emitStringSearch` (a one-line delegation to `emitStringIndexOf`); `emitStringLocaleCompare` (`strcmp` + a `select`-chain normalizing the raw result to `-1`/`0`/`1`).
- `codegen/llvm/emit_call.go`: `charAt`/`codePointAt`/`search`/`localeCompare` added to the flat `mem.Property ==` dispatch chain, right after the existing `charCodeAt` case.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s string-method switch gained `codePointAt`/`search`/`localeCompare` (→ `TypeI64`) and `charAt` (→ `TypePtr`).

## Side effects discovered

The `emitStringCharAt` naming collision described in Investigation — caught immediately by `go build` failing, not by manual review; fixed by renaming before it ever reached a test run.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, including the 8 new tests below), `make examples` all green (52/52 — one `examples/fetch/fetch.ts` failure during one run was the already-documented httpbin.org flakiness, unrelated, and passed cleanly on retry).
- Manual repros, each actually compiled and run: `.charAt` at an in-range index, past the end, and at a negative index (confirming no wraparound, unlike `.at(-1)`), and confirming `.charAt(i) === s[i]` for an in-range `i`; `.codePointAt(0) === .charCodeAt(0)`; `.search` matching `.indexOf`'s result for both a found and a not-found needle; `.localeCompare` returning `-1`/`1`/`0` for a before/after/equal pair, and used directly as an array-sort comparator (`words.sort((a, b) => a.localeCompare(b))`).
- New tests in `compiler_test.go`: `TestE2EStringCharAt`, `TestE2EStringCharAtWrongArgCountRejected`, `TestE2EStringCodePointAt`, `TestE2EStringSearch`, `TestE2EStringLocaleCompare`, `TestE2EStringLocaleCompareWrongArgCountRejected`.
- Extended `examples/strings/char_code.ts` (`.codePointAt`) and `examples/strings/string_indexing.ts` (`.charAt` alongside bracket indexing, for direct comparison); new `examples/strings/search_compare.ts` (`.search`/`.localeCompare`, including the sort-comparator usage).
- `STATUS.md`: flipped four rows from ❌ to ✅ with their scope narrowing spelled out; added explicit deferral reasons to `.normalize()`/`.match()`/`.matchAll()`/`String.raw`'s existing ❌ rows; removed the now-done `String.prototype.charAt(i)` item from "What Is NOT Implemented"; recomputed the String methods coverage row (22/33 → 26/33, ~79%) and the Core Language total (found and fixed a second, unrelated stale figure while re-summing every category row directly: 289 total items was already correct, but implemented had drifted to 204 instead of the ADR-00026-era Global-functions fix's actual carry-through value; corrected to 209/289, ~72%).
