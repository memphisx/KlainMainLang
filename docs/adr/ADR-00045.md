# ADR-00045: Reject const reassignment with a Symbol.IsConst check in emitAssign

- **Date**: 2026-07-13

## Context

Found during the same bug-hunting pass as `ADR-00044.md`'s array-bounds fix: `const x = 5; x = 10; console.log(x)` compiled and printed `10`, instead of the compile-time error real TS/JS gives (`TypeError: Assignment to constant variable`). Flagged in `STATUS.md`'s Known Limitations table, picked up as its own piece of work this session, second of the four bugs found in that pass.

## Investigation

`Symbol` (`codegen/llvm/emitter.go`) had no field recording a binding's declared kind at all. `decl.Kind` (`"let"`/`"const"`/`"var"`, on `ast.VarDeclaration`) is parsed and already available at declaration time, but was never propagated onto the `Symbol` created for it — so `emitAssign`'s scalar-identifier assignment path (`codegen/llvm/emit_exprs.go`) had nothing to check a reassignment against.

Checked every call site that constructs a `Symbol` from a `VarDeclaration` to find where `Kind` needed to be threaded through: the plain-scalar path (`emitVarDecl`), and the three other top-level declaration forms that also bind a name directly to a `const`/`let`/`var` (`emitArrayVarDecl`, `emitObjectVarDecl`, `emitMapVarDecl`/`emitSetVarDecl`) — all four take `v *ast.VarDeclaration` already, so `v.Kind` was available at every one without new plumbing. Deliberately left out of scope: `for...of`/`for...in` loop variables and function/catch parameters — none of those go through `ast.VarDeclaration` at all (they have their own `Kind` fields on different AST node types), and the original bug report was specifically about a plain `const` declaration's own binding, not loop-variable reassignment (a real but separate gap, not chased here).

Also found, while wiring this up, that `IsConst` needed to survive two symbol-rewriting paths that already exist for closures: `emitClosureFunc`'s per-capture definition inside the closure body (`emit_func.go`, defines a fresh `Symbol` for the captured name in the new function's scope) and the capture-time promotion-to-boxed-heap-cell path (`emit_func.go`, `updateSymbolInPlace` when a variable is captured by reference for the first time) — both previously constructed a new `Symbol` from scratch without copying `IsConst` over from the original, which would have silently dropped the check for any `const` captured by a closure. Confirmed by direct test: `const x = 5; const f = () => { x = 10 }` correctly rejects only once both paths copy `IsConst` from `cap.Sym.IsConst`.

## Decision

Added `Symbol.IsConst bool`, set to `v.Kind == "const"` at all four top-level `VarDeclaration`-backed definition sites. Added a single check at the top of `emitAssign`'s scalar-identifier path, right after the symbol lookup and before the `=`/compound-operator branch (so it rejects both `x = 10` and `x += 1` uniformly, rather than just plain `=`): `if sym.IsConst { return error }`.

Array-element writes (`arr[i] = ...`) and object-field writes (`obj.field = ...`) go through different code paths (`emitIndexPtr`/the object-field GEP-and-store branch, not the scalar-identifier branch) and are deliberately unaffected — `const` in TS/JS only freezes the binding itself, not the contents of what it points to, so `const arr = [1,2,3]; arr[0] = 99` and `const p = {x:1}; p.x = 99` must both continue to work. Confirmed both still work after the fix.

## Implementation notes

- `codegen/llvm/emitter.go`: `Symbol.IsConst bool` field added.
- `codegen/llvm/emit_exprs.go`: `emitVarDecl`'s scalar `e.define` call sets `IsConst: v.Kind == "const"`; `emitAssign`'s scalar-identifier path gets the new check.
- `codegen/llvm/emit_arrays.go`, `emit_objects.go`, `emit_collections.go`: same `IsConst: v.Kind == "const"` addition to each of `emitArrayVarDecl`/`emitObjectVarDecl`/`emitMapVarDecl`/`emitSetVarDecl`'s `e.define` call.
- `codegen/llvm/emit_func.go`: `emitClosureFunc`'s capture-boxing path (`updateSymbolInPlace`) and the closure-body capture-definition path (`e.define(cap.Name, ...)`) both now copy `IsConst: cap.Sym.IsConst` instead of leaving it defaulted to `false`.

## Side effects discovered

None beyond the closure-capture gap described above, which was found and fixed within this same piece of work (not a separately deferred bug).

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including seven new tests below) all green; `make examples` 56/56.
- Manual repros: plain `const` scalar reassignment (`x = 10`) and compound assignment (`x += 1`) both rejected at compile time; `const` object rebinding (`p = {...}`) rejected; a `const` captured by a closure and reassigned inside it (`const f = () => { x = 10 }`) rejected. Confirmed unaffected: `let`/`var` reassignment still works, `const`-bound array element mutation (`arr[0] = 99`) still works, `const`-bound object field mutation (`p.x = 99`) still works.
- New tests in `compiler_test.go`: `TestE2EConstScalarReassignmentRejected`, `TestE2EConstCompoundAssignmentRejected`, `TestE2EConstObjectRebindingRejected`, `TestE2EConstCapturedByClosureReassignmentRejected`, `TestE2ELetReassignmentStillWorks`, `TestE2EConstArrayElementMutationStillWorks`, `TestE2EConstObjectFieldMutationStillWorks`.
- `examples/basics/const.ts`: updated the file's own top-of-file comment, which previously (and incorrectly, as of this fix) stated "the compiler treats const and let identically at the IR level" — ran the updated example manually, output unchanged and correct (it never actually rebinds a const, only mutates array elements/object fields, both still legal).
- `STATUS.md`: removed the now-fixed row from Known Limitations & Bugs and the Roadmap's "Next up" table.
