# TDD-00007: Coerce object literal fields against their declared type

- **Status**: Not Started
- **Date**: 2026-07-13

## Context

Found while investigating a much older, vaguer memory of "issues with JSON.parse/JSON.stringify around objects more than one level deep." Tracing it down, nesting depth turned out to be irrelevant, and it isn't a JSON bug at all: `emitObjectLiteral` (`codegen/llvm/emit_objects.go`) infers an object literal's field types purely from the literal's own values (`inferObjectType`), with zero awareness of any declared/expected type from the surrounding context (an interface annotation, a function parameter's declared type, a return type, an array's element type, or an outer object literal field's own declared type).

Confirmed directly, not assumed: for a completely flat, one-level interface,

```ts
interface Point { x: number; y: number }
const p: Point = { x: 1, y: 40.6 };
console.log(p.y); // prints 4630910759336725709
```

`4630910759336725709` is the exact IEEE-754 bit pattern of `40.6` as a `double`, reinterpreted as an `i64`. This is not a truncation (which would print `40`) and not a rejection: it's silent bit-reinterpretation corruption, readable through *any* subsequent field access, not just `JSON.stringify`. Also confirmed bidirectional: a `@type {float64}`-annotated field given an integer literal (`score: 5`) corrupts the same way in reverse, printing `2.47033e-323`.

**Why this looked fixed before, three separate times, without ever actually being found:**
- `ADR-00007` (`JSON.parse`→object) noted a field "can only become float-typed via literal inference in a plain object literal, never via an explicit interface declaration" — correct, but framed as "can't declare," not "corrupts on mismatch."
- `ADR-00035` (`JSON.stringify` float fix) explicitly tested the interface case, found it "always resolves to `i64` regardless of the assigned value," and moved on treating it as an already-tracked, separate, out-of-scope limitation — without checking *what* the stored value actually was.
- `ADR-00040` (JSDoc `@type` on interface fields) fixed the ability to correctly *annotate* a field as `float64`, sidestepping the mismatch by making the declared and literal-inferred types agree from the start. It never touched the underlying corruption for the case where someone doesn't use the annotation.
- The existing nested-object test (`TestE2EJSONStringifyNestedObject`) never had a chance to catch this either: it uses an untyped literal (`const person = {...}`, no interface) with only strings and integers, so there was never a declared-type-vs-literal-type mismatch to trigger it.

**The direct connection worth naming**: this is the same failure shape as `TDD-00005`/`ADR-00042` (an unannotated function parameter defaulting to `i64` with no check against what's actually passed, causing identical bit-reinterpretation for a string argument). That got a real fix: a clean check at every call site consuming the value. Object literal construction is a separate code path that never got the same treatment; nobody made the connection until now.

**Why coercion, not rejection, is the right fix here (unlike `ADR-00042`)**: `ADR-00042` rejected mismatched arguments to unannotated parameters, because "no annotation" genuinely means the correct type isn't knowable — any argument type could be intended. Here, the correct type *is* knowable: the interface says `y: number`, which this compiler resolves to `i64`. `const p: Point = { x: 1, y: 40.6 }` is completely unremarkable, valid TypeScript that any real TS compiler accepts without complaint (`number` covers both integers and floats in real JS/TS). Rejecting it would be wrong; silently coercing `40.6` down to `40` (matching what a real `i64`-backed `number` field means here) is the correct behavior.

## Design

**Root cause, precisely**: `emitObjectLiteral`'s `storeField` closure already does the right thing when given the right input: `val = e.coerce(val, fieldTy)` before storing. The bug is that `fieldTy` comes from `ty.FieldIndex(name)`, where `ty := e.inferObjectType(lit)` is the literal's *own* self-inferred struct type, not any externally declared one. Coercing a value against a type inferred from that same value is a no-op by construction. The coercion mechanism isn't broken; it's just never given the correct target.

**The fix**: give `emitObjectLiteral` an optional expected-type hint (mirroring two patterns this project already has proven, not inventing a new one):
- `resolveCallbackWithHints` (`emit_func.go`) already threads a hint down into `emitArrowFunctionWithHints` for exactly this reason (an unannotated HOF callback parameter's real type comes from the array's element type, not from the callback literal alone).
- `emitJSONParseObject` (`emit_call.go`) already threads a `targetTy` parameter through object construction from a JSON string, for the identical reason (each field's expected type has to come from the declaration being parsed into, not from the untyped JSON text).

Concretely: a new `emitObjectLiteralWithHint(lit *ast.ObjectLiteral, hint *Type) (Value, error)` (or an added parameter on the existing function, since every current caller can pass `nil` unchanged). When `hint` is non-nil and `IsObject`, `storeField` looks up each field's type from `hint.FieldIndex(name)` instead of the literal's own self-inferred `ty`, and coerces against that. When `hint` is nil (no known expected type: e.g. the untyped `const x = {...}` case), behavior is unchanged.

**Call sites needing the hint threaded through** (this is the actual work, not the coercion itself):
1. **Variable declarations**: `const p: Point = {...}`. Easiest case: `emitObjectVarDecl` already has the declared `ty` in hand; it currently discards it by calling the hint-less `emitObjectLiteral`.
2. **Function call arguments**: `someFn({ x: 1, y: 40.6 })` where the parameter is declared `Point`. The callee's declared parameter types are already available at the call site (`FuncSig.ParamTypes` for named functions, `ty.FuncParams` for closures) — the same data `ADR-00042`'s parameter-typing fix already reads.
3. **Return statements**: `function f(): Point { return { x: 1, y: 40.6 } }`. The function's declared return type is already tracked in `e.currentRetType` during body emission.
4. **Array literal elements**: `const pts: Point[] = [{x:1,y:2}, {x:3,y:4}]`. The array's declared element type needs to be passed as the hint for each element literal.
5. **Nested object literal fields**: an outer literal's field whose own declared type is another interface (`address: { ..., coords: { lat: 40.6, lng: 22.9 } }`, where `Address.coords: Coords`). The hint has to propagate recursively: when a field's own hinted type is itself an object type, that becomes the hint for a nested literal assigned to it.
6. **Object spread**: likely already fine as-is, since a spread's source is an already-typed, already-correctly-stored object (`srcVal.Ty.Fields`) — worth confirming directly during implementation rather than assumed.

**Deliberately out of scope**: validating/rejecting genuinely incompatible mismatches (e.g. a string literal assigned to a `number`-declared field). Real TypeScript already rejects that at the type-checking level, and this compiler doesn't independently re-verify type compatibility anywhere else either (consistent with its existing "trust the source" posture) — this fix is about *not silently corrupting a value TypeScript itself considers perfectly valid* (`number` covering both int- and float-shaped literals), not about becoming a general type checker.

## Prerequisites

| Dependency | Status | Notes |
|---|---|---|
| `e.coerce(val, targetType)` correctly handles int↔float conversion | ✅ already done | Already used by `ADR-00035`'s fix and elsewhere; the coercion logic itself isn't in question, only what type it's called with |
| A proven "pass an expected-type hint into literal construction" pattern | ✅ already proven, twice | `resolveCallbackWithHints` (HOF callback params) and `emitJSONParseObject`'s `targetTy` threading (JSON→object) are both direct, working precedents for the same shape of problem |
| Declared parameter types available at a call site | ✅ already done | `FuncSig.ParamTypes`/`ty.FuncParams`, the same data `ADR-00042` already reads |
| Declared return type available during body emission | ✅ already done | `e.currentRetType` |
| Declared array element type available at a literal's declaration site | ✅ already done | Already resolved wherever array-typed variables are declared |
| Recursive hint propagation into nested object literal fields | ❌ not designed | The one genuinely new piece of plumbing — hint propagation for the other four call sites is a straightforward "pass the already-known type," but nesting requires the hint itself to recurse |

## Open questions

- Whether to land all five call sites in one pass or ship the variable-declaration case first (the most common shape, and the one this bug was actually found through) and extend to the others incrementally, the same way `fetch`/`execFileSync`/other features in this project have grown their scope in stages.
- Whether object spread genuinely needs no changes, or whether a spread source that itself came from an unhinted literal could still carry a corrupted field forward — worth checking directly rather than assuming during implementation.
