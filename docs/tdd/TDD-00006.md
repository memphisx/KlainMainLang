# TDD-00006: Event Loop

- **Status**: Not Started
- **Date**: 2026-07-13

## Context

Tracked as this project's single biggest structural gap relative to its stated microservice direction. Written up in full because the reason the gap exists at all matters for how to close it: `async`/`await` (`emit_async.go`) was deliberately built as a fast, syntax-compatible stand-in for `Promise`-shaped code — malloc a slot, run synchronously, hand back a pointer to it — specifically to get real programs writing `async`/`await`/`Promise<T>` working quickly, not as a faithful implementation of what those keywords actually mean. The file's own header comment says so plainly: *"No LLVM coroutine intrinsics are used."* That was the right call at the time (getting the syntax working end-to-end mattered more than getting concurrency right immediately), but it means every "async" program this compiler has ever produced has actually run in strict, blocking, top-to-bottom order — this document exists because that stand-in now needs to be replaced with the real thing.

### Why this is two problems, not one

That splits "event loop" into two pieces of very different size:

1. **An I/O-multiplexing wait loop** (`select`/`poll`/`epoll`/`kqueue`) that lets the process wait on sockets and timers together instead of blocking on one thing at a time. This alone is enough to make `TDD-00004` (HTTP Server)'s own V1 scope (single-threaded, one connection at a time) actually buildable, and reuses a lot that already exists.
2. **Real suspension** — actually letting `await fetch(...)` yield control back to the loop while a request is in flight, letting `Promise.all` run several awaits concurrently, letting one slow HTTP request not block every other client. This needs a fundamental rewrite of how `async`/`await` is compiled, not an addition next to it.

Part 2 cannot do anything useful without Part 1 (something still has to notice "this socket is ready now" and know which suspended computation to resume) — so Part 1 is the correct thing to build first regardless of which direction Part 2 eventually takes, and is worth doing even if Part 2 never happens (it still unlocks the HTTP server).

## Design

### Part 1: I/O-Multiplexing Loop

- **Mechanism**: plain POSIX `select()` for a first cut, not `epoll`/`kqueue`. `select()` is available identically on both macOS and Linux with zero platform-conditional codegen needed (unlike most of this project's other platform differences), and its `FD_SETSIZE` (typically 1024) / O(n)-scan limitations are a non-issue at the scale this project actually operates at — the same "simplicity now, revisit only if it needs to scale" reasoning `TDD-00002` (Timers) already used for a linear-scan queue instead of a real heap. `epoll` (Linux) / `kqueue` (macOS/BSD) are the natural follow-up once/if a program genuinely needs more than a handful of concurrent connections.
- **Merges with the existing timer queue, generalizing the drain loop** Timers already added: instead of "sleep until the next timer is due," compute that same next-due timestamp but pass it to `select()` as a timeout instead of `nanosleep`, alongside a set of file descriptors to watch. Whichever wakes `select()` first — a timer coming due, or a socket becoming readable/writable — gets handled, then the loop repeats. One unified loop, not two competing ones.
- **fd registration is an internal runtime concern, not a new user-facing TS API for V1** — no direct "register this socket for readiness" call exposed to user code, the same way the timer queue itself was never a user-visible data structure either. The HTTP server's accept loop (and, later, a non-blocking `fetch`) would register file descriptors internally.
- **Calling back into user code on readiness reuses the same closure-trampoline convention** already proven three times over (`qsort` comparators, timer callbacks, and `TDD-00004`'s own planned dispatch) — no new calling-convention work needed here specifically.

**Recommended sequencing**: buildable independently of Part 2 — the first piece to pick up regardless of Part 2's eventual direction, since it's needed either way and is far more tractable.

### Part 2: Real Async Suspension

Without this, Part 1 alone still leaves every "async" program running fully synchronously under the hood — `await fetch(...)` blocks the entire process exactly as it does today, `Promise.all` has nothing meaningful to parallelize, and an HTTP server built on Part 1 alone can multiplex *accepting* connections but still has to finish one request's handler, synchronously, top-to-bottom, before starting the next. Real concurrency needs a function to actually stop executing at an `await`, hand control back to the loop, and resume later, picking up exactly where it left off — a fundamentally different execution model than "call a function, it returns when it's done," which is the only model this compiler (and LLVM IR's own `call`/`ret` instructions) work with today.

**Options considered (no decision made):**

1. **LLVM coroutine intrinsics** (`llvm.coro.id`/`coro.begin`/`coro.suspend`/`coro.end`, the mechanism Rust's `async`/`await`, Swift's concurrency, and clang's own C++20 coroutines are built on). LLVM's own coroutine-splitting pass does the hard part — transforming a function into a resumable state machine with a heap-allocated frame — automatically, once the right intrinsic calls are emitted at the right points. The most battle-tested, well-documented option, and doesn't add a new *external* dependency (it's part of LLVM itself, already required) — but it's a wholly new corner of LLVM IR generation this project has zero prior experience with, and every existing `await` call site and the entire async codegen path would need rewriting from scratch. Interactions with this compiler's own `setjmp`/`longjmp`-based exception handling and closure-capture model around suspend points are unstudied.
2. **Continuation-passing-style (CPS) transformation done by this compiler itself**, not by LLVM: split an async function's body at every `await` into separate continuation functions, each capturing exactly the local variables still needed — conceptually similar to this compiler's *own* existing closure-capture/heap-boxing machinery (reusing a pattern it already knows), rather than learning a new LLVM subsystem. But doing this correctly needs a real compiler pass this project has never built: a control-flow-graph-aware split with genuine live-variable analysis (which variables are alive across which suspend point) — a serious, standalone piece of compiler engineering, comparable in kind to the borrow-checker-scale effort `TDD-00001` (Memory Management) already declined to build for exactly this reason.
3. **Hand-rolled fibers/green threads** (separate heap-allocated stacks per concurrent task, switching between them via a small amount of architecture-specific assembly saving/restoring the stack pointer and callee-saved registers — the same shape as `boost::context`, `libco`, or Lua's coroutines). The least invasive option to *existing* codegen — an ordinary function body needs no special transformation at all, since `await` just becomes "park this fiber, switch to another, resume later" with the native call stack fully intact. The cost moves elsewhere: a genuinely new kind of low-level, per-architecture (arm64 vs x86-64) runtime primitive this project has never written, and every blocking libc call a suspended task might make (`recv`, `connect`, `read`) would need a non-blocking replacement wired into Part 1's loop. Also needs auditing whether the exception system's `setjmp`/`longjmp` jump-buffer stack is safe across a stack switch — likely not, without changes.

**Unlike `TDD-00001` (Memory Management), none of these three is an obvious standout the way Boehm GC was there.** Each trades a different kind of unfamiliar, substantial work for a different kind of safety/reuse, and this project has zero direct experience with any of the three mechanisms involved (LLVM coroutines, CFG-splitting compiler passes, or hand-written fiber context-switching). A real decision here should follow a small prototyping spike on at least the leading 1–2 candidates, not be made from a design doc alone.

## Prerequisites

### Part 1

| Dependency | Status | Notes |
|---|---|---|
| Timer queue + drain loop (`TDD-00002`) | ✅ already done | Generalizes directly — same loop, `select()`'s timeout replaces `nanosleep`, fds get added alongside the existing timer-due check |
| Closure-calling-from-C-trampoline pattern | ✅ already proven | `qsort` comparators and timer callbacks are direct precedents |
| Raw POSIX sockets | ❌ not yet used anywhere | Needed by both this and `TDD-00004`; no blocker, just not written yet |
| Monotonic clock for computing the `select()` timeout | ✅ already done | Same `clock_gettime(CLOCK_MONOTONIC)` Timers/`performance.now()` already use |

### Part 2

| Dependency | Status | Notes |
|---|---|---|
| Part 1 (I/O-multiplexing loop) | ❌ not started | A hard prerequisite — nothing here is useful without something to notice readiness and resume the right task |
| Any experience with LLVM coroutine intrinsics | ❌ none | Relevant only to option 1; would need to be learned from scratch |
| A control-flow/liveness-analysis compiler pass | ❌ none built yet | Relevant only to option 2; this compiler's existing analysis is all shallow, single-expression-at-a-time inference, not full CFG-level |
| Architecture-specific context-switching primitive | ❌ none written | Relevant only to option 3; a genuinely new category of low-level code for this project |
| Exception system (`setjmp`/`longjmp`) compatibility with suspension | ⚠️ unstudied | Whichever option is chosen, needs a direct check — a suspended computation's jump-buffer validity across a real yield point isn't obviously safe under any of the three options as-is |
| Garbage collection / memory management (`TDD-00001`) | ⚠️ same caveat as `TDD-00004` | Long-running concurrent tasks that never free anything is the same accumulating-memory problem, just with more tasks alive at once |

## Open questions

- Which of Part 2's three options to pursue — genuinely undecided, and probably shouldn't be decided without a small throwaway prototype of the top 1–2 candidates first.
- Whether Part 1 should target `select()` permanently for this project's realistic scale, or whether `epoll`/`kqueue` should be built from the start — leaning toward `select()` first, revisit only if it becomes a real bottleneck.
