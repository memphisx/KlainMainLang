# ADR-00021: Implement fetch(url) and Response (GET only, V1)

- **Date**: 2026-07-12

## Context

Picked this up as the next feature after the link-flags plumbing (`ADR-00020`), which existed specifically to unblock this — `fetch` is one of the most broadly useful missing Web/Platform APIs, and squarely serves this project's stated CLI/microservice priority (REST API interaction).

## Investigation

- **Verified libcurl's exact option/info numeric constants against the real header, rather than trusting memory.** `curl_easy_setopt`/`curl_easy_getinfo` take enum values that look like arbitrary numbers in hand-written IR (`10002`, `20011`, `2097154`, …) — getting one wrong would compile fine and fail silently or behave subtly wrong at run time, not raise a clean error. Grepped the actual system `curl.h` (`CURLOPTTYPE_LONG=0`, `CURLOPTTYPE_OBJECTPOINT=10000`, `CURLOPTTYPE_FUNCTIONPOINT=20000`, then each option's own offset) and confirmed every constant used (`CURLOPT_URL=10002`, `CURLOPT_WRITEFUNCTION=20011`, `CURLOPT_WRITEDATA=10001`, `CURLOPT_FOLLOWLOCATION=52`, `CURLOPT_TIMEOUT=13`, `CURLOPT_NOSIGNAL=99`, `CURLINFO_RESPONSE_CODE=2097154`) directly rather than recalling them — curl's own versioning policy freezes these permanently for ABI compatibility, so hardcoding them here (rather than needing `curl.h` present at KML-compile time) is safe long-term, not just today.
- **Prototyped the entire design in plain C first, before writing a single line of hand-written LLVM IR.** Wrote a small C program doing exactly what the eventual `__kml_fetch` runtime helper needed to do (init, set the write-callback and options, perform, read the status, clean up), compiled and ran it against a real local server. This caught nothing that needed fixing, but de-risked the translation to IR considerably — by the time I wrote the actual `define { i64, ptr } @__kml_fetch(...)` IR, every piece of logic in it had already been runtime-verified once, so any bug found afterward could be attributed to the *translation*, not the underlying design.
- **Verified real network behavior directly, not assumed from the libcurl docs.** Confirmed against a mix of `httpbin.org` and a local test server: a 3xx redirect is followed automatically (`CURLOPT_FOLLOWLOCATION`) and the *final* status is what comes back; a 404/500 response resolves normally with the correct status and `ok=false`; a genuine network failure (unresolvable host) makes `curl_easy_perform` return nonzero with a human-readable `curl_easy_strerror` message.
- **The real design question: what should "async" mean here, given there's no event loop.** `fetch(url)` needs to return `Promise<Response>`, but this compiler's async/await (`emit_async.go`) is already a "synchronous V1" — an async function mallocs a slot and returns its pointer immediately, with no actual suspension. Rather than block on building an event loop first, implemented `fetch` as a blocking `curl_easy_perform` call whose result is immediately wrapped in an already-resolved Promise slot, using exactly the same `{malloc slot, store resolved value, return slot pointer tagged IsPromise}` shape `emitAsyncEpilogue` already produces for real async functions — meaning `await fetch(...)` needed zero changes to `emit_async.go` at all; `emitAwait` already handles any correctly-tagged Promise value uniformly, regardless of whether it came from user async code or a builtin. The honest, necessary caveat: two `fetch` calls issued before either is awaited run one after another, not concurrently, since there's no event loop to interleave them — documented explicitly rather than left as a surprise, the same way every other deliberate simplification in this compiler gets called out.
- **What real fetch's own async/reject semantics are, and which parts are worth replicating.** Real `fetch`'s Promise only rejects on a genuine network-level failure (DNS, connection refused, TLS, timeout) — a 404 or 500 still *resolves*, with the caller expected to check `.ok`/`.status` themselves. This distinction is meaningful and cheap to preserve exactly: `curl_easy_perform`'s return code already tells the two cases apart, so a nonzero result calls the existing `@__kml_throw` runtime helper (`emit_exceptions.go`) directly — a network failure surfaces as a catchable KML `Error`, exactly like a hand-written `throw new Error(...)`, using the exact same mechanism (including its existing "uncaught → print and exit 1" behavior for code with no surrounding `try`/`catch`) rather than inventing a parallel error-reporting path.
- **How to represent `Response` without `class`.** Following the precedent already established for `Date`/`Map`/`Set` (a special built-in type with hand-written method dispatch, not a general user-defined-methods feature): `status`/`ok`/`body` are modeled as **plain fields** on an ordinary heap object (reusing `ObjectType`'s existing malloc-and-GEP machinery directly, so field reads need zero new dispatch code — the general "object field read" fallback in `inferExprType` already added for `Date` in `ADR-00014` covers them for free), with only `text()`/`json()` needing actual method dispatch (a new `IsResponse` marker, mirroring `IsDate`). Considered hiding the raw body behind a "private" field name — rejected, since this compiler has no real access-modifier system to enforce that anyway; naming it plainly (`body`) and documenting `text()`/`json()` as the intended API surface is more honest than a naming convention pretending to be privacy.
- **`response.json()` reuses `JSON.parse` end to end, including its exact scope and known limitations.** Rather than write a second JSON-parsing path, factored `emitJSONParse`'s core into a new `emitJSONParseValue(val Value, targetTy Type, pos)` that works on an already-evaluated string `Value` (not just a literal call argument), and had `response.json()` call it directly after extracting the buffered body. This means `.json()` inherits `JSON.parse`'s existing declaration-context special-casing (the target type comes from `const x: T = response.json()`'s own annotation, exactly like bare `JSON.parse(s)` already works) and its existing, already-documented scope gap — nested object fields aren't supported — for free, with the exact same fix (if it ever happens) automatically applying to both call sites at once.
- **Two real, `ADR-00014`-style pre-existing-gap bugs found and fixed while wiring up the natural `await fetch(...)` usage pattern, not left for later.** (1) `emitVarDecl`'s untyped-inference switch had no case recognizing a bare `fetch(...)` call (its `*ast.CallExpression` branch only special-cased user-defined functions registered in `e.funcs`, which a builtin like `fetch` isn't) — `const p = fetch(url)` would have silently inferred as plain `i64`. (2) `emitObjectVarDecl`'s switch (for object-typed declarations) had no case for `*ast.AwaitExpression` at all — `const r: Response = await fetch(url)`, an extremely natural way to write this, would have hit the switch's default `"object variable must be initialized with an object literal or function call"` error, since `await ...` doesn't match either of the two cases that switch already had. Both fixed in this same pass, following the standing rule to fix bugs found during development rather than deferring them, exactly like the analogous `Date`/`Error` gaps found in `ADR-00014`.
- **Found, and deliberately did not treat as a `fetch`-specific bug, a truncation issue with binary response bodies.** Fetching 50,000 bytes of random binary data came back as a 383-byte string. Traced this to the pre-existing fact that every string in this compiler is a plain null-terminated C string (`.length` is `strlen`-based) — *any* string value with an embedded `\0` reads short, regardless of where it came from; `fetch` just makes this reachable from live, uncontrolled external data for the first time. Verified the actual growable-buffer/`realloc` logic in the write-callback is correct by re-testing with a large all-text (no embedded nulls) body, which came back complete and byte-exact — ruling out a bug in the new code, isolating it to the pre-existing string representation. Logged as a known limitation (`STATUS.md`) rather than fixed here, since a real fix needs `ArrayBuffer`/`TypedArrays` (0% implemented, its own separate, larger piece of work).

## Decision

`fetch(url: string): Promise<Response>` — GET only, no custom method/headers/request body, no `Request`/`Headers` objects (all explicitly out of scope for this pass). `Response` = `{ status: number, ok: boolean, body: string }` plus `text()` (returns `body` as-is) and `json()` (parses `body` via the same machinery `JSON.parse` uses, into whatever type the declaration context provides, defaulting to a plain string otherwise). A non-2xx status still resolves normally; a network-level failure throws a catchable `Error`.

## Implementation notes

- `codegen/llvm/types.go`: `IsResponse` field on `Type`; `ResponseType()` constructor; `"Response"` added to `ResolveTypeName`'s switch (found missing during manual testing — without it, `const r: Response = ...` failed with `field access on non-object`, since the type name simply didn't resolve to anything object-shaped at all).
- `codegen/llvm/runtime.go`: `ensureFetch()` — declares the curl C API, a `__kml_curl_write_cb` growable-buffer callback (realloc-doubling, floor 64 bytes, always kept null-terminated), and the `__kml_fetch` driver function; calls `e.requireLink("curl")` (`ADR-00020`). Lazy one-time `curl_global_init` guarded by a runtime global flag, since KML programs have no explicit startup-hook phase to call it from unconditionally.
- `codegen/llvm/emit_fetch.go` (new file): `isResponseMethodName`, `emitFetch` (builds the Response struct from `__kml_fetch`'s result, wraps it as an already-resolved `Promise<Response>` by hand — **must** use `emitInstr`, not `emitAlloca`, for the promise-slot `malloc`, since unlike `emitAsyncPrologue` (called exactly once at a real async function's entry), `fetch()` can appear anywhere, including inside a loop or conditional; using `emitAlloca` would have hoisted the malloc to the function's unconditional entry block regardless of how many times or where `fetch` is actually called — caught and fixed before it shipped, not found later), `emitResponseBody`, `emitResponseCall` (generic dispatch for `text()`/`json()` outside a declaration context), `emitResponseJSON` (the declaration-context path, taking a target type).
- `codegen/llvm/emit_call.go`: `emitJSONParse` split into a thin wrapper plus the reusable `emitJSONParseValue`; dispatch added for bare `fetch(...)` and for `.text()`/`.json()` calls (guarded by `IsResponse`), following the exact placement/shape of the existing `Date` dispatch block.
- `codegen/llvm/emit_exprs.go`: `inferExprType` gained a `fetch` case (→ `PromiseOf(ResponseType())`) and `text`/`json` cases (→ `TypePtr`, matching bare `JSON.parse`'s own default); the untyped-declaration switch and the scalar-declaration `JSON.parse`-special-case block both gained the two fixes described in Investigation.
- `codegen/llvm/emit_objects.go`: `emitObjectVarDecl` gained the `*ast.AwaitExpression` case (also described in Investigation) and a `response.json()` case mirroring the existing `JSON.parse` one.
- `main.go` / `compiler_test.go`: both updated to append `-l<lib>` flags from the new `LinkLibs()` (`ADR-00020`) — required for this feature's own tests to link at all.

## Side effects discovered

The three gaps described in Investigation (untyped `fetch(...)` inference, `AwaitExpression` in object declarations, the binary-body truncation) — the first two fixed in this same pass; the third logged as a known, documented limitation rather than fixed (out of scope — needs `ArrayBuffer`/`TypedArrays`).

## Verification

- `go build .`, `go vet ./...`, `go test ./...`, `make examples` (all pass, examples 44/44).
- Manual repros against both `httpbin.org` and a local Go `httptest`-backed server during development: status/ok for 200/404/500/redirect-followed responses; `.text()`/`.json()` (flat objects) both correct; untyped `const p = fetch(url); const r = await p` infers correctly; `fetch()` called repeatedly inside a `for...of` loop returns distinct, correct results per iteration (the `emitAlloca`-vs-`emitInstr` fix, specifically verified); a genuine DNS failure throws and is catchable via `try`/`catch`, and prints `"Uncaught: ..."` + exits 1 when not caught; a 40,000-byte all-text body round-trips exactly, isolating the binary-truncation issue to string representation, not the new buffer-growth code.
- New tests in `compiler_test.go`, using a local `httptest.Server` (not a real external URL, so the suite stays deterministic and offline-capable while still exercising the real libcurl path against a real local TCP server): `TestE2EFetchStatusAndText`, `TestE2EFetchNotFoundHasOkFalse`, `TestE2EFetchServerErrorHasOkFalse`, `TestE2EFetchFollowsRedirects`, `TestE2EFetchJSONIntoTypedTarget`, `TestE2EFetchUntypedInference`, `TestE2EFetchLargeBody`, `TestE2EFetchNetworkFailureThrows`, `TestE2EFetchWrongArgCountRejected`, `TestE2EFetchFieldAccessOnNonResponseRejected`.
- New example `examples/fetch/fetch.ts` — the one example in this repo that needs real network access to run (documented explicitly in its own header comment as an expected, not-a-bug source of `make examples` failure on an offline machine), verified against the real `httpbin.org` service.
- `STATUS.md`: `fetch(url)` and `Response` flipped to ✅ in the Networking table (with `Request`/`Headers`/custom method/headers/body explicitly still ❌); removed `fetch` from the Phase 5 roadmap; coverage numbers updated; the binary-body-truncation limitation added to Known Limitations & Bugs; top summary note added for the new conditional `libcurl` build dependency.
