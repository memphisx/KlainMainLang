# ADR-00031: Implement setTimeout/clearTimeout/setInterval/clearInterval

- **Relations**: Implements TDD-00002
- **Date**: 2026-07-12

## Context

Looked into how to move forward with `setTimeout`/`setInterval` â€” `STATUS.md`'s own Timers row had always said "requires an event loop," lumping it in with the much bigger, still-unstarted general-purpose (I/O-multiplexing) event loop the HTTP Server scoping section calls out as its own biggest blocker. Concluded timers only need a much smaller, self-contained piece of infrastructure (a sleep-until-next-due queue, no socket/file-descriptor polling involved at all), wrote that scoping up in `STATUS.md` first (`Timers â€” Design Notes`), and built it once the scope was confirmed.

## Investigation

- **Confirmed the exact heap/queue shape before writing any IR**: a global growable array of fixed-size 32-byte entries (`{ i64 id, i64 fireAtNs, i64 intervalMs, ptr closureHdr }`), reusing the same `{ ptr, i64, i64 }` realloc-doubling growable-buffer shape already proven three times over (`__kml_fetch`'s curl write callback, `__kml_exec_file_sync`'s stdout capture, `__kml_fs_readdir`'s entry list) â€” this is the first use of that shape for fixed-size *structs* rather than raw bytes or `ptr`s, but a mechanical extension, not new ground. `intervalMs = -1` doubles as the "cancelled / done, never consider again" sentinel, chosen over a separate flag field specifically so every field stays a plain i64/ptr with zero padding ambiguity to reason about (unlike, say, `struct dirent`'s cross-platform layout in `ADR-00027`, this is this compiler's *own* struct, but keeping it padding-free by construction was still worth the small design discipline).
- **Where the drain loop plugs in**: `EmitProgram` (`emitter.go:409`, at the time) emitted the top-level statement list directly followed by `ret i32 0`. Added a call to the drain function immediately before that `ret`, gated behind a `usedTimers` flag (via `emitInstr`, which already no-ops on dead code) so a program that never calls a timer function pays nothing, and so a program that calls `process.exit()` as its last top-level statement correctly skips draining â€” matching real Node's own behavior of never running pending timers after an explicit exit.
- **A real, previously-undiscovered bug, caught immediately by the first smoke test â€” entirely unrelated to timers.** `console.log("tick " + count)` (a string literal concatenated with a `number`) crashed at the `clang` verification step (`'%tN' defined with type 'i64' but expected 'ptr'`). Root cause, traced in `emitBinary` (`emit_exprs.go`): the generic `right = e.coerce(right, left.Ty)` step assumed both operands of `+` were already the same representation, so a string-typed left operand with a numeric right operand just reinterpreted the raw `i64` as if it were already a `ptr` â€” passing it straight into `strlen()` â€” instead of stringifying it first. Confirmed both directions were broken (`"x" + 5` and `5 + "x"`), and that `` `tick ${count}` `` (template literals) already did this correctly via `emitValueToString`, confirming the fix should reuse that exact conversion. Fixed by detecting "exactly one operand is string-typed" before the generic coerce step and stringifying the other operand first, for `"+"` specifically (real JS's own rule â€” other operators like `<`/`-` have different coercion semantics, out of scope for this fix).
- **A second, deeper, also entirely unrelated bug â€” found while writing the idiomatic self-cancelling-interval test.** `const id = setInterval(() => { ...; if (done) clearInterval(id) }, ms)` compiled and ran, but `clearInterval(id)` silently never cancelled anything â€” the interval ran away indefinitely (confirmed directly: an unbounded test run produced 380KB of output before being killed). Traced to an interaction between `emitVarDecl` and `ADR-00001`'s closure-capture-promotion mechanism: `emitVarDecl` allocates `id`'s storage and calls `e.define()` *before* evaluating the initializer (necessary â€” otherwise a closure inside the initializer couldn't resolve `id` as an identifier at all), then evaluates the initializer (which, in evaluating `setInterval`'s closure argument, captures `id` â€” triggering `ADR-00001`'s boxing: a *new* heap cell is malloc'd, the *current* â€” at this point still uninitialized â€” value of `id`'s original alloca is copied into it, and the scope's own `id` symbol entry is updated in place to point at this new cell instead), and only afterward stores the initializer's real result â€” but into `ptrName`, a Go-level variable captured *before* the initializer ran and the promotion happened, i.e. the now-orphaned original alloca nothing reads from anymore. The real `setInterval` return value silently landed somewhere unreachable; the closure's captured cell forever held the pre-initialization garbage it was seeded with. Fixed by re-resolving `v.Name`'s current `Symbol` via a fresh `e.lookup()` immediately before the final store, instead of trusting the `ptrName` captured before the initializer ran.

## Decision

`setTimeout(callback: () => void, delayMs?: number): number`, `clearTimeout(id: number): void`, and the same shape for `setInterval`/`clearInterval` â€” bare global functions, not a namespace (matching real JS/Node). Callback must be a zero-argument, `void`-returning closure; a bare reference to a top-level named function isn't supported as a value yet (a pre-existing, general limitation of this compiler â€” no callback-accepting API here supports that today, not something specific to timers). An active `setInterval` that's never cleared keeps the process running indefinitely, matching real Node â€” the first time that's true anywhere in this compiler. Also fixed, as bugs found along the way: `"+"` with a mixed string/non-string operand pair, and a variable's own initializer capturing itself in a closure.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureTimerRuntime()` â€” the four global queue variables, `__kml_monotonic_ns` (reusing the existing `ensureClockGettime()`/`monotonicClockID()` singletons from `performance.now()`), `__kml_timer_schedule`, `__kml_timer_clear`, `__kml_timer_drain` (the linear-scan-for-soonest-due loop, `nanosleep`, the `call void (ptr) %fp(ptr %ep)` closure-call convention already proven by the `qsort` trampolines, and the reload-everything-after-the-call-returns care needed since the callback may itself grow/realloc the queue or cancel/reschedule entries).
- `codegen/llvm/emitter.go`: `usedTimers` flag; `EmitProgram`'s drain-loop call site before the final `ret i32 0`.
- `codegen/llvm/emit_timers.go` (new file): `emitSetTimeout`/`emitSetInterval`/`emitClearTimer`, plus shared `timerCallbackPtr` (validates the zero-arg/void-return callback shape) and `timerDelayArg` (optional-argument resolution, defaulting to `0`).
- `codegen/llvm/emit_call.go`: dispatch added to the existing bare-global-function switch (alongside `fetch`/`btoa`/`parseInt`).
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s bare-identifier-call switch gained `setTimeout`/`setInterval` â†’ `TypeI64`; `emitBinary` gained the mixed-string-operand stringify-then-concatenate branch (the first bug fix); `emitVarDecl`'s general initializer-store path now re-resolves the variable's current `Symbol` before its final `store` instead of trusting the pre-initializer-evaluation pointer (the second bug fix).

## Side effects discovered

Both bugs described in Investigation above, fixed in this same pass:
1. `"+"` silently producing invalid IR for exactly one string-typed operand â€” affects any program mixing a string and a non-string with `+`, unrelated to timers except that writing a natural timer example (printing a tick count) immediately exercised it.
2. A variable's own initializer capturing itself in a closure silently never seeing the real assigned value â€” affects the general "self-referential closure in a declaration" pattern, not just timers, but the idiomatic self-cancelling-interval pattern is exactly the shape that surfaces it, so this ADR is where it was found and fixed.
