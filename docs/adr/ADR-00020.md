# ADR-00020: Link-flags plumbing (compiled programs can depend on external libraries)

- **Status**: Completed
- **Date**: 2026-07-12

## Context

Picked this up as the prerequisite for `fetch` (`ADR-00021`), once I'd settled on `libcurl` as the HTTP client library rather than shelling out to a helper process or embedding a cgo-wrapped Go `net/http` shim (considered and rejected — see the Investigation below).

## Investigation

- **Checked what actually links compiled programs today, rather than assuming.** `main.go`'s build step was fully hardcoded: `exec.Command("clang", "-O2", llFile, "-o", outBin)` — no `-l` flags, ever, for anything. Every C dependency so far (`malloc`, `sscanf`, `gmtime`, `sprintf`, …) is plain libc, implicitly linked by clang's default driver behavior. `fetch` needs `libcurl` — the first feature needing an external, non-libc library — so this gap had to be closed before writing any `fetch` codegen, not discovered partway through it.
- **Considered whether "use Go's own `net/http`" could sidestep this entirely** (a genuine question raised before committing to `libcurl`). Two ways that could work: (1) embed a cgo-wrapped Go HTTP shim as a C-shared library — rejected, since `net/http` can't be separated from the full Go runtime (goroutine scheduler, GC), meaning every compiled program using `fetch` would embed an entire second runtime just to make one HTTP call; (2) shell out to a companion Go binary via `popen` at run time — genuinely avoids needing any new link-flag mechanism at all (`popen` is plain libc), but trades it for a *worse* dependency: the compiled binary would need to locate and execute a second executable at run time, on whatever machine it ends up running on, which is a new kind of fragility nothing else this compiler produces has, plus real per-request fork/exec latency (a bad fit for the microservice use case this project prioritizes). Chose `libcurl` — a lean, single-purpose C library, linked once at compile time, keeping "one compiled binary, fully self-contained" true.
- **Decided this should be a general mechanism, not a `libcurl`-specific special case.** `fetch` won't be the last feature needing a native library — `WebSocket`, `crypto.subtle`, and others are already sitting in `STATUS.md`'s roadmap with the same "needs linking a library" note. Building a one-off "if fetch is used, hardcode `-lcurl`" branch in `main.go` would mean redoing this same design exercise for the next one. Instead: a general `requireLink(lib string)` method on `Emitter`, called by whichever `ensure*()` helper needs it (today, only `ensureFetch()`), collecting into a set; `main.go` reads it back via a new `LinkLibs()` accessor after `EmitProgram` and appends `-l<lib>` for each. This follows the same "declare a dependency once, from wherever it's actually needed" shape the existing `ensure*()`/`used*` pattern already established for IR declarations — just extended to also cover link-time flags, not only compile-time text.
- **Considered, and rejected, unconditionally adding `-lcurl` to every build.** That would break compiling *any* program on a machine without `libcurl`'s library/headers installed, even one that never calls `fetch` — a real regression for the "plain libc, no dependencies" property every other program has enjoyed until now. Gating it per-program (only when `ensureFetch()` actually ran) keeps that property intact for everyone except programs that opt in by using `fetch`.
- **Found the test harness has its own, separate hardcoded clang invocation.** `compiler_test.go`'s `buildBinary` (used by every E2E test, not just new ones) calls `clang` directly, independently of `main.go` — meaning fixing only `main.go` would have left every future `fetch`-using E2E test failing to link with undefined `curl_*` symbol errors, without a single line of test code hinting at why. Fixed both call sites in the same pass, in the same way (read `LinkLibs()`, append `-l<lib>` per entry) — necessary for `ADR-00021`'s own tests to work at all, not a separate follow-up.

## Decision

`Emitter.requireLink(lib string)` collects needed external libraries into a set; `Emitter.LinkLibs() []string` returns them sorted (for a reproducible build command). `main.go` and `compiler_test.go`'s `buildBinary` both read `LinkLibs()` after `EmitProgram`/emitting IR and append `-l<lib>` to their respective `clang` invocations. No library is ever linked unless something in the compiled program actually required it.

## Implementation notes

- `codegen/llvm/emitter.go`: new `linkLibs map[string]bool` field, `requireLink`/`LinkLibs` methods (added `sort` to the file's imports for `LinkLibs`'s deterministic ordering).
- `main.go`: build the `clang` argument slice dynamically (`-O2`, the `.ll` file, `-o`, the output name, then one `-l<lib>` per entry in `em.LinkLibs()`) instead of the old fixed four-argument call.
- `compiler_test.go`: identical change to `buildBinary`'s `clang` invocation.
- `--emit-llvm` mode (prints IR to stdout, skips `clang` entirely) is unaffected — a user piping that output into their own build step is already responsible for their own link flags, same as before.

## Side effects discovered

None beyond the test-harness gap already described above (found and fixed in the same pass, since `ADR-00021`'s tests depend on it working).

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- Confirmed directly with `otool -L` on two compiled binaries: a program that never calls `fetch` has zero `libcurl` link dependency; a program that does links it correctly.
- Confirmed (via `./KlainMainLang --emit-llvm`) that a non-`fetch` program's IR contains zero `curl` references at all — the gating is real, not just a no-op flag.
