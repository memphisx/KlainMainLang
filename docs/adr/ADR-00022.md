# ADR-00022: Implement import/export (named, declarations-only, V1)

- **Status**: Completed
- **Date**: 2026-07-12

## Context

Picked this up as the next feature after `fetch` (`ADR-00021`) — multi-file compilation was flagged in `STATUS.md`'s own roadmap as "the biggest architectural change" this compiler had left to make, since every feature up to this point assumed a single file, a single flat global scope, and a single emitted `main()`.

## Investigation

- **The first real decision: separate compilation units (linked at the object/binary level) vs. whole-program, single-IR-module compilation.** The former is how C/Rust/Go actually do multi-file compilation — each file becomes its own translation unit, with a name-mangling scheme and a real link step resolving cross-file symbol references. Considered it and rejected it for this project specifically: it needs stable external linkage names, forward declarations of symbols defined in other files, and an actual multi-step build orchestration (N `.ll` files → N `.o` files → one link step) replacing today's single `clang` invocation — a large jump in build-pipeline complexity for a project with no actual need for separate compilation (no incremental-build story exists today either, so this wouldn't be a regression, just not a win). Chose whole-program compilation instead: parse every file reachable from the entry file's imports, merge them into one combined `*ast.Program`, and hand that to the *entirely unmodified* `codegen/llvm` emitter — one `.ll`, one `clang` call, one `main()`, exactly like today, just fed a bigger merged AST. This is architecturally the cheapest option, and it's exactly how a bundler resolves imports, just done at the AST level here instead of producing JS text.
- **The second, more consequential decision, made explicitly rather than defaulted into: should imported files support real ES-module execution semantics (top-level side effects, running once, in dependency order) or be restricted to declarations only for now?** Confirmed this specific question directly before writing any code, since it changes what the feature even *is*: real ES modules run a file's top-level code exactly once, the first time it's imported, in dependency order, with the usual caveats around circular imports and temporal-dead-zone-style access-before-initialization. Building that correctly is a real, separate piece of design work (run-once guards, dependency-ordered execution, deciding what happens when a cycle means a module is used before its own top-level code has finished running). Decided — deliberately, not as an oversight — to restrict imported (non-entry) files to declarations only for this pass: functions, `const`/`let`/`var`, interfaces, type aliases, enums, and their own imports; no bare executable statements. Only the entry file's own top-level statements become the program's actual runtime behavior. This sidesteps the entire module-execution-order design space for now, in exchange for the (very common) case of sharing types/functions across files working correctly today. **Explicitly flagged as revisit-later**: build the fuller, real-ES-modules-shaped version eventually, possibly gated behind a compiler configuration flag so callers can choose between today's simpler behavior and full module-execution semantics once both exist — not simply "declared out of scope forever."
- **The third decision, which fell naturally out of the second: how should cross-file name resolution and scoping work?** A fully faithful implementation would give each file its own private scope (so two unrelated files could each declare a same-named helper function without colliding), requiring per-file symbol registries, internally-mangled LLVM names, and an explicit alias-resolution step for `import { a } from './x'` binding the name `a` into the importing file's own scope. Given the declarations-only restriction removes the temporal hazard that makes module scoping *matter most* in real JS (no "did this run yet" question to answer), chose a much simpler model instead: **one flat, globally-unique namespace across every merged file**. Concretely, this means `import` statements do no renaming or scoping work at all — they exist purely as a compile-time *validation* step (does the referenced file exist? does it actually declare and export that name?), and the actual cross-file reference just works because the name is already correct and already unique in the merged program. This is a real, meaningful simplification compared to true per-file module scope (documented explicitly in `STATUS.md` as a **revisit-later** item, not silently accepted as "how modules work now") — but it made the whole feature dramatically smaller: **zero changes needed anywhere in `codegen/llvm`.** The entire feature lives in a new pipeline stage that runs between parsing and codegen.
- **Given the flat-namespace design, aliasing (`import { a as b }`) needed a decision of its own.** For `b` to work inside the importing file, *something* needs to make the identifier `b` resolve to whatever `a` refers to — the cleanest mechanism would be an AST-level rename of every occurrence of `b` in the importing file. Rejected doing this naively: a blind textual/AST rename risks colliding with local shadowing (a parameter or local variable also happening to be named `b` for unrelated reasons) unless the rename is properly scope-aware — real complexity, for a feature (renaming an import for readability) that isn't essential to "basic import/export functionality." Chose to parse `as` correctly (so real TS-shaped code gets a clear, specific error rather than a confusing raw parse failure) but reject it at the resolver stage with an explicit "not yet supported" message, rather than attempt an unsafe rename.
- **Verified circular imports actually work under this design, rather than assuming or banning them.** Since names are flat and global (no temporal "has this module run yet" question under the declarations-only restriction), two files that mutually import each other's exported functions should, in principle, "just work" via the same two-pass forward-reference registration the single-file emitter already does for functions declared in any order. Verified this directly rather than trusting the reasoning alone: two files, each importing and calling a function from the other, compiled and ran correctly, printing the expected values from both directions. Also verified the "diamond" case (two files both importing from a shared third file) — the resolver's path-memoization (keyed by absolute file path, not by import specifier text) means the shared file is parsed and merged exactly once, not duplicated or dropped.
- **`from`/`as` were kept contextual (matched as plain `IDENT` tokens with a literal-string check in the parser), not added as reserved lexer keywords** — following the exact precedent this parser already established for `interface`/`type`/`enum` (all contextual, not reserved). Real TypeScript/JS also treats `from`/`as` as valid ordinary identifiers everywhere outside an import/export clause; reserving them fully would have been a small, avoidable deviation from real JS with no upside. `import`/`export` themselves, by contrast, *are* fully reserved in real JS too, so adding them as true keyword tokens carries no such risk.

## Decision

**Scope**: `export function`/`const`/`let`/`var`/`interface`/`type`/`enum` (a declaration-level modifier only); `import { a, b } from './relative/path'` (named imports only, relative paths only, `.ts` auto-appended if omitted, resolved against the importing file's own directory). Whole-program compilation: every file transitively reachable from the entry file's imports is parsed and merged into one `*ast.Program`; only the entry file's own top-level statements execute; every other merged file may only contain declarations and imports. All top-level declaration names must be unique across every merged file (no true per-file scope yet). Import aliasing is parsed but rejected with a clear error. `export default`, `import * as ns`, re-exports, and bare/package-style imports are all out of scope.

## Implementation notes

- `lexer/token.go`: `IMPORT`/`EXPORT` added as real keyword tokens; `from`/`as` deliberately *not* added — matched contextually in the parser instead.
- `ast/nodes.go`: `ExportDeclaration{Decl Statement}` (a pure wrapper, unwrapped by the resolver before codegen ever runs); `ImportSpecifier{Imported, Local string}` and `ImportDeclaration{Specifiers []ImportSpecifier, Source string}`. Both node types are documented at the type level as being fully consumed by the resolver — `codegen/llvm` never receives either.
- `parser/parser.go`: `parseImportDeclaration` (specifier list, contextual `from`/`as` handling, string literal source); `parseExportDeclaration` (parses whatever follows via the existing `parseStatement` dispatch, then requires the result to be one of the five declaration types, erroring otherwise — e.g. `export console.log(...)` is rejected at parse time, not silently accepted).
- `resolver/resolver.go` (new package): `ResolveProgram(entryPath string) (*ast.Program, error)` — the entire feature's logic. Visits the entry file and its transitive imports (memoized by absolute path, safely handling cycles by treating an in-progress file as "already being handled" rather than re-descending or erroring); validates non-entry files are declarations-only; validates each import's specifiers exist and are exported in the resolved file; checks global name-uniqueness across every merged file; merges everything (non-entry files' declarations first, in visitation order, then the entry file's own full statement list), dropping `ImportDeclaration` nodes and unwrapping `ExportDeclaration` nodes along the way. The package doc spells out the full V1 scope list (mirrored into `STATUS.md`'s new Modules section) so it doesn't only live in this ADR.
- `main.go`: replaced the direct `os.ReadFile` + `parser.Parse` call with `resolver.ResolveProgram(inFile)` — the resolver does its own file reading internally, starting from the entry path.
- `compiler_test.go`: found and fixed the same class of gap `ADR-00020` already found once for `fetch` — `buildBinary`'s hardcoded `clang` invocation would have been a second, independent place needing the `LinkLibs()` fix, but it already had it from that ADR, so no new gap here. Added `writeMultiFile`/`resolveMultiFile`/`buildBinaryMultiFile`/`assertMultiFileOutput` test helpers (write a `map[string]string` of relative-path→source into a temp directory, then resolve/build/run from there) and factored `assertOutput`'s line-by-line comparison into a shared `compareLines` helper so both single-file and multi-file assertions use identical, already-proven comparison logic.

## Side effects discovered

None — this feature was deliberately designed (see Investigation) to avoid touching `codegen/llvm` at all, specifically to keep its blast radius contained to new code (`resolver/`) plus small, additive changes to the lexer/parser/AST/`main.go`/test harness. No existing behavior needed to change.
