# ADR-00053: Implement Map.entries()/.forEach()/.clear() and Set.forEach()/.clear()

- **Date**: 2026-07-14

## Context

`STATUS.md`'s Object/Collections table tracked five `Map`/`Set` methods as ❌: `Map.entries()`, `Map.forEach()`, `Map.clear()`, `Set.forEach()`, `Set.clear()`. All five were self-contained, low-complexity gaps in an otherwise-complete `Map`/`Set` implementation (`set`/`get`/`has`/`delete`/`keys`/`values`/`size` were already done) — a natural "quick wins" pass to round out the collection.

## Investigation

Read `codegen/llvm/emit_collections.go` (`emitMapCall`/`emitSetCall`, the existing method-dispatch switches) and the backing runtime helpers in `runtime.go` (`ensureMapStrHelpers`/`ensureMapNumHelpers`). Both `Map` and `Set` share one underlying representation: a heap struct `{i64 size, i64 cap, ptr keys, ptr vals}` (`Set` stores its elements as map keys with an unused `0` value, per the existing `add` implementation). `keys()`/`vals()` already exist as runtime helpers returning `{ptr, i64}` arrays typed to the map's actual key/value types — enough to build all five new methods without any new runtime IR helper functions, just new Go-side codegen that walks those same arrays.

Two non-obvious things surfaced while designing `entries()`:
- This compiler has no tuple type, so a real JS `[key, value]` pair isn't representable. `Object.entries(obj)` (`emit_objects.go`'s `emitObjectEntries`) already solved the identical problem by returning `{key: string, value: string}[]` — a heap-allocated `{key, value}` object per entry, iterated with `for (const e of entries)` then `e.key`/`e.value`. Reused that exact convention for `Map.entries()`, with one difference: `Object.entries` stringifies every value (since an object's fields can be heterogeneously typed), while `Map.entries()` keeps the map's actual, uniform `K`/`V` types (no stringification needed, since a `Map<K,V>`'s keys/values are already homogeneous).
- `Object.entries`'s implementation is a **compile-time** Go `for` loop, since an object's field list is known at compile time. A `Map`'s size is only known at runtime, so `Map.entries()` needed a genuine IR loop (same `alloca`/`freshLabel`-based shape `emitArrayForEach` already uses) walking the `keys()`/`vals()` arrays in lockstep — not a copy of `Object.entries`'s codegen, just the same target shape reached a different way.

A real name collision was found and fixed while wiring up dispatch: `codegen/llvm/emit_call.go` checked `mem.Property == "forEach"` (routing to `emitArrayForEach`) *before* checking whether the receiver was actually a `Map`/`Set` symbol, so `m.forEach(...)` would have silently gone through the array codegen path and failed (`resolveArrayForHOF` on a non-array symbol) instead of reaching the new `Map`/`Set` handling at all. Fixed by moving the `IsMap`/`IsSet` dispatch block before the `"forEach"` name check — the two other places that guard against the exact same kind of array/string method-name collision (e.g. `"slice"`, `"indexOf"`, `"at"`) already use an inline `e.inferExprType(mem.Object).IsArray` guard instead, since those are checked once, ungrouped; `forEach` needed a reordering instead because the `Map`/`Set` dispatch was already a self-contained block placed after it, not a single `if`.

## Decision

Added, in `codegen/llvm/emit_collections.go`:
- `mapKeysAndVals(mapPtr, strKey) (keysPtr, keysLen, valsPtr string)` — a small shared helper wrapping the two `keys()`/`vals()` runtime calls and their `extractvalue`s, since both `emitMapEntries` and `emitMapForEach` need the same two parallel arrays.
- `emitMapEntries(mapPtr, strKey, keyTy, valTy) (Value, error)` — builds `{key: K, value: V}[]` via a runtime loop over the `keys()`/`vals()` arrays, mallocing one entry struct per pair (same `ObjectType([]Field{...})` + `StructIR()`/`StructSize()` machinery `emitObjectEntries` already established).
- `emitMapForEach(mapPtr, strKey, keyTy, valTy, cb Callback) (Value, error)` — same loop shape, calling the callback as `fn(value, key)` per entry (real JS's order, with the 3rd `map` argument dropped — mirroring `Array.forEach`'s own `(elem, index)` simplification, which already drops the 3rd `array` argument real JS also passes).
- `emitSetForEach(setPtr, strElem, elemTy, cb Callback) (Value, error)` — loops over the set's `keys()` array (a set's only backing array), calling `fn(element[, element])` — real JS's actual `Set.prototype.forEach(value, value, set)` signature passes the value twice; mirrored here as a 2nd argument when the callback declares one.
- `.clear()` on both `Map` and `Set`: no new runtime helper needed — `store i64 0, ptr %mapPtr, align 8` directly resets the `size` field in place. Doesn't free or reallocate the backing `keys`/`vals` arrays (consistent with this compiler's established "leak by design, `Memory.free(x)` is the only manual escape hatch" model — see `STATUS.md`'s Memory Management section); the map/set is immediately reusable afterward since `cap` is untouched and the next `set`/`add` just overwrites from index 0.

`inferExprType` (`emit_exprs.go`) gained a `case "entries"` in the existing `IsMap` switch, returning the same `{key, value}[]` type, so `const arr = m.entries()`-style inference works. `forEach`/`clear` didn't need cases there — they return `void`, and no other void-returning `Map`/`Set`/`Array` method has one either (confirmed `Array.forEach` has no `inferExprType` case).

## Implementation notes

- Files touched: `codegen/llvm/emit_collections.go` (new methods + 3 new switch cases each for `Map`/`Set`), `codegen/llvm/emit_call.go` (dispatch reordering fix), `codegen/llvm/emit_exprs.go` (`entries()` type inference).
- `Set.entries()` was deliberately **not** added — it was never tracked as ❌ in `STATUS.md` in the first place (only `Map.entries()` was), and real JS's `Set.prototype.entries()` returns `[value, value]` pairs, which would need the same `{key, value}`-shaped workaround for arguably little benefit over the already-existing `.values()`/`for...of`. Left out of scope, not silently dropped — noting it here since it's the one real JS `Map`/`Set` method still genuinely missing after this pass.
- `Map.forEach`'s callback hint is `[]Type{valTy, keyTy}` (value first, matching real JS) — easy to get backwards, double-checked against real JS's own `Map.prototype.forEach(callbackFn)` signature (`value` is the callback's first parameter) before implementing.

## Side effects discovered

None beyond the `forEach` dispatch-ordering bug above, which was caught and fixed during implementation (not left as a known issue) before it could ship.
