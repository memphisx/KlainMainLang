# ADR-00055: Implement Object.freeze(obj) / Object.seal(obj)

- **Date**: 2026-07-14

## Context

`STATUS.md` tracked `Object.freeze()`/`.seal()` as ❌ — the next natural item after `ADR-00054`'s `Object.assign`. Unlike the other recent Object.* quick wins, this one had a real scope question worth resolving before writing any code: this compiler's objects are fixed-shape heap structs (an interface's field list, and therefore its `malloc` size and GEP layout, is fixed at compile time), so there is no dynamic property add/delete at the language level *at all*, for any object. That means `seal`'s core guarantee ("no new/deleted properties") already holds unconditionally today — the only thing either method could meaningfully add is *blocking writes to existing fields*, and doing that correctly requires tracking "is this particular object frozen" at runtime, by the object's own identity, not by whichever variable happened to call `freeze`.

Weighed three options before writing any code: (a) real runtime enforcement via a side-table checked at every field-write site, (b) accepting freeze/seal as documented no-ops, or (c) deferring the whole thing as a TDD. Went with (a) — a no-op would be exactly the "silently wrong behavior instead of a clean rejection" failure mode this project's standing rules try to avoid, and the enforcement mechanism itself turned out simple enough (reusing the existing `Map`/`Set` runtime helpers, see Investigation) that deferring to a TDD wasn't warranted. Noted a *fully* JS-compatible dynamic property bag (real runtime add/delete) as a reasonable future direction, not a blocker for this pass — "something to be aware of," not something to research or design now.

## Investigation

- Confirmed there's no existing runtime "track a set of heap pointers" primitive to reuse directly, but `Map`/`Set`'s own backing representation (`codegen/llvm/runtime.go`'s `ensureMapNumHelpers`) already is exactly that: a linear-scan `{size, cap, ptr keys, ptr vals}` structure keyed on `i64`. A frozen-object tracker is just a `Set<number>` keyed on `ptrtoint(obj)`, so no new hand-rolled data structure was needed — just one more `ensure*()` wrapper (`ensureFrozenSet`) creating one global instance of the already-existing map-num helpers.
- Found the one place all direct object-field writes funnel through: `emitAssign`'s "Object field assignment" branch (`emit_exprs.go:482`, handling both `obj.field = val` and compound forms like `obj.field += val`, since compound assignment loads-then-stores through the same GEP). This is also the branch that already handles `pts[i].field = val` (the index expression is evaluated first via `emitExpr`, so the frozen check — keyed on the *resolved* object pointer — naturally applies per-instance regardless of how that pointer was reached).
- A second write path was found and needed the same guard: `ADR-00054`'s `emitObjectAssign` (`Object.assign(target, ...)`) stores directly into `target`'s struct via its own GEP/store, bypassing `emitAssign` entirely. Without an explicit check there, `Object.assign(frozenObj, ...)` would have silently succeeded — exactly the kind of gap this feature exists to close. Confirmed no other write path exists (spread and `Object.entries`/`.values` only *read* fields into freshly allocated structures, never mutate an existing one).
- Verified the "per-value, not per-variable" requirement directly: a repro with a `mutate(pt: Point)` function that takes a frozen object as a parameter and writes to `pt.x` — confirmed the write is blocked exactly because the check is keyed on the object's own pointer (computed from whatever expression currently holds it), not on any particular symbol.

## Decision

Added, in `codegen/llvm/runtime.go`: `ensureFrozenSet()`, declaring `@__kml_frozen_set` (a lazily-initialized global `ptr`, `null` until first use) and `__kml_frozen_set_get() -> ptr` (creates the one global `Set<number>`-shaped structure via `__kml_map_num_create` on first call, returns the existing one otherwise).

Added, in `codegen/llvm/emit_objects.go`:
- `emitObjectFreeze(args, pos)` — validates the argument is an object, `ptrtoint`s its pointer, calls `__kml_map_num_set(frozenSet, ptrAsInt, 1)`, returns the object unchanged (same pointer/type, matching real JS returning the same reference).
- `emitObjectSeal(args, pos)` — validates the argument is an object, returns it unchanged. A genuine no-op, not an approximation of one: seal's only real guarantee (no add/delete) already holds unconditionally for every object in this compiler.
- `emitFrozenCheck(ptrRef string)` — the shared guard: `__kml_map_num_has(frozenSet, ptrtoint(ptrRef))`, branching to either a `__kml_throw`-based catchable Error ("Cannot assign to read only property of a frozen object", mirroring `ADR-00044`'s array-bounds-check shape exactly — same `ensureExceptionHelpers`/`malloc`/`store`/`__kml_throw`/`unreachable` pattern) or falling through to the normal write.

Wired `emitFrozenCheck` into both write sites: `emitAssign`'s object-field-assignment branch (checked once, right after resolving the target object and field index, before either the plain-`=` or compound-op branches — covering both uniformly with one check) and `emitObjectAssign`'s target (checked once per call, only when there's at least one source — `Object.assign(frozenObj)` with zero sources never attempts a write, matching real JS not throwing for that case either).

The check is **unconditional**: every object-field write in every KML program now pays one `O(n)` lookup against the (possibly empty) global frozen set, whether or not that program ever calls `Object.freeze`. This is the same trade-off `ADR-00044`'s array bounds check already made (an always-on correctness check, not something gated behind whether the specific feature is used elsewhere in the program) — consistent with this project's general posture of favoring correctness over micro-optimizing the common case.

**Deliberately out of scope**: a real dynamic property bag (runtime property add/delete on any object, toggleable via a future compiler flag for full JS compatibility). Not designed, not researched — noted only as a `STATUS.md` roadmap entry ("Opt-in dynamic property add/delete on objects," marked Speculative) so the gap from 100% JS-compatible `freeze`/`seal` semantics is visible without implying it's queued up next.

## Implementation notes

- Files touched: `codegen/llvm/runtime.go` (`ensureFrozenSet`), `codegen/llvm/emitter.go` (`usedFrozenSet` flag), `codegen/llvm/emit_objects.go` (`emitObjectFreeze`/`emitObjectSeal`/`emitFrozenCheck`, plus the one-line call site added to `emitObjectAssign`), `codegen/llvm/emit_exprs.go` (`emitFrozenCheck` call site in `emitAssign`'s object-field branch; `inferExprType`'s `Object.<method>` switch gained `"freeze"`/`"seal"` alongside the existing `"assign"` case), `codegen/llvm/emit_call.go` (dispatch cases).
- Freezing is shallow, matching real JS: the check is keyed on the exact object pointer being written to, so freezing an outer object doesn't propagate to a nested object field — `outer.inner.x = 5` resolves `outer.inner`'s own pointer first and checks *that*, independent of whether `outer` itself is frozen.

## Side effects discovered

None beyond the `Object.assign`-bypasses-the-check gap described above, which was identified and fixed during this same implementation pass, not left as a known issue.
