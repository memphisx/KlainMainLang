# ADR-00047: Fix bitwise shift operators to use JS's 32-bit semantics

- **Relations**: Implements TDD-00008 (partially — first real Test262 test ports)
- **Date**: 2026-07-13

## Context

Last of the four bugs found in the bug-hunting pass that also produced `ADR-00044.md`/`ADR-00045.md`/`ADR-00046.md`: `1 << 63` printed `-9223372036854775808` (raw 64-bit shift), while real JS gives `-2147483648`, since the spec requires converting both shift operands to 32-bit integers (`ToInt32`/`ToUint32`) and masking the shift count to 0-31 before shifting. Flagged in `STATUS.md`'s Known Limitations table, picked up last this session, once the other three were already fixed.

## Investigation

`emit_exprs.go`'s `<<`/`>>`/`>>>` cases — duplicated identically in both `emitBinary` (plain binary expressions) and `emitArith` (compound-assignment operators, `<<=`/`>>=`/`>>>=`) — coerced both operands straight to `i64` and emitted plain `shl`/`ashr`/`lshr` with no truncation step at all, so results diverged from real JS for any shift count ≥ 32 or any operand outside the 32-bit range.

Confirmed the exact scope of the divergence, and the correct target values, using Test262's shift-operator test files already scanned in a prior session (`docs/testing/CONFORMANCE-COVERAGE.md`, `language/expressions/{left-shift,right-shift,unsigned-right-shift}`) rather than hand-deriving expected values: `S9.5_A2.1_T1.js` gives concrete `ToInt32` wraparound values for the left operand (e.g. `2147483648 << 0 === -2147483648`, `-4294967296 << 0 === 0`), `S9.6_A2.1.js` gives the equivalent `ToUint32` values for `>>>` (`-1 >>> 0 === 4294967295`, `4294967296 >>> 0 === 0`), and `S11.7.1_A4_T2.js` gives ~250 more of the same shape — used directly as this fix's verification values rather than re-deriving them by hand.

Worked out the exact JS semantics needed: for `<<` and `>>`, the left operand is `ToInt32`'d (truncate to 32 bits, keep sign); for `>>>`, it's `ToUint32`'d (same 32-bit truncation, but the result is always non-negative). The shift count (right operand) is always `ToUint32`'d then masked with `& 0x1F`, regardless of which shift operator — meaning a shift count of `32` wraps to `0` (not "shift everything out"), and negative shift counts wrap the same way (`ToUint32(-1) & 0x1F === 31`).

## Decision

Factored the identical three-case logic (previously duplicated in both `emitBinary` and `emitArith`) into one new shared helper, `emitBitShift(op string, left, right Value) (Value, error)` (`codegen/llvm/emit_exprs.go`) — named to avoid colliding with the pre-existing `emitShift` in `emit_arrays.go`, which implements `Array.prototype.shift()`, an unrelated method.

Implementation, entirely in terms of native LLVM `i32` operations (no runtime helper needed — the 32-bit truncation and masking are both compile-time-representable as plain `trunc`/`and`/shift instructions):
1. Coerce both operands to `i64` (as before, reusing the existing `coerce` path for e.g. a float-typed operand).
2. `trunc i64 ... to i32` both operands. For the left operand, this is exactly `ToInt32`'s mod-2^32 wraparound — truncation keeps the low 32 bits regardless of the original value's sign or magnitude, and interpreting those bits as a signed `i32` *is* `ToInt32`'s result by construction.
3. `and i32 <right-trunc>, 31` to mask the shift count to 0-31 — masking only depends on the low 5 bits of the bit pattern, so this gives the correct result regardless of whether the right operand's `i32` truncation is interpreted as signed or unsigned, with no separate `ToUint32` step needed.
4. Shift as a 32-bit operation: `shl`/`ashr`/`lshr i32` (the same three LLVM opcodes as before, just at 32-bit width instead of 64-bit, and with the masked shift amount instead of the raw one).
5. Extend the `i32` result back to this compiler's native `i64` `number` representation: `sext` for `<<`/`>>` (JS results are `Int32`, so e.g. `1 << 31 === -2147483648` needs to stay negative at `i64` width too), `zext` for `>>>` (JS results are always a non-negative `Uint32`, so e.g. `-1 >>> 0 === 4294967295` must not sign-extend into a negative `i64`).

`emitBinary`'s and `emitArith`'s `<<`/`>>`/`>>>` cases were both replaced with a single `case "<<", ">>", ">>>": return e.emitBitShift(...)`, eliminating the prior duplication.

## Implementation notes

- `codegen/llvm/emit_exprs.go`: new `emitBitShift` function; both `emitBinary`'s and `emitArith`'s three shift cases collapsed into one delegating case each.
- No new `Type`/`Symbol` fields, no runtime helper (`ensure*()`) needed — the fix is pure LLVM-IR-level integer-width manipulation.
- Result type stays `TypeI64` in both the `<<`/`>>` (sign-extended) and `>>>` (zero-extended) cases, consistent with how shift results were already typed before this fix — only the *value* computed changes, not its representation.

## Side effects discovered

None — this was a self-contained fix to the two duplicated shift-case blocks, as anticipated in `STATUS.md`'s original bug entry (which had already correctly scoped the fix to "both operands truncated to 32-bit, shift count masked to 0-31").

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including six new tests below) all green; `make examples` 57/57.
- Manual repros, verified directly against Test262's own pre-computed expected values: `1 << 63 === -2147483648` (not `-9223372036854775808`), `1 << 31 === -2147483648`, `-1 >>> 0 === 4294967295`, `4294967296 >>> 0 === 0`, `2147483648 << 0 === -2147483648`, `-4294967296 << 0 === 0`, `-8 >> 1 === -4` (arithmetic shift sign-extends), shift-count masking (`1 << 32 === 1`, `1 << 33 === 2`, `8 >> 33 === 4`), and the compound-assignment path (`x = 1; x <<= 31` gives `-2147483648`, same as the plain-binary path).
- New tests in `compiler_test.go`: `TestE2ELeftShiftInt32Overflow`, `TestE2EUnsignedRightShiftUint32`, `TestE2ELeftShiftOperandToInt32Wraparound`, `TestE2EShiftCountMaskedTo5Bits`, `TestE2ERightShiftArithmeticSignExtends`, `TestE2EShiftCompoundAssignmentUsesInt32Semantics`.
- Extended `examples/basics/bitwise.ts` with a new section demonstrating the 32-bit-overflow and shift-count-masking behavior explicitly (`1 << 31`, `-1 >>> 0`, `1 << 32`, `1 << 33`); confirmed the file's pre-existing small in-range shift values (which never exceeded 32-bit range, so were already correct under the old 64-bit code) are unaffected — ran the full updated example manually, all output correct.
- `STATUS.md`: removed the now-fixed row from Known Limitations & Bugs and the Roadmap's "Next up" table — the last of the four bugs from this pass, closing it out.
