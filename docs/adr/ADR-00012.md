# ADR-00012: Implement shorthand object properties { x }

- **Status**: Completed — extended by ADR-00041
- **Date**: 2026-07-11

## Context

Picked this up as the final remaining Phase 2 roadmap item, closing out the phase (`ADR-00001` through this one).

## Investigation

- **Parser location.** `parseObjectLiteral` (`parser/parser.go`) unconditionally expected `IDENT COLON <value>` for every property — no path for a bare `IDENT` meaning "value is the identifier itself." A minimal change: after consuming the key `IDENT`, check whether the next token is `:`; if not, build the value as an `Identifier` node of the same name instead of requiring a colon.
- **Confirmed no downstream changes needed.** Since the fix produces exactly the same AST shape as the existing `{ x: x }` form (an `ObjectProperty{Key: "x", Value: &ast.Identifier{Name: "x"}}`), every codegen path that already handles object literals (`emitObjectLiteral`, `inferObjectType`, `emitJSONStringifyObject`, etc.) needed zero changes — this really is pure parser sugar, as the roadmap entry said.
- **A pre-existing, unrelated limitation surfaced while testing.** `function makePoint(x, y) { return { x, y } }` (no explicit return-type annotation) fails with `field access on non-object` when the caller reads a field off the result. Isolated this precisely: replacing the shorthand `{ x, y }` with the old explicit `{ x: x, y: y }` syntax fails identically, confirming this is a pre-existing type-inference gap (a function with no declared return type can't have its inferred return type recognized as an object shape by callers), not something introduced by shorthand properties. It's related to, but distinct from, the already-tracked "closure returned from an unannotated arrow function" limitation (`STATUS.md`) — that one is specifically about closures; this one is about plain object literals. Not fixed here — it's a separate, pre-existing issue, and every one of this ADR's own tests and the new example use an explicit return-type annotation (`function f(...): Point { return { x, y } }`), which works correctly.
- **Also confirmed while auditing `STATUS.md` for this change:** a markdown formatting slip from an earlier ADR's edit this session (a table row missing its leading `|`, breaking that row's rendering) — fixed alongside this change since it was in the same section I was already editing, not left for a future pass to rediscover.

## Decision

`parseObjectLiteral` (`parser/parser.go`): after consuming the property-key `IDENT`, check for `:`. If present, parse the value as before (`parseAssignment`). If absent, the value is synthesized as `ast.NewIdentifier(keyTok.Literal, posOf(keyTok))` — the same node a hand-written `{ x: x }` would produce, so shorthand and explicit properties can be freely mixed (`{ a: 1, b, c: 3 }`) since they build identical AST shapes either way.

## Implementation notes

Files touched: `parser/parser.go` only (`parseObjectLiteral`).

- A markdown table-row formatting bug from an earlier edit this session (missing leading `|`) — fixed in the same `STATUS.md` pass.
