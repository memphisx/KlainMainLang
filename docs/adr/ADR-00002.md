# ADR-00002: Implement process.argv, process.exit(code), process.env

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked up as the next feature after the project's prioritization guidance called out terminal/process interaction as one of three capability areas the CLI/microservice long-term goal depends on (alongside REST API interaction and file I/O). `STATUS.md`'s "Process / CLI I/O" section (added as part of documenting that prioritization) tracked `process.argv`, `process.exit(code)`, and `process.env` as near-zero-effort, unimplemented items — this ADR implements those three (stdin reading and file read/write remain unimplemented, tracked separately).

## Investigation

- **`main()`'s signature.** `codegen/llvm/emitter.go`'s `EmitProgram` hard-coded `define i32 @main() {` — zero parameters. `process.argv` needs the C runtime's `argc`/`argv`, which the OS/CRT startup code always passes to `main` in the standard calling-convention registers (SysV/AAPCS), regardless of whether `main`'s own declared signature reads them. Confirmed this makes changing the declared signature to `define i32 @main(i32 %argc, ptr %argv)` **always safe** — no feature-detection or gating needed, since the CRT always supplies these values whether or not the callee's prototype declares params for them.
- **Existing namespace-dispatch pattern.** `codegen/llvm/emit_exprs.go`'s `emitMember` already special-cases bare-identifier-object member access for compile-time constants (`Math.PI`, `Number.MAX_SAFE_INTEGER`, enum members), and `codegen/llvm/emit_call.go`'s `emitCall` already does the same 2-level `if id, ok := mem.Object.(*ast.Identifier); ok && id.Name == "X"` dispatch for calls (`console.log`, `Math.floor`, `JSON.stringify`, etc.). Both were reused as the model for `process.argv` (member access) and `process.exit` (call).
- **Array value duality.** `string[]` is represented as `{ptr, i64}` — a data pointer plus a length. C's `argv` (`char**` / opaque `ptr`) paired with `argc` is bit-identical to that representation. This meant `process.argv` could alias the CRT's own `argv` buffer directly with zero copying, rather than allocating and populating a new array.
- **Generic fallbacks for non-identifier array expressions.** Checked `.length` (`emit_exprs.go`, the "any other expression: evaluate it, then dispatch on the result type" branch), `resolveArrayForHOF` (`emit_arrays.go`), `emitIndexPtr`/`emitIndex`, and `emitForOf` (`emit_stmts.go`) — all already have a fallback path for "any expression that evaluates to a `{ptr,i64}` aggregate," not just named identifiers. This meant that once `process.argv` returns a correctly-typed `Value{Ty: ArrayOf(TypePtr)}`, `.length`, indexing, `.map()`/`.forEach()`/etc., and `for...of` all worked immediately with **no further special-casing** — confirmed by testing each directly.
- **Nullable-return convention.** `emitArrayFind` (`emit_arrays.go`) returns a plain `elemTy` with a runtime-null sentinel when nothing matches, rather than marking the `Value`'s `Type` with the `Nullable` flag (which turned out to mostly serve `resolveType`-driven annotations and `emitValueToString`, not runtime null-checks or `??`/`typeof`). Followed the same convention for `process.env`'s `getenv()` result — a plain `TypePtr`, letting `??` and `=== null` work against the actual runtime pointer value, consistent with existing code rather than introducing new type-system behavior.
- **`@exit` was already declared elsewhere.** `runtime.go`'s `ensureExceptionHelpers()` (used for uncaught-exception termination via `setjmp`/`longjmp`) already inline-emitted `declare void @exit(i32) noreturn`, gated behind `usedExceptionHelpers`, not behind a dedicated flag. A naive independent `ensureExit()` would double-declare `@exit` (an LLVM error) for any program using both `try`/`catch` and `process.exit`.
- **`inferExprType` needed updating too, found via a real build failure.** The first working version of `process.env.HOME` compiled but failed at the `clang` step with `error: '%t13' defined with type 'ptr' but expected 'i64'` when assigned to `const home: string = process.env.HOME`. Root cause: `VarDeclaration` codegen picks the variable's alloca type via `e.inferExprType(init)`, which didn't have a case for `process.argv`/`process.env.X`/`process.env["X"]` and fell through to a numeric default. Fixed by adding matching cases to `inferExprType`'s existing `*ast.MemberExpression` and `*ast.IndexExpression` switch arms (had to merge into the **existing** `*ast.IndexExpression` case rather than add a second one — Go doesn't allow duplicate `case` types in the same `switch`, caught immediately by `go build`).

## Decision

1. **`process.argv`**: `EmitProgram` (`emitter.go`) now unconditionally declares two internal globals, `@__argv_ptr` (`ptr`) and `@__argv_len` (`i64`), and — as the very first instructions of `main`'s body, before any user statement — `zext`s `%argc` to `i64` and stores both parameters into these globals. `main`'s signature is unconditionally `define i32 @main(i32 %argc, ptr %argv)`. Doing this unconditionally (rather than gating behind "does the program actually reference `process.argv`") avoids needing a whole-program pre-scan; the cost (two stores, a fixed signature) is negligible and always safe per the investigation above. Any expression anywhere in the program — top-level code, a named function, or a closure — can read `process.argv` by loading these two globals, since LLVM globals aren't scoped to a single function.

   `process.argv` itself (`emitProcessArgv` in the new `emit_process.go`) loads both globals and builds a `{ptr, i64}` aggregate directly — no allocation, no copy, since C's `argv` layout is already exactly this compiler's own `string[]` layout. **Deliberate deviation from Node.js**: `argv[0]` is the compiled binary's own path (plain C convention), not Node's two-element prefix (interpreter path + script path) — there's no interpreter/script distinction for a native compiled binary, so replicating Node's convention would be actively misleading. Documented in `STATUS.md`.

2. **`process.exit(code)`** (`emitProcessExit` in `emit_process.go`, dispatched from `emit_call.go`'s `emitCall` alongside the existing `console`/`Math`/etc. 2-level dispatch): evaluates the argument, coerces to `i64`, truncates to `i32` (C's `exit(int)`), calls `@exit`, then emits `unreachable` — correct and intentional here, unlike the `unreachable`-after-non-terminating-code bug fixed in ADR-00001, because `exit()` genuinely never returns. Code textually after `process.exit(...)` in the same block is silently dropped by the existing `emitTerminator`/`blockDone` invariant, exactly the desired "never returns" semantics.

   To avoid double-declaring `@exit`, added a shared `ensureExit()` in `runtime.go` (new `usedExit` flag, same singleton pattern as every other `ensure*()`), and changed `ensureExceptionHelpers()` to call `e.ensureExit()` instead of inlining its own `declare void @exit(i32) noreturn`.

3. **`process.env.KEY` / `process.env["KEY"]`**: new `ensureGetenv()` (new `usedGetenv` flag) declares `declare ptr @getenv(ptr noundef)` once. `isProcessEnvExpr(expr)` (new helper in `emit_process.go`) recognizes the exact pattern `process.env` (non-optional `MemberExpression` on identifier `process`, property `env`). Dot notation (`emitProcessEnvGetStatic`) interns the property name as a compile-time string constant; bracket notation (`emitProcessEnvGetDynamic`) evaluates the index expression at runtime — both funnel into one `emitGetenvCall` that calls `@getenv` and returns a plain `TypePtr` (nullable via runtime null, per the `.find()` convention above), so `process.env.X ?? "default"` composes with the existing `??` operator with no extra work.

## Implementation notes

Files touched:
- `codegen/llvm/emitter.go` — `usedExit`/`usedGetenv` fields; `EmitProgram`'s `main()` signature and argc/argv-to-globals prelude.
- `codegen/llvm/runtime.go` — new `ensureExit()`/`ensureGetenv()`; `ensureExceptionHelpers()` now calls `ensureExit()` instead of inlining the `@exit` declare.
- `codegen/llvm/emit_process.go` (new) — `isProcessEnvExpr`, `emitProcessArgv`, `emitProcessExit`, `emitGetenvCall`, `emitProcessEnvGetStatic`, `emitProcessEnvGetDynamic`.
- `codegen/llvm/emit_call.go` — `process.exit` dispatch in `emitCall`, same shape as the existing `console`/`Math`/`JSON`/etc. blocks.
- `codegen/llvm/emit_exprs.go` — `process.argv`/`process.env.KEY` cases in `emitMember`; `process.env["KEY"]` case in `emitIndex`; matching cases added to `inferExprType`'s `*ast.MemberExpression` case and merged into its existing `*ast.IndexExpression` case.

## Side effects discovered

- The `inferExprType` gap above was caught and fixed as part of this same change (not deferred) — it's necessary plumbing for the feature to actually work for typed `const`/`let` declarations, not a separate bug.
- Verified no `@exit` double-declaration regression via a combined test: a program using both `try`/`catch` (uncaught-exception path) and `process.exit(...)` compiles and runs correctly (see Verification).

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including new `examples/process/process.ts`.
- Manual repros:
  - `process.argv` — `.length`, indexing, and `for...of` all work directly on the member-expression result (no need to assign to a local first); run with `./bin hello world` prints `3`, own path, `hello`, `world` as expected.
  - `process.exit(42)` — prints `"before"`, exits with code 42, `"after"` never printed.
  - `process.env.HOME` (dot), `process.env[key]` (bracket, dynamic key), and `process.env.MISSING ?? "default"` (nullish coalescing over a truly-unset var) all produce correct results.
  - Combined `try`/`catch` (triggering the uncaught-exception path's `@exit` declaration) + `process.exit(7)` in the same program compiles and runs without a duplicate-declaration error, confirming the `ensureExit()` refactor is correct.
- New tests in `compiler_test.go`: `TestE2EProcessArgv`, `TestE2EProcessExit`, `TestE2EProcessEnv`. These needed new test-harness helpers (existing `assertOutput`/`compileAndRun` assumed no CLI args and a zero exit code): `buildBinary` (extracted from `compileAndRun`), `compileAndRunWithArgs` (passes CLI args through), and `compileAndRunExpectExit` (captures stdout and a non-zero exit code instead of failing the test on one).
- `STATUS.md`'s "Process / CLI I/O" section and coverage tables updated (`process.argv`/`.exit`/`.env` flipped to ✅; stdin and file read/write remain ❌); Phase 5 roadmap entries for the three implemented items struck through with a pointer to this ADR.
