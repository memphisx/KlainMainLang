# ADR-00016: Implement Date setters (setFullYear, setMonth, setDate, setHours, setMinutes, setSeconds, setMilliseconds, setTime)

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the last piece explicitly called out as not-yet-implemented in the `Date` feature (`ADR-00014`), right after `Date.parse` (`ADR-00015`).

## Investigation

- **The real design fork: value semantics vs. reference semantics.** Real JS's `Date` is a heap object — calling `d.setFullYear(y)` mutates that object in place, and every other variable/field/array slot holding a reference to the *same* Date object sees the change. This compiler's `Date` (`ADR-00014`) is deliberately a plain `i64` value, copied by value everywhere exactly like `number` — there is no shared heap identity to mutate. That's a real, consequential divergence from JS semantics, not just an implementation detail, so it needed a concrete scope decision rather than being silently glossed over.
- **What "mutate in place" can mean given value semantics.** The only thing that can be mutated is a variable's own storage (its `alloca`). Checked how the existing codebase already handles this exact class of problem — array mutation. `emitPush` (`emit_arrays.go:483`) requires its receiver to be a plain `*ast.Identifier` resolved to a named `Symbol`, and returns a clean compile error ("push requires an array variable") for anything else (a field access, a call result, a literal). This is the same shape of problem: an operation that needs to write back to a storage location, not just read a value. Followed that exact precedent for Date setters rather than inventing a new pattern.
- **What happens to a setter call on a non-identifier receiver, then.** Considered silently treating it as a no-op that still returns a computed value (i.e., not erroring, just not persisting the mutation) — rejected, since that would silently diverge from real JS behavior (which *does* mutate object fields, array elements, etc. in place) in a way a program author would have no signal about. Chose to fail with a clear compile-time error instead, naming the specific receiver shape needed, consistent with this project's standing preference for failing cleanly over producing quietly-wrong behavior.
- **Calendar-field math: reusing, not duplicating, `Date.parse`'s logic.** A setter's job is: decompose the current timestamp into its 8 calendar fields (already available via `emitDateDecompose`, `ADR-00014`), override one field with the new argument, then recompose. `ADR-00015`'s `__kml_date_parse` already contained exactly this "recompose calendar fields into a timestamp" logic inline. Rather than copy it, factored it out into a new shared runtime helper, `__kml_date_compose`, called by both `__kml_date_parse` and the new setters — reducing the calendar-math logic that needs to be correct to exactly one place.
- **Whether the day/month-overflow-and-carry behavior real JS has (e.g. `setMonth(12)` rolling into next January, `setDate(32)` on a 31-day month rolling into next month) would work automatically or need extra handling.** This wasn't obvious up front — Howard Hinnant's `days_from_civil` algorithm (already in use for `Date.parse`, `ADR-00015`) is documented by its author as correct for arbitrary integer month/day values, not just the conventional `[1,12]`/`[1,31]` ranges, but this was verified directly rather than trusted on faith: compiled and ran `new Date(0)` (Jan 1 1970) then `.setMonth(12)` → got `1971-01-01T00:00:00.000Z` (correct: month 12, 0-indexed, is January of the following year), and separately `.setDate(32)` on the same starting date → got `1970-02-01T00:00:00.000Z` (correct: January has 31 days, so day 32 rolls into February 1st). Both match real JS's own rollover behavior exactly, with zero extra normalization code needed — a pleasant confirmation, not an assumption.
- **Month-indexing convention mismatch to watch for.** `emitDateDecompose`'s `month` field is 0-indexed (JS's `getMonth()` convention: January = 0), but `__kml_date_compose`/`__kml_days_from_civil` (both first written for `Date.parse`, which feeds them a 1-indexed month straight from an ISO string's `"MM"` field) expect a 1-indexed month. Missing this in the setter path would have silently shifted every date by one month. Handled by keeping the decomposed/overridden month value 0-indexed throughout the setter's own logic (matching `setMonth`'s own argument convention) and only converting to 1-indexed with a single `add i64 %month0, 1` immediately before the call to `__kml_date_compose` — the same conversion `emitDateToISOString` already does for the same reason (`ADR-00014`).
- **Scope: single-argument setters only.** Real JS also allows optional trailing arguments on several setters (`setFullYear(y, m, d)`, `setHours(h, m, s, ms)`, etc.). Scoped these out for this pass — each would need its own field-mapping logic beyond the single-field-override shape implemented here — and made this a clean compile error ("multi-argument overloads are not supported") rather than silently ignoring the extra arguments or misreading them.

## Decision

Implemented `setFullYear`, `setMonth`, `setDate`, `setHours`, `setMinutes`, `setSeconds`, `setMilliseconds` (each: decompose → override one field → recompose → store back), and `setTime` (direct store, no decompose/recompose needed). All: single-argument only, UTC-only (matching `Date` throughout), require a named-variable receiver, and return the new timestamp as a plain `number` (matching real JS's setter return value).

## Implementation notes

- `codegen/llvm/emit_date.go`: `dateSetterFieldIndex` (setter name → decomposed-field index, mirroring the existing `dateDecomposeFieldIndex`), `isDateSetterName`, and `emitDateSetterCall(mem, method, args, pos)` — resolves the identifier receiver (erroring per the shapes described above), loads the current value, decomposes it (for all but `setTime`), overrides the one targeted field, converts month back to 1-indexed, calls `__kml_date_compose`, stores the result back into the symbol's `alloca`, and returns it.
- `codegen/llvm/emit_call.go`: dispatch checked *before* the existing getter dispatch (`isDateSetterName(mem.Property) && e.inferExprType(mem.Object).IsDate`) — there's no name overlap between getters and setters, but ordering it first keeps the setter's own identifier-receiver error path in control rather than falling through to the getter path's more permissive (any-expression) receiver handling.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s existing Date-method-returns-`TypeI64` branch extended to include the setter names, so `const t = d.setFullYear(2020)` (or a setter used as an argument, in a function return, etc.) infers correctly. Not re-litigating `ADR-00014`'s three type-inference gaps here since setters don't introduce a new *construction* form (no `NewXExpression`-style AST node), only a new *call* shape, which already routes through the general `CallExpression`/`MemberExpression` inference path.
- `codegen/llvm/runtime.go`: new `ensureDateCompose()` (the extracted, shared calendar-recompose helper) and a refactor of `ensureDateParse()`'s `__kml_date_parse` to call it instead of duplicating the math inline.
- `codegen/llvm/emitter.go`: new `usedDateCompose` singleton-guard flag.

## Side effects discovered

None. The Hinnant-algorithm overflow/rollover behavior (see Investigation) worked correctly on first try without needing any additional carry logic — verified, not assumed.
