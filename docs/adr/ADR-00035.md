# ADR-00035: Fix JSON.stringify truncating float-typed values to integers

- **Status**: Completed
- **Date**: 2026-07-12

## Context

Tracked in `STATUS.md`'s Known Limitations & Bugs since the `JSON.parse`→object work (`docs/adr/ADR-00007.md`'s later investigation): `JSON.stringify({ score: 9.5 })` printed `{"score":9}` instead of `{"score":9.5}`. Picked up now as a small, well-scoped fix to exercise the new release automation (`.github/workflows/release.yml`) end-to-end after fixing its `allow-initial-development-versions` misconfiguration.

## Investigation

- `emitJSONStringifyValue` (`codegen/llvm/emit_call.go`) — the shared per-value JSON encoder used for object fields and array elements — has a `switch val.Ty.IR` with cases for `i1` (boolean) and `ptr` (string), and a `default` case for everything else that unconditionally does `e.coerce(val, TypeI64)` before calling `@__kml_json_str_num`. A `double`-typed value falls into this `default` branch and gets truncated by the `i64` coercion (`fptosi`) before it's ever formatted — the bug is the coercion, not the formatting.
- Found a **second, independent copy of the identical bug** while fixing the first: `emitJSONStringify` (the top-level `JSON.stringify(x)` entry point, same file) has its own near-identical `switch val.Ty.IR` block for the case where `x` isn't an object or array — meaning `JSON.stringify(9.5)` directly hit the same truncation, un-tracked in `STATUS.md` since the original bug report only exercised the object-field path.
- `emitValueToString` (`codegen/llvm/emit_exprs.go:1034`, backs template literals) already has a correct `v.Ty.Float` case using `sprintf("%g", ...)` — reused directly rather than writing a new runtime helper.
- Manually confirmed, while writing the repro, that the bug's own reproduction shape matters: an interface-typed field (e.g. `score: number` on an `interface Point`) always resolves to `i64` regardless of the assigned value — a separate, already-documented limitation (`STATUS.md`'s "Interface fields cannot be declared as `float64`/`float32`" row). Only a *plain, inline object literal* (`{ score: 9.5 }`, no interface annotation) infers the field as `Float` — confirmed this is exactly the shape `STATUS.md`'s original bug report used, and the one this fix actually addresses.

## Decision

Added a `val.Ty.Float` check ahead of the `default` case's unconditional `i64` coercion, delegating to the existing `emitValueToString` instead of a new dedicated helper — one code path already does the right thing, no reason for a second. Additionally, simplified `emitJSONStringify` to delegate its non-array, non-object case directly to `emitJSONStringifyValue` instead of duplicating the same switch a second time, so a future fix to this logic only has one place to land instead of two silently drifting copies (which is exactly how the second copy of this bug went unnoticed the first time).

## Implementation notes

- `codegen/llvm/emit_call.go`, `emitJSONStringifyValue`: added `if val.Ty.Float { return e.emitValueToString(val) }` as the first check inside the `default` case.
- `codegen/llvm/emit_call.go`, `emitJSONStringify`: removed its duplicate `switch val.Ty.IR` block entirely (the `i1`/`ptr`/`default` cases and the `val.Ty.IsObject` check ahead of it), replaced with a single `return e.emitJSONStringifyValue(val)` — `emitJSONStringifyValue` already checks `IsObject` itself, so no behavior changed, just de-duplicated.

## Side effects discovered

- The interface-field-can't-be-float limitation (`STATUS.md`) blocks this fix from applying to interface-typed object fields specifically — confirmed by testing, not assumed. Left as-is; already tracked separately, out of scope here.
- `JSON.stringify` on array literals passed directly as a call argument (`JSON.stringify([1.5, 2.25])`) isn't accepted at all (`array literal must be used in a variable declaration`) — an existing, separate parser/codegen restriction on inline array-literal arguments generally, not specific to `JSON.stringify` or this fix. Not investigated further here; noted for whoever picks up array-literal-as-argument support generally.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` (full suite, including two new tests below) all green; `make examples` 55/55.
- New tests in `compiler_test.go`: `TestE2EJSONStringifyObjectFloat` (`{ score: 9.5 }` → `{"score":9.5}`), `TestE2EJSONStringifyFloatDirect` (`JSON.stringify(9.5)` → `9.5`).
- Extended `examples/json/json_obj.ts` with both cases; manually ran the compiled binary and confirmed correct output for all six `console.log` lines, including the three pre-existing ones (no regression).
