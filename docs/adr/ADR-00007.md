# ADR-00007: Implement JSON.parse(s) → object (flat objects, primitive fields)

- **Date**: 2026-07-11

## Context

Found via the same audit of every `❌`-marked `STATUS.md` item that surfaced the `Map.size`/`Set.size` staleness and the `JSON.stringify` array bugs (`ADR-00006`). `const p: Point = JSON.parse('{"x":1,"y":2}')` compiled and ran, but a field read afterward (`p.x`) returned garbage instead of `1` — worse than a clean rejection.

## Investigation

- **Root cause.** `emitJSONParse` (`codegen/llvm/emit_call.go`) only special-cased number targets (`atoll`); everything else — including object targets — fell through to `@__kml_json_parse_str`, a helper that strips the outer two characters of the input and returns a raw string pointer. For an object target, the caller then treated that string pointer as if it pointed to a struct matching the target's field layout — a fundamental type mismatch, explaining the garbage reads.
- **A second, deeper bug found while wiring the fix in.** Fixing `emitJSONParse` alone wasn't enough — testing still produced garbage. Traced to `emitVarDecl`'s object-typed branch (`emit_exprs.go:1469-1471`) delegating to `emitObjectVarDecl` (`emit_objects.go`) *before* the scalar path's existing JSON.parse special-case (`emit_exprs.go:1484-1497`) is ever reached. `emitObjectVarDecl`'s `*ast.CallExpression` branch called the fully generic `emitExpr(init)`, which dispatches through `emitCall`'s JSON.parse case with a hardcoded `TypePtr` target (correct only for bare, non-declaration uses of `JSON.parse`) — meaning the object's *actual* target type never reached `emitJSONParse` at all for any object-typed declaration, regardless of what `emitJSONParse` itself did. Confirmed via `--emit-llvm`: the generated IR called `@__kml_json_parse_str` even after the `emitJSONParse` fix was in place, and produced a garbage-but-plausible-looking large integer when the resulting `{i64,i64}`-shaped struct field was read — a `strlen`-truncated string pointer's bytes reinterpreted as struct data.
- **Existing precedent to reuse.** `emitObjectLiteral` (`emit_objects.go:13-34`) and `emitJSONStringifyObject` (`emit_call.go`, the mirror-image "serialize an object" function) both already do the "malloc a struct sized to `targetTy.StructSize()`, then for each field in `targetTy.Fields` (fully resolved at compile time from the type annotation, including for nested interfaces), GEP via `FieldIndex` and store" shape — directly reusable for construction, just sourcing each field's value from parsed JSON text instead of a literal expression.
- **Scope decision: flat objects only, nested object fields deferred.** A field-finder helper that just does `strstr(json, "\"key\":")` is correct for a flat object, but incorrect once nested objects are involved: searching for a key inside a nested sub-object's raw text without first isolating that sub-object's `{...}` extent (via brace-matching) risks matching a same-named key belonging to a *later sibling* object elsewhere in the full JSON text. Properly supporting nested objects needs that brace-matching step, which is meaningfully more work than the flat case. Chose to detect any `IsObject` field in the target type upfront and return a clean compile-time error ("JSON.parse into a nested object field is not yet supported") rather than attempt broken codegen for that shape — consistent with this session's established pattern of failing cleanly for explicitly out-of-scope shapes.
- **Two more issues found and *not* fixed, tangential to this task.** (1) Interface/object field type annotations have no path to become `float`-typed — they resolve through the plain type-name mapping, which has no JSDoc-override mechanism (unlike variable declarations, where `/** @type {float64} */` works); a field can only become float-typed via literal inference in a plain object literal (`{ score: 9.5 }`), never via an explicit interface declaration. This means the float-field branch implemented here (see Decision) is currently unreachable through any interface-typed `JSON.parse` target, despite being implemented correctly — confirmed by testing that `interface X { /** @type {float64} */ score: number }` fails to parse the annotation at all. (2) `JSON.stringify(object)` truncates float-typed fields to integers (`{ score: 9.5 }` stringifies to `{"score":9}`) — `emitJSONStringifyValue`'s per-field switch has no case for a `Float`-typed value, silently falling through to the integer-formatting default. Both are separate, pre-existing gaps unrelated to `JSON.parse` — documented in `STATUS.md`'s Known Limitations rather than fixed here.

## Decision

1. **`emitJSONParse`** (`emit_call.go`): added `if targetTy.IsObject { return e.emitJSONParseObject(val, targetTy, pos) }`, checked before the existing number/string branches.
2. **`emitObjectVarDecl`**'s `*ast.CallExpression` branch (`emit_objects.go`): added the same JSON.parse special-case the scalar declaration path already has — detect `JSON.parse(...)` specifically and call `emitJSONParse(args, ty, pos)` with the *actual* resolved object type, instead of falling through to the generic `emitExpr(init)` path that loses all type context.
3. **`emitJSONParseObject`** (new, `emit_call.go`): upfront, walks `targetTy.Fields` checking for any `IsObject` field and errors cleanly if found (per the nested-object scope decision). Otherwise: `malloc`s the target struct (`targetTy.StructSize()`), then for each field (compile-time Go loop over `targetTy.Fields`, mirroring `emitJSONStringifyObject`'s existing field loop): calls a new `@__kml_json_find_value` runtime helper with that field's compile-time-interned `"name":` search pattern; branches on whether the result is `null` (key missing → `zeroRef(field.Ty)` default, reusing the existing zero-value convention rather than inventing a JSON-specific one) or found (parse via `emitJSONParseFieldValue`, dispatching on the field's compile-time type: `i1` → `strncmp` against `"true"`; `Float` → `strtod`; integer → `atoll`, matching the existing `JSON.parse(s)` → number behavior; string → the new `@__kml_json_parse_field_str` helper); merges the two branches via a `phi`; GEPs via `FieldIndex` and stores.
4. **Two new runtime helpers** (`runtime.go`, standard singleton `ensure*()` pattern): `@__kml_json_find_value(ptr json, ptr pattern) -> ptr` — `strstr`s for the pattern, then skips whitespace forward (a small in-place loop, not reusing `@__kml_trim_start` since that mallocs a copy, an unneeded allocation here), returning a pointer into the *original* text (no allocation) or `null`; downstream parsers (`atoll`/`strtod`/`strncmp`/the new string parser) each naturally stop at their own end (digit run, closing quote, etc.) without needing the value's extent bounded up front. `@__kml_json_parse_field_str(ptr valueStart) -> ptr` — the reverse of `@__kml_json_str_str`'s escaping loop: a two-pass (count-then-copy) scan from just after the opening `"` to an unescaped closing `"`, decoding `\n`/`\t`/`\r` explicitly (the decode `select` chain's default case already correctly passes through `\"`→`"` and `\\`→`\`, since the raw escaped byte already equals the decoded byte for those two).

## Implementation notes

Files touched:
- `codegen/llvm/emit_call.go` — `emitJSONParse`'s new object branch; new `emitJSONParseObject`, `emitJSONParseFieldValue`.
- `codegen/llvm/emit_objects.go` — `emitObjectVarDecl`'s `CallExpression` branch now special-cases `JSON.parse` (this was the actual root cause of garbage output persisting even after the `emitJSONParse` fix — see Investigation).
- `codegen/llvm/runtime.go` — `ensureJSONFindValue`/`@__kml_json_find_value`, `ensureJSONParseFieldStr`/`@__kml_json_parse_field_str`.
- `codegen/llvm/emitter.go` — `usedJSONFindValue`/`usedJSONParseFieldStr` flags.

A bug in the first draft of `@__kml_json_parse_field_str`, caught before landing: the escape-decode block initially read the character to decode from the *backslash's own* offset (`%s0, i64 %fi`) rather than one past it (`%s0, i64 %fi + 1`), and separately referenced an out-of-scope register (`%ci`, from the earlier count-loop, not the fill-loop's `%fi`) — a straightforward copy-paste slip while translating the count-pass logic into the fill-pass, caught immediately since the wrong-scope register reference would have been an LLVM verifier error, and fixed before the first test run.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including the extended `examples/json/json_methods.ts`.
- Manual repros: `{"x":1,"y":2}` → `Point{x:1,y:2}` (previously garbage); mixed field types (string/number/boolean) in one object; a missing key falling back to its type's zero value; a string field with `\n` and escaped `\"` decoding correctly; a nested-object field annotation producing the clean compile-time error instead of broken codegen (confirmed, not just designed).
- New tests in `compiler_test.go`: `TestE2EJSONParseObject`, `TestE2EJSONParseObjectMixedFields`, `TestE2EJSONParseObjectMissingField`, `TestE2EJSONParseObjectEscapedString`.
- `examples/json/json_methods.ts` extended with flat-object parse cases including the missing-key default.
- `STATUS.md`: `JSON.parse(s)` → object flipped to ✅ (noting the flat-objects-only scope); removed its entry from Known Limitations & Bugs; added the two tangential findings (interface fields can't be `float`-typed; `JSON.stringify(object)` truncates floats) as new, separate, not-fixed-here limitations; coverage numbers updated (JSON category now 100%).
