# ADR-00014: Implement Date (UTC-only)

- **Date**: 2026-07-11

## Context

Picked this up as the next Phase 3 item, following object spread (`ADR-00013`).

## Investigation

- **The consequential design fork: UTC vs. local time.** Real JS's `Date.getFullYear()`/`getHours()`/etc. default to the *local system timezone*. For a compiled native binary that can run on any machine or CI runner, that makes output non-deterministic depending on where it executes. Confirmed this explicitly rather than picking one silently: chose UTC-only (via `gmtime()`, not `localtime()`) for every getter, documented clearly as a deviation from real JS. This also simplifies the design — there's no need for separate `getUTCFullYear()`-style variants when there's only one behavior.
- **Representation.** Unlike `Map`/`Set` (heap-allocated, need a `Symbol{Ptr, LenPtr}`-style declaration path), a JS `Date` is fundamentally just a single number — milliseconds since the Unix epoch. Modeled it as a plain `i64` with a new `Type.IsDate` flag (`TypeDate`), the same storage as `number`, distinguished only for method dispatch. This meant `new Date(...)` could be dispatched directly through `emitExpr` (mirroring `NewErrorExpression`, which is also a single-value construct with no declaration-time special-casing), and Date method calls could work on *any* expression producing a Date value — not just named variables — since there's no `Symbol`/heap-pointer resolution needed, just a plain SSA register.
- **Platform portability for the two C calls needed.** For `Date.now()`/`new Date()`, chose `clock_gettime(CLOCK_REALTIME, struct timespec*)` over `gettimeofday(struct timeval*)` specifically because `struct timespec`'s two fields (`time_t tv_sec`, `long tv_nsec`) are *both* 64-bit on every LP64 target this compiler supports (macOS ARM64, Linux x86-64/aarch64) — whereas `struct timeval`'s `tv_usec` is a 32-bit `suseconds_t` on macOS/BSD but 64-bit on Linux, meaning a hardcoded `{i64,i64}` GEP layout for it would silently misread fields on macOS. `CLOCK_REALTIME` is `0` on both platforms, safe to inline without a header-derived constant. For calendar-field decomposition, confirmed `struct tm`'s first 7 fields (`tm_sec` through `tm_wday`) are `int` (i32) in the same order on both glibc and Darwin/BSD (the standard POSIX layout) — reading only those, not the platform-varying tail fields like `tm_gmtoff`, keeps the hardcoded GEP struct portable.
- **Verified the tricky calendar math directly, not just trusted the C library.** Ran the actual compiled output against known reference values: `new Date(0)` → `1970-01-01T00:00:00.000Z`, weekday `4` (Thursday — Jan 1 1970 is a well-known reference date), and `new Date(1700000000000)` → `2023-11-14T22:13:20.000Z`, matching the well-known "epoch 1700000000" reference value exactly. This wasn't assumed correct just because it compiled.
- **A latent bug found while testing untyped declarations, then generalized.** `const d = new Date(0)` (no explicit annotation) initially fell back to a plain `i64` default — `emitVarDecl`'s untyped-type-inference switch (a hand-written `switch init := v.Init.(type)`) had no `case *ast.NewDateExpression` at all, so `ty` stayed at `resolveType(nil)`'s `TypeI64` fallback, silently discarding the `IsDate` flag. Before just adding a `NewDateExpression` case, checked whether the same switch was missing a case for `*ast.NewErrorExpression` too, since it's the closest existing analog — it was, confirmed via a standalone repro (`const e = new Error("oops"); console.log(e.message)` failed with `field access on non-object`, identical failure mode). Fixed both in the same pass, since the fix is the same one-line pattern and it's a real, previously-silent bug, not just something adjacent to Date.
- **A second, chained instance of the same class of gap, for `Error` specifically.** Fixing the type-inference switch surfaced a *second* layer: `emitVarDecl` routes object-typed declarations to `emitObjectVarDecl`, which has its *own* separate switch on `v.Init`'s AST type (`ObjectLiteral` / `CallExpression` / default-error) — and that switch also had no case for `*ast.NewErrorExpression`, so even after the type-inference fix, `new Error(...)` in an untyped declaration failed one layer further in with `object variable must be initialized with an object literal or function call`. Fixed by adding a `NewErrorExpression` case there too, mirroring the existing `ObjectLiteral` case's shape.
- **A third, broader gap found while wiring up chained field access (`ev.when.getFullYear()`).** My Date-method dispatch (in `emit_call.go`) uses `e.inferExprType(mem.Object)` as a side-effect-free guard before committing to evaluate the receiver. For `ev.when` (a field access on an object), this returned the wrong type — tracing `inferExprType`'s `*ast.MemberExpression` case revealed it has several special-cased namespaces (enum members, `Map`/`Set.size`, `Math`/`Number`/`process` constants) but **no general fallback** for a plain object field read at all, silently defaulting past all of them. This is broader than Date — it would misinfer the type of *any* chained field access on an object-typed field, for any type, not just Date. Fixed by adding a general fallback: if the object's inferred type `IsObject`, look up the field via the existing `FieldIndex` and return its declared type.
- **A fourth instance, in the untyped-declaration path for function calls.** `const e = epoch()` (an untyped `const` assigned from a call to a user-defined function returning `Date`) also lost the return type — the `case *ast.CallExpression:` branch of the same untyped-inference switch only propagated a called function's declared return type when it was `IsArray || IsObject || IsFunc`, not `IsDate`. Fixed by adding `|| sig.RetType.IsDate` to that condition.

## Decision

**Scope**: `new Date()` (current time) and `new Date(ms)` (explicit timestamp); static `Date.now()`; instance methods `getFullYear`, `getMonth` (0-indexed), `getDate`, `getDay` (0=Sunday), `getHours`, `getMinutes`, `getSeconds`, `getMilliseconds`, `getTime`/`valueOf`, `toISOString`. All UTC. Explicitly not implemented: `Date.parse(string)` (needs a date-string parser), setters (`setFullYear`, etc. — Date is effectively read-only here), and any local-time variant.

**Representation**: `TypeDate = Type{IR: "i64", Signed: true, IsDate: true}` (`codegen/llvm/types.go`); `"Date"` added to `ResolveTypeName`'s switch (so explicit `: Date` annotations, function params/returns, and object/interface fields all resolve correctly, not just inferred untyped declarations).

**New file `codegen/llvm/emit_date.go`**: `emitNewDate` (dispatches the two constructor forms), `emitDateNow` (the static call), `emitDateCall` (instance method dispatch, keyed by a small field-index map for the 8 getters plus explicit handling for `getTime`/`valueOf`/`toISOString`), `emitDateDecompose` (calls the runtime helper once, shared by every getter and by `toISOString`), `emitDateToISOString` (formats via `sprintf`, converting the 0-indexed month to ISO's 1-indexed convention), and `isDateMethodName` (the cheap pre-check used alongside `inferExprType` at the call site so a same-named method on an unrelated type is never mistakenly evaluated).

**New runtime helpers** (`codegen/llvm/runtime.go`, standard `ensure*()` singleton pattern): `ensureDateNow`/`@__kml_date_now` (the `clock_gettime`-based current-time helper) and `ensureDateDecompose`/`@__kml_date_decompose` (the `gmtime`-based calendar-field decomposition, returning `{year, month, day, weekday, hour, min, sec, millis}` as one `{i64 x 8}` aggregate so every getter and `toISOString` share one call site's worth of C-level logic instead of each doing its own `gmtime` call).

**Dispatch** (`codegen/llvm/emit_call.go`): `Date.now()` is a plain identifier-based static check (`id.Name == "Date" && mem.Property == "now"`), matching the existing `Math`/`JSON` static-dispatch convention. Instance methods are dispatched more generally — `isDateMethodName(mem.Property) && e.inferExprType(mem.Object).IsDate` — evaluating the receiver only after confirming via the side-effect-free type guard, so this works uniformly whether the receiver is a named variable, a field access, a function call result, or a fresh `new Date(...)` literal, unlike `Map`/`Set`'s existing (pre-existing, unrelated) restriction to named-identifier receivers only.

## Implementation notes

Files touched: `ast/nodes.go` (`NewDateExpression`), `parser/parser.go` (`parseNewDateBody`, `parseNew`'s dispatch), `codegen/llvm/types.go` (`IsDate`, `TypeDate`, `ResolveTypeName` case), `codegen/llvm/emitter.go` (`usedDateNow`/`usedDateDecompose` flags), `codegen/llvm/emit_date.go` (new), `codegen/llvm/runtime.go` (the two new helpers), `codegen/llvm/emit_call.go` (dispatch), `codegen/llvm/emit_exprs.go` (the three type-inference fixes described in Investigation: the untyped-declaration switch gaining `NewErrorExpression`/`NewDateExpression` cases, the `CallExpression` branch's `IsDate` condition, and the general object-field fallback in `inferExprType`'s `MemberExpression` case), `codegen/llvm/emit_objects.go` (`emitObjectVarDecl` gaining a `NewErrorExpression` case).

## Side effects discovered

All four gaps described in Investigation were found and fixed in this same pass, not deferred — they were directly in the path of getting Date working end-to-end (as an untyped declaration, as a function param/return, and via chained field access), and each fix was small and low-risk (adding a missing `case`/condition to an existing switch, following the exact shape of the adjacent cases already there).
