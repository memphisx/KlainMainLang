# ADR-00048: select()-based event loop (TDD-00006 Part 1) and a minimal HTTP server (TDD-00004 V1)

- **Relations**: Implements TDD-00004, TDD-00006 (Part 1 only — Part 2, real async suspension, still not started)
- **Date**: 2026-07-13

## Context

Picked up as this project's next big, structural piece of work: the event loop is this project's own stated biggest structural gap relative to its microservice direction, and `docs/tdd/TDD-00006.md` already scoped it in detail across two differently-sized pieces — Part 1 (a `select()`-based I/O-multiplexing wait loop, generalizing the existing timer drain loop) and Part 2 (real `async`/`await` suspension, three candidate mechanisms sketched, no decision made). Part 1 was explicitly called out as "the first piece to pick up regardless of Part 2's eventual direction," and is also `docs/tdd/TDD-00004.md` (HTTP Server)'s own ceiling on how good a first version could be — so both were built together in this pass: Part 1 alone, plus the HTTP server V1 on top of it, exactly as both TDDs already designed.

## Investigation

Read both TDDs in full before writing any code (both were already thorough design documents from a prior session, so this was implementation, not fresh scoping). Confirmed directly against the actual codebase, not assumed:

- The existing timer queue (`ensureTimerRuntime`, `runtime.go`) already had exactly the shape TDD-00006 Part 1 wanted to generalize: a linear-scan-for-earliest-due-entry loop, currently blocking via `nanosleep()`. Confirmed its exact IR (`__kml_timer_drain`) to reuse the scan logic verbatim inside a new, separate function rather than risk modifying the already-shipping one.
- Confirmed `emitClosureCallByPtr` (`emit_func.go`) and the `qsort` comparator trampolines (`ensureSortTrampolineI64`/`F64`, `runtime.go`) are exactly the "closure stored in a global slot, called back from hand-written IR" pattern TDD-00004 predicted would be reusable for dispatching into the HTTP handler — verified directly, not just cited from the TDD.
- Confirmed no C helper files are ever compiled alongside the generated `.ll` (`main.go`'s `clangArgs` is just `[-O2, llFile, -o, outBin]` plus `-l` flags) — meaning `struct sockaddr_in`/`fd_set`/`struct timeval` all had to be built as raw byte-level LLVM IR, with real platform ABI differences to get right by construction rather than by testing on both platforms (this sandbox only has macOS available): `SOL_SOCKET`/`SO_REUSEADDR` differ numerically between Linux (1, 2) and Darwin/BSD (0xffff, 4); `sockaddr_in`'s first two bytes differ in *meaning* (Linux: a plain 2-byte `sin_family`; Darwin: 1-byte `sin_len` + 1-byte `sin_family`) though never in total struct size (16 bytes either way). Resolved with the same `runtime.GOOS`-branch-at-Go-compile-time approach already established for `monotonicClockID()`/`errnoAccessor()` (this compiler always builds and runs on the same host, so a Go-side branch baked into the emitted constants is sufficient — no IR-level conditional needed). `fd_set`/`timeval`, by contrast, turned out to need **no** platform branching at all once implemented as raw byte-level bit manipulation (`byte = fd/8; bit = 1 << (fd%8)`, matching what `FD_SET`/`FD_ISSET` do internally on both platforms regardless of whether the underlying "word" they optimize over is 32-bit (Darwin) or 64-bit (Linux)) rather than trying to replicate either platform's specific struct/word layout — and `timeval`'s only genuine difference (Darwin's `tv_usec` is `int32_t`, not `int64_t`) turned out to not matter either: writing an over-sized `i64` at the right offset only ever overwrites trailing padding bytes on Darwin, never anything the kernel reads.

## Decision

**Event loop (Part 1)**: a new `__kml_event_loop_run()` (`runtime.go`), left entirely separate from the existing `__kml_timer_drain()` rather than modifying it in place — a program that never calls `http.listen` still uses the original, unmodified `nanosleep()`-based drain path with zero behavior change; `__kml_event_loop_run` is only ever called by `http.listen` itself. Each iteration: scans the timer queue for the earliest-due entry (identical logic to `__kml_timer_drain`'s own scan, duplicated rather than shared, to avoid coupling the new code to the old function's internals), builds a 128-byte (1024-bit) `fd_set` bitmap containing the registered listener fd (`@__kml_listen_fd`, `-1` when none), and calls `select()` with a timeout computed from the earliest-due timer — blocking indefinitely if a listener is registered but no timer is pending (a listener alone has no natural "timeout" the way a timer does), returning immediately once neither a timer nor a listener is registered (queue exhausted, matching `__kml_timer_drain`'s own exit condition). On wake: dispatches through `@__kml_listen_dispatch` (a stored function pointer, `void()`) if the listener is ready, then fires/reschedules/retires the due timer exactly like the original drain loop.

**HTTP server (V1)**: `http.listen(port, handler)` — a bare global function (not a namespace/object with methods), matching TDD-00004's design exactly: never returns (same category as `process.exit()`, `unreachable` emitted right after the call site), single listener only (no `.close()`), single connection handled fully synchronously per accept (no concurrent request handling — that's Part 2's job specifically). `Request` (`method`/`path` string fields) is a new special built-in type (`types.go`'s `RequestType()`, mirroring `ResponseType()`'s exact shape), resolved via `ResolveTypeName("Request")`. The handler's *response* type needs no special type at all — checked structurally at the call site (`emit_http.go`'s `emitHTTPListen`: must be `IsObject` with `status`/`body` fields) and read via the same GEP-with-Go-known-offsets pattern every other object field read in this compiler already uses.

The one place this genuinely couldn't reuse a fully generic hand-written-IR helper (unlike the timer trampoline, which never needs per-call-site specialization): reading `status`/`body` off the handler's *user-declared* return type needs the field offsets baked in at the specific `http.listen` call site, not at the shared runtime-helper level. Solved by building the request dispatcher (`@__kml_http_dispatch`) as an ordinary Go-emitted top-level function, specialized per call site — using the exact same save-and-restore-emitter-state pattern `emitClosureFunc` already uses for closures (fresh `allocas`/`body`/`regCtr`/`labelCtr`/scope, restored after). A fixed function name is safe: only one `http.listen` call is ever reachable per program in V1, since the first one's own `unreachable` makes any second call dead code.

Everything else — `socket`/`setsockopt`/`bind`/`listen` (`__kml_http_bind_and_listen`, throws a catchable Error via a new `__kml_http_throw` mirroring `ensureFsThrow`'s errno/strerror shape, just without a path argument), `accept`+`read`+`sscanf`-parse-the-request-line (`__kml_http_accept_and_read`, using `sscanf`'s existing `%s` whitespace-delimited-token conversion rather than hand-rolling a scanner), and format-response+`write`+`close` (`__kml_http_send_response`) — are fully generic hand-written IR helpers, following the exact "big multi-block `define` string, C functions called in sequence, branch on error" style already established by `ensureFetch`'s `__kml_fetch`.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureMemset()` (new, alongside the existing `ensureMemcpy()`); `httpSockConstants()`/`httpSockaddrFamilyBytes()` (Go-side `runtime.GOOS` branches, same shape as `monotonicClockID()`); `ensureHTTPThrow()` (mirrors `ensureFsThrow()`, reusing the already-shared `ensureErrnoAccessor()`/`ensureStrerror()` singletons — deliberately, since this codebase already hit and fixed a duplicate-declaration bug once from two features independently declaring the same errno accessor); `ensureHTTPRuntime()` (socket primitives, the three hand-written IR helpers, `__kml_event_loop_run`, new globals `@__kml_listen_fd`/`@__kml_listen_dispatch`/`@__kml_listen_handler`).
- `codegen/llvm/types.go`: `RequestType()`; `"Request"` case in `ResolveTypeName`.
- `codegen/llvm/emitter.go`: `usedMemset`/`usedHTTP`/`usedHTTPThrow` flags.
- `codegen/llvm/emit_http.go` (new): `emitHTTPListen` (arg/type validation, bind+listen, registers the dispatcher, calls the event loop, `unreachable`); `buildHTTPDispatcher` (the per-call-site-specialized dispatcher function).
- `codegen/llvm/emit_call.go`: new `http.listen` dispatch case, alongside the existing `process`/`fs`/`crypto`/etc. namespace checks.
- No changes to `emitter.go`'s `EmitProgram` — `http.listen` is entirely self-contained (registers everything, calls the loop, marks `unreachable`), the same way `process.exit()` needs no special handling from `EmitProgram` either.

## Side effects discovered

None new — every platform-ABI question (the genuine risk area for this feature, given no C helper files and no Linux machine available to test on directly) was investigated and resolved by construction (see Investigation above) before writing the IR, rather than discovered as a bug afterward.
