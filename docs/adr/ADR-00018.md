# ADR-00018: Implement Date arithmetic (adding/subtracting durations)

- **Status**: Completed
- **Date**: 2026-07-11

## Context

Picked this up as the next `Date` follow-on after setters (`ADR-00016`) and `Date.parse` offsets (`ADR-00017`).

## Investigation

- **Established what already worked, and what didn't, before writing anything.** Since `Date` (`ADR-00014`) is represented as a plain `i64`, identical in storage to `number`, generic `+`/`-`/comparison codegen already operates on it correctly at the raw-value level without any Date-specific handling. Confirmed directly: `d1 - d2` and `d1 < d2` already produced numerically correct results with no changes needed. The actual gap was narrower than "implement Date arithmetic from scratch": the *value* computed by `d1 + 500` was already numerically correct, but the *type* tag on that value didn't stay `IsDate` through the operation, so `(d1 + 500).getFullYear()` failed with a generic, unhelpful `"only simple function calls are supported"` error — the expression losing its Date-ness, not the arithmetic being wrong.
- **Traced the type-loss to two independent, need-both-fixed spots.** `emitBinary`'s `"+"`/`"-"` cases set the result type via `ty := left.Ty` unconditionally — so `Date + number` (left is Date) happened to already tag the result `IsDate` by accident, but `number + Date` (left is a plain number) didn't, and `Date - Date` (left is Date) *incorrectly* tagged its result `IsDate` too, when it should be a plain number (this particular bug was invisible in earlier testing only because `inferExprType`'s separate, compile-time-only copy of this logic defaulted to `TypeI64` for every case, coincidentally masking it for untyped variable declarations — but an inline expression like `(d1 - d2).getFullYear()` would need to consult `inferExprType` directly, and did **not** silently misbehave, only by that separate accident). Both `emitBinary` (the actual runtime codegen) and `inferExprType` (the compile-time mirror used for method dispatch and untyped-declaration inference, per `ADR-00014`'s three analogous gaps) needed matching, deliberate Date-aware rules — not just leaving one to accidentally cover for the other.
- **Found a real pre-existing bug while confirming the "already works" claim for comparisons and subtraction.** Nothing wrong there — but worth stating: this was checked directly (compiled and ran probes) rather than assumed from reading the code, consistent with this project's pattern of verifying claims about existing behavior before building on them.
- **The real design question: what should `Date + number` and `Date - number` even mean, given real JS's actual (surprising) behavior.** Real JS's `+` operator on a Date uses the "default" `ToPrimitive` hint, and `Date` is one of the few built-ins whose default hint is `"string"`, not `"number"` — so `someDate + 5` in real JS produces a bizarre string concatenation (the Date's `.toString()` output with `"5"` appended), not numeric addition. `-`, by contrast, always uses the numeric hint regardless of type, so `someDate - 5` in real JS *does* give a plain number (the timestamp minus 5) — meaning real JS's own `+`/`-` behavior on Date is already asymmetric and one side (`+`) is a well-known footgun, not a design worth preserving. Decided to treat `Date ± number` uniformly as numeric duration arithmetic, producing a **new Date** — genuinely more useful than either real JS behavior, and a natural fit for this compiler's plain-`i64` Date representation, but a documented, deliberate deviation, called out explicitly (not silently reproducing JS's `+` quirk, and not silently making `-` inconsistent with `+`).
- **What about `Date + Date` and `number - Date`, which don't have a sensible "duration" reading?** `Date + Date` sums two absolute epoch timestamps — not meaningful under any interpretation (real JS's own actual behavior there, string concatenation, isn't something worth replicating either). `number - Date` has no sensible "duration" meaning in this compiler's model (there's no operation like "a plain count of milliseconds minus an absolute timestamp, producing a new Date") — note this one **is** technically valid, number-producing JS (since `-` always numeric-coerces regardless of operand order), so rejecting it is a real, deliberate scope-narrowing rather than matching a bug ; both were rejected with a clear compile-time error rather than silently producing a nonsense value, consistent with this project's standing preference for failing cleanly over guessing.
- **Compound assignment (`d += ms`) needed a separate check, not just the `emitBinary` fix.** Traced how `+=`/`-=` is actually implemented: `emitAssign` calls a distinct function, `emitArith`, passing the *already-coerced* RHS value — meaning by the time `emitArith` runs, `right.Ty` has already been overwritten to match the target's declared type (via `e.coerce(rhsVal, targetTy)` in the caller), so a plain-number RHS being compound-assigned into a Date-typed variable would *already* show `Ty.IsDate == true` after coercion, indistinguishable from a genuinely Date-typed RHS. This meant the "reject Date `+=`/`-=` Date" check had to happen in `emitAssign`, using the RHS's type *before* coercion, not inside `emitArith` itself — a subtlety that would have produced a wrong result (either silently allowing `d += otherDate`, or incorrectly rejecting the ordinary `d += 1000` case) if implemented naively.

## Decision

- `Date - Date` → a plain number (milliseconds difference), matching real JS. Unchanged from before this ADR at the value level; fixed at the type level (`emitBinary`, `inferExprType`) so it no longer incorrectly stays tagged as a Date.
- `Date + number` / `number + Date` / `Date - number` → a new Date (duration arithmetic). A deliberate, documented deviation from real JS.
- `Date + Date` and `number - Date` → rejected at compile time with a clear error.
- `d += durationMs` / `d -= durationMs` (compound assignment on a Date-typed variable/field/array element) → mutates in place, same duration-arithmetic semantics as the binary form (this already worked correctly at the value level before this ADR, since `emitArith` is called with the target's own declared type; only the new Date+Date rejection is new here). `d += otherDate` / `d -= otherDate` → rejected at compile time.

## Implementation notes

- `codegen/llvm/emit_exprs.go`, `emitBinary`: captured `leftIsDate`/`rightIsDate` *before* the existing `right = e.coerce(right, left.Ty)` line (which would otherwise overwrite `right.Ty`, losing whether it was originally a Date — the same class of subtlety as the compound-assignment case below, just within a single function this time). `"+"` rejects `Date+Date`; otherwise resolves to `TypeDate` if exactly one side `IsDate`. `"-"` rejects `number-Date` (`!leftIsDate && rightIsDate`); resolves to `TypeI64` for `Date-Date`, `TypeDate` for `Date-number`.
- `codegen/llvm/emit_exprs.go`, `inferExprType`'s `*ast.BinaryExpression` case: added the mirrored Date-aware rules to the existing `"+"` case, and added a new `"-"` case (previously absent entirely, silently falling through to the function's blanket `TypeI64` default — benign for `Date-Date`, wrong for `Date-number`).
- `codegen/llvm/emit_exprs.go`, new `dateCompoundAssignGuard(op string, targetIsDate, rhsIsDate bool) error` helper, called from all three `emitAssign` compound-assignment branches (scalar identifier, object field, array element) — each call site passes the RHS's type *before* its own subsequent `e.coerce(rhsVal, targetTy)` line, per the Investigation finding above.
- No changes needed to `emitArith` itself, or to `emit_date.go`/`emit_call.go` — the fix is entirely in type-tagging and the two new rejection checks; the underlying arithmetic (`add`/`sub` on `i64`) was already correct.

## Side effects discovered

The `Date - Date` mistagging described in Investigation (would have incorrectly returned `IsDate: true` instead of a plain number) — fixed as part of this same change, not deferred, since it's a direct product of the same `emitBinary` code path being rewritten here.
