# TDD-00005: Unannotated parameter typing

- **Status**: Partially Implemented — option 1 (clean rejection at call sites) is done (see `docs/adr/ADR-00042.md`); options 2 and 3 are still not started
- **Date**: 2026-07-13

## Context

Found while verifying `docs/adr/ADR-00041.md`'s return-type inference fix: an unannotated function/arrow function parameter defaulted to `i64` (`registerFunctions`'s `pty := TypeI64` fallback) with no check or coercion against what was actually passed at a call site. `function log(msg) { console.log(msg) }; log("hello")` used to compile clean and print a garbage number instead of `hello` — the string's pointer got bit-reinterpreted as an `i64` inside the callee. Worse than a clean rejection or a crash: silent data corruption.

**Why this is a harder problem than the return-type fix it was found next to**: return-type inference only ever needed to look *inward*, at a single function's own body — self-contained, one pass, done. Parameter-type inference needs to look *outward*, at every call site, which might live in a different file, might pass different argument types in genuinely different, intentional ways (the closest thing this compiler has to ad-hoc polymorphism, given it has no user-defined generics), or might never appear as a direct named call at all if the function is only ever used as a passed-around callback value. That's a meaningfully bigger problem than what `ADR-00041` solved.

## Design

Three options considered:

1. **Clean rejection at call sites.** Keep the `i64` default, but check each call site's actual argument type against it; if they don't match and the parameter has no annotation, fail at compile time with a clear message instead of silently corrupting data. Smallest, safest option — converts the silent-corruption bug into a clean failure. Doesn't reduce annotation burden at all: code that only ever passes numbers to an unannotated parameter keeps working exactly as before; everything else now demands an annotation instead of quietly breaking.
2. **Best-effort inference from the first call site**, mirroring `ADR-00041`'s own approach for return types: a single top-to-bottom pass would find the first direct call to a given function and use its argument types as the parameter types, falling back to `i64` (with option 1's check still in force) if none is found. Would reduce annotation burden further than option 1, but inherits the same "first occurrence wins, forward references and callback-only usage aren't seen" caveats `ADR-00041` already accepted for return types — and unlike return types, a function genuinely called with different argument shapes in different places is a much more plausible real case here, not just a theoretical corner (this compiler has no generics to reach for instead, so writing one loosely-typed helper called several different ways is a natural thing to want to do).
3. **Model real TypeScript's implicit-`any` semantics directly**: default an unannotated parameter to the boxed `any` type (`docs/adr/ADR-00008.md`) instead of `i64`, dispatching every operation on it at runtime — exactly what real TS does today (an unannotated parameter *is* `any`, and only errors under the `noImplicitAny` compiler option most modern `tsconfig.json` defaults now enable). The most conceptually correct option, and the one closest to "just match real TypeScript" — but also the biggest lift by a wide margin: `any`/`unknown` is explicitly rejected as a parameter type today (Staged V1, `ADR-00008`), so this would mean lifting that restriction and wiring boxing/unboxing through every arithmetic operator, call site, and coercion path a parameter can reach — closer to a new feature than a bug fix.

**The `noImplicitAny`-vs-JS-interop tension, worth being explicit about**: option 3's "just match real TS" framing is attractive in principle — plenty of real-world TS projects now default to `noImplicitAny`-strict, i.e. exactly "reject anything unannotated" (closer in spirit to option 1 than option 3, ironically). But the moment this project ever wants to consume genuinely untyped, plain-JS code (an imported package with no types at all, not written in TS), a strict "everything must be annotated" stance stops being purely a style preference and starts actively fighting the interop use case — plain JS has no annotations to give it at all. This project's `import`/`export` support today is TS-only, relative-path-only declarations (`docs/adr/ADR-00022.md`), so this tension isn't actively blocking anything yet — but it's worth naming now rather than rediscovering later: **this compiler is already, deliberately, a bit stricter about `any` than real TypeScript is** (real TS's implicit `any` is permissive-by-default; this compiler's Staged V1 `any` is opt-in and restricted to a handful of contexts). That's an acceptable, even reasonable, position for a project that leans toward the statically-typed end of the TS spectrum on purpose — if genuine dynamic, untyped, JS-style code is what someone wants to write, that's arguably a sign they want JS, not TS, and this project doesn't need to chase that use case just because real TS technically allows it.

**Recommended direction**: option 1 was picked as the immediate fix — a permanent safety net, not an inference system, and the smallest, safest thing that closes the actual silent-corruption hole. Option 3 (real `any` semantics) is the intended eventual destination, with `noImplicitAny`-style strict rejection as the *default* once it exists — i.e. an unannotated parameter should eventually be a compile error by default, not a silent `number`. Given the interop tension above, option 1's behavior (default to `number`, reject non-numeric arguments) is worth keeping available too, most likely behind an opt-in flag, rather than removed outright once option 3 ships — the exact shape of that flag (and whether it's the same flag that would eventually help with untyped-JS interop) is undecided and deliberately deferred; not worth designing until option 3 itself is actually being built.

## Prerequisites

| Dependency | Status | Notes |
|---|---|---|
| Return-type inference precedent (`ADR-00041`) | ✅ already done | Proved the "temporary scope + inferExprType" mechanics work; parameter inference (option 2) would reuse the same shape, just scanning call sites instead of return statements |
| `any`/`unknown` boxed runtime representation (`ADR-00008`) | ✅ already done, but restricted | Exists and works for variables; explicitly rejected as a function parameter or return type today — option 3 needs that restriction lifted, not just referenced |
| Call-site argument type inference | ✅ already done | `inferExprType` on a call's arguments already works; options 1 and 2 both reuse it, just applied at a new point (checking/inferring parameters, not just return values) |
| A real multi-pass or fixed-point inference system for genuinely polymorphic call patterns | ❌ not attempted, not designed | Would be needed to fully resolve option 2's "different call sites, different argument types" case correctly rather than just picking the first one seen |

## Open questions

- Exact shape of the eventual "keep today's number-default behavior" opt-in flag once option 3 ships, and whether it doubles as a JS-interop escape hatch — deliberately deferred until option 3 is actually being built.
- Whether option 2 (call-site inference) is worth building at all before option 3, given it's a smaller ergonomic win with the same forward-reference caveats `ADR-00041` already accepted for return types.
