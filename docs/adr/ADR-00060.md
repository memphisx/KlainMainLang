# ADR-00060: Fix for...in and return-of-an-array only recognizing a plain named variable

- **Relations**: Extends ADR-00059
- **Date**: 2026-07-14

## Context

Direct follow-on to `ADR-00059`: after generalizing Map/Set method dispatch beyond plain named variables, swept the rest of `codegen/llvm` for the same "identifier-only" pattern (`mem.Object.(*ast.Identifier)` / `s.Object.(*ast.Identifier)` gating a symbol lookup) to check for other genuine instances, rather than assuming the sweep was already complete.

## Investigation

Found roughly 50 identifier-type-assertions across `codegen/llvm`. Most are legitimate and not this bug: namespace dispatch (`id.Name == "Math"`/`"JSON"`/`"process"`/etc. — checking a fixed global name, not a variable's dynamic type), assignment-target-shape checks, and named-function-call resolution (`ex.Callee.(*ast.Identifier)`, which genuinely needs a compile-time name to look up a registered signature).

A distinct, larger group — every array-length-mutating method (`push`/`pop`/`shift`/`unshift`/`splice`) plus `crypto.getRandomValues` and spread-in-array-literal — also requires a named identifier, but for a real architectural reason, not an oversight: these need to write a *new*, possibly-reallocated pointer and/or length back into the array's own two-part storage (`Symbol.Ptr`/`Symbol.LenPtr`), and a general expression's result has nowhere to write that back to (unlike `Map`/`Set`, whose mutating methods never change what the outer pointer points to — the map struct's own internal buffers grow in place, so `ADR-00059`'s fix was safe to generalize broadly in a way these can't be). All of these already fail with a clear, on-topic error message ("push requires an array variable") rather than falling through to something confusing — the correct, honest failure mode for a genuine restriction, not the silent-fallthrough bug `ADR-00059` fixed. Left alone.

Two real, fixable instances of the same bug turned up:

1. **`for...in`** (`emit_stmts.go`): required a named object variable, but the only thing it actually uses from the resolved symbol is `sym.Ty.Fields` — a purely compile-time field-name list. The object's runtime value/pointer is never touched at all (for-in only iterates keys). Confirmed directly: `for (const k in c.point)` (a field access) failed with `"for...in requires a named object variable"` even though nothing about the fix needs `c.point`'s runtime value.
2. **`return` of an array-typed expression** (`emit_stmts.go`'s `emitReturn`): required a named array variable, unconditionally rejecting anything else with `"can only return a named array variable from a function"`. But an array-producing expression that isn't a named variable — `arr.slice(1)`, another function's array-typed return value — already evaluates to exactly the `{ptr, i64}` aggregate a function's own array return needs (confirmed by reading `LLVMRetType()`, which already renders array types as `{ptr,i64}` for return position). Confirmed directly: `return a.slice(1)` failed with the named-variable-only error before the fix.

**A third, separate, deeper bug was found while verifying the `return`-array fix, not caused by it.** Testing `return c.items` (an array-typed *object field*, as opposed to a call result) produced a hard `clang`-stage IR type-mismatch crash rather than a clean result. Traced this to something the `return` fix only *exposed*, not introduced: confirmed directly, on the unmodified pre-session binary, that `c.items.length` alone (no `return` involved at all) already crashes the same way — `codegen/llvm/types.go`'s `ArrayOf()` gives an array type `IR: "ptr"`, and nothing in `emit_objects.go`'s object-literal field-store path special-cases array-typed fields, so constructing `{ items: someArray }` silently keeps only the data pointer and drops the length entirely (no length is stored anywhere in the object's own heap struct for an array-typed field). Every read path — `.length`, indexing, `for...of`, and now `return` — either crashes at the `clang` stage or would silently misbehave once it *does* have a `{ptr, i64}`-shaped value to work with, because the length was never there to recover in the first place. This is not the same bug as `ADR-00059`'s (a dispatch-recognition gap); it's a genuine storage-representation gap for array-typed object/interface fields specifically, unrelated to whether the receiver is a named variable. **Deliberately not fixed here** — it's sizable (a real struct-layout question: does an array field need a second, implicit length slot, or does the field itself need to become a `{ptr,i64}` aggregate the way return values already are, with all the `StructIR()`/`StructSize()`/GEP-codegen consequences that implies?) and not something to fix as a rushed tangent to this ADR's actual scope. Flagged in `STATUS.md`'s Known Limitations for a deliberate follow-up decision, not silently dropped.

## Decision

**`for...in`**: replaced the `s.Object.(*ast.Identifier)` + `e.lookup` gate with `e.inferExprType(s.Object)`, using the inferred type's `.Fields` directly — no runtime evaluation of `s.Object` needed at all, since (confirmed above) nothing downstream ever touches the object's actual pointer.

**`return <array>`**: kept the named-variable path unchanged (still the fast path, reusing existing `Ptr`/`LenPtr` allocas), and added a fallback — mirroring the exact "named variable vs. arbitrary expression" split `resolveArrayForHOF` (arrays) and `resolveMapOrSetForCall` (`ADR-00059`) already use — that evaluates the expression via `emitExpr` and returns its `{ptr, i64}` aggregate directly when it isn't a plain identifier, rejecting only if the evaluated value genuinely isn't array-typed.

## Implementation notes

- Files touched: `codegen/llvm/emit_stmts.go` (`emitForOf`'s for-in handling — note: despite the name, this project's for-in and for-of both live in statement-emission code near each other; the specific function touched here is the for-in one, not `emitForOf` itself — and `emitReturn`).
- No changes to `emit_objects.go`/`types.go` — the array-typed-object-field storage bug found along the way is explicitly out of scope for this ADR.

## Side effects discovered

The array-typed object/interface field length-loss bug described above — real, confirmed via direct reproduction on the pre-session binary (not introduced by this ADR's changes), sizable enough to need its own dedicated design/fix pass rather than a rushed inline patch. Tracked in `STATUS.md`'s Known Limitations, not silently dropped.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green.
- Manual repros: `for (const k in c.point)` (a field access) now correctly iterates the field's keys; `return a.slice(1)` and a two-level named-function relay (`return makeNums()` called from another function that itself does `return relay()`) both now work and return correct, indexable results.
- Confirmed the named-variable paths for both fixes are unchanged and still work (existing `TestE2EForIn*` tests, existing array-return-via-named-variable tests all still pass).
- Confirmed, deliberately, that `return c.items` (array-typed object field) still fails — now at the `clang` stage instead of a clean compile-time rejection, which is a real regression in *failure quality* (a worse error message) for that one specific sub-case, though not a new *crash class* — `.length` on the same field already crashed the same way before this ADR touched anything. Accepted as a known consequence of fixing the common cases first and deferring the deeper storage bug; not silently ignored.
- New E2E tests: `tests/stmts_test.go`'s `TestE2EForInFieldAccess`; `tests/arrays_test.go`'s `TestE2EReturnArrayMethodResult`, `TestE2EReturnArrayFromNestedFunctionCall`.
- `make examples` — full suite passes (one transient, unrelated `httpbin.org` network flake on an early run, clean on immediate re-run — the same known flakiness already documented in `examples/fetch/fetch.ts`'s own comment).
