# ADR-00041: Infer return types for unannotated functions and arrow functions

- **Relations**: Extends ADR-00012, ADR-00037
- **Date**: 2026-07-13

## Context

The last "Medium" item in `STATUS.md`'s tracked bug list, after `docs/adr/ADR-00040.md`: two related, previously-separate entries, both really symptoms of the same root cause —

1. `function makePoint(x, y) { return { x: x, y: y } }` (no return-type annotation) — accessing `.x`/`.y` on the caller's result fails with `field access on non-object` (`docs/adr/ADR-00012.md`).
2. `const middle = (): (() => void) => { ...; return innerFn }; const inner = middle(); inner()` fails with `undefined function or closure 'inner'`, regardless of the return-type annotation, *specifically* when the outer binding is a `const`-bound arrow function rather than a `function` declaration (`docs/adr/ADR-00037.md`).

## Investigation

- Traced symptom 2 first (already partially investigated in `ADR-00037`): `emitClosureCallByPtr` (`emit_func.go`) correctly dispatches a call by trusting the callee variable's own stored `Symbol.Ty.FuncRetType` — so the actual bug isn't in call *dispatch*, it's in how `middle`'s own type gets computed when `const middle = ...` is declared. `emitArrowFunctionWithHints` (`emit_func.go`) already resolves `af.RetType` correctly when present, and its expression-bodied branch (`af.Body != nil`) already infers correctly too — but its **block**-bodied branch (`af.Block`, no annotation) unconditionally did `retTy = TypeI64 // block body: default, caller may override via annotation`, regardless of what was actually returned.
- Traced symptom 1: `registerFunctions` (`emitter.go`) registers every top-level `function` declaration's signature into `e.funcs` *before* any body is emitted (the existing two-pass setup that lets functions forward-reference each other). For an unannotated function, it unconditionally set `retType := TypeVoid` — never inspecting the body at all — so every caller's `inferExprType`/`emitVarDecl` correctly *looked up* `e.funcs[name].RetType`, but that registered type was simply wrong from the start.
- Found, while fixing symptom 2, that `inferExprType`'s own `*ast.ArrowFunction` case (`emit_exprs.go`) has an **independent, third copy** of this exact "block body → blind default" logic — used specifically when something needs to know an arrow function *expression's* type before any closure value exists yet (e.g. `emitVarDecl`'s `case *ast.ArrowFunction: ty = e.inferExprType(init)`, which is what actually determines a `const`-bound arrow function's own declared symbol type). This duplicate hadn't been touched by `ADR-00037`'s fix to `emitArrowFunctionWithHints`, so the two computations could (and did) disagree: the closure's *real* emitted body correctly returned an object struct, while the *variable's declared type* still said `i64` — a mismatch that doesn't get caught by any verifier, since indirect closure calls are typed by whatever the caller-side code believes, not checked against the callee's actual signature. This produced silently wrong output (a raw pointer bit-reinterpreted as a number), not a clean failure — confirmed directly by testing `console.log(JSON.stringify(makePoint(5, 6)))` (no intermediate variable) before finding and fixing this third copy.
- Fixing symptom 1 similarly meant discovering `emitFunctionDecl` (`emit_func.go`) had **its own, fourth independent copy** of "compute this function's return type" — recomputing `TypeVoid`/`resolveType(decl.ReturnType)` from scratch instead of reusing whatever `registerFunctions` had already decided, so even after teaching `registerFunctions` to infer correctly, the function's own emitted LLVM signature still said `void` — a hard `clang`-stage type mismatch (`value doesn't match function result type 'void'`), not the field-access error, since this is the function's *own* definition disagreeing with its *callers'* now-correct expectations.
- While verifying the fix against deliberately adversarial cases (recursion, plain scalar returns, no return at all), found the same class of bug applies beyond object/array/closure/Date: `function addOne(n) { return n + 1 }` (a plain, unannotated function returning a number) hit the *identical* `clang`-stage mismatch — confirmed this reproduces identically on the pre-`ADR-00041` compiler (`git stash` + rebuild), i.e. a genuinely separate, pre-existing bug, not something introduced while fixing the other two. Since the fix machinery already existed and the only change needed was removing an artificial "only object/array/func/Date count" restriction, this was fixed in the same pass rather than left half-done.
- **Found a second, separate, more severe bug while stress-testing, deliberately left unfixed and flagged instead of folded in**: `function log(msg) { console.log(msg) }; log("hello")` prints a garbage number instead of `hello`. Root cause is unrelated to return types — unannotated *parameters* default to `i64` (`registerFunctions`'s existing, unchanged `pty := TypeI64` fallback) with no coercion or check against what's actually passed at a call site, so a string argument's pointer gets bit-reinterpreted as a number inside the callee. Confirmed pre-existing (reproduces identically on the pre-`ADR-00041` compiler) and confirmed fixed by an explicit `msg: string` annotation. This is a *substantially* bigger and more concerning problem than anything actually fixed here — silent data corruption, not a clean rejection — and needs its own design decision (infer parameter types from call-site arguments? require annotations for non-numeric parameters and reject cleanly otherwise?). Documented in `STATUS.md`'s Known Limitations & Bugs and Roadmap as a new, separate, high-effort item; not attempted here.
- Also confirmed and accepted, as a known boundary rather than something to chase further: a function calling *another, unannotated function declared later in the same file* doesn't get its return type inferred correctly, since `registerFunctions`'s single top-to-bottom pass hasn't registered the later function yet when the earlier one's inference runs. Fails cleanly (`field access on non-object`), not silently — reordering the declarations or adding an explicit annotation works around it. A real fixed-point/multi-pass inference system could close this, but that's a materially bigger undertaking than this ADR's scope; documented in `STATUS.md` instead.

## Decision

One shared helper, `inferUnannotatedReturnType(block, paramNames, paramTypes)` (`emit_func.go`), used by all three places that used to independently (and inconsistently) default an unannotated block-bodied function/arrow function's return type: `registerFunctions`, `emitArrowFunctionWithHints`, and `inferExprType`'s `*ast.ArrowFunction` case. It pushes the function's own parameters into a temporary scope (safe before the real function body exists, since `inferExprType` and everything it calls are verified side-effect-free — no `emitInstr`/`freshReg` anywhere in their call graph), finds the first reachable `return` statement's expression via a new `firstReturnExprInBlock`/`firstReturnExprInStmt` pair (mirroring the existing `blockHasReturn`/`stmtHasReturn` traversal shape exactly), infers its type in that scope, and returns it as-is — scalars included, not just object/array/closure/Date. A function with multiple, differently-shaped returns still only considers the first one; this compiler has no general union-type support beyond `T | null`, so a function genuinely returning different types on different paths was never a designed-for case.

Additionally, `emitFunctionDecl` no longer recomputes a function's return type independently — it reuses `e.funcs[decl.Name].RetType`, which `registerFunctions` already resolved (explicit or inferred) before any function body is emitted, guaranteeing the function's own signature always matches what callers expect.

## Implementation notes

- `codegen/llvm/emit_func.go`: new `firstReturnExprInBlock`/`firstReturnExprInStmt` (placed next to `blockHasReturn`/`stmtHasReturn`, identical recursion shape); new `inferUnannotatedReturnType`; `emitArrowFunctionWithHints`'s block-bodied branch and `emitFunctionDecl`'s return-type computation both updated to use them.
- `codegen/llvm/emitter.go`, `registerFunctions`: calls `inferUnannotatedReturnType` for any `fd.ReturnType == nil` function.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s `*ast.ArrowFunction` case updated to use the same helper instead of its own independent (and, until now, undiscovered-as-inconsistent) copy; `inferExprType`'s `*ast.CallExpression` case and `emitVarDecl`'s duplicate inline `*ast.CallExpression` switch both gained an `e.lookup(callee.Name)` fallback for closure-typed *variables* (not just `e.funcs`-registered named declarations) — the same fallback pattern `resolveCallback` (`emit_func.go`) already used, just missing from these two call-result-type-inference sites.
- `STATUS.md`: removed both now-fixed rows from Known Limitations & Bugs and the Roadmap's "Next up" table; added two new rows for the genuinely separate issues found along the way (the forward-reference boundary, and the parameter-type bug); `docs/adr/ADR-00012.md` and `docs/adr/ADR-00037.md` cross-referenced as extended by this ADR (later reworked into the `Relations` field convention, see `docs/adr/README.md`).

## Side effects discovered

- The parameter-type-defaulting bug described in Investigation — the most significant thing found this session, deliberately not fixed here. Tracked in `STATUS.md` as its own, separate, high-effort item.
- The forward-reference boundary described in Investigation — an accepted limitation of this ADR's single-pass inference, not a bug introduced by it (the previous behavior for this shape was already broken, just via `TypeVoid` instead of a clean field-access error either way).
