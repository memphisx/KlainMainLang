# ADR-00052: Fix `alloca`s inside loop bodies causing unbounded stack growth

- **Relations**: Extends ADR-00049, ADR-00050
- **Date**: 2026-07-14

## Context

A GitHub Actions run flagged `examples/fetch/wttr_weather.ts` as `FAIL` inside `make examples`. That file already carries its own long-standing comment saying it's expected to fail without network access or when wttr.in itself is briefly down — but the actual failure was a SIGSEGV (exit 139), not a clean network error, so it needed to be run down rather than dismissed.

Wrapping the fetch in `try`/`catch` (in case the failure was a legitimate network-level throw) didn't help — the crash still happened. A `try`/`catch` in this compiler is implemented via `setjmp`/`longjmp` (`emit_exceptions.go`), which cannot intercept a SIGSEGV; that ruled out "uncaught network exception" immediately and confirmed this was a real memory-safety bug.

Reproducing it directly against the live `wttr.in` endpoint was unreliable (crashed once in ~6 runs one day, 0 times in 40 runs right after). That pointed at a timing-dependent bug rather than a deterministic one.

## Investigation

- Compiled a minimal repro: a bare top-level `await fetch('https://httpbin.org/delay/3')` (deliberately slow, no `http.listen`, no fiber). This crashed **reliably** (3/3 runs, exit 139, ~0.6s — far faster than the 3s the endpoint takes to respond) once network latency was introduced, while the same fetch against a fast endpoint never crashed.
- Read `__kml_await_fetch` (`runtime.go`, then around line 2489) and its `busyspin:` block: when a fetch is awaited outside any connection fiber (`@__kml_current_conn_idx < 0`, true for any top-level/non-`http.listen` code), it loops calling `curl_multi_perform` + `__kml_curl_drain_messages()` with **no delay whatsoever** — a genuine tight spin, easily millions of iterations/second.
- Found the root cause: `%runningp = alloca i32, align 4` sat inside the `busyspin:` block itself, which is a loop body (reached repeatedly via `checkloop` → `maybeyield` → `busyspin` → back to `checkloop`), not the function's `entry:` block. In LLVM IR, an `alloca` reached multiple times via a loop allocates **fresh, never-freed stack space on every execution** — space that isn't reclaimed until the enclosing function returns, unlike `malloc`/`free`. A spin loop with no delay hitting this on every iteration overflows the stack in well under a second once the loop needs more than a handful of iterations to see `done` become true — which only happens when the transfer takes any real time, exactly matching the "crashes sometimes, not always" symptom (a fast local/cached response finishes on the very first `checkloop` check, before the loop ever spins).
- `__kml_curl_drain_messages` (same function group) had the identical pattern: `%msgsleft = alloca i32` in `drainloop:` (a loop head), and `%privslot`/`%statusslot` in `handledone:` (a block that loops back to `drainloop`).
- **Confirmed the mechanism, not just the theory**, before deciding this was the whole story: built the pre-fix compiler as a separate binary and ran the `httpbin.org/delay/3` repro 3 more times — crashed 100% of the time (exit 139, ~0.6-0.9s). After hoisting the two `alloca`s to `entry:` (see Decision), the identical repro passed 100% of the time (3/3, ~3.9s, matching the real network delay) and used normal (not spiking) CPU.
- Given `busyspin`/`__kml_curl_drain_messages` had this bug, checked whether any other hand-written IR function in `runtime.go` has the same pattern (every function body in this file is raw LLVM IR text, not run through the Go-side two-builder `e.emitAlloca`/`e.emitInstr` split that normally keeps allocas in the entry block automatically — so this class of bug can only be caught by manually auditing each hand-written block). A full-file, block-by-block control-flow audit (every `define`, every label, every backward branch) turned up two more, more severe, instances:
  - **`__kml_timer_drain`** (backs any `setTimeout`/`setInterval` program without `http.listen`): `outerloop:` itself contains `%besti`/`%bestfire`/`%scani` (all `alloca i64`), and `dosleep:` (also part of the loop, reached every time a timer fire needs to wait) contains `%ts = alloca { i64, i64 }`.
  - **`__kml_event_loop_run`** (the core `select()`-based dispatch loop backing every `http.listen` server — TDD-00006's central loop): `outerloop:` has the same `%besti`/`%bestfire`/`%scani`; `dowork:` (run on every loop pass with any pending work) has `%fdset`/`%wfdset`/`%efdset` (three `[128 x i8]` fd_sets) and `%maxfd`; `skipsetfd:` has `%fsi`; `mergecurlfds:` has `%curlmaxfd`; `timeoutpath:` has `%tv`; `docurlperform:` has `%runningp2`; `scanconn:` has `%rsi`. All of these execute on **every single `select()` wake**, i.e. every incoming request or timer fire, for the entire lifetime of any long-running KML server — the worst-case instance of this bug in the file, since it directly undermines this project's ambition of supporting long-running microservices.
  - The audit also confirmed every *other* `alloca` in the file (in `__kml_fetch`, `__kml_fetch_async`, `__kml_crypto_random_uuid`, `__kml_read_line_sync`, `__kml_exec_file_sync`, `__kml_monotonic_ns`, `__kml_timer_clear`, `__kml_http_bind_and_listen`, and elsewhere) sits in a block that executes at most once per call — entry blocks or one-shot branch targets with no back-edge — so those were left untouched.
- **Quantified and directly confirmed the `__kml_event_loop_run` instance separately**, since its per-iteration leak (roughly ~450-500 bytes across the eleven hoisted allocas, most from the three 128-byte fd_sets) predicted a crash around 8MB / ~500B ≈ 16,000 requests against the default macOS stack — a number easily reached by ordinary sustained traffic, not an edge case. Wrote a minimal `http.listen` server, compiled it with the pre-fix binary, and hammered it with Apache Bench (`ab -n 40000 -c 10`): it crashed with `Connection reset by peer` after **20,773 requests**, matching the prediction almost exactly. The post-fix binary, same load pattern, survived `ab -n 100000 -c 10` (100,000 requests, 0 failures, ~14,189 req/s, server still running afterward).
- `__kml_timer_drain`'s leak rate (~24-40 bytes/iteration) is much smaller and, more importantly, each iteration is naturally throttled by real `nanosleep` waits for the actual timer interval, so forcing enough iterations to overflow within a practical test timeout wasn't feasible (at 1ms intervals, reaching the several-hundred-thousand iterations needed would take minutes). Given the mechanism is identical, already confirmed twice (fetch busyspin, `__kml_event_loop_run`) via direct before/after reproduction, and the fix is the same one-line-per-alloca hoist, this instance was fixed on the strength of the (deterministic, not probabilistic) LLVM semantics and the passing existing timer test suite, without a separate forced-overflow repro.

## Decision

For all four functions (`__kml_curl_drain_messages`, `__kml_await_fetch`, `__kml_timer_drain`, `__kml_event_loop_run`), hoisted every `alloca` that sits inside a loop-body block to the function's `entry:` block, leaving the `store`/`memset` initialization calls that reset each slot's value in place at the top of each loop iteration (an `alloca` only needs to execute once per function call; re-initializing its *contents* every iteration is still correct and necessary). No other functions in `runtime.go` needed this treatment (confirmed by the full-file audit).

This is the same fix shape ADR-00051 already used for a different class of bug in the same neighborhood (fiber/event-loop internals) — narrow, mechanical, and doesn't change any function's observable behavior, only where its scratch stack slots live.

Rejected alternative: switching these scratch values to `malloc`/`free` instead of `alloca`. Rejected because it's strictly more code (paired `free` calls on every exit path, several of which are early-return-shaped) for no benefit — a hoisted `alloca` in the entry block is exactly the right tool for a fixed-size scratch value reused every iteration, which is what the two-builder pattern (`e.emitAlloca`/`e.emitInstr`, `emitter.go`) already establishes as the convention for Go-emitted code; these hand-written IR blocks just hadn't been following it.

## Implementation notes

- `codegen/llvm/runtime.go`:
  - `__kml_curl_drain_messages`: `%msgsleft`, `%privslot`, `%statusslot` moved to `entry:`.
  - `__kml_await_fetch`: `%runningp` moved to `entry:`.
  - `__kml_timer_drain`: `%besti`, `%bestfire`, `%scani`, `%ts` moved to `entry:`.
  - `__kml_event_loop_run`: `%besti`, `%bestfire`, `%scani`, `%fdset`, `%wfdset`, `%efdset`, `%maxfd`, `%fsi`, `%curlmaxfd`, `%tv`, `%runningp2`, `%rsi` moved to `entry:`.
- No signature or call-site changes anywhere — this is purely an internal stack-layout fix within four already-existing function bodies.
- Worth remembering for any *future* hand-written IR block in this file: since these bodies bypass the Go-side two-builder pattern that normally prevents this automatically, any new loop needs its scratch `alloca`s manually checked for the same mistake.

## Side effects discovered

- This is very likely the same underlying failure mode (or a contributing one) behind at least some of the `connection reset by peer` / hangs originally attributed entirely to ADR-00051's `ucontext_t` layout bug — that bug was real and independently confirmed, but `__kml_event_loop_run`'s stack leak was live in the exact same code path (every `http.listen` request) the whole time and could equally well produce the same symptom under enough load. Both are now fixed; no further action needed, noting it here only so the history isn't misread as "the ucontext fix was incomplete."
- None of this affects fiber-path (`doyield`) fetches — the fiber/`swapcontext` mechanism itself has no allocas in its own loop-adjacent blocks, so `http.listen` handlers awaiting `fetch()` were never exposed to the `busyspin`-specific instance, only to the always-present `__kml_event_loop_run` one.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green after the fix (full suite, ~128s).
- `make examples` — all 58 examples pass, including `examples/fetch/wttr_weather.ts` (re-run 5x standalone with 0 crashes, plus wrapped in `try`/`catch` per a separate, unrelated hardening — see the file's own comment).
- **Direct before/after reproduction, `__kml_await_fetch`/`__kml_curl_drain_messages`**: a 3-second-delay fetch crashed 3/3 runs pre-fix (~0.6-0.9s to crash) and passed 3/3 runs post-fix (~3.9s, correct output).
- **Direct before/after reproduction, `__kml_event_loop_run`**: a trivial `http.listen` server hammered with Apache Bench crashed the pre-fix binary at 20,773 requests (`ab -n 40000 -c 10`, `Connection reset by peer`); the post-fix binary handled `ab -n 100000 -c 10` with 0 failed requests and was still alive and serving afterward.
- New regression tests: `tests/fetch_test.go`'s `TestE2EFetchTopLevelBusySpinDoesNotOverflowStack` (a local httptest `/slow` endpoint with a 300ms delay, exercised via a bare top-level `await fetch` so it takes the `busyspin` path, not the fiber path) and `tests/http_test.go`'s `TestE2EHTTPListenManyRequestsDoesNotLeakStack` (30,000 sequential requests against a local `http.listen` server, comfortably past the ~20,000-request crash threshold observed above).
