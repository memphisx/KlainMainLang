# ADR-00034: Fix missing -lm link flag for Math builtins on Linux

- **Status**: Completed
- **Date**: 2026-07-12

## Context

Found via the very first run of `go test ./...` under the new GitHub Actions release workflow (`.github/workflows/release.yml`, `ADR` not yet written for the versioning setup itself since it's process/tooling, not a compiler feature) — the first time this project's test suite ever ran on Linux instead of the macOS dev machine. Three tests failed at the `clang` link step, not codegen: `TestE2EInterface`, `TestE2EMathHyperbolic`, `TestE2EMathCbrtExpm1Log1p`, each with `undefined reference to` `sqrt`/`floor`/`tanh`/`cbrt`/`expm1`/`log1p`.

## Investigation

- `ensureMathFuncs()` (`codegen/llvm/runtime.go`) declares all Math-backed C functions (`sqrt`, `floor`, `tanh`, `cbrt`, `expm1`, `log1p`, etc.) via `declare double @fn(...)`, but never called `e.requireLink(...)` for any library — every prior Math example happened to compile and link cleanly on this project's own macOS dev machine, so the gap was invisible locally.
- Root cause: macOS's `libSystem` folds `libm`'s symbols in with `libc`, so no explicit `-lm` is ever needed there. Linux (glibc) keeps `libm` as its own shared object — `clang`'s default link step doesn't pull it in automatically, producing exactly the `undefined reference to sqrt` class of error seen in CI. This is the same category of macOS-vs-Linux link-behavior gap `ADR-00032`/`ADR-00033` already ran into with `-static` and `libcurl`, just for a builtin (libm) instead of an opt-in feature.
- `requireLink(lib string)` (`emitter.go`) already existed as the general mechanism for exactly this — `fetch`/`libcurl` (`ADR-00021`) already uses it, and `main.go` already turns `LinkLibs()` into `-l<lib>` flags unconditionally. `ensureMathFuncs` simply never called it.

## Decision

Add `e.requireLink("m")` inside `ensureMathFuncs()`, called unconditionally — no `runtime.GOOS` branch, unlike `--static`. `-lm` is a standard, harmless flag on macOS too (accepted as a no-op there, since the symbols are already present via `libSystem`), so there's no platform split to encode here, unlike the `--static` case where macOS's linker categorically refuses the flag. Any program that calls `ensureMathFuncs()` (i.e. uses any Math method beyond the handful implemented as compiler intrinsics) now links `-lm` automatically on every platform.

## Implementation notes

- `codegen/llvm/runtime.go`, inside `ensureMathFuncs()`: one new line, `e.requireLink("m")`, right after `e.usedMathFuncs = true`.
- No change to `main.go` — `LinkLibs()`/the `-l<lib>` loop already existed and needed no modification, since this reuses the same mechanism `fetch` already exercises.

## Side effects discovered

None — this was the exact, isolated bug the first Linux CI run was looking for.

## Verification

- Locally on macOS (dev machine): `go build .`, `go vet ./...`, `go test ./...` (full suite, including the three previously-failing-on-Linux tests, all pass — expected, since `-lm` was already a no-op there), `make examples` (55/55 passed).
- The actual regression this fixes is Linux-specific and could only be confirmed by the next push through the GitHub Actions workflow, which is what originally caught it.
