# ADR-00004: Fix empty-string-argument bugs in .split(), .padStart(), .padEnd()

- **Date**: 2026-07-11

## Context

`ADR-00003` flagged, but deliberately did not fix, a confirmed hang in `.split("")` (empty separator) — discovered as a side effect while implementing `.replaceAll()`, since both share a "scan the string via `strstr`, advance a cursor by the search string's length" shape, and `.replaceAll()`'s new helper had to guard against the same zero-length-match risk. Before fixing `.split()` in isolation, this ADR generalizes the question: **what else shares that same failure shape — a loop that assumes a user-supplied string argument has nonzero length?** — and fixes everything found, rather than patching one call site at a time as each is separately reported.

## Investigation

- **Confirmed the `.split("")` hang is real, not theoretical.** Compiled and ran `"abc".split("")` with a wall-clock timeout: the process never returns. Root cause (`@__kml_split` in `codegen/llvm/runtime.go`): both its count and fill loops advance the scan cursor via `getelementptr i8, ptr %found, i64 %sep_len` after each `strstr` match; `strstr(s, "")` always matches at the current position with a zero-length match, so `%sep_len` is `0` and the cursor never advances.
- **Searched for every other loop with the same shape.** `grep`'d all `@strstr` call sites across `codegen/llvm/*.go`: two are single, non-looping calls (`.indexOf`/`.includes` in `emit_strings.go` — no risk, they don't loop). The remaining occurrences are `@__kml_replace` (single-shot, already correct for an empty search — inserts at position 0, matching real JS `"abc".replace("", "X")` → `"Xabc"`), `@__kml_replace_all` (already guarded in `ADR-00003`), and `@__kml_split`'s two loops (the confirmed bug).
- **Generalized beyond `strstr` to any "assumes nonzero step" loop and found a second, worse bug.** `emitStringPad` (`codegen/llvm/emit_strings.go`, backs both `.padStart`/`.padEnd`) fills the pad region character-by-character with `modIdx = srem(j, fillPLen)`, cycling through the fill string. If the fill string is empty, `fillPLen` is `0` and `srem(j, 0)` is undefined behavior at the LLVM IR level.
  - Tested `"ab".padStart(5, "")`: did **not** crash (ARM64 doesn't trap on integer division/remainder by zero — `sdiv`/`srem` by zero is architecturally defined to return `0` rather than raising a signal, unlike x86), but silently produced an **empty string** instead of the correct `"ab"`. Inspected the raw output bytes to confirm: the `srem`-by-zero evaluated to `0` on this target, so every fill-loop iteration read the fill string's own NUL terminator and wrote NUL bytes into the pad region; for `padStart`, those NUL bytes land *before* the real content, so `strlen`-based printing sees an empty string immediately.
  - Tested `"ab".padEnd(5, "")`: printed the correct `"ab"`, but only *by coincidence* — the same NUL-byte corruption happens, but lands *after* the real content (`padEnd` copies the string first, then fills after it), so `strlen` still stops at the right place. The buffer itself is equally corrupted (still writes UB `srem` results into memory); it just doesn't happen to be visible through a `strlen`-based read. Confirmed this is not "working," just silently lucky, by reasoning through the same code path `padStart` takes.

## Decision

Fixed both at the root cause rather than special-casing the visible symptom:

1. **`.split("")`** (`@__kml_split`, `runtime.go`): added a dedicated `is_empty_sep` branch, checked once at function entry, that produces the actual correct result — one single-character string per byte of the input (`"abc".split("")` → `["a","b","c"]`, `"".split("")` → `[]`) — matching real JS semantics exactly, rather than merely avoiding the hang with a fallback value. This was barely more work than a guard clause, so there was no reason to settle for a documented deviation the way `.replaceAll()`'s empty-search case did.
2. **`.padStart`/`.padEnd` with an empty fill string`** (`emitStringPad`, `emit_strings.go`): reordered so the fill string is resolved *before* `padLen` is finalized, then force `padLen` to `0` whenever the fill string's length is `0` (`padLen = isEmptyFill ? 0 : max(0, targetLen - sLen)`), matching JS's actual behavior (an empty fill string means no padding is added at all — return the original string unchanged). Because the fill loop's bound is `padLen`, forcing it to `0` means the loop body — and its `srem` — never executes, eliminating the UB entirely rather than just the symptom that happened to be visible for `padStart`.

## Implementation notes

Files touched:
- `codegen/llvm/runtime.go` — `@__kml_split`: new `char_split`/`char_loop`/`char_body`/`char_done` blocks, branched to from `entry` when `sep_len == 0`, before the pre-existing `cnt_loop`/`fill_loop` path (left unchanged for non-empty separators).
- `codegen/llvm/emit_strings.go` — `emitStringPad`: moved fill-string resolution earlier; `padLen` computation now factors in `isEmptyFill` via a `select`.

## Side effects discovered

None beyond what's described above — this ADR exists specifically to close out both once found, rather than deferring either.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including the updated `examples/strings/string_methods.ts`.
- Manual repros:
  - `"abc".split("")` → `["a","b","c"]` (length 3, correct elements); `"".split("")` → `[]` (length 0).
  - Re-ran the same hang check from `ADR-00003` with a wall-clock timeout: the process now terminates immediately instead of hanging.
  - `"ab".padStart(5, "")` → `"ab"` (previously silently produced an empty string).
  - `"ab".padEnd(5, "")` → `"ab"` (previously happened to look correct despite corrupted memory; now genuinely correct with no UB).
  - Non-empty-separator/non-empty-fill cases re-verified unchanged: `"a,b,c".split(",")`, `"5".padStart(3,"0")`, `"hi".padEnd(6,"!-")`, etc.
- New tests in `compiler_test.go`: `TestE2EStringSplitEmptySeparator`, `TestE2EStringPadEmptyFill`.
- `examples/strings/string_methods.ts` extended with both cases.
- `STATUS.md`: removed the now-fixed `.split("")` entry from Known Limitations & Bugs; added notes to the `.split(sep)`/`.padStart`/`.padEnd` rows pointing here.
