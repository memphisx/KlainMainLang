# ADR-00003: Implement String.prototype.replaceAll(from, to)

- **Date**: 2026-07-11

## Context

Picked this up as the next Phase 2 "quick win" from `STATUS.md`'s roadmap, which listed a "loop over `replace`" approach as low-complexity. This ADR implements it, but not via that literal approach — see Investigation for why.

## Investigation

- **Existing `.replace()`.** `emitStringReplace` (`codegen/llvm/emit_strings.go`) delegates to a C-level LLVM helper, `@__kml_replace` (`codegen/llvm/runtime.go`), which finds the first occurrence of the search string via `strstr` and replaces just that one match — matching JS's `.replace(string, string)` semantics (first occurrence only).
- **Why "loop over `replace`" is the wrong approach.** Repeatedly calling `@__kml_replace` on its own output until no match remains would rescan the *already-inserted replacement text* on each iteration. If the replacement string itself contains the search string — e.g. `"a".replaceAll("a", "aa")`, or more subtly `"aaa".replaceAll("a", "aa")` — this either infinite-loops or produces runaway/incorrect output, because real `replaceAll` semantics require scanning the **original** string exactly once, left to right, non-overlapping, and never re-matching against inserted replacement text.
- **A working precedent for the actual shape needed.** `@__kml_split`'s existing C-level helper (`runtime.go`) already solves a structurally similar problem — needing to know a dynamic output size before allocating — via two passes: a `strstr`-based counting loop, then a second `strstr`-based loop that builds the result. This two-pass shape was reused for `replaceAll`, but scanning the original string in both passes (never the output) and tracking read/write cursors independently, which is what actually prevents the "replacement contains search" problem above.
- **Found while designing this: `.split()` has a related, pre-existing bug.** `@__kml_split`'s count and fill loops advance the scan cursor by `getelementptr i8, ptr %found, i64 %sep_len` after each match. If the separator is an empty string, `strstr(s, "")` always matches at the current position with a zero-length match, so `sep_len` is `0` and the cursor never advances — an infinite loop for `"abc".split("")`. This is pre-existing and unrelated to this change (not introduced here); not fixed in this ADR since it's a separate, `.split()`-specific issue — flagged in `STATUS.md`/here so it doesn't get silently lost, and deliberately guarded against introducing the *same* class of bug in the new `replaceAll` helper (see Decision).

## Decision

Implemented `@__kml_replace_all(ptr %s, ptr %search, ptr %rep)` (`runtime.go`, gated by a new `ensureStringReplaceAll()`/`usedStringReplaceAll` pair, same singleton-declare convention as every other `ensure*()`), structured as:

1. **Empty-search guard.** If `%search` is empty, return a copy of `%s` unchanged. Real JS `"abc".replaceAll("", "-")` inserts between every character (`"-a-b-c-"`), which would need a different loop shape entirely (advance-by-one-even-on-empty-match) — treating it as a no-op instead is a deliberate, minor deviation, and — more importantly — avoids the exact zero-length-match-makes-no-progress bug identified in `.split()` above.
2. **Count phase.** A `strstr`-based loop over the original `%s` counts non-overlapping matches (identical shape to `@__kml_split`'s `cnt_loop`).
3. **Size the output** from the original length, match count, search length, and replacement length (`slen - cnt*search_len + cnt*rep_len + 1`), and `malloc` it once.
4. **Build phase.** A second `strstr`-based loop over the *original* `%s` (a fresh `cur_f`/`found_f` pair, never touching the output buffer being built) copies each prefix segment plus the replacement into the output at an independently-advancing write cursor (`out_f`), then continues scanning the original string past the match. Because the scan cursor only ever walks the original input and the write cursor only ever walks the output, a replacement containing the search string cannot cause re-matching or runaway growth.

Wired up via `emitStringReplaceAll` (`codegen/llvm/emit_strings.go`, same shape as `emitStringReplace`) and dispatched from the existing string-method if-chain in `codegen/llvm/emit_call.go`.

## Implementation notes

Files touched:
- `codegen/llvm/emitter.go` — new `usedStringReplaceAll` field.
- `codegen/llvm/runtime.go` — new `ensureStringReplaceAll()` and the `@__kml_replace_all` IR definition.

- `codegen/llvm/emit_call.go` — `replaceAll` dispatch alongside the existing `replace`/`split` entries.

## Side effects discovered

- `.split("")` (empty separator) hangs due to a zero-length-match infinite loop in `@__kml_split`, described above. Not fixed here — it's a separate, `.split()`-specific issue, not something this change touches or introduces. Tracked in `STATUS.md` so it isn't lost silently.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 41/41 pass, including the updated `examples/strings/string_methods.ts`.
- Manual repros, all matching real JS `replaceAll` semantics exactly:
  - `"aaa".replaceAll("a", "bb")` → `"bbbbbb"`
  - `"hello world hello".replaceAll("hello", "hi")` → `"hi world hi"`
  - `"no match here".replaceAll("xyz", "abc")` → unchanged
  - `"abc".replaceAll("", "-")` → `"abc"` unchanged (documented deviation from JS, see Decision)
  - `"aaa".replaceAll("a", "a")` → `"aaa"` unchanged — proves the replacement-contains-search case does not infinite-loop or over-replace
  - `"banana".replaceAll("ana", "ANA")` → `"bANAna"` — confirms non-overlapping, original-string-only scanning matches JS exactly
- New test `TestE2EStringReplaceAll` in `compiler_test.go` covers the above cases.
- `STATUS.md` updated: `.replaceAll()` flipped to ✅ (with the empty-search deviation noted), coverage numbers updated, removed from the Phase 2 roadmap and the "not implemented" table.
