# ADR-00056: Fix Array.prototype.splice's out-of-bounds read and missing insert-item support

- **Date**: 2026-07-14

## Context

Picked up `.splice(start, delete?, ...items)`'s `...items` insertion as part of a broader Array-methods pass — `STATUS.md` already marked the whole row ✅, but `emitSplice` (`codegen/llvm/emit_arrays.go`) only ever accepted exactly two arguments (`start`, `deleteCount`); the `...items` insertion form was never implemented despite the row's own claim. Before extending it, tested the existing two-argument path against an edge case that's easy to hit by accident: a `deleteCount` larger than what's actually left in the array.

## Investigation

`a.splice(1, 10)` on a 3-element array produced `removed.length == 10` and, worse, `a.length == -7` after the call — a negative length, silently accepted, that then made every subsequent operation on `a` behave unpredictably (a `for...of` over it printed nothing rather than throwing). Traced the root cause directly: the old implementation computed `resultPtr := calloc(deleteCount, elemSize)` and then `memmove`d `deleteCount * elemSize` bytes starting at `ptr + start`, with **no clamp at all** on `deleteCount` against the array's actual remaining length (`len - start`). A `deleteCount` of 10 against 2 remaining elements read 8 elements' worth of bytes past the end of a 3-element (24-byte) heap allocation — a genuine out-of-bounds read, not just a wrong-answer bug, and the corrupted `newLen = curLen - deleteCount` (`3 - 10 = -7`) then got stored back into the array's own length field and used, unchecked, by every later operation on that array.

Real JS clamps `deleteCount` to `[0, len - start]` and normalizes negative `start` the same way `.slice()`/`.at()` already do in this compiler (via `emitNormalizeSliceIdx`) — neither of those was implemented either, confirming the original `splice` was a minimal, unguarded implementation that happened to work for the narrow "well-formed, in-range arguments" case every existing test/example happened to use (a repo-wide search turned up zero direct tests for `splice` at all — only example usage, always with valid bounds, so this never surfaced before).

## Decision

Rewrote `emitSplice` to, in order: normalize `start` via the existing `emitNormalizeSliceIdx` (same negative-index handling `.at()`/`.slice()` already have); clamp `deleteCount` to `[0, len - start]` when given, defaulting to "everything from `start` to the end" when omitted (matching real JS); copy the about-to-be-removed slice into the result array *before* anything else mutates the buffer; grow the backing allocation via `realloc` first if the net effect is more elements than before (`numInserted > deleteCount`), so the tail shift below always lands in valid memory; `memmove` the tail from its old position to `start + numInserted` (handles both grow and shrink correctly, since `memmove` is overlap-safe in either direction); then write the inserted items into `[start, start+numInserted)`.

One implementation pitfall caught before it shipped: the grow/no-grow branch needs the buffer pointer as a `phi` node (realloc'd pointer on the grow path, original pointer otherwise). A first draft hardcoded the second `phi` predecessor as `%entry`, which is wrong in general — `emitSplice` runs mid-function, inline into whatever block was active when the user's `.splice(...)` call was reached, not necessarily literally the function's entry block. Fixed by explicitly branching into a freshly created label *before* the conditional, so both `phi` predecessors are labels this function created and controls, never assumed.

`emitArrayToSpliced` (`ADR-00057`, built in the same pass) reuses the same start/deleteCount normalization and clamping logic, but doesn't share the grow/shift machinery — building a fresh three-segment result (head + inserted items + tail) into one correctly-sized allocation from the start is simpler than replicating splice's in-place buffer growth for a case that never needs to preserve or reuse the original buffer.

## Implementation notes

- Files touched: `codegen/llvm/emit_arrays.go` (`emitSplice` rewritten in place; `emit_call.go`'s dispatch already routed `"splice"` here, unchanged).
- Known, accepted narrow gap: argument expressions for inserted items are evaluated *after* the tail shift, so `arr.splice(0, 1, arr[2])` (an item expression reading from the very array being spliced) could read an already-shifted value rather than the pre-splice one. Real JS evaluates arguments left-to-right before the splice operation conceptually happens. This is a rare, self-referential edge case; not fixed here, not claimed as spec-exact — flagged so it isn't mistaken for full compliance.

## Side effects discovered

None beyond the negative-length corruption already described — the fix and the `...items` extension were designed and implemented together, since both changes touch the same start/deleteCount computation.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green.
- Manual repros: `deleteCount` larger than remaining length (used to corrupt length to `-7`, now clamps to the actual removed count and leaves a valid, correctly-shrunk array); negative `deleteCount` (clamps to 0, no-op deletion); omitted `deleteCount` (deletes to the end); negative `start`; inserting more items than deleted (array grows); inserting with `deleteCount` 0 (pure insertion); inserting fewer/equal items to deleted count (array shrinks or stays the same size) — all produced the exact real-JS-equivalent result, checked element-by-element.
- New E2E tests in `tests/arrays_test.go`: `TestE2ESpliceBasic`, `TestE2ESpliceDeleteCountClampedToAvailable`, `TestE2ESpliceNegativeDeleteCountClampsToZero`, `TestE2ESpliceOmittedDeleteCountDeletesToEnd`, `TestE2ESpliceNegativeStart`, `TestE2ESpliceInsertItemsReplacingDeleted`, `TestE2ESpliceInsertItemsWithZeroDeleteCount`, `TestE2ESpliceInsertMoreItemsThanDeletedGrowsArray` — the previous complete absence of direct `splice` tests (only example coverage) is exactly why this bug went unnoticed; these close that gap.
- `examples/arrays/array_methods.ts` extended with an insert-items section, an omitted-`deleteCount` example, and a zero-`deleteCount` pure-insertion example — each verified against actual `./KlainMainLang` output.
- `make examples` — full suite still passes.
