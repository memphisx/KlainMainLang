# ADR-00057: Implement findLast/findLastIndex, toSorted/toReversed/toSpliced, with, keys/values/entries, copyWithin, and Array.of

- **Relations**: Extends ADR-00056
- **Date**: 2026-07-14

## Context

`STATUS.md`'s Array Methods table tracked nine remaining ❌ items after `Object.assign`/`.freeze`/`.seal` wrapped up the Object/Collections gaps: `.flat`/`.flatMap`, `.findLast`/`.findLastIndex`, `.toSorted`/`.toReversed`/`.toSpliced`, `.with`, `.keys`/`.values`/`.entries`, `.copyWithin`, `Array.from`, `Array.of`. Scoped this pass to whatever didn't need new type-system machinery first, deferring the rest with a documented reason rather than attempting a partial/broken version.

## Investigation

- Checked whether nested arrays (`number[][]`) work at all, since `.flat()`/`.flatMap()` need them: `const a: number[][] = [[1,2],[3,4]]` fails to compile with "array literal must be used in a variable declaration" — the inner `[1,2]` literal, appearing as an *element* of the outer literal rather than directly in variable-declaration position, hits the same restriction `emitExpr`'s `*ast.ArrayLiteral` case already documents (array literals can currently only be evaluated in the one specific codegen path `emitArrayVarDecl` handles, not as a general sub-expression). Fixing this is real, separate work — deferred, not attempted as a partial/silently-wrong implementation. `Array.from(iterable)` was deferred for a related but distinct reason: this compiler has no general iterable protocol (only arrays and the specially-cased `Map`/`Set` are directly iterable), so even the "array-like" overload `STATUS.md`'s roadmap already scoped as the minimal first step doesn't have anywhere to plug into yet.
- For everything else, checked for reusable patterns before writing new codegen, and found direct precedent for every piece:
  - `emitArraySort`/`emitArrayReverse` already do exactly what `.toSorted()`/`.toReversed()` need, just mutating the caller's own buffer instead of a copy — extracted their comparator-resolution and swap-loop logic into `emitQsortCall`/`emitReverseInPlace` so both the mutating and non-mutating versions share one implementation, rather than duplicating ~80 lines of qsort/comparator-trampoline setup or ~40 lines of swap-loop IR twice each.
  - `emitArraySlice`'s malloc+memcpy full-array copy is the exact shape `.toSorted()`/`.toReversed()`/`.with()`/`.values()` all need before they can mutate independently of the original — factored into a shared `emitArrayCopy(ptrReg, lenReg, elemTy) string` helper.
  - `emitArrayAt`'s negative-index normalization (`emitNormalizeSliceIdx`) is exactly what `.with()`/`.copyWithin()` need for their own index arguments.
  - `Object.entries`/`Map.entries` (`ADR-00053`, `ADR-00054`) already established the "no tuple type, so return `{key, value}[]` (or here, `{index, value}[]`) instead" convention — reused verbatim for `.entries()`, with `index` instead of `key` since that's the natural name for an array position.
  - `.shift()`/`.unshift()`/the fixed `.splice()` (`ADR-00056`) already rely on `memmove`'s overlap-safety for in-place shifts — the same property `.copyWithin()` needs, since `arr.copyWithin(0, 3)` on a short array is a self-overlapping copy by construction.
  - `Array.isArray`'s existing "static call on the `Array` identifier" dispatch shape is exactly where `Array.of` slots in.
- One real memory-safety concern was checked and ruled out before trusting `.findLast`/`.findLastIndex`'s reverse loop: unlike a forward scan that just keeps overwriting "last match found," real JS's `Array.prototype.findLast`/`findLastIndex` are spec'd to invoke the callback starting from the *last* index and moving backward — observable via side effects (e.g. a `console.log` inside the predicate), not just via the returned value. Implemented as a genuine reverse loop (index starts at `len-1`, decrements, `icmp slt i64 idx, 0` as the exit condition) rather than the cheaper-looking "forward scan, keep the last true result," and added a dedicated test asserting the call order.

## Decision

Added, in `codegen/llvm/emit_arrays.go`:
- `emitQsortCall`/`emitReverseInPlace` (extracted from `emitArraySort`/`emitArrayReverse`, behavior unchanged) and `emitArrayCopy` (new shared helper) — the reuse backbone for everything else in this pass.
- `emitArrayFindLast`/`emitArrayFindLastIndex` — genuine reverse loops, same shape as the existing forward `.find()`/`.findIndex()` but counting down from `len-1`.
- `emitArrayToSorted`/`emitArrayToReversed` — copy via `emitArrayCopy`, then run the same `emitQsortCall`/`emitReverseInPlace` the mutating versions use, on the copy instead of the original.
- `emitArrayWith(index, val)` — copies via `emitArrayCopy`, normalizes `index` via `emitNormalizeSliceIdx`, and throws a catchable Error (same `ensureExceptionHelpers`/`__kml_throw` shape `ADR-00044`'s array-bounds check already established) if still out of range after normalization, matching real JS's `RangeError`.
- `emitArrayKeys`/`emitArrayValues`/`emitArrayEntries` — materialized-array results (no lazy iterator exists in this compiler), matching the convention `Map`/`Set`'s own `.keys()`/`.values()` already use; `.entries()` returns `{index: number, value: T}[]`.
- `emitArrayOf(...items)` — unlike an array literal `[...]` (which can only appear in variable-declaration position today), this is a plain call expression usable anywhere; element type inferred from the first argument, mirroring `inferArrayType`'s own first-element rule for `[...]` literals.
- `emitArrayCopyWithin(target, start?, end?)` — normalizes all three indices, clamps the copy count to `min(end-start, len-target)` so it can never read or write past the array's own bounds, and uses `memmove` for the actual copy.

Dispatch (`emit_call.go`): `Array.of` added alongside `Array.isArray` under the existing `"Array"`-identifier switch. `findLast`/`findLastIndex`/`toSorted`/`toReversed`/`toSpliced`/`with`/`copyWithin` have no name collisions with any other receiver type, so they're plain unconditional `mem.Property ==` checks, same as `.reverse()`/`.fill()` already are. `keys`/`values`/`entries` *do* collide with `Map`'s own same-named methods (`ADR-00053`), so they're guarded with an explicit `e.inferExprType(mem.Object).IsArray` check and placed after the Map/Set dispatch block, mirroring the exact fix `ADR-00053` already made for `forEach`'s own Array/Map/Set collision.

`inferExprType` (`emit_exprs.go`) gained matching cases: `findLast`/`at` share a case (both return the element type); `findLastIndex` joins `findIndex` (both return `i64`); `toReversed`/`toSorted`/`toSpliced`/`with`/`copyWithin`/`values` join `concat`/`reverse`/`fill` (all return the receiver's own array type); `keys` returns `number[]`; `entries` returns the `{index, value}[]` shape; `Array.of` returns `ArrayOf(inferExprType(firstArg))`.

## Implementation notes

- Files touched: `codegen/llvm/emit_arrays.go` (all new/extracted functions above), `codegen/llvm/emit_call.go` (dispatch), `codegen/llvm/emit_exprs.go` (`inferExprType` cases).
- `.values()` returns a *copy*, not the same backing array — matches the "materialized array, not a live view" convention already used for `Map`/`Set`'s iterator-shaped methods, and avoids a caller mutating what looks like a fresh result and unexpectedly affecting the original.
- `Set.entries()` was **not** touched by this ADR — `Array.prototype.entries()` and `Map.prototype.entries()` are unrelated features that happen to share a name; `Set.entries()` remains out of scope for the reason `ADR-00053` already gave (never tracked as ❌, real JS's `[value, value]`-pair shape wouldn't add anything over the existing `.values()`).

## Side effects discovered

None beyond the nested-array-literal limitation described in Investigation, which was already a pre-existing, undocumented boundary — now written down in `STATUS.md` (the `.flat`/`.flatMap` rows) rather than left silent.
