# ADR-00027: Complete the fs.* API (mkdirSync/rmdirSync/renameSync/copyFileSync/readdirSync)

- **Relations**: Extends ADR-00023
- **Date**: 2026-07-12

## Context

Picked up finishing off the `fs.*` API — `STATUS.md` had `mkdirSync`/`readdirSync` explicitly flagged as "not implemented yet, a natural next step" since `ADR-00023`, and `renameSync`/`copyFileSync` had come up as quick wins in an earlier difficulty assessment but hadn't been picked up yet. Built all five (plus `rmdirSync`, added along the way — see Side effects discovered).

## Investigation

- **`mkdirSync`/`renameSync`/`rmdirSync`** needed no new design at all — each is exactly the same shape as `unlinkSync` (`ADR-00023`): one POSIX syscall, throw the existing `"<opDesc> '<path>': <reason>"` `Error` via `ensureFsThrow` on failure. `renameSync` just has two path arguments instead of one; `rmdirSync` uses `rmdir()` specifically rather than reusing `unlinkSync`'s `remove()` (`remove()` also happily deletes an empty directory on POSIX, but that would silently accept a plain file too if called on one — real Node's `fs.rmdirSync` is directory-only and fails with `ENOTDIR` on a file, so a dedicated `rmdir()` call matches that, not `remove()`'s more permissive behavior).
- **`copyFileSync` needed no new C-level I/O at all** — composed the existing `readFileSync`/`writeFileSync` runtime helpers directly at the codegen level (`emitFsCopyFileSync` just calls `@__kml_fs_read_file` then `@__kml_fs_write_file` back to back). Inherits `readFileSync`'s existing text-only limitation as a natural, already-documented consequence, not a new gap.
- **`readdirSync` was the one genuinely new problem: `struct dirent`'s field layout is not portable across libc implementations**, and POSIX only guarantees "`d_name` is a null-terminated `char` array somewhere in there" — nothing about its byte offset. Every real directory-iteration API (`readdir`, `scandir`, `fts_read`) ultimately returns this same struct, so there's no way to get a filename out without knowing its offset (the one exception, `nftw()`, hands the callback an already-formatted path string instead of a struct — but avoiding recursion into subdirectories portably needs `FTW_ACTIONRETVAL`, whose cross-platform support wasn't confirmed, so it didn't end up looking simpler than just handling the offset directly).
  - **Verified the Darwin offset directly, not from memory**: compiled and ran a real C program on this project's own dev machine calling `offsetof(struct dirent, d_name)` against Xcode's actual `<dirent.h>` — confirmed `21`.
  - **Verified the Linux offset by reading glibc's actual upstream source**, not by recalling it: fetched `sysdeps/unix/sysv/linux/bits/dirent.h` from glibc's own repository (the Linux-specific header — a generic "4.4BSD/Generic version" `bits/dirent.h` also exists in the same tree for non-Linux glibc targets like Hurd, and has a *different* layout, so finding the right file mattered). Confirmed `struct dirent`/`dirent64` on 64-bit Linux is `{ __ino64_t d_ino (8); __off64_t d_off (8); unsigned short d_reclen (2); unsigned char d_type (1); char d_name[256]; }` — offset `19`, no padding needed since `d_name` is 1-byte-aligned `char` data immediately after another 1-byte field. Not independently compiled on a real Linux machine in this sandbox (unlike the Darwin number) — flagged honestly in the code comment rather than presented with the same confidence as the directly-verified Darwin value.
  - **The rest of the growable-array shape reuses precedent directly**: the same `{ ptr, i64, i64 }` realloc-doubling growable buffer `__kml_fetch`/`__kml_exec_file_sync` already use, just growing an array of `ptr`-sized name slots instead of raw bytes. Each returned name is `strdup()`'d — independent of the OS's own `dirent` buffer, which `readdir()` is free to overwrite on the very next call.
  - **`.`/`..` filtering**: POSIX's `readdir()` returns these as real entries (unlike Node's `fs.readdirSync`, which never does) — filtered them out inline via `strcmp` against interned `"."`/`".."` constants before appending to the result.

## Decision

`fs.mkdirSync(path)`, `fs.rmdirSync(path)`, `fs.renameSync(oldPath, newPath)`, `fs.copyFileSync(src, dest)`, `fs.readdirSync(path): string[]` — all synchronous, all throwing the existing catchable `Error` shape on OS-level failure (`existsSync`, unchanged, remains the one exception that reports "doesn't exist" as `false`). No `{recursive: true}` on `mkdirSync`, no recursive delete on `rmdirSync` — both plain, single-level POSIX operations, the same incremental-scope pattern every `fs.*`/`fetch` feature here has started with.

## Implementation notes

- `codegen/llvm/runtime.go`: `ensureFsMkdir`/`ensureFsRmdir`/`ensureFsRename` (each a thin POSIX-syscall-plus-throw wrapper, following `ensureFsUnlink`'s exact shape); `direntNameOffset()` (the `runtime.GOOS`-conditional offset, `21`/`19`); `ensureFsReaddir` (`opendir`/`readdir`/`closedir`/`strdup`, the growable `ptr`-array buffer, `.`/`..` filtering).
- `codegen/llvm/emitter.go`: `usedFsMkdir`/`usedFsRmdir`/`usedFsRename`/`usedFsReaddir` flags.
- `codegen/llvm/emit_fs.go`: `emitFsMkdirSync`/`emitFsRmdirSync`/`emitFsRenameSync`/`emitFsReaddirSync` (each validates its own argument count, coerces to `TypePtr`, calls the matching runtime helper); `emitFsCopyFileSync` (no dedicated runtime helper — composes `readFileSync`+`writeFileSync` directly).
- `codegen/llvm/emit_call.go`: `mkdirSync`/`rmdirSync`/`renameSync`/`copyFileSync`/`readdirSync` added to the existing `fs` dispatch switch.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s `fs` case gained `readdirSync` → `ArrayOf(TypePtr)` (the other four return `void` and need no case, same as `writeFileSync`/`unlinkSync` already didn't).

## Side effects discovered

None — no pre-existing bugs surfaced. One deliberate scope addition beyond the original list: `fs.rmdirSync` wasn't originally planned, but shipping `mkdirSync` with no way to remove the directory it creates would leave the API asymmetric, so it was added in the same pass for parity.

## Verification

- `go build .`, `go vet ./...`, `go test .` (full suite, including the 22 new tests below), `make examples` all green.
- Manual repros, each actually compiled and run: `mkdirSync` + `existsSync` + a second `mkdirSync` on the same path throwing (`EEXIST`); `writeFileSync` two files into a fresh directory, `readdirSync` listing exactly those two (sorted for a deterministic printout, confirming `.`/`..` are never present) and correctly returning an empty array for a genuinely empty directory (confirming the never-allocated-`null`-data path is safe when nothing is ever appended); `renameSync` moving a file and the old path no longer existing; `copyFileSync` leaving both files independently readable with identical content; `rmdirSync` removing an empty directory and throwing on a non-empty one; `readdirSync`/`renameSync`/`mkdirSync`/`rmdirSync` all throwing cleanly on a nonexistent path.
- New tests in `compiler_test.go`: `TestE2EFsMkdirSyncCreatesDirectory`, `TestE2EFsMkdirSyncAlreadyExistsThrows`, `TestE2EFsMkdirSyncWrongArgCountRejected`, `TestE2EFsRmdirSyncRemovesEmptyDirectory`, `TestE2EFsRmdirSyncNonEmptyThrows`, `TestE2EFsRmdirSyncWrongArgCountRejected`, `TestE2EFsRenameSyncMovesFile`, `TestE2EFsRenameSyncNonexistentThrows`, `TestE2EFsRenameSyncWrongArgCountRejected`, `TestE2EFsCopyFileSyncCopiesContent`, `TestE2EFsCopyFileSyncWrongArgCountRejected`, `TestE2EFsReaddirSyncListsEntriesExcludingDotAndDotDot`, `TestE2EFsReaddirSyncEmptyDirectory`, `TestE2EFsReaddirSyncNonexistentThrows`, `TestE2EFsReaddirSyncWrongArgCountRejected`.
- Extended `examples/fs/fs.ts` with the full directory lifecycle: create, populate, list (sorted), rename, copy, duplicate-create throwing, clean up, remove.
- `STATUS.md`: flipped the `fs.mkdirSync(path)` / `fs.readdirSync(path)` combined row (previously ❌) into five separate ✅ rows; removed the now-done item from the Roadmap's Web Platform & Node.js APIs backlog; updated the Node.js APIs coverage table (File System (fs) 5/8 → 10/12, ~83%; Node.js total 16/20 → 21/24, ~88%).
