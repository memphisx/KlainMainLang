# ADR-00061: Fix array-typed object/interface fields losing their length entirely

- **Relations**: Extends ADR-00060
- **Date**: 2026-07-14

## Context

`ADR-00060` deferred a deeper bug found while verifying its `return`-array fix: `interface Container { items: number[] }; const c: Container = { items: someArray }; console.log(c.items.length)` crashes at the `clang` stage with an IR type mismatch, confirmed at the time to predate that ADR's own changes (reproduced on a pre-session binary via `git stash`). Picking that up now, per direct instruction to fix it.

## Investigation

Confirmed the root cause described in `ADR-00060`: array types resolve to a plain `ptr` IR type (`ArrayOf()`, `codegen/llvm/types.go`), and `StructIR()`/`StructSize()` (also `types.go`) — which drive both a struct's GEP-addressable layout and its `malloc()` size — route every field through `ty.IR`/`ty.Align()` uniformly. For an array-typed field that reserves exactly 8 bytes (a data pointer) with no room for the length at all, so `emitObjectLiteral`'s field store silently drops the length the moment it's written.

While designing the fix, found the bug is broader than "object fields": `const b = arr; console.log(b.length)` (no object involved) *also* crashes. Traced to `emitIdent` (`emit_exprs.go`) — the generic identifier-evaluation path used whenever a bare array-typed variable is evaluated as an ordinary expression value (not through one of the several existing call sites that special-case identifiers first, e.g. `resolveArrayForHOF`, `emitReturn`'s named-variable branch, `emitIndexPtr`'s identifier branch). It did a single naive `load sym.Ty.IR from sym.Ptr`, recovering only the data pointer and silently discarding `sym.LenPtr`. This reframed the fix: `emitIdent` is the actual root cause for the general case, and the struct-field storage width (`StructIR`/`StructSize`) is a second, independent gap specific to object/interface fields.

This compiler already has three established, mutually-consistent conventions for representing an array value depending on context: a named local variable is two separate allocas (`Symbol{Ptr, LenPtr}`); a function return value or any other general expression result (`.slice()`, HOF output, `Array.of(...)`, etc.) is a single `{ptr, i64}` aggregate register (confirmed via `LLVMRetType()`); a function parameter expands to two separate LLVM parameters, stored into a 2-alloca `Symbol` identical in shape to a local (`emit_func.go`). A struct field storing only 8 bytes was a fourth, broken, undersized convention — the fix adds a correct fifth one (struct field = `{ptr, i64}`-sized slot) without touching the three that already work.

Swept every call site across `codegen/llvm` that reads or writes a struct field by GEP: object literal construction/spread, destructuring, `Object.assign`/`.values`/`.entries`, the three member-access paths (`emitMember` for reads, `emitAssign`'s field-assignment branch for writes, `emitOptionalMember` for `?.`), and `resolveObjectPtr`. Also checked `emit_call.go`'s `JSON.parse`-into-object path (already excludes array fields via an existing `!f.Ty.IsArray` filter, consistent with `JSON.parse`'s documented flat-objects/primitive-fields-only scope — left unchanged, not part of this bug), `emit_fetch.go`/`emit_http.go`/`emit_async.go` (fixed, hardcoded `Response`/`Request` struct shapes with no array-typed fields, ever — irrelevant), and `emit_collections.go`/`emit_arrays.go`'s `Map.entries()`/`Array.entries()` synthetic `{key/index, value}` entry structs. The latter do build a genuine struct field via GEP (`i32 0, i32 N`), but the "value" field's own contents (`kElem`/`vElem`/`elemVal`) are read from the map/array's raw internal element buffer using the *same* under-sized convention one level down (`keyTy.IR`/`valTy.IR`/`elemTy.IR`), so patching just the entry-struct store without also fixing the raw buffer read would produce an SSA-register/store-type mismatch for an array-valued case (e.g. `Map<string, number[]>`) that clang would reject outright — worse than the current silent behavior, for a scenario with no evidence of ever having worked. Left unchanged, same reasoning `ADR-00060` used to defer nested arrays and `JSON.parse` used to defer nested objects: a separate, deeper, already out-of-scope limitation, not this bug.

## Decision

Added two small helpers to `types.go` rather than changing `ArrayOf()`'s `IR` field globally — `Type.IR` staying `"ptr"` everywhere is correct and load-bearing for the three conventions that already work; only the struct-field storage slot itself needed widening:

```go
func StructFieldIR(ty Type) string {
    if ty.IsArray {
        return "{ptr, i64}"
    }
    return ty.IR
}

func StructFieldSize(ty Type) int64 {
    if ty.IsArray {
        return 16
    }
    return int64(ty.Align())
}
```

`StructIR()`/`StructSize()` now route every field through these — a no-op for every non-array field (returns exactly what it returned before), so existing struct layouts are provably unaffected. LLVM's own type-driven GEP offset computation then makes addressing automatically correct for array fields once the struct's declared type reserves the right shape.

Fixed `emitIdent` to special-case `IsArray` symbols: load both `Ptr` and `LenPtr`, rebuild the `{ptr, i64}` aggregate via two `insertvalue`s, instead of the old single undersized load. This alone fixes every generic-expression-evaluation call site "for free," since they already correctly expect a `{ptr, i64}` aggregate once they have one — only the symbol-to-value step was wrong.

Every read/write call site enumerated above now uses `StructFieldIR(fieldTy)` in place of `fieldTy.IR` for the field's own load/store instruction. `emitObjectDestructuring` needed one additional piece beyond the load-width fix: a destructured array-typed field must produce a proper 2-alloca `Symbol{Ptr, LenPtr}` (matching every other array local), not a single alloca holding the raw aggregate — otherwise a later mutating call like `.push()` on the destructured binding would have nowhere valid to write an updated length. `emitOptionalMember`'s null-branch needed a matching fix: `zeroRef()` returns the bare literal `"null"` for any `ptr`-IR type, which is not a valid `{ptr, i64}` store value — the null branch for an array-typed optional-chain result now builds a proper zero aggregate (`insertvalue {ptr,i64} undef, ptr null, 0` then `insertvalue ..., i64 0, 1`) instead.

`coerce()` needed no change: its first check (`if v.Ty.IR == target.IR { return v }`) already returns unchanged whenever both sides report `IR == "ptr"`, which is true for any array-to-array coercion regardless of this fix, so it never interferes with the aggregate shape.

## Implementation notes

Files touched:
- `codegen/llvm/types.go` — `StructFieldIR`/`StructFieldSize`, wired into `StructIR()`/`StructSize()`.
- `codegen/llvm/emit_exprs.go` — `emitIdent` (root-cause fix), `emitAssign`'s field-assignment branch, `emitOptionalMember`, `emitMember`.
- `codegen/llvm/emit_objects.go` — `emitObjectLiteral` (field store and spread-read), `emitObjectDestructuring` (2-alloca Symbol construction for array fields), `resolveObjectPtr`'s `*ast.ObjectLiteral` case, `Object.values`, `Object.entries`, `emitObjectAssign`.

No changes needed in `emit_call.go` (`JSON.parse` already correctly scoped), `emit_fetch.go`/`emit_http.go`/`emit_async.go` (no array-typed fields in those fixed shapes), or `emit_collections.go`/`emit_arrays.go` (entry-struct value field storage is gated by a separate, deeper, unfixed limitation in the map/array's own raw-buffer element storage for array-typed values — see Side effects below).

## Side effects discovered

`Map<K, V>`/`Array.entries()`'s synthetic entry structs can, in principle, hold an array-typed "value" field (`Map<string, number[]>`, or an array of arrays via `Array.entries()`), but the map/array's own internal element buffer storage — one level below the entry-struct construction touched here — uses the same undersized single-register convention this ADR fixes for object fields, and was *not* fixed here (see Investigation). No evidence this ever worked before; not a regression. Not tracked as a new `STATUS.md` Known Limitation on its own, since it falls under the pre-existing, already-documented nested-array limitation (`Array<Array<T>>`) for the `Array.entries()` case, and is a narrower, more exotic variant of it for the `Map` case.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` — all green.
- `make examples` — 58/58 passed.
- Manual repro (the exact one deferred by `ADR-00060`): `interface Container { items: number[] }; const c: Container = { items: someArray }; console.log(c.items.length); console.log(c.items[1]); for (const x of c.items) { console.log(x) }` — now prints the correct length, indexed value, and iterates correctly (previously crashed at the `clang` stage).
- Additional manual repros, all correct: object spread with an array field, `Object.assign` overwriting an array field, destructuring an array field into a binding that supports `.push()` without corrupting the source object's own length, optional chaining (`cc?.items.length` on both a present and a `null` receiver), and `return cc.items` (the exact case `ADR-00060` left failing).
- New E2E tests in `tests/objects_test.go`: `TestE2EArrayTypedFieldLengthAndIndex`, `TestE2EArrayTypedFieldForOf`, `TestE2EArrayTypedFieldSpread`, `TestE2EArrayTypedFieldDestructuring`, `TestE2EArrayTypedFieldObjectAssign`, `TestE2EArrayTypedFieldOptionalChaining`, `TestE2EArrayTypedFieldReturnedFromFunction`.
