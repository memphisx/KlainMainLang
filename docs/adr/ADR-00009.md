# ADR-00009: Implement Math.asin/acos/atan/atan2, sinh/cosh/tanh, cbrt/expm1/log1p

- **Date**: 2026-07-11

## Context

The next Phase 2 roadmap item in `STATUS.md`, picked up as a self-paced continuation of this session's work. All ten functions were already flagged as low-complexity "forward to C `libm`" wrappers with no design ambiguity.

## Investigation

- `emitMathCall` (`codegen/llvm/emit_call.go`) already has fully generic single-argument (`emitMathUnaryFloat`) and two-argument (`emitMathBinaryFloat`) helpers backing the existing `sqrt`/`log`/`log2`/`log10`/`sin`/`cos`/`tan` (unary) and `pow`/`hypot` (binary) cases — each just evaluates the argument(s), coerces to `double`, and emits `call double @<fn>(double ...)`. All ten new functions (`asin`, `acos`, `atan`, `sinh`, `cosh`, `tanh`, `cbrt`, `expm1`, `log1p` — all unary; `atan2` — binary) fit these exact same shapes with zero new codegen logic needed.
- `ensureMathFuncs` (`runtime.go`) already bundles every libm `declare` into one singleton-gated function (not one flag per function, unlike most other `ensure*()` helpers in this codebase) — the new functions' `declare`s were added to this same function.
- `inferExprType`'s `Math.*` call-type table (`emit_exprs.go`) needed the same new names added, so an untyped `const x = Math.atan2(1, 1)` infers `TypeF64` correctly instead of falling through to a wrong default.

## Decision

Added the ten new names to `emitMathCall`'s switch (nine routed through the existing `emitMathUnaryFloat`, one — `atan2` — through `emitMathBinaryFloat`), added their `declare double @fn(...)` lines to `ensureMathFuncs`, and added the same names to `inferExprType`'s `Math.*` type table. No new functions, no new files — this is a pure "extend an existing generic dispatch table" change.

## Implementation notes

Files touched: `codegen/llvm/emit_call.go` (`emitMathCall` switch), `codegen/llvm/runtime.go` (`ensureMathFuncs`), `codegen/llvm/emit_exprs.go` (`inferExprType`'s `Math.*` case).

## Side effects discovered

None.

## Verification

- `go build .`, `go vet ./...`, `go test ./...` all pass.
- `make examples` — 42/42 pass, including the extended `examples/math/math_methods.ts`.
- Manual repros, all numerically correct: `asin(1)`/`atan(1)`/`atan2(1,1)` all approximate `π/2`/`π/4`/`π/4` respectively; `acos(1) = 0`; `sinh(0)=0`, `cosh(0)=1`, `tanh(0)=0`; `cbrt(27)=3`; `expm1(0)=0`; `log1p(0)=0`; an untyped `const x = Math.atan2(1,1)` correctly infers as a float and prints the right value.
- New tests in `compiler_test.go`: `TestE2EMathTrigInverse`, `TestE2EMathHyperbolic`, `TestE2EMathCbrtExpm1Log1p` (this was also the first-ever `TestE2EMath*` test in the suite — none existed before, despite `Math` being extensively used elsewhere).
- `examples/math/math_methods.ts` extended with all ten new functions.
- `STATUS.md`: all three corresponding rows flipped to ✅, removed from the "not implemented" table and the Phase 2 roadmap list, coverage numbers updated. `Math.clz32`/`fround`/`imul` remain the only unimplemented Number/Math items.
