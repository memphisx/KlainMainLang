# ADR-00038: Fix new Date(aStringLiteral) crashing instead of parsing

- **Date**: 2026-07-13

## Context

Tracked in `STATUS.md`'s Known Limitations & Bugs as the next pick after `docs/adr/ADR-00037.md` — the only remaining crash (as opposed to a clean rejection or a type-inference gap) in the tracked bug list. `new Date("2023-11-14T00:00:00.000Z")` compiled to invalid LLVM IR and crashed at the `clang` stage, instead of either working (like real JS, which routes a string argument through the same parsing `Date.parse` uses) or failing with a clear KlainMainLang-level error.

## Investigation

- `emitNewDate` (`codegen/llvm/emit_date.go:75`) unconditionally did `Value{Ref: e.coerce(val, TypeI64).Ref, Ty: TypeDate}` for any non-empty constructor argument. `coerce` (`codegen/llvm/emit_exprs.go:1578`) has no `ptr`→`i64` conversion case at all — its `default` branch is "no known coercion, return as-is," meaning a string argument's `Value` (IR type `ptr`) passed straight through **unchanged**, then got relabeled `Ty: TypeDate` (IR type `i64`) without actually being converted. The resulting `Value` claimed to be an `i64` while its underlying SSA register was actually a `ptr` (e.g. a global string constant reference) — invalid by construction the moment it's used anywhere an `i64` is expected, which is exactly what `clang`'s LLVM parser rejected.
- `emitDateParse` (`emit_date.go:214`, backing the static `Date.parse(s)`) already has the correct logic: `call i64 @__kml_date_parse(ptr %s)`. It took `args []ast.Expression` and evaluated `args[0]` itself, so it couldn't be called directly from `emitNewDate`, which already has its argument evaluated as a `Value`.

## Decision

In `emitNewDate`, check `val.Ty.IR == "ptr"` after evaluating the constructor argument; if true, route it through a new `emitDateParseValue(strVal Value) (Value, error)` — the same `@__kml_date_parse` call `emitDateParse` already makes, factored out so both the `Date.parse(s)` call site and this one share the same logic instead of duplicating it. `emitDateParse` itself now just evaluates its argument and delegates to `emitDateParseValue`. This mirrors the existing `emitJSONParseValue`/`emitJSONParse` split (`emit_call.go`, `ADR-00007`), which exists for the identical reason — `Response.json()` needing to parse an already-evaluated `Value` rather than a fresh `ast.Expression`.

An invalid/unparseable string argument now returns `-1`, the same documented sentinel `Date.parse` already uses for unparseable input (this compiler's `Date` is a plain `i64` with no `NaN` representation) — consistent behavior between `new Date(s)` and `new Date(Date.parse(s))`, which were already equivalent in real JS and are now equivalent here too.

## Implementation notes

- `codegen/llvm/emit_date.go`, `emitNewDate`: added the `val.Ty.IR == "ptr"` branch calling `emitDateParseValue`, ahead of the existing `coerce(val, TypeI64)` numeric path.
- `codegen/llvm/emit_date.go`, `emitDateParse`: reduced to argument evaluation + delegation to the new `emitDateParseValue`.
- `codegen/llvm/emit_date.go`, `emitDateParseValue` (new): the actual `@__kml_date_parse` call, factored out of `emitDateParse`.
- `STATUS.md`: removed the crash row from Known Limitations & Bugs and the Roadmap's "Next up" table; updated the `Date` feature-matrix row to mention the string-constructor form; fixed a dangling cross-reference in the multi-argument-constructor row that used to point at this (now-fixed) crash entry.

## Side effects discovered

None new — no other callers of `coerce` hit the same `ptr`→numeric-target gap in a way that produces invalid IR (every other numeric-coercion call site is only ever reached with an already-numeric source value).
