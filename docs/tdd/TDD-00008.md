# TDD-00008: External conformance suites (TypeScript + Test262) as a test-coverage benchmark

- **Status**: Partially Implemented (ADR-00047)
- **Date**: 2026-07-13

## Context

Looked into whether Microsoft's official TypeScript conformance test suite (`microsoft/TypeScript`, `tests/cases/conformance/`) could be used directly to stress-test this compiler, or at least mined for portable test cases, after the last bug-hunting pass (which found four real bugs — array out-of-bounds read/write, silent `const` reassignment, JS-incompatible bitwise shift semantics, and a parser gap on `(FuncType)[]` — all hand-invented, ad hoc, with no external reference point). The appeal isn't just more bugs — it's a recognized, external benchmark to track test coverage against and set future goals with, as opposed to coverage numbers this project invented for itself.

That investigation (below) found the TypeScript suite itself isn't directly usable, which raised the follow-up question of whether TC39's Test262 (`tc39/test262`, the official ECMAScript *engine* conformance suite) fares better — this compiler is deliberately stricter than real JS in places (static typing, no prototype chain, no dynamic array growth), so it isn't a target for Test262 compliance the way a JS engine would be, but that doesn't rule out mining it the same way.

## Investigation: TypeScript conformance suite

**Investigated directly, not assumed**: sparse-cloned `tests/cases/conformance/{expressions,statements,types,controlFlow}` from the official repo (1,477 `.ts` files across just these four categories) and read representative files from `expressions/binaryOperators/{arithmeticOperator,additionOperator,comparisonOperator}`, `statements/{switchStatements,forStatements,tryStatements}`.

- **The suite is a type-checker conformance suite, not a runtime-behavior suite.** Every test file's ground truth lives in a separate, much larger `tests/baselines/reference/` directory (not fetched) as a `.types`/`.errors` baseline — a record of exactly what type string or diagnostic TSC produces for each expression. There is no expected-stdout oracle anywhere in `tests/cases/`.
- Confirmed by direct measurement: `console.log` (the only way a `.ts` file could carry a runtime-observable assertion at all) appears in only 25 of the 1,477 files sampled (~1.7%), and every one sampled uses it incidentally — to test whether the type checker permits a member access inside the call — not to assert what value gets printed. Example: `statements/tryStatements/catchClauseWithTypeAnnotation.ts` is entirely `// should be OK` / `// error in the body` comments next to `try`/`catch` shape variations; nothing in it is meant to actually run and produce output.
- Most files declare variables with types but no concrete, executed values — `var b: number; var r = a * b;` with `b` never assigned — because the point is testing the *type* assigned to the expression `a * b`, not its runtime result. `statements/forStatements/forStatements.ts` is a sequence of `for(var x: T = someValue;;){}` with empty loop bodies, purely exercising declaration-type compatibility.
- A large fraction of even "in scope"-sounding categories lean on features this compiler doesn't implement at all: 244 of the 1,477 files in just `expressions`+`statements` use `class`, `namespace`, or generics (`<T>`) directly, often incidentally (e.g. `forStatements.ts` declares a generic `class D<T>` purely as one more type to test loop-variable compatibility against).
- **Conclusion**: the suite cannot be used directly, even restricted to nominally in-scope categories. There's no expected-output oracle to run against, and the values used are frequently not concrete or executable in the first place. Only its *category taxonomy* is reusable, as a checklist of edge-case ideas — see Design.

## Investigation: Test262

**Investigated directly, not assumed**: sparse-cloned `test/built-ins/Array`, `test/language/expressions`, and `harness/` from `tc39/test262` and read representative files from `left-shift`, `right-shift`, `unsigned-right-shift`, `bitwise-and`/`bitwise-or`/`bitwise-xor`/`bitwise-not`, and `Array/length`.

- **Test262 is an execution-based suite, unlike the TypeScript one** — its tests actually run code and assert concrete runtime values (`assert.sameValue(actual, expected, message)`, or the older Sputnik-era style, `if (actual !== expected) throw new Test262Error(...)`), which is a direct structural match for this project's own E2E convention (compile, run, assert stdout). This is the opposite conclusion from the TypeScript suite.
- Measured directly across the shift/bitwise operator categories (`left-shift`, `right-shift`, `unsigned-right-shift`, `bitwise-and/or/xor/not`): 168 of 233 files (~72%) use the simple, self-contained `if (x !== y) throw new Test262Error(...)` pattern with zero harness dependency — trivially portable, just swap the throw for a `console.log` of the actual value and compare against the expected value already stated in the file. A further ~20 files per category need the `assert.js`/`sta.js` harness (small: 28 and 183 lines respectively, both simple enough to not need porting themselves, just reading to understand what `assert.sameValue` does). ~6 files per category are BigInt-specific — out of scope, this compiler has no `BigInt`.
- Confirmed the shift/`ToInt32`/`ToUint32` operator files are exactly on-target for the bug already found and tracked in `STATUS.md`: e.g. `left-shift/S9.5_A2.1_T1.js` directly asserts `(2147483648 << 0) !== -2147483648` is false (i.e. `2147483648 << 0 === -2147483648`) and `(-4294967296 << 0) !== 0` is false — precisely the 32-bit-wraparound values this compiler's current 64-bit-native `shl`/`ashr`/`lshr` gets wrong. `left-shift/S11.7.1_A4_T2.js` alone is ~250 mechanically-identical assertions of the same shape (`-N << 16 !== expected`) — a rich, already-computed source of ground-truth values for whenever the shift-semantics bug gets fixed, without having to hand-compute expected 32-bit results.
- **`Array/length` turned out to be the wrong category to have picked for arrays**: 13 of its 30 files test JS's *dynamic, writable* `.length` property directly (`[].length = 4294967296` throwing `RangeError`, `defineProperty`/`writable`/`configurable` semantics) — behavior that doesn't map onto this compiler's arrays at all, since they're fixed-buffer-plus-length-field with no writable `.length` property, no sparse holes, and no `RangeError` concept. This is exactly the "stricter, deliberately different" case flagged as a risk before starting: Test262 here asserts JS's dynamic-growth-on-assignment semantics, which this compiler's out-of-bounds-write bug should almost certainly *not* be fixed to replicate (bounds-checking + rejecting, not silently growing the buffer, is the sane fix for a static, native-compiled array). `Array/from` and `Array/isArray` turned out similarly low-value: 26 of 47 `Array/from` files are `Symbol`/`Proxy`/`Iterator`-protocol-dependent (`Array.from` itself isn't implemented at all, per `STATUS.md`), and `isArray` tests a dynamic-typing runtime check this compiler doesn't need the same way (array-ness is a static type, not a runtime tag, on every variable).
- **Conclusion**: genuinely useful, unlike the TypeScript suite — but only for categories exercising fixed, spec-mandated *value* semantics that this compiler actually intends to match (operator conversions, numeric edge cases). Categories built around JS's dynamic/mutable object model (writable array length, property descriptors, prototype-based `Array.from`/`isArray`) are the wrong place to look, since this compiler deliberately doesn't have that model at all — pulling from those categories would mean chasing behavior that isn't a bug for this project, just a different, intentional design.

## Design

Two different tracks, since the two suites turned out to need different treatment:

**TypeScript conformance suite** — reframe from "use the suite" to "use the suite's *category taxonomy and edge-case ideas* as a checklist," writing this project's own runtime-observable, concrete-value E2E tests inspired by category names, not ports of the files themselves (see Investigation above for why the files aren't executable as-is).

**Test262** — directly portable for the right categories: pick a file, strip the `Test262Error`/`assert.sameValue` harness call, keep the concrete literal expression and its already-computed expected value, and turn each into (or fold several into) a `compiler_test.go` E2E test that `console.log`s the actual value and asserts it against the value the upstream file already states. The main judgment call per category is **whether the tested behavior is something this compiler intends to match** (spec-mandated numeric/operator semantics: yes) **or an artifact of JS's dynamic object/prototype model this compiler deliberately doesn't have** (writable array length, property descriptors, `Proxy`/`Symbol`-based iteration: no) — categories in the second group should be skipped, not chased, per the Investigation findings on `Array/length`/`from`/`isArray`.

Shared infrastructure for both tracks:
- `docs/testing/CONFORMANCE-COVERAGE.md`: one row per category from *either* suite judged relevant, with a column identifying which suite it came from, in/out-of-scope call and why, status, and number of E2E tests derived.
- Ongoing practice, not a one-time pass: revisit categories as new language features land (e.g. once `class` lands, both `tests/cases/conformance/classes/` and Test262's `test/language/statements/class/` become newly relevant).
- Recommended first slice: Test262's `left-shift`/`right-shift`/`unsigned-right-shift` (directly portable, directly targets the already-found shift bug, ~250+ pre-computed 32-bit-wraparound assertions ready to draw from once that bug is fixed) and `bitwise-and`/`bitwise-or`/`bitwise-xor`/`bitwise-not` (same portability profile); from the TypeScript suite, `types/any`/`types/unknown` (highest overlap with what's already implemented) and the remaining control-flow statement categories.

## Open questions

- Whether to vendor a pinned copy (or record a commit SHA) of the upstream categories mined, for reproducibility, vs. just citing category names/relative upstream paths as loose inspiration with no pinned copy. Leaning toward citing-only for the TypeScript suite (files aren't reusable as-is anyway) but this is less clear-cut for Test262, where the actual expected values (not just category names) get hand-copied into new tests — worth deciding before porting the first real batch, so the provenance of each ported value is traceable.
- How many categories to prioritize and what effort budget this gets, relative to actually fixing the four bugs already found — not yet decided.
- Whether upstream category coverage becomes a tracked percentage (mirroring `STATUS.md`'s existing feature-coverage tables) once enough categories have been scanned, or stays a qualitative checklist indefinitely.
- For Test262 specifically: whether to also scan `test/language/statements` (JS-side control flow, likely a better match than the TypeScript suite's own `statements/` since it's execution-based) before committing to a first-slice priority order.
