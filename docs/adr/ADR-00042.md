# ADR-00042: Reject non-numeric arguments to unannotated parameters at call sites

- **Relations**: Implements TDD-00005
- **Date**: 2026-07-13

## Context

Found while verifying `docs/adr/ADR-00041.md`'s return-type inference fix, and deliberately not folded into it: an unannotated function/arrow function parameter defaults to `i64` (`registerFunctions`'s `pty := TypeI64` fallback) with no check or coercion against what's actually passed at a call site. `function log(msg) { console.log(msg) }; log("hello")` compiled clean (exit code 0, no `clang` warning or error) and printed a garbage number instead of `hello` — the string's pointer got bit-reinterpreted as an `i64` inside the callee. Worse than every other tracked limitation in `STATUS.md`: silent data corruption, not a clean rejection or a crash.

Three options were scoped in `STATUS.md`'s "Unannotated Parameter Typing" section: (1) clean rejection at call sites, (2) best-effort inference from the first call site, (3) modeling real TypeScript's implicit-`any` semantics. Option 1 was chosen — smallest, safest, and a permanent safety net regardless of whether options 2 or 3 are ever built on top of it later.

## Investigation

- Confirmed directly why `clang`/LLVM never catches this: this compiler requires opaque pointers (LLVM 15+). With opaque pointers, a global function symbol like `@log` has no static callee-signature type attached to it at the IR level — every `call` instruction carries its *own* explicit signature (`call void @log(ptr @.s3)`), and LLVM's verifier has no way to cross-check that signature against `@log`'s *actual* definition (`define void @log(i64 %p_msg)`). Verified this isn't a parsing fluke by compiling the exact generated `.ll` directly with `clang -O2` (exit 0, no diagnostic at all) — the mismatch is only observable as undefined behavior at the ABI level once the program actually runs. This means `clang` structurally cannot help catch this class of bug; only KlainMainLang's own codegen can.
- Traced the actual corruption: `emitCall`'s named-function call path (`emit_call.go`) evaluates each argument via `emitExpr`, then unconditionally calls `e.coerce(val, paramTy)`. `coerce`'s `default` case ("no known coercion") returns the value **unchanged** when asked to convert a `ptr`-typed value (a string, in this repro) to `i64` — so the argument gets passed through with its real `ptr` IR type at the call site, while the callee's parameter alloca and `define` line both say `i64` (built from the same blind `TypeI64` default). Two independently-computed types, silently disagreeing, exactly the same class of bug `ADR-00041` fixed for return types — just on the parameter side, which (as scoped in `STATUS.md`) is structurally harder to fully solve since it means looking at every call site instead of just the function's own body.
- Confirmed the identical failure exists for closure calls (arrow functions with unannotated parameters), via `emitClosureCallByPtr` (`emit_func.go`) doing the same unconditional `e.coerce(val, ty.FuncParams[i])`.
- Also found, while stress-testing the fix against real HOF usage, a **separate, unrelated, pre-existing bug**: `nums.forEach((n) => console.log(n))` fails at the `clang` stage (`value doesn't match function result type 'i64'`). Root cause: `inferExprType`'s fallback for a `*ast.CallExpression` it doesn't specifically recognize (any call whose callee isn't a registered function, a closure-typed variable, or a known static method — `console.log` included) is `TypeI64`, not `TypeVoid`, so an expression-bodied arrow whose implicit return is such a call gets built expecting to return a number that's never produced. Confirmed pre-existing via `git stash` against the last commit (reproduces identically) and confirmed no existing test/example ever exercised this exact shape (every `forEach`/`map`/`filter` usage in the corpus either uses a block body or an expression `inferExprType` already recognizes, like a `BinaryExpression`). Documented in `STATUS.md`, not fixed here — a different bug in a different part of `inferExprType`, found incidentally while verifying this one.

## Decision

Added `Type.Inferred bool`, marking a parameter type that defaulted to `TypeI64` purely because no annotation was given (as opposed to a real `number`/`int32`/etc. annotation that happens to also resolve to `i64`). Every place that used to silently default an unannotated parameter to `TypeI64` now sets this flag: `registerFunctions` (top-level functions), `emitArrowFunctionWithHints` (closures, when no hint is available either), and `inferExprType`'s `*ast.ArrowFunction` case (the type a `const`-bound arrow function variable gets declared with).

At both call-site paths — `emitCall`'s named-function branch and `emitClosureCallByPtr` — after evaluating an argument, check: if the target parameter's type `Inferred` and the argument's actual type isn't safely numeric (`isSafeNumericArg`, new helper in `types.go`: integer or float, and explicitly *not* the boxed `any`/`unknown` representation, which is a non-`ptr`, non-`float` aggregate type that would otherwise slip past a naive `IsInteger()` check), fail with a clear compile-time error naming the parameter and the fix (add an explicit annotation) instead of coercing and proceeding. A numeric argument (including `boolean`, which real JS/this compiler's own `coerce` already treats as a valid `0`/`1` number) still passes through exactly as before — this is a safety net, not a new restriction on already-working code.

`emitFunctionDecl` was also simplified to reuse `e.funcs[decl.Name].ParamTypes` (the already-registered signature) instead of an independent, third copy of "compute this parameter's type from scratch" — the same duplicate-computation risk `ADR-00041` found and fixed for return types, caught here before it could cause the same kind of silent divergence.

## Implementation notes

- `codegen/llvm/types.go`: new `Type.Inferred bool` field; new `isSafeNumericArg(t Type) bool` helper; `FuncSig` gained `ParamNames []string` (for error messages only).
- `codegen/llvm/emitter.go`, `registerFunctions`: marks `Inferred: true` for any parameter with no type annotation and no rest marker; populates `sig.ParamNames`.
- `codegen/llvm/emit_func.go`: `emitFunctionDecl`'s parameter-type loop now indexes into `sig.ParamTypes` (from the pre-registered signature) instead of recomputing; `emitArrowFunctionWithHints` marks `Inferred: true` when a parameter has no annotation and no HOF-supplied hint; `emitClosureCallByPtr` gained the call-site check.
- `codegen/llvm/emit_exprs.go`: `inferExprType`'s `*ast.ArrowFunction` case marks `Inferred: true` the same way, so a `const`-bound arrow function's own declared parameter types agree with what its actual closure body was built with; `emitCall`'s (`emit_call.go`) named-function scalar-argument path gained the call-site check.

## Side effects discovered

- The `forEach`/expression-bodied-void-callback bug described in Investigation — confirmed pre-existing and unrelated, not fixed here, documented in `STATUS.md`'s Known Limitations & Bugs and Roadmap as its own item.
